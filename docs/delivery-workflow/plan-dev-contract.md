# スキル間インターフェース契約: plan-issue → implementation

本ドキュメントは、`plan-issue` スキルと `implementation` スキルを「データ変換器」として扱い、両者の境界を流れる入出力の「型」を契約として定める。各スキルの `SKILL.md` は内部ロジック（手順・判断・出力フォーマット）に集中し、スキル間でやり取りするデータの構造は本契約を参照する。これにより、一方のスキルの内部実装が変わっても、入出力の型が本契約に適合する限り他方は影響を受けない。

契約の対象は次の変換である。

- `plan-issue`: 外部入力（Issue 情報）を受け取り、プランファイル（`PlanFile`）を生成する。
- `implementation`: 4 種別の入力（Issue / 計画ファイル / PR レビュー指摘 / 会話コンテキスト）のいずれかを受け取り、実装を駆動する。入力種別とパス判定（正規 / 簡易）からレビュー契約を導出する。

## 記法について

型定義の記法は、リポジトリ既存の関数型ドメインモデリング日本語擬似言語ガイド `domain-model-notation.md` に準拠する。使用する記号は `=`（型の定義）、`AND`（直積・全フィールド必須）、`OR`（排他選択・いずれか1つ）、`:`（フィールドの型注釈）、`( )`（自然言語の説明注記）に限る。矢印記号や `of`・`opaque`・`type`、縦棒によるOR表現といった言語固有・他流派の記号は使わない（禁止記号の一覧は記法ガイドおよび `scripts/lint-domain-doc.sh` を参照）。

本契約はスキルを変換器として扱うため、ガイドの「ワークフロー」型を流用し、スキルの出力（成果物）を `生成:` フィールドで表す。`生成:` はガイドでは値構築の生成値を指すが、本契約では「当該スキルが出力する成果物の型」を表すものとして用いる。

階層構造はインデントで示し、各セクションの型は契約の粒度（セクション構造まで）にとどめ、内部の詳細フィールドまでは展開しない。したがって本契約はセクション粒度の適合のみを要求し、各セクション内部のフィールドの充足は問わない（構造を一部欠く計画ファイルは、`implementation` 側のパス判定で `簡易` に倒れることで吸収される）。

## 外部入力型

特定の Issue 管理ツール・レビューツールに依存しないよう、外部から渡される入力は型名のみを定義し、内部構造（ツール固有のフィールド等）は定義しない。

```
IssueInfo (外部入力。型名のみ定義し内部構造は持たない。特定のIssue管理ツールに非依存)

PRレビュー指摘 (外部入力。型名のみ定義し内部構造は持たない。PRコメント等の集合を抽象化)

会話コンテキスト (外部入力。型名のみ定義し内部構造は持たない。直前の会話から抽出した要件・指摘を抽象化)
```

これらは「必ず外部から与えられる」前提で扱い、本契約の内部では構造に立ち入らない。

## plan-issue 出力型: PlanFile

`plan-issue` は外部入力 `IssueInfo` を受け取り、プランファイル `PlanFile` を生成するワークフローである。

```
プランファイルを作成する:
    入力: IssueInfo
    生成: PlanFile
```

`PlanFile` は、確定済みの出力フォーマット `plan-output-format.md` の実セクション構成に対応する直積型（AND 束ね）として定義する。構成要素はタイトル行を含む 8 要素であり、各要素はセクションを表す型名として表現する（内部フィールドは展開しない）。

```
PlanFile =
    タイトル行 (実装プランの表題。Issue番号とタイトルを含む)
    AND 概要 (Issue番号・ベースブランチ・スコープ)
    AND 判断依頼 (高リスクの設計判断・前提確認事項。該当なしを許容)
    AND 検証方針 (テストレベル AND 検証すべき振る舞い)
    AND テストケース対応表 (AC とテストケース・観点・技法・レベルの対応)
    AND 実装設計 (変更概要・データフロー・変更対象ファイル等)
    AND タスク分解 (1コミット粒度のタスクの一覧)
    AND 参照ドキュメント (関連ドキュメント・Issue・外部仕様の一覧)
```

各構成要素は `plan-output-format.md` の `## ` 見出しに 1 対 1 で対応する（タイトル行は冒頭の `# 実装プラン:` 行に対応）。`検証方針` はガイドに従い `テストレベル AND 検証すべき振る舞い` の直積として内訳を示す。

## implementation 入力型

`implementation` の入力は、「インプット種別」と「パス判定」という直交する 2 つの関心事を持つ。両者を概念として分離しつつ、種別がパスを制約する依存（存在しない組み合わせを表現不能にする）を型で表す。

### パス判定

パスは入力種別とは独立した関心事として定義する。

```
パス = 正規 OR 簡易
```

- `正規`: 計画ファイルに検証すべき振る舞いが具体的に定義されている場合のパス（`Issue入力` の場合は探索された計画ファイル `docs/plans/issue-{番号}.md` を指す）。
- `簡易`: それ以外のパス。

### インプット種別

`implementation` 入力は 4 種別の排他選択（OR 型）である。`Issue入力` と `計画ファイル入力` はパスを保持し（正規・簡易のいずれも取りうる）、`PRレビュー指摘入力` と `会話コンテキスト入力` は簡易パス固定であり、パスフィールドを持たない。これにより「PR レビュー指摘 × 正規パス」「会話コンテキスト × 正規パス」という存在しない組み合わせを型で表現不能にする。

```
実装入力 =
    Issue入力
    OR 計画ファイル入力
    OR PRレビュー指摘入力
    OR 会話コンテキスト入力

Issue入力 =
    IssueInfo
    AND パス

計画ファイル入力 =
    PlanFile
    AND パス

PRレビュー指摘入力 =
    PRレビュー指摘
    (簡易パス固定。パスフィールドを持たない)

会話コンテキスト入力 =
    会話コンテキスト
    (簡易パス固定。パスフィールドを持たない)
```

`PRレビュー指摘入力` と `会話コンテキスト入力` がパスフィールドを持たないことから、これらが正規パスを取りえない（簡易固定である）ことが型構造から読み取れる。この制約は `ADR-20260407` の正規 / 簡易パス分岐の設計判断、および `implementation` SKILL.md Phase 0「正規パス / 簡易パスの判定」表と整合する。

### レビュー契約の導出

`implementation` は入力からレビュー契約を導出し、セルフレビュー（code-reviewer）の機能要件検証基準として用いる。レビュー契約は、入力の受け入れ条件・完了条件から導いた検証可能なチェック項目の集合である。

```
レビュー契約項目 = チェック項目 (差分から検証可能な記述)

レビュー契約 = レビュー契約項目の一覧（1個以上）

レビュー契約を導出する:
    入力: 実装入力
    生成: レビュー契約
```

導出規則はパスによって分岐する（`implementation` SKILL.md Phase 0 および `ADR-20260406` と整合）。

- `正規`: 検証方針（検証すべき振る舞い）由来のチェック項目を主とし、Issue AC のうち未カバーの項目を補完として追加する。
- `簡易`: 入力の受け入れ条件があれば各条件をチェック項目に変換し、なければ作業リストの各項目から「何が完了していれば正しいか」を導出する。

導出規則の詳細は `implementation` SKILL.md の Phase 0 を正とし、本契約は入出力の型シグネチャのみを定める。
