# 実現マッピング（コマンド/ポリシー粒度）

開発コンテキストのドメインモデル（[domain-model.md](../development/domain-model.md) / [event-storming.md](../development/event-storming.md)）の各コマンド・ポリシーに、実現の担い手を1対1（主担い手ちょうど1つ）で割り当てた表。汎用実装メカニクスを superpowers へ委譲し、dev-workflow を Discovery（堀）＋接続契約に縮退する方針（[ADR-20260531](../adr/ADR-20260531-superpowers-delegation-boundary.md) / [ADR-20260606-3](../adr/ADR-20260606-3-superpowers-soft-delegation.md)、Epic #261）の実現単位を、ツール非依存のモデル粒度で de-risk することを目的とする。

スキル粒度の概観は [skill-phase-mapping.md](./skill-phase-mapping.md) を参照。本書はそれをコマンド/ポリシー粒度に分解した実現ビューである。

> **位置づけ**: 本書は Issue #263 の成果物。接続契約シームの抽出までを担う。個別メカニクスへの参照機構 ②/③（② メインループ Skill 呼び出し／③ サブエージェント preload）の割当、撤去/保持の線引き、および単位妥当性判定は #264 で確定済み（**ADR-20260607**）。本書のシーム一覧はその入力であった。確定結果は本書末尾「#264 で確定済みの論点」および ADR-20260607 を参照。

## 担い手カテゴリ定義

| カテゴリ | 定義 |
|---|---|
| **superpowers** | 汎用実装メカニクス。既存 superpowers スキルへ委譲する |
| **dev-workflow 接続契約** | dev-workflow 固有の堀／接続契約。汎用メカニクスに委譲できず、dev-workflow が保持する |
| **人間** | 人間アクターの意思決定が必須（型で表せない不変条件の担保点） |
| **ハーネス** | CI・システム内部コマンド・ポリシーの自動駆動（イベント→状態反映、自動実行） |
| **対象外** | 開発コンテキストモデルには現れるが、本 Epic（implementation / plan-issue の委譲再構成）のスコープ外 |

**接続契約シーム**: 主担い手が superpowers / ハーネスのコマンドのうち、dev-workflow が入力供給・結果受領の接続契約を要する結合点。主担い手が「dev-workflow 接続契約」のコマンドはそれ自体がシーム。シーム一覧（後掲）は #264 の入力であった（確定結果は ADR-20260607、後掲シーム表に②/③列として反映）。

> **技術的意思決定集約（#19-22）の「対象外」注記**: これらは開発コンテキストのドメインモデルに正規に属するコマンドである（モデルから漏れているのではない）。ただし ADR ライフサイクル（提案・承認・廃止・上書き）は人間アクター（提案者・承認者）が担う独立した運用であり、本 Epic #261 が対象とする implementation / plan-issue の superpowers 委譲再構成のスコープには含まれない。この意味で「対象外」と分類する。

未割当: 0（全 22 コマンドに主担い手ちょうど1つ）。

> **「人間」割当（#2 / #6 / #16 / #17）の根拠**: いずれも event-storming.md アクター注記「以下のコマンドは人間開発者のみが発行する: 実装プランを確定する / 回答を受理する / PRをマージする / PRを破棄する」が一次根拠。出典の所在は2系統に分かれる:
> - **#2 / #6**: アクター注記に加え、domain-model.md の型不変条件テーブルにも明示（実装プラン不変条件1「確定の判断主体は人間開発者のみ」/ 判断依頼不変条件4「回答受理は人間開発者のみ」）。
> - **#16 / #17**: アクター注記が唯一の明示根拠。型不変条件テーブルには人間限定の項目を持たない（PRレビュー不変条件4 は「マージ済み状態の終端性」であって発行主体を規定しない）。「不可逆操作だから人間」という推論ではなく、アクター注記の明示規定に基づく割当である。

## AC追跡関連項目

開発コンテキストのドメインモデル内に「AC追跡」に該当するコマンド/ポリシーは存在しない（domain-model.md / event-storming.md を検索、該当ゼロ）。Epic #261 の OUT「AC追跡接続契約の実装は需要未検証のため遅延」に対応する実体がモデル粒度に未出現のため、本表では **対象なし** と記録する。需要が顕在化しモデルに項目が追加された段階で、担い手割当のみ行い「実装遅延」と明記する。

## 接続契約シーム一覧（#264 で②/③確定）

dev-workflow が接続契約を担う結合点。各シームについて superpowers 委譲先と dev-workflow が保持する接続契約、および②/③参照機構の割当を示す。②/③ 参照機構の割当・撤去/保持の線引きは #264 で確定済み（ADR-20260607 Decision 項4）。割当規則: メインループの implementation / plan-issue が直接呼ぶ process/規律スキル→②、ディスパッチ先サブエージェント内で走るメカニクス→③ preload。

| シーム | 対象コマンド | superpowers 委譲先 | 参照機構 | dev-workflow 接続契約 |
|---|---|---|:---:|---|
| S1 計画起動 | プランニングを開始する | `writing-plans` | ③ | Issue/Discovery を計画入力へ変換、検証方針・判断依頼を計画へ織込み（Plan サブエージェントへ preload） |
| S2 判断依頼 | 判断依頼を発行する | （委譲なし。dev-workflow 固有） | — | 判断必要事項をチケット管理（GitHub Issue 等）へ問い合わせる堀（②/③対象外） |
| S3 実装起動 | 実装を開始する | `subagent-driven-development`（主）／`executing-plans`（フォールバック） | ② | 確定プランを実装実行へ受け渡す起動契約。委譲先選択基準＝サブエージェント利用可否（SKILL.md と整合） |
| S4 TDD 実行 | タスクを完了する | `test-driven-development` | ② | タスク単位を TDD 小サイクルへ受け渡す |
| S5 検証ゲート | 検証ゲートを実行する | `verification-before-completion`（規律） | ② | 検証方針の供給、ゲート結果のワークフロー反映 |
| S6 PR 化 | PRを作成する | `finishing-a-development-branch` | ② | 実装完了状態を PR 化へ受け渡す |
| S7 レビュー契約 | レビュー依頼をする / 変更を要求する / レビューを承認する | `requesting-code-review` | ② | レビュー依頼の起動、レビュー結果（承認/変更要求）のワークフロー反映、Issue エスカレーション |

シーム数: 7（対象コマンド 9 件。S7 は 3 コマンドを束ねる）。

## 補足

- **ワークフロー（複合）**: domain-model.md「ワークフロー」節（実装プランを確定する / PRをマージしてビルド成果物を公開する 等）は上記コマンドの合成であり、コマンド/ポリシー粒度の新規割当項目を生まないため本表では個別行を立てない。
- **コンテキスト境界の外部コマンド**: 「判断回答を返す」（企画コンテキスト発）は開発コンテキスト外のため対象外。

## 割当判断の記録

### 確定事項

- **技術的意思決定集約（#19-22）= 対象外**。開発コンテキストに属するが、ADR ライフサイクルは本 Epic の委譲再構成スコープ外（上記注記参照）。
- **レビュー系（#13-15）= superpowers 主担い**。レビュー実行メカニクスは `requesting-code-review` に委譲し、レビュー契約（依頼起動・結果反映・エスカレーション）はシーム S7 として dev-workflow が保持する。

### #264 で確定済みの論点（ADR-20260607）

以下は本表だけでは決めきれず #264（単位妥当性判定）へ申し送っていたが、ADR-20260607 で確定した。

- **単位判定**: implementation / plan-issue とも**現状維持**（再分割しない）。担い手境界は「どのスキルか」を分ける横線ではなく各フェーズ内の「superpowers コア vs dev-workflow 接続契約シェル」を分ける縦線であり、スキル単位ではなく内部メカニクス/接続契約境界に宿るため。
- **L1: 検証ゲートを実行する（#11）の主担い = superpowers** `verification-before-completion`（上表#11に反映）。
- **L2: ポリシー実現方式 = 委譲先 superpowers 実行スキルによるオーケストレーション**（上記ポリシー表注記に反映）。
- **参照機構②/③割当**: S1=③、S3〜S7=②（S4 は #265＝PR #276 で ② に最終固定）。詳細は ADR-20260607 Decision 項4（ADR 本文への #265 追従は #278）。
