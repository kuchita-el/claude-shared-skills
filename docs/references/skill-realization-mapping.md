# 実現マッピング（コマンド/ポリシー粒度）

開発コンテキストのドメインモデル（[domain-model.md](../development/domain-model.md) / [event-storming.md](../development/event-storming.md)）の各コマンド・ポリシーに、実現の担い手を1対1（主担い手ちょうど1つ）で割り当てた表。汎用実装メカニクスを superpowers へ委譲し、dev-workflow を Discovery（堀）＋接続契約に縮退する方針（[ADR-20260531](../adr/ADR-20260531-superpowers-delegation-boundary.md) / [ADR-20260606-3](../adr/ADR-20260606-3-superpowers-soft-delegation.md)、Epic #261）の実現単位を、ツール非依存のモデル粒度で de-risk することを目的とする。

スキル粒度の概観は [skill-phase-mapping.md](./skill-phase-mapping.md) を参照。本書はそれをコマンド/ポリシー粒度に分解した実現ビューである。

> **位置づけ**: 本書は Issue #263 の成果物。接続契約シームの抽出までを担う。個別メカニクスへの参照機構 ②/③（② メインループ Skill 呼び出し／③ サブエージェント preload）の割当、および撤去/保持の線引きは #264（単位妥当性判定）で確定する（ADR-20260606-3 項4）。本書のシーム一覧は #264 の入力。

## 担い手カテゴリ定義

| カテゴリ | 定義 |
|---|---|
| **superpowers** | 汎用実装メカニクス。既存 superpowers スキルへ委譲する |
| **dev-workflow 接続契約** | dev-workflow 固有の堀／接続契約。汎用メカニクスに委譲できず、dev-workflow が保持する |
| **人間** | 人間アクターの意思決定が必須（型で表せない不変条件の担保点） |
| **ハーネス** | CI・システム内部コマンド・ポリシーの自動駆動（イベント→状態反映、自動実行） |
| **対象外** | 開発コンテキストモデルには現れるが、本 Epic（dev-loop / plan-issue の委譲再構成）のスコープ外 |

**接続契約シーム**: 主担い手が superpowers / ハーネスのコマンドのうち、dev-workflow が入力供給・結果受領の接続契約を要する結合点。主担い手が「dev-workflow 接続契約」のコマンドはそれ自体がシーム。シーム一覧（後掲）は #264 の入力となる。

> **「人間」割当（#2 / #6 / #16 / #17）の根拠**: いずれも event-storming.md アクター注記「以下のコマンドは人間開発者のみが発行する: 実装プランを確定する / 回答を受理する / PRをマージする / PRを破棄する」が一次根拠。出典の所在は2系統に分かれる:

> **ポリシー主担いは全て暫定**。「ハーネス」割当は、ポリシー（イベント→コマンドの自動反応）を**イベント駆動方式**（イベント発生→ハーネス/メインループが自動反応）で実現する前提に立つ。これを**オーケストレーション方式**（dev-workflow が明示的に次ステップを駆動）で実現する選択肢もあり、方式の選択と各ポリシーの主担い確定は #264 で行う（論点 L2）。方針 C の「最小インラインをフォールバック保持」と P7 等の統治ロジックの撤去/保持の線引きも #264 / #265 に連動する。

## AC追跡関連項目

開発コンテキストのドメインモデル内に「AC追跡」に該当するコマンド/ポリシーは存在しない（domain-model.md / event-storming.md を検索、該当ゼロ）。Epic #261 の OUT「AC追跡接続契約の実装は需要未検証のため遅延」に対応する実体がモデル粒度に未出現のため、本表では **対象なし** と記録する。需要が顕在化しモデルに項目が追加された段階で、担い手割当のみ行い「実装遅延」と明記する。

## 接続契約シーム一覧（#264 の入力）

dev-workflow が接続契約を担う結合点。各シームについて superpowers 委譲先と dev-workflow が保持する接続契約を示す。②/③ 参照機構の割当・撤去/保持の線引きは #264 で確定する。

| シーム | 対象コマンド | superpowers 委譲先 | dev-workflow 接続契約 |
|---|---|---|---|
| S1 計画起動 | プランニングを開始する | `writing-plans` | Issue/Discovery を計画入力へ変換、検証方針・判断依頼を計画へ織込み |
| S2 判断依頼 | 判断依頼を発行する | （委譲なし。dev-workflow 固有） | 判断必要事項をチケット管理（GitHub Issue 等）へ問い合わせる堀 |
| S3 実装起動 | 実装を開始する | `executing-plans` / `subagent-driven-development` | 確定プランを実装実行へ受け渡す起動契約 |
| S4 TDD 実行 | タスクを完了する | `test-driven-development` | タスク単位を TDD 小サイクルへ受け渡す |
| S5 検証ゲート | 検証ゲートを実行する | `verification-before-completion`（規律） | 検証方針の供給、ゲート結果のワークフロー反映 |
| S6 PR 化 | PRを作成する | `finishing-a-development-branch` | 実装完了状態を PR 化へ受け渡す |
| S7 レビュー契約 | レビュー依頼をする / 変更を要求する / レビューを承認する | `requesting-code-review` | レビュー依頼の起動、レビュー結果（承認/変更要求）のワークフロー反映、Issue エスカレーション |

シーム数: 7（対象コマンド 9 件。S7 は 3 コマンドを束ねる）。

## 補足

- **ワークフロー（複合）**: domain-model.md「ワークフロー」節（実装プランを確定する / PRをマージしてビルド成果物を公開する 等）は上記コマンドの合成であり、コマンド/ポリシー粒度の新規割当項目を生まないため本表では個別行を立てない。
- **コンテキスト境界の外部コマンド**: 「判断回答を返す」（企画コンテキスト発）は開発コンテキスト外のため対象外。

## 割当判断の記録

### 確定事項

- **技術的意思決定集約（#19-22）= 対象外**。開発コンテキストに属するが、ADR ライフサイクルは本 Epic の委譲再構成スコープ外（上記注記参照）。
- **レビュー系（#13-15）= superpowers 主担い**。レビュー実行メカニクスは `requesting-code-review` に委譲し、レビュー契約（依頼起動・結果反映・エスカレーション）はシーム S7 として dev-workflow が保持する。

### #264 へ申し送る未確定論点

以下は本表（コマンド/ポリシー粒度）だけでは決めきれず、#264（単位妥当性判定）で全体を俯瞰して確定する。それまで暫定割当とする。

- **L1: 検証ゲートを実行する（#11）の主担い**。実行主体は CI / test runner（ハーネス）だが、「検証してから完了と言う」規律は superpowers `verification-before-completion`。主担いをハーネス（暫定）とするか superpowers とするかは、検証ゲートの単位を俯瞰して #264 で確定。
- **L2: ポリシー実現方式とポリシー主担い**。イベント駆動方式（ハーネス）か オーケストレーション方式（dev-workflow が明示駆動）かの選択。方式により全ポリシーの主担いが変わる。方針 C の統治ロジック（P7 のリトライ等）の撤去/保持の線引きと合わせ #264 / #265 で確定。
