# 自律度判定ルーブリック

## 位置づけ

本書は、ある判断・作業に対して AI と人間のどちらがどこまで主導・確定するか（自律度レベル L0–L3）を、5つの要因のスコアリングによって一意に判定するためのルーブリックである。親トラッカー Issue #232 の決定⑤に対応する。

- **自律度レベル L0–L3 の正式定義（名称・判定基準）の唯一の情報源（SoT）は規約 ADR（#238）である。** 本書は #238 が確定するまで [docs/principles.md](../principles.md)「承認ゲート軸（自律度レベル別）」の暫定定義を判定対象として参照し、正式定義を上書きしない。#238 確定後に齟齬があれば本書を同期する。
- 自律度を責任分担マトリクス（Explanation 層）で表現し、ドメインモデルへ持ち込まない方針の根拠は [ADR-20260601](../adr/ADR-20260601-autonomy-approval-gate-alignment.md) を参照。
- 本ルーブリックの5要因は、ADR 粒度判定の4項目チェックリスト（[docs/adr/README.md](../adr/README.md)）を発想元とする（後述。厳密な同型ではない）。

## 5要因スコアリング

自律度は次の5要因で評価する。**各要因を 0–3 点で採点し、5要因の単純合算（合計 0–15 点）でスコアを求める。**

**スコアの向き：** すべての要因は「点数が高いほど人間の関与を強める（＝自律度を下げる）方向」に揃える。リスクや不確実性が大きいほど高得点になる。検証可能性は直感と逆になりやすいため、**「検証の困難さ」として採点する**（容易に検証できる＝0点、検証できない＝3点）点に注意する。

要因間に重み付けや優先・支配ルールは設けない。すべての要因を対等に合算する。

### 各要因のスコア基準

#### 1. 不可逆性（後戻りのしにくさ）

| 点 | 基準 |
|---|---|
| 0 | 完全に元へ戻せる（`git revert`、設定の戻し等で痕跡なく復元可能） |
| 1 | 戻せるが手間がかかる（複数手順・関係者連絡が必要） |
| 2 | 部分的にしか戻せない（一部のデータ・状態が失われる） |
| 3 | 実質的に戻せない（本番データ削除、外部公開済み等） |

#### 2. 影響範囲（波及の広さ）

| 点 | 基準 |
|---|---|
| 0 | 単一ファイル・単一モジュール内で閉じる |
| 1 | 同一コンポーネント内の複数モジュールに及ぶ |
| 2 | 複数コンポーネント・複数開発者に波及する |
| 3 | システム全体・外部利用者・複数チームに波及する |

#### 3. 説明責任（誤判断時に問われる責任の重さ）

| 点 | 基準 |
|---|---|
| 0 | 誰が判断しても結果が変わらず、責任が問われない |
| 1 | 軽微な責任（チーム内の手戻りに留まる） |
| 2 | 相応の責任（ユーザー影響・対外説明が生じうる） |
| 3 | 重大な責任（法務・契約・安全・信用に関わる） |

#### 4. 検証可能性（検証の困難さ ※スコアは困難なほど高い）

| 点 | 基準 |
|---|---|
| 0 | 自動テスト・CI で正しさを機械的に判定できる |
| 1 | 目視・手動確認で短時間に判定できる |
| 2 | 検証はできるが専門知識・長時間・特殊環境を要する |
| 3 | 事後に正しさを確認する手段が乏しい／実質的に検証不能 |

#### 5. 曖昧さ（要件・正解の不確かさ）

| 点 | 基準 |
|---|---|
| 0 | 要件が一意で、正解が明確に定まる |
| 1 | わずかな解釈の幅があるが、妥当な選択肢は1つに収束する |
| 2 | 複数の妥当な選択肢があり、トレードオフ判断を要する |
| 3 | 要件自体が未確定で、何が正解か合意が必要 |

## スコア → L0–L3 判定境界

5要因の合計スコア（0–15）を、次の3つのしきい値で L0–L3 の4区分に一意に写像する。**合計が高い（リスクが大きい）ほど自律度を下げる。**

| 合計スコア | 自律度レベル | 意味（暫定・SoT は #238） |
|---|---|---|
| 0–3 | **L3（AI自律）** | AI が提案から確定まで自律実行する |
| 4–7 | **L2（AI主導・人承認）** | AI が提案を主導し、人間が承認ゲートを通す |
| 8–11 | **L1（人主導・AI補助）** | 人間が主導し、AI が補助する |
| 12–15 | **L0（人のみ）** | 人間が直接実施し、AI 提案を介さない |

**境界の性質：**

- 各区分は連続した4点幅の整数範囲（0–3／4–7／8–11／12–15）で定義され、全域 0–15 を重複・欠落なく覆う。しきい値は 3↔4・7↔8・11↔12 の3点。
- 合計スコアは 0–3 点の整数の和（必ず整数）であるため小数境界は生じず、任意の合計スコアは厳密に1区分にのみ属する。
- 隣り合う区分の帰属は重複しない（例：合計7は L2、合計8は L1。合計11は L1、合計12は L0）。

## サンプル適用（3ケース）

各ケースで5要因を採点し、合計から L 区分が一意に決まることを示す。

### ケース1：ドキュメントの typo 修正（低リスク）

| 要因 | 点 | 根拠 |
|---|---|---|
| 不可逆性 | 0 | `git revert` で戻せる |
| 影響範囲 | 0 | 単一ファイル内で閉じる |
| 説明責任 | 0 | 結果に責任が伴わない |
| 検証可能性 | 0 | 目視・CI で即判定できる |
| 曖昧さ | 0 | 修正内容が一意 |
| **合計** | **0** | → **L3（AI自律）** |

### ケース2：公開 API のレスポンス項目追加（要因拮抗：高不可逆 × 検証容易）

| 要因 | 点 | 根拠 |
|---|---|---|
| 不可逆性 | 3 | 公開インターフェースで後方互換の制約が強い |
| 影響範囲 | 1 | 当該エンドポイントの直接利用に限定 |
| 説明責任 | 1 | 軽微（チーム内の手戻りに留まる） |
| 検証可能性 | 0 | 契約テスト・自動テストで完全に検証可能 |
| 曖昧さ | 0 | 追加項目の仕様が明確 |
| **合計** | **5** | → **L2（AI主導・人承認）** |

不可逆性が最大（3）でも、検証が容易（0）で影響が限定的なら合計は L2 に収まる。仮に検証が困難（検証可能性=3）であれば合計は8となり L1 に下がる。単純合算が要因の拮抗を吸収し、検証可能性の差が1段の自律度差として現れる。

### ケース3：本番 DB スキーマの破壊的マイグレーション（高リスク・境界付近）

| 要因 | 点 | 根拠 |
|---|---|---|
| 不可逆性 | 3 | 本番データを失う可能性、ロールバック困難 |
| 影響範囲 | 3 | 複数サービス・全社に波及 |
| 説明責任 | 3 | 障害時の責任が重大 |
| 検証可能性 | 2 | 事前検証はできるが本番影響は事後にしか分からない部分がある |
| 曖昧さ | 1 | 移行手順に解釈の幅がわずかにある |
| **合計** | **12** | → **L0（人のみ）** |

合計12は L1（8–11）と L0（12–15）の境界に接するが、境界値の帰属が重複しない規約により L0 に一意に決まる。仮に検証可能性が1点低ければ合計は11となり L1 に移る。境界をまたぐ際も区分は常に一意である。

## ADR 粒度判定との発想上の対応

本ルーブリックの5要因は、ADR 化要否を判定する4項目チェックリスト（[docs/adr/README.md](../adr/README.md)「粒度判定基準」）を発想元としている。両者は「後戻りコストと波及の広さで関与の度合いを決める」という発想を共有するが、**厳密な1対1の同型ではない**（同型性の証明は本書の目的としない）。

| 5要因 | ADR 粒度判定の対応項目 | 対応の有無 |
|---|---|---|
| 不可逆性 | 後戻りコストが高い | 対応する |
| 影響範囲 | 複数モジュール・複数開発者に波及する | 対応する |
| 検証可能性 | ツールで自動強制できない（≒機械的に正しさを担保できない） | 対応する |
| 説明責任 | （直接対応する項目なし） | 対応しない |
| 曖昧さ | （直接対応する項目なし） | 対応しない |

- **対応する3要因**：不可逆性・影響範囲・検証可能性は、ADR 粒度判定の項目1・2・4にそれぞれ発想上対応する。
- **対応しない2要因**：説明責任・曖昧さは、ADR 粒度判定に直接対応する項目を持たない。これらは自律度判断に固有の観点であり、本ルーブリックでは他3要因と同じ 0–3 点軸で対等に採点するが、粒度判定との対応関係は主張しない。
- **逆方向の非対応**：ADR 粒度判定の項目3（採用理由が時間経過で揮発しやすい）には、直接対応する要因を置いていない。

この非対称性が、両者を同型とみなさず「発想元として参照する」に留める理由である。
