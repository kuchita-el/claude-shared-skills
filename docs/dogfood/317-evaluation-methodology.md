> **【2026-07-16 追記】上記は初回反映時（2026-06-21）の目的である。実機評価の結果、Sonnet 固定を維持できたのは 6 件中 1 件（`refactorer`）にとどまった。**「機械的タスクは Sonnet で十分」という前提は、実測では精査系の 4 エージェントいずれについても成立しなかった。加えてコスト前提にも誤りがあった（「実施状況」節参照）。ただし `inherit` からの固定値化そのものは無意味ではない。親セッションが Sonnet の場合でも Opus を保証する効果が残る。

1. **評価メソドロジー**: 降格対象エージェントについて、Opus 版と Sonnet 版を比較し降格可否を判定するための手順・指標・閾値。
2. **初回反映の暫定判定**: 文書ベース推論で導いた初期 `model` 値と根拠。

実機並行評価は 2026-07-16 に全 5 エージェントで実施済み。判定結果と現行 `model` 値は「実施状況」節を参照する。

## 判定指標

各エージェントについて、Opus 版と Sonnet 版を同一入力で並列実行した結果を以下 4 軸で比較する。

1. **ブロッカー指摘の見落とし件数**: Opus 版が出した「ブロッカー」相当の指摘のうち、Sonnet 版が出せなかった件数。中庸方針では重要観点（バグ/セキュリティ・要件充足）の見落とし 1 件で降格不可と判定する。
2. **観点カバレッジ**: 各エージェントの責務に含まれる観点（例: code-reviewer は「要件充足・バグ/セキュリティ・規約違反・スコープ逸脱・コード重複・テスト品質」の 6 観点）について、Sonnet 版が触れた観点の割合。50% 未満で降格不可、50%〜80% は要警戒（降格は可だが後続ドッグフードで早期再評価）、80% 以上は許容範囲。
3. **妥当な指摘の総数と比率**: Sonnet 版が出した指摘総数のうち、人間が「妥当」と判定する比率。誤検知が増えていないかの確認。
4. **質的評価（Opus 版にあって Sonnet 版にない指摘の評価）**: Opus 版にしかない指摘のうち**指標 1 のブロッカー相当を除いた残余**について、「重要・あれば望ましい・なくてもよい」の 3 段で人間が分類する。重要 1 件以上で降格不可。指標 1 と 4 は重複しない集合に対する評価で、二重計上を避ける（指標 1 はブロッカー相当の差集合、指標 4 はそれ以外の差集合）。

## 判定閾値（中庸方針）

降格可否は以下のフローで決定する。

```
【第0段: 責務ゲート】
Sonnet 版が割り当て文脈で有効な成果物を出しているか
  → 出していない（責務未達）→ 降格不可（指標を適用せず確定）
↓ 出している
ブロッカー指摘の見落とし（指標 1）が重要観点で 1 件以上 → 降格不可
↓ なし
質的評価（指標 4）で「重要」分類が 1 件以上 → 降格不可
↓ なし
観点カバレッジ（指標 2）が 50% 未満 → 降格不可
↓ 50% 以上
観点カバレッジ（指標 2）が 50%〜80% → 降格可・要警戒（後続ドッグフードで早期再評価）
↓ 80% 以上
降格可（Sonnet で運用）
```

**第0段（責務ゲート）**: 指標 1〜4 はいずれも「両版が有効な成果物を出している」ことを暗黙の前提とし、その差を測る。前提が崩れている場合は指標を適用しても意味がないため、指標より前に判定する。責務未達とは次を指す。

- 割り当てられた作業ディレクトリの範囲外を参照し、**割り当て文脈では成立しない成果物**を出した。
- エージェントの責務そのもの（例: test-designer なら「要件を満たすテスト仕様の設計」）を果たしていない。

責務未達は見落としより重い。見落としは有効な成果物の中の欠落だが、責務未達は成果物自体が使えないことを意味する。

**指標が写像しないエージェントの扱い**: 各指標は特定の出力形態を前提とする（指標 1・4 は「指摘」、指標 2 は「観点」）。出力形態が異なるエージェントには当該指標を適用せず、**スキップして次段へ進む**（例: refactorer は指摘ではなく変更を出力するため、指標 1・4 は写像しない）。全指標をスキップした場合は、当該エージェントの契約が定める性質（例: refactorer なら「外部から見た振る舞いを変えない」「テストが通る状態を維持する」）を実測して判定し、その根拠を記録する。

「規約違反等の見落とし」「文体・表現の差異」「指摘の網羅性が Opus 版より低い」のみで降格不可とはしない（中庸方針の許容範囲）。

## 評価手順

実機並行評価を実施する場合の標準手順。

1. **対象エージェントの `model` を `sonnet` に変更**: `plugins/dev-workflow/agents/<name>.md` のフロントマター `model` を一時的に `sonnet` に書き換える。
2. **比較用ブランチで `model: inherit`（Opus 親）を別途準備**: 同一サンプル入力に対する Opus 版出力を得るため、別ブランチまたは別 worktree で `model: inherit` のままにする。
3. **Agent ツールで両エージェントを起動し、同一サンプル入力を渡す**: 出力をテキストとして保存。
4. **判定指標 1〜4 を記録**: 評価条件（入力・正解キー・両版のプロンプトが同一であること）と測定結果を記録する。記録先は当該変更の PR 本文とする（検証結果は git 管理下に置かない。`records/.gitignore` を参照）。指標 4 は定義どおり人間が分類する。
5. **判定閾値フローに従って降格可否を決定**: 「降格不可」となったエージェントは `model` を `opus` または `inherit` に巻き戻す。

本手順は 2026-07-16 に全 5 エージェントへ適用済み（PR #506）。実施条件と測定結果は同 PR 本文を参照する。

## 文書ベース暫定判定（初回反映の根拠）

実機評価を実施する前の段階で、各エージェントの責務とタスクの機械性から Sonnet 降格可否を文書ベースで推論する。

### code-reviewer → sonnet 採用

- 責務は「要件充足・バグ・規約違反・スコープ逸脱・テスト品質」の精査。
- 規約違反検出・スコープ逸脱検出は機械的（型化された判定）。要件充足判定は推論寄りだが、入力（プラン・Issue 本文）が構造化されているため Sonnet で十分対応可能と推定。
- バグ検出のうち高難度のロジック誤りは Opus 優位だが、本エージェントの主用途は implementation Phase 2 のセルフレビューであり、後段の人間レビューが直列に続く。見落とし耐性は中庸方針内に収まると判断。
- リスク: バグ検出能力で Opus と差が出る可能性。
- **【2026-07-16 実機評価: 降格不可】** このリスク予測は的中した。正解が既知の実バグを含む差分に対し、Sonnet 版はブロッカー 0 件・「バグ/セキュリティ ✅」と報告した（指標 1: 重要観点の見落とし 1 件）。あわせて上記「後段の人間レビューが直列に続く」という耐性の想定が成立しないことも判明した（「実施状況」節参照）。`opus` へ巻き戻し済み。

### plan-reviewer → sonnet 採用

- 責務はファイル実在検証 + プランの懐疑的レビュー。実在検証は Grep/Read による機械的タスク（Sonnet で十分）。
- プラン懐疑レビューは観点ベース（10 観点のチェックリスト）で運用されており、観点を逐一照合する処理は Sonnet 単価で十分推定。
- 既存運用で plan-reviewer は PASS/FAIL の二値出力で、判定根拠は構造化済。
- リスク: 「観点間の関係性に基づく深い推論」が要る場面で Opus 優位の可能性。
- **【2026-07-16 実機評価: 降格不可】** このリスク予測は的中した。Opus 版のみが、判断依頼で未決とされた論点が実装設計節では決め打ちされているという**セクション間の矛盾**を検出した（指標 4:「重要」1 件）。まさに「観点間の関係性に基づく推論」が要る所見である。`opus` へ巻き戻し済み。

### test-designer → sonnet 採用

- 責務は「入力ソースからテスト仕様を設計」（テストコード非生成）。
- 観点（典型/境界/異常/状態遷移/組み合わせ）と採用技法（同値分割/境界値/ディシジョン/状態遷移/ユースケース）はクローズドリスト化されており、入力 AC から型化された変換が可能。
- ドメイン特化の境界値発見は Opus 優位の可能性があるが、本エージェントの観点クローズドリストはそれを Sonnet にも追随可能にしている。
- リスク: ドメイン依存の境界値見落とし。
- **【2026-07-16 実機評価: 降格不可（第 0 段 責務ゲート）】** ただし**このリスク予測は外れ、想定と異なる理由で降格不可となった**。Sonnet 版は割り当てられた作業ディレクトリを無視して絶対パスでメイン作業ツリーを探索し、実装前状態を対象とする指示に対し「既に実装済み」という割り当て文脈では成立しない結論を報告した（責務未達）。境界値の網羅性以前の問題である。逆に Opus 版は要件のみから、後に実バグとして顕在化する 2 箇所（空要素を含むリスト値・重複要素）を「判断を要する未定義境界」として予測した。**観点のクローズドリスト化があれば Sonnet でも追随可能という想定は、参照範囲の逸脱という別軸のリスクを捉えていなかった。** `opus` へ巻き戻し済み。
- **本判定は他 3 件と異なり、指標 1・4 による独立した根拠を持たない**（第 0 段のみで確定）。#509 で参照範囲の拘束手段が導入されたら再評価すること（「実施状況」節の再評価トリガー参照）。なお指標 4 での代替判定は成立しない。Sonnet 版の仕様は上記境界値 2 件を名目上カバーしており（実装済み fixture を読んだため）、差集合が Opus 側に寄らない。それを無効と論じるには結局「割り当て範囲外の参照」を持ち出すことになり、第 0 段と同じ根拠へ帰着する。

### test-spec-validator → sonnet 採用

- 責務は test-designer の出力を独立視点で検証（カバレッジ漏れ・観点漏れ・誤読検出）。
- 独立視点による「漏れ検出」は推論寄りだが、test-designer のクローズドリスト準拠を前提とすれば、観点照合は機械的。
- リスク: 観点漏れの質的判定（重要な漏れか軽微な漏れか）で Opus 優位の可能性。中庸方針で許容範囲と判断。
- **【2026-07-16 実機評価: 降格不可】** このリスク予測は的中し、**「中庸方針で許容範囲」という判断の方が誤りだった**。Opus 版のみが仕様の誤読（テストケースの対応要件が AC の意図と食い違う）を検出した（指標 4:「重要」1 件）。「仕様の誤読」は本エージェントの 3 大検出カテゴリの一つであり、その見落としは看板責務の欠落にあたる。加えて Sonnet 版は境界値の指摘をマージ済み PR の差分を参照して得ており、実運用時点では存在しない情報に依存していた。`opus` へ巻き戻し済み。

### refactorer → sonnet 採用（リスク中）

- 責務は確定済みコードに対するリファクタリング適用（外部振る舞い不変）。
- 内部構造の改善判断は推論寄り。コードの設計意図を読み解いてリファクタリング案を提示する性質上、Opus 優位の可能性は他 4 件より高い。
- ただし、リファクタは本質的に HOW 寄り（プロジェクト DoR でも「適正な抽象度の厳しさ＝低」と整理されている）。Sonnet でも「振る舞い不変＋テスト維持」の制約内で十分機能すると推定。
- リスク: 大規模リファクタの設計品質。
- **【2026-07-16 実機評価: 降格可（sonnet 維持）】** **「リスク中・最優先で注視」という当初の見立ては外れ、5 エージェント中で唯一 sonnet を維持できた。** Sonnet 版はヘルパー抽出を行い、テスト 48/48 を維持。振る舞い不変を 3 種の corpus（末尾カンマ / カンマのみ / 無改変）で実測し、出力と終了コードが前後で全一致することを確認した。所要時間は Opus の約半分。なお指標 1（ブロッカー指摘の見落とし）は、指摘ではなく変更を出力する本エージェントには写像しない。指標 4 の対象（Opus のみが出した指摘）は契約外のバグエスカレーションのみであり、対象外と判断した。

### plan → opus 固定（維持・評価対象外）

- 責務は writing-plans メソドロジーで計画骨格を生成する設計の本丸。
- CLAUDE.md 原則「設計・難所のみ Opus」の典型適用例。
- 本 Issue のプラン生成自体に使われており、出力品質が直接利用される（メタ循環）。
- 評価コストを払わない選択をした理由: Issue 制約「降格不可なら Opus 維持」を尊重し、設計品質リスクを優先。

## 巻き戻し条件と手順

後続実機評価で「降格不可」判定が出たエージェントは以下手順で巻き戻す。

1. `plugins/dev-workflow/agents/<name>.md` の `model` を `opus`（または `inherit`）に書き換え。
2. 巻き戻し理由・評価条件・測定結果を PR 本文に記録し、本ファイルの「実施状況」節へ PR 番号を追記する（検証結果は git 管理下に置かない。`records/.gitignore` を参照）。
3. プラグインバージョンを上げる。モデル固定値の変更はコスト・速度・品質特性の観測可能な変化を伴うため、PR #324 の前例に倣い MINOR とする（同 PR は `inherit` → 固定値の変更を同じ理由で MINOR に格上げした）。

### 2026-07-16 評価で判明した前提の誤り

- **コスト前提**: Opus 4.8 対 Sonnet 5 の単価差は 1.67 倍にとどまる（$5/$25 対 $3/$15）。両モデルとも入出力の単価比が 5:1 で共通のため、コスト比は内訳によらず一意に定まる。実測では Sonnet 側がトークンを多く消費し、実質 1.38 倍まで目減りした。所要時間の優位も観測されなかった。**降格の経済的動機は初回反映時の想定より小さい。**この誤りは初回反映時点（Opus 4.7 対 Sonnet 4.6 も同じ 1.67 倍）から存在していた。

### 測定設計上の既知の制約

- **detached worktree は同一リポジトリの全履歴を共有する**ため、評価対象時点より後のコミットを参照できてしまう。2026-07-16 の評価では 4 件中 2 件が実際に修正コミットを参照した。実運用時点では存在しない情報であり、当該判断の独立性を損なう。**漏洩源が存在しない題材（未着手の Open Issue）を選ぶことで回避する。**
- **メイン作業ツリーが同ディスク上に実在する**ため、cwd を worktree に置いても絶対パス探索は防げない。2026-07-16 の評価では 2 件がこれを行い、割り当て文脈では成立しない結論に至った。
