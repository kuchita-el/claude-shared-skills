# Issue #317 サブエージェント実行モデル選定メソドロジー

dev-workflow プラグインの 6 サブエージェントについて、`model` フロントマター値を役割別に最適化する際の評価メソドロジーと初回反映の暫定判定根拠を記録する。AC2（評価方法・サンプルタスク・判定指標の定義）と AC1（選定根拠の文書化）の一次資料。

## メタ

- 対象リポ: kuchita-el/claude-shared-skills
- 対象 Issue: https://github.com/kuchita-el/claude-shared-skills/issues/317
- 実施日: 2026-06-21
- 担当: kuchita-el + Claude（メインループは model: opus-4-7）
- 表記形式: エイリアス形式（`opus` / `sonnet` / `haiku` / `inherit`）。具体ID形式は不採用（モデル世代更新時の保守責務最小化のため）。本 Issue では `haiku` 採用エージェントはなく、評価は Opus / Sonnet の 2 値比較に絞る。
- 判定閾値: 中庸方針（重要観点＝バグ/セキュリティ・要件充足の見落とし 1 件で降格不可、規約違反等の見落としは許容）。

## 目的

dev-workflow の 6 サブエージェントは現状全件 `model: inherit` で親（多くの場合 Opus）を継承し、機械的タスクまで Opus 単価で実行されている。これを役割別に最適化し、Sonnet で十分な精査タスクは Sonnet 固定、設計の本丸は Opus 固定へ移行する。本文書は次の 2 点を提供する。

1. **評価メソドロジー**: 降格対象エージェントについて、Opus 版と Sonnet 版を比較し降格可否を判定するための手順・指標・閾値。
2. **初回反映の暫定判定**: 文書ベース推論で導いた初期 `model` 値と根拠。実機並行評価は後続ドッグフードで継続。

## 評価対象エージェント

| エージェント | 役割（要約） | 初期 `model` 値 | 判定区分 |
|---|---|---|---|
| code-reviewer | 差分の精査（要件充足・バグ/セキュリティ・規約違反・スコープ逸脱・コード重複・テスト品質の 6 観点） | sonnet | 降格対象（文書ベース判定） |
| plan-reviewer | プランのファイル実在検証＋初見・独立視点の懐疑レビュー | sonnet | 降格対象（文書ベース判定） |
| test-designer | 入力ソースからテスト仕様を設計（テストコード非生成） | sonnet | 降格対象（文書ベース判定） |
| test-spec-validator | test-designer の仕様を独立視点で検証（カバレッジ漏れ・観点漏れ・誤読検出） | sonnet | 降格対象（文書ベース判定） |

| plan | writing-plans メソドロジーで計画骨格を生成（設計の本丸） | opus | 維持確定（評価対象外） |

## 判定指標

各エージェントについて、Opus 版と Sonnet 版を同一入力で並列実行した結果を以下 4 軸で比較する。

1. **ブロッカー指摘の見落とし件数**: Opus 版が出した「ブロッカー」相当の指摘のうち、Sonnet 版が出せなかった件数。中庸方針では重要観点（バグ/セキュリティ・要件充足）の見落とし 1 件で降格不可と判定する。
2. **観点カバレッジ**: 各エージェントの責務に含まれる観点（例: code-reviewer は「要件充足・バグ/セキュリティ・規約違反・スコープ逸脱・コード重複・テスト品質」の 6 観点）について、Sonnet 版が触れた観点の割合。50% 未満で降格不可、50%〜80% は要警戒（降格は可だが後続ドッグフードで早期再評価）、80% 以上は許容範囲。
3. **妥当な指摘の総数と比率**: Sonnet 版が出した指摘総数のうち、人間が「妥当」と判定する比率。誤検知が増えていないかの確認。
4. **質的評価（Opus 版にあって Sonnet 版にない指摘の評価）**: Opus 版にしかない指摘のうち**指標 1 のブロッカー相当を除いた残余**について、「重要・あれば望ましい・なくてもよい」の 3 段で人間が分類する。重要 1 件以上で降格不可。指標 1 と 4 は重複しない集合に対する評価で、二重計上を避ける（指標 1 はブロッカー相当の差集合、指標 4 はそれ以外の差集合）。

## 判定閾値（中庸方針）

降格可否は以下のフローで決定する。

```
ブロッカー指摘の見落とし（指標 1）が重要観点で 1 件以上 → 降格不可
↓ なし
質的評価（指標 4）で「重要」分類が 1 件以上 → 降格不可
↓ なし
観点カバレッジ（指標 2）が 50% 未満 → 降格不可
↓ 50% 以上
観点カバレッジ（指標 2）が 50%〜80% → 降格可・要警戒（後続ドッグフードで早期再評価）
↓ 80% 以上
降格可（Sonnet で運用）
```

「規約違反等の見落とし」「文体・表現の差異」「指摘の網羅性が Opus 版より低い」のみで降格不可とはしない（中庸方針の許容範囲）。

## 評価手順

実機並行評価を実施する場合の標準手順。

1. **対象エージェントの `model` を `sonnet` に変更**: `plugins/dev-workflow/agents/<name>.md` のフロントマター `model` を一時的に `sonnet` に書き換える。
2. **比較用ブランチで `model: inherit`（Opus 親）を別途準備**: 同一サンプル入力に対する Opus 版出力を得るため、別ブランチまたは別 worktree で `model: inherit` のままにする。
3. **Agent ツールで両エージェントを起動し、同一サンプル入力を渡す**: 出力をテキストとして保存。
4. **判定指標 1〜4 を記録**: 評価条件（入力・正解キー・両版のプロンプトが同一であること）と測定結果を記録する。記録先は当該変更の PR 本文とする（検証結果は git 管理下に置かない。本ディレクトリの `.gitignore` を参照）。
5. **判定閾値フローに従って降格可否を決定**: 「降格不可」となったエージェントは `model` を `opus` または `inherit` に巻き戻す。

実機評価は本 Issue では実施せず、後続ドッグフードで継続する（リソース現実性のため）。

## 文書ベース暫定判定（初回反映の根拠）

実機評価を実施する前の段階で、各エージェントの責務とタスクの機械性から Sonnet 降格可否を文書ベースで推論する。

### code-reviewer → sonnet 採用

- 責務は「要件充足・バグ・規約違反・スコープ逸脱・テスト品質」の精査。
- 規約違反検出・スコープ逸脱検出は機械的（型化された判定）。要件充足判定は推論寄りだが、入力（プラン・Issue 本文）が構造化されているため Sonnet で十分対応可能と推定。
- バグ検出のうち高難度のロジック誤りは Opus 優位だが、本エージェントの主用途は implementation Phase 2 のセルフレビューであり、後段の人間レビューが直列に続く。見落とし耐性は中庸方針内に収まると判断。
- リスク: バグ検出能力で Opus と差が出る可能性。後続実機評価で再検証。

### plan-reviewer → sonnet 採用

- 責務はファイル実在検証 + プランの懐疑的レビュー。実在検証は Grep/Read による機械的タスク（Sonnet で十分）。
- プラン懐疑レビューは観点ベース（10 観点のチェックリスト）で運用されており、観点を逐一照合する処理は Sonnet 単価で十分推定。
- 既存運用で plan-reviewer は PASS/FAIL の二値出力で、判定根拠は構造化済。
- リスク: 「観点間の関係性に基づく深い推論」が要る場面で Opus 優位の可能性。後続実機評価で再検証。

### test-designer → sonnet 採用

- 責務は「入力ソースからテスト仕様を設計」（テストコード非生成）。
- 観点（典型/境界/異常/状態遷移/組み合わせ）と採用技法（同値分割/境界値/ディシジョン/状態遷移/ユースケース）はクローズドリスト化されており、入力 AC から型化された変換が可能。
- ドメイン特化の境界値発見は Opus 優位の可能性があるが、本エージェントの観点クローズドリストはそれを Sonnet にも追随可能にしている。
- リスク: ドメイン依存の境界値見落とし。後続実機評価で再検証。

### test-spec-validator → sonnet 採用

- 責務は test-designer の出力を独立視点で検証（カバレッジ漏れ・観点漏れ・誤読検出）。
- 独立視点による「漏れ検出」は推論寄りだが、test-designer のクローズドリスト準拠を前提とすれば、観点照合は機械的。
- リスク: 観点漏れの質的判定（重要な漏れか軽微な漏れか）で Opus 優位の可能性。中庸方針で許容範囲と判断。後続実機評価で再検証。

### refactorer → sonnet 採用（リスク中）

- 責務は確定済みコードに対するリファクタリング適用（外部振る舞い不変）。
- 内部構造の改善判断は推論寄り。コードの設計意図を読み解いてリファクタリング案を提示する性質上、Opus 優位の可能性は他 4 件より高い。
- ただし、リファクタは本質的に HOW 寄り（プロジェクト DoR でも「適正な抽象度の厳しさ＝低」と整理されている）。Sonnet でも「振る舞い不変＋テスト維持」の制約内で十分機能すると推定。
- リスク: 大規模リファクタの設計品質。後続実機評価では特にこのエージェントの結果を注視。

### plan → opus 固定（維持・評価対象外）

- 責務は writing-plans メソドロジーで計画骨格を生成する設計の本丸。
- CLAUDE.md 原則「設計・難所のみ Opus」の典型適用例。
- 本 Issue のプラン生成自体に使われており、出力品質が直接利用される（メタ循環）。
- 評価コストを払わない選択をした理由: Issue 制約「降格不可なら Opus 維持」を尊重し、設計品質リスクを優先。

## 巻き戻し条件と手順

後続実機評価で「降格不可」判定が出たエージェントは以下手順で巻き戻す。

1. `plugins/dev-workflow/agents/<name>.md` の `model` を `opus`（または `inherit`）に書き換え。
2. 巻き戻し理由・評価条件・測定結果を PR 本文に記録する（検証結果は git 管理下に置かない。本ディレクトリの `.gitignore` を参照）。
3. プラグインバージョンを上げる。モデル固定値の変更はコスト・速度・品質特性の観測可能な変化を伴うため、PR #324 の前例に倣い MINOR とする（同 PR は `inherit` → 固定値の変更を同じ理由で MINOR に格上げした）。
4. PR で巻き戻しの根拠を明示。

## 後続実機評価の計画

本 Issue では実機評価を実施しないが、以下の前提でドッグフードを継続する。

- 担当: 各エージェントを実運用で使った後、Opus と Sonnet の差を当該変更の PR 本文に記録する。
- タイミング: 各エージェントが 3 回以上実運用で起動された時点で初回評価。
- 判定: 本ファイルの「判定閾値（中庸方針）」フローに従う。
- 巻き戻し優先度: refactorer > code-reviewer > test-spec-validator > test-designer > plan-reviewer の順で観察。
