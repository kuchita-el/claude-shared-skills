# 開発の原則 — AI×人間協働ワークフローの根拠
## ── コーディングエージェントを基盤とした開発ライフサイクルの「なぜ」

---

本書は AI エージェントと人間の協働による開発ワークフローの**根拠・哲学**を扱う Explanation 文書（根拠ハブ）である。構造（集約・状態・ポリシー）の唯一の情報源は `docs/development/event-storming.md` とドメインモデル、具体的な手順は各 `SKILL.md`、個別の技術的意思決定は `docs/adr/` が持つ。本書はこれらを再記述せず一方向に参照する（根拠 → 構造/操作）。文書類型の役割分担は ADR-20260531-2 を参照。

## スコープと位置づけ

**対象 = build スライス（Discovery→Delivery）。** 本書が深く扱うのは「何を作るか」を固める Discovery と、それを継続的に届ける Delivery。ライフサイクル全体の広さ（企画〜運用保守）は [`docs/big-picture.md`](big-picture.md) が持つため、本書では再記述せずリンクで委譲する。

**ワークフローの骨格。** Discovery で 問題定義（`docs/problem-statement.md`：誰のどんな問題か・成功指標・スコープ外）→ ドメイン構造（`event-storming` / `domain-modeling`）→ デリバリーアイテム定義（ユースケース仕様 `spec.md` + Issue）を固め、Delivery で 技術設計+タスク分解（`plan-issue`）→ 実装+セルフレビュー+PR（`dev-loop`）→ 自動AIレビュー → 人間レビュー → マージ と進める。各フェーズの担い手スキル・入出力・状態は [スキルとフェーズの対応](references/skill-phase-mapping.md) を参照。Discovery から Delivery への移行可否は、問題定義・主要ドメインのドメイン構造・最初のデリバリーアイテムのユースケース仕様が揃っているかで判断する。

**実装メカニクスは superpowers に委譲する。** TDDサイクル・検証ゲート・コードレビュー・worktree並列・subagent駆動実装といった汎用実装メカニクスは OSS プラグイン superpowers に委譲し、dev-workflow は Discovery（ドメイン設計）と接続契約（検証方針・判断依頼・レビュー契約・エスカレーション）に縮退する。委譲境界の根拠は [ADR-20260531](adr/ADR-20260531-superpowers-delegation-boundary.md) を参照。

**結合方向 = 根拠 → 構造/操作の一方向。** 構造（集約・状態・ポリシー）と操作（具体的手順・設定例）の唯一の SoT は別文書が持つ。本書はそれらを再記述せず参照する。検証ゲートの構成定義を [event-storming.md「検証実行」集約](development/event-storming.md#検証実行) へ参照委譲しているのはこの適用例。

**スコープ外。** チーム規模・チーム構成に依存する合意形成プロセス（レビュー承認フロー、複数ステークホルダー間の意思決定等）は含めない（[ADR-20260513](adr/ADR-20260513-workflow-design-scope-team.md)）。

**継続的に更新するドキュメント。** 以下は Discovery で骨格を作り、Delivery を通じてボトムアップに育てる。Discovery 完了時に完備されている必要はない。

- **CLAUDE.md:** プロジェクト開始時に最低限の記述（言語・フレームワーク・ディレクトリ構造）を置き、レビューで発見したパターンを都度追記する
- **ADR:** 技術スタック選定等の初期判断があれば記録する。以降はPRの設計判断が複数機能で参照されるようになった時点で昇格させる

## 基本原則

本ワークフローは3つの原則に基づく。

**原則①：計画と実装を分離する**
AIにいきなりコードを書かせない。仕様（Spec）を先に作り、人間がそれを承認してから実装に入る。仕様が「信頼できる唯一の情報源（Single Source of Truth）」となり、AIの実装を検証する基準になる。ACでは自明でない振る舞いは、`plan-issue` の検証方針セクションで設計レベルの振る舞い仕様として記述し、人間が承認する（詳細は `skills/plan-issue/SKILL.md`）。

**原則②：フェーズゲートで品質を積み上げる**
実装中の数百のマイクロ判断を逐一レビューするのではなく、フェーズの境界（ゲート）で集中的にレビューする。検証ゲート（機械的品質チェック）→ セルフレビュー → 自動AIレビュー → 人間レビュー → マージ、と段階的にゲートを通す。これにより、レビューの認知負荷を下げつつ品質を維持できる。

**原則③：AIにAIを監視させ、人間は判断に集中する**
書いたAI（セッション）とは別のAI（セッション）にレビューさせ、自己評価バイアスを排除する。人間はAIが苦手な「文脈を踏まえた判断」に集中する。

## なぜ事前にドメイン構造が必要か（70%の見通し）

ドメインの全体像と内部構造は、デリバリーアイテムの実装に先立って作成する。デリバリーアイテムを個別に切り出してAIに実装させると、AIは各デリバリーアイテムの局所的な情報だけで設計判断を下す。たとえば契約管理ドメインで「契約を新規作成できる」だけを渡すと、AIはその時点で必要な構造だけでモデルを作る。次の「契約を変更できる」で `amendment` の概念が登場し、さらに「契約を解約できる」「自動更新する」とデリバリーアイテムが進むたびにモデルの拡張・再設計が発生し、最悪の場合、数アイテム後に「ドメインモデルをゼロから作り直したい」事態に陥る。

これを防ぐために、デリバリーアイテムの実装に先立ってドメインの全体像を「70%の見通し」として確保する。完璧な事前設計ではなく、デリバリーアイテムの実装を通じて発見した知見はドメイン構造にフィードバックする。ドメイン構造（ドメインイベント・コマンド・集約・状態遷移・データ型・ワークフロー型、集約境界、コンテキスト間の依存方向）の洗い出しは `event-storming` / `domain-modeling` スキルが担い、その出力（`docs/{domain}/event-storming.md` 等）が構造の SoT となる（[スキルとフェーズの対応](references/skill-phase-mapping.md)）。

## 情報設計の原則

### ストック情報とフロー情報

ドキュメントを「ストック」と「フロー」に区別する。

| 分類 | 性質 | 例 |
|------|------|-----|
| ストック | 蓄積され、繰り返し参照される | problem-statement.md, event-storming.md, domain-modeling.md, ユースケース仕様（spec.md）, ADR |
| フロー | 作業を駆動し、完了後は参照頻度が下がる | Issue, PR, 実装プラン（docs/plans/） |

**原則：ストックとフローは異なる整理原理に従う。** ストック情報はソフトウェアアーキテクチャの構造（ドメイン > ユースケース）で整理する。フロー情報はプロジェクト管理の構造（テーマ > デリバリーアイテム > タスク）で整理する。両者はデリバリーアイテム（Issue）がどのユースケース仕様を参照するか、で接続される。

**原則：フェーズではなくドメインで分類する。** Discovery/Deliveryは時間軸（フェーズ）であり、ディレクトリ構造（空間軸）に反映すべきではない。`docs/discovery/` のようなフェーズ名のディレクトリは作らない。ドメイン構造もユースケース仕様もDiscoveryで作成されるが、Deliveryで継続的に参照・更新される。

### ドキュメント間の結合度

**原則：ドメイン内は密結合OK、ドメイン間は疎結合にする。**

各ドメインのドキュメント（event-storming.md、domain-modeling.md、ユースケース仕様）は同一ドメイン内で密に参照し合ってよい。一方、あるドメインのドキュメントが別ドメインの内部ドキュメントを直接参照してはならない。ドメイン間の接点は `docs/big-picture.md`（コンテキストマップ）に集約する。この原則により、ドメイン数が増えてもドキュメント間の整合性コストがドメイン数の二乗に比例して増大することを防ぐ。

### 具体例

ディレクトリ構成・外部ツールとの連携の具体例は [ドキュメント構成の具体例](references/documentation-structure-examples.md) を参照。作業単位の階層（テーマ／デリバリーアイテム／タスク）とSprint運用は [プロジェクト管理とSprint運用](references/sprint-management-examples.md) を参照。

## テスト方針

Discovery（AC作成）とDelivery（テスト設計・TDD実行）の両フェーズで参照する横断的ガイドライン。プロジェクト固有の追加方針（使用フレームワーク、カバレッジ基準、モック方針等）はプロジェクトのCLAUDE.mdに定義する。

**デフォルトで考慮する観点：**

- 正常系: 主要なユースケースパスの網羅
- 異常系: 不正入力、権限不足、外部サービス障害等
- 境界値: 上限・下限、空・null・ゼロ、型の境界
- 状態遷移: ドメイン構造で定義された遷移パスと禁止遷移の検証

**観点の具体化：** デフォルト観点はplan-issueの検証方針セクションで、選択した技術アプローチに応じたBDD的記述として具体化される。test-designerはこの検証方針セクションをコードレベルのテストケースに変換する。

## 変更パターン別の運用

すべての変更にフルプロセスを適用するとオーバーヘッドが過大になる。変更の性質に応じて、適用するプロセスのレベルを使い分ける。

```
変更の要求が来た
    │
    ├─ ドメインの概念（状態遷移、エンティティ構造）が変わる？
    │     │
    │     YES → パターンC: ドメイン構造更新 → ユースケース仕様作成/更新 → 実装
    │
    ├─ 既存デリバリーアイテムの振る舞い（受け入れ基準）が変わる？
    │     │
    │     YES → パターンA: ユースケース仕様更新 → 実装
    │
    └─ 上記いずれでもない（内部的な修正、バグ修正、軽微な調整）
          │
          └─ パターンB: 直接実装（Issue/チケットで管理すれば十分）
```

**パターンA：既存デリバリーアイテムの拡張（ユースケース仕様を更新する）**

例：契約一覧に「契約金額の範囲」で検索できるフィルターを追加する場合。既存のユースケース仕様に受け入れ基準を追記し、実装プランを更新してから実装する。ドメイン構造の更新は不要（契約のライフサイクルやエンティティ構造は変わらないため）。ユースケース仕様更新時は変更履歴をドキュメント内（spec.md の `## 変更履歴`）に残す。

**パターンB：軽微な修正（ユースケース仕様の更新不要）**

例：一覧のデフォルトソート順変更、バグ修正、リファクタリングなど。受け入れ基準が変わらない変更にはSpecプロセスを適用しない。既存テストの期待値修正と実装変更をコミットすれば十分。判断基準：受け入れ基準が変わるならユースケース仕様を更新、内部挙動だけが変わるなら更新不要。

**パターンC：ドメインモデルに影響する変更（ドメイン構造も更新する）**

例：契約に多段階の承認ワークフローを追加する場合。状態遷移が変わり、新しい集約（Approval）が追加される。更新の順序が重要：ドメイン構造更新（`event-storming` スキルの再実行、人間が承認）→ ユースケース仕様作成/更新 → 実装。

**日常の開発では大半がパターンAとBであり、パターンCは月に数回程度。** ユースケース仕様の運用が重くなりすぎないためには「受け入れ基準が変わるときだけユースケース仕様を更新する」くらいのルールが現実的である。

## AIと人間の責任分担

### 責任分担マトリクス

```
              AIが主導        AI+人間の協働       人間が主導
            ─────────────  ────────────────  ─────────────
計画        仕様書の起草     要件の曖昧性解消     ビジネス判断
            タスク分解       設計のトレードオフ   優先順位決定

実装        コード生成       複雑な設計判断       新規アーキテクチャ
            テスト生成       エッジケース特定     ライブラリ選定
            リファクタリング  パフォーマンス最適化

検証        lint/型チェック   テスト戦略設計       受け入れテスト
            脆弱性スキャン   テスト十分性判断     探索的テスト
            回帰テスト実行   ビジネスロジック検証

レビュー    コード規約チェック ロジックの正しさ     アーキテクチャ判断
            PRサマリー生成   セキュリティ影響評価  マージ可否決定
            重複コード検出   影響範囲の評価       チーム方針との整合
```

### 承認ゲート軸（自律度レベル別）

上記マトリクスの3段（AIが主導／AI+人間の協働／人間が主導）は「誰が主導するか」を表す。これに直交する軸として、**判断を確定する前に人間の承認ゲートを通すか否か** で自律度レベルを区別する。担い手はフェーズではなく判断の性質で決まる。

| 自律度レベル | 概要 | 承認ゲート |
|---|---|---|
| L0（人のみ） | 人間が直接実施し、AI による提案を介さない | 縮退形（提案なし） |
| L1（人主導・AI補助） | 人間が主導し、AI が補助する | 人間の判断内（明示ゲートなし） |
| L2（AI主導・人承認） | AI が提案を主導し、人間が承認する | **有り**（提案 → 承認の二段） |
| L3（AI自律） | AI が提案から確定まで自律実行する | **無し**（縮退形） |

L2 と L3 の差は承認ゲートの有無で表現する。この二段表現（提案 → 承認）は技術的意思決定集約のライフサイクル（Proposed → Accepted、[ADR-20260511](adr/ADR-20260511-technical-decision-aggregate-foundation.md)）と同型であり、L0／L3 はその縮退形にあたる。

自律度を表現する軸を責任分担マトリクス（本 Explanation 文書）に置き、ドメインモデル（`docs/development/event-storming.md`）には持ち込まない方針の根拠は [ADR-20260601](adr/ADR-20260601-autonomy-approval-gate-alignment.md) を参照（ADR-20260421 と非重複）。L0–L3 各レベルの正式定義・判定基準は規約 ADR（#238）および判定ルーブリック（#239）で確定する。

### レビューの役割分担（自動AIレビュー ↔ 人間）

PR は自動AIレビュー（CodeRabbit / PR-Agent 等）を一次フィルターとし、人間が最終判断する。観点ごとの担い手は以下。

| レビュー観点 | 担当 | 検出方法 |
|------------|------|---------|
| 構文エラー・型エラー | CI（自動） | コンパイラ, リンター |
| コーディング規約違反 | CI（自動） | リンター, フォーマッター |
| 既知の脆弱性パターン | AIボット | SAST + LLMパターンマッチ |
| テストカバレッジ不足 | AIボット | カバレッジレポート + LLM分析 |
| 命名の一貫性 | AIボット | コードベース全体との比較 |
| ロジックの正しさ | AIボット + 人間 | LLM推論 + 人間の判断 |
| 仕様との整合性 | 人間 | ユースケース仕様 / ACとの突合 |
| アーキテクチャ適合性 | 人間 | 設計原則との照合 |
| ビジネス価値の判断 | 人間 | ドメイン知識 |

### 人間がレビューすべき観点（AIに任せられないもの）

**計画（plan-issue のゲート）：** 既存アーキテクチャとの一貫性、ドメイン構造で定義された集約・状態遷移への準拠、検証方針の妥当性（重要な振る舞いが漏れなく識別されているか）、テストレベルの選択、タスク粒度・順序・依存関係。

**コード（人間レビューのゲート）：**

- この変更は本当に必要か？（要件の妥当性）
- チームの暗黙知と整合しているか？
- 将来の拡張性に問題はないか？
- ユーザー体験への影響は？
- 他のチームへの影響は？

AIレビューボットが生成するPRサマリーとウォークスルーを起点に読むことで、大きな差分でも変更の意図と影響範囲を素早く把握できる。人間はAIが「OK」と判断した部分は軽く確認し、AIが指摘した部分と、AIが判断できない高次の観点に集中する。CIがすべて通り、AIレビューに重大な指摘がなく、人間レビュアーがApproveしたらマージ可能。

## アンチパターンと対策

| アンチパターン | なぜ危険か | 対策 |
|-------------|----------|------|
| ドメイン構造なしでデリバリーアイテムを積み上げる | データモデルが断片的になり、数アイテム後に破壊的な作り直しが発生 | デリバリーアイテムの実装に先立ってドメイン構造を作成し、全体像を確保する |
| ドメイン構造で完璧を目指す | ウォーターフォール化し、着手が遅れる | 「70%の見通し」で十分。残り30%はデリバリーアイテムの実装を通じて発見・反映する |
| すべての変更にフルプロセスを適用 | 軽微な修正でもオーバーヘッドが発生し、チームが疲弊する | パターンA/B/Cの判断フローに従い、受け入れ基準が変わるときだけユースケース仕様を更新 |
| 仕様なしでいきなり実装させる | AIの解釈が要件とずれ、大規模な手戻りが発生 | 必ず技術設計+タスク分解を経てから実装に入る |
| 巨大なタスクを1つのセッションで | コンテキスト枯渇で品質が劣化 | 1タスク＝1コミットの粒度でサブエージェントに委任 |
| デリバリーアイテムの粒度が大きすぎる | PRが肥大化しレビュー品質が低下、AIレビューの精度も落ちる | 受け入れ基準3〜7個、レビュアーが30分で理解できるサイズに分割 |
| AIレビューだけで人間レビューを省略 | ビジネスロジックの誤りやアーキテクチャ劣化を見逃す | AIレビューは「フィルター」、人間レビューは「判断」 |
| テスト生成をAIに丸投げ | ハッピーパスのみのテストになりがち | テスト観点（異常系、境界値）を明示的に指示 |
| CLAUDE.mdを更新しない | 同じ間違いが繰り返される | レビューで発見したパターンを都度追記 |
| AIの出力を無条件に信頼 | 微妙なロジックエラーが本番に混入 | セルフレビュー＋人間レビューの二段構え |
| ストック情報をIssueだけに書く | Issueがcloseされると埋もれ、後続アイテムで過去のスコープが不明になる | ユースケース仕様にストック情報を残し、IssueはACの転記+作業駆動用として使う |
