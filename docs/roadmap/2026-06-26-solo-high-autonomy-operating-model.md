## 0. 課題設定

一人の開発者が、ソフトウェア開発のライフサイクル全体——企画・要件擦り合わせ・実装・運用——を Claude Code とともに回そうとしている。本人の問題意識は「一人では厳しい。人＋Claude の2並列でも無理」。

ボトルネックは特定フェーズではなく**全フェーズ**。さらに本人が自前で5つ目を追加した——「実装されたソフトウェアの仕様把握や要求とのトレース」。

自律性の方針は **原則委任（高自律）** を選択。ガードレールと検証が通る限り Claude に任せ、本人は例外・警告・方針ずれの時だけ介入する。

## 1. 運転モデルの核心

### 1.1 一人開発で本当に失われるもの

失われるのは「各フェーズの速さ」ではなく、**チームが暗黙に提供していた4つの機能**である。

- **外部記憶** — 他人が覚えていてくれた経緯・決定理由。一人だと全部自分の頭に乗り、乗り切らない
- **独立レビュー** — 自分の盲点を別の目が潰す。一人だと盲点がそのまま通る
- **並列の手** — 分担。一人だと全タスクが自分を直列で通過する
- **説明の強制力** — 他人に説明するために考えが整理される。一人だと曖昧なまま進む

本人が追加した「トレース」は、この**外部記憶**と**4機能を貫く一本の糸**そのもの。

したがって課題は「Claude を各フェーズで速くする」ことではない。**消えたチーム機能を、本人の周りにシステムとして作り直す**ことである。

### 1.2 役割の再定義と「希少資源の移動」

高自律にすると、本人のボトルネックは作業から**2つ**に移動する。これが運転モデルの核心。

- **委任できない資源 ＝ 何を作るかの判断（企画の taste・優先順位）**
  これは本人の戦略・美意識そのもの。Claude に渡せない。むしろ高自律にすることで、ここに時間を**集中させる**のが目的。
- **委任してはいけない方法で消耗してはならない資源 ＝ 検証**
  「全部自分で読んでレビューする」は直列ボトルネックに逆戻り。高自律下の検証は**敵対的レビューエージェント＋CI＋DoRゲート**に肩代わりさせ、本人は**例外と最終ゲートだけ**見る。

本人の新しい役割は **director（方向付け）＋例外処理＋重要ゲートの最終承認**。Claude は自律チーム。この運転モデルを名付ければ **例外管理（management by exception）**。

### 1.3 鉄則：検証強度が自律の上限を決める

高自律には鉄則がある。**検証が自動・敵対的に回っていない領域に、自律を上げてはいけない。** 検証の強度が、許してよい自律の上限を決める。

### 1.4 スループットの正体：「判断を文章に書き出す＝委任できる判断が増える」

高自律で一人＋Claude が回る量を決めるのは「Claude の速さ」ではない。**本人を直列で通過しなければならない判断の数**である。これを減らす唯一の方法は、**本人の判断基準を文章に書き出すこと**。

書き出された判断は、以後 Claude が「本人として」下せる。CLAUDE.md・ADR・DoRプロファイル・spec は機能の整理ではなく、**本人の taste の外部化＝委任可能化**である。

裏を返すと——**自律の上限を上げる行為とは、本人の判断を文章に書き出す行為**にほかならない。

## 2. 管制OS（レイヤー0）— 少人数で全体を監督するための土台

これが無いと「なんでもやる」は破綻する。高自律を統治する仕組み。

> 本レイヤー（管制OS）は §1.2 の例外管理（management by exception）の運転実体であり、§8.3 **G2**／#360 の「例外管理OS」と同一を指す。以降、本書では「管制OS＝例外管理OS＝G2」を同義として扱う。

- **例外キュー** — Claude が「本人を必要とする件」だけを積む。割り込み駆動をやめ、非同期バッチで捌く
- **ゲート定義** — 人手承認が要る点を明文化（例: spec確定／PRマージ／リリースの3点だけ）。それ以外は自動通過
- **通知・エスカレーション方針** — スマホ通知等で、止血が要る時だけ呼ばれる
- **権限・サンドボックス** — allowlist、不可逆操作の禁止域を設定
- **監査証跡** — 本人は全部読まず、抜き取りで spot-check する前提を作る

## 4. 運用・保守・監視（レイヤー5）— 本人が名指しした弱点

ここが一番「やれることが残っている」。高自律運用の本丸。

| やれること |
|---|
| **監視→例外サーフェス**: alert / log / dashboard を scheduled agent が定期巡回し、異常だけを例外キューへ |
| **インシデント自動対応**: alert を Claude が読む→診断→ガードレール内で修正PR or 自動止血→本人は承認だけ |
| **問い合わせ自動トリアージ**: 受信→分類→FAQ自動回答 or Issue化 |
| **runbook のスキル化**: 定型運用手順をスキルにし、Claude が実行する（手順書＝委任可能な判断） |
| **保守の自動PR**: 依存更新（dependency-check ＋定期実行）、軽微修正を自動起票・自動PR |
| **テレメトリ分析**: 利用データを定期 agent が要約し、企画レイヤーへ還流 |

## 5. 複利ループ（レイヤー6）— 「一人で全部」を可能にする唯一の倍率

> **［重複・SoT は別文書］** 本節は `plugins/growth/DESIGN.md`（Capture→Distill→Route→Promote→Distribute→Measure/Retire の完全ループ・7原理・二段ゲート・個人store＝生捕捉→committed昇格・フリート配布）の劣化要約である。原型はスナップショットとして残すが、**今後の更新はロードマップ側ではなくリンク先（`plugins/growth/DESIGN.md`）で行う**。差異が生じた場合はリンク先を正とする。本書側に残す固有の論点は「複利＝solo運転の前提条件」という位置づけのみ。

**「なんでもやる」を一人で実現する正攻法は、全部を手で建てることではない——システムが自分自身に機能を足し続ける状態を作ること。**

- すべての修正・指摘・インシデントを **内省→学び→policy更新** に流す
- 学びは CLAUDE.md / ADR / 新スキル として外部化＝以後 Claude が「本人として」その判断を下す
- 結果、**本人の介入1回が、以後の介入を1回減らす**。手数が複利で増える

これが無いと、機能を足すほど監督負荷が線形に増えて、結局回らなくなる。複利ループは「なんでもやりたい」の前提条件である。

## 6. 到達戦略

### 6.1 専門家としての本音

「やれることは全部」は正しい野心だが、**全部を同時に建てるのは、検証されていない巨大並列作業で、運転モデルそのものに反する**（鉄則: 検証強度＞自律。本人の哲学: 可逆性優先・迷ったら止まる）。

到達法は2つしかない。

1. **管制OS（レイヤー0）と複利ループ（レイヤー6）を先に作る**
2. **残りをシステム自身に足させる**

順番を守れば「全部」は射程に入る。守らなければ機能の山に潰される。

### 6.2 着手の3案

| 案 | 先に切る直列 | 効く相手 | 弱点 |
|---|---|---|---|
| **A. 検証先行** | 「自分で全部読んでレビュー」 | 敵対的レビュー＋CI＋DoRゲートを load-bearing 化し、検証を本人の注意から剥がす | 検証の定義が無い領域（例: markdown中心の本repo）では「何を以て検証済みか」から決める必要 |
| **B. トレース先行** | 「経緯を頭で保持」 | 企画→Issue→spec→plan→PR→ADR を ID で一本化。5つ目の課題を直接潰す | 検証が弱いと「ゴミを忠実にトレース」するだけになる |
| **C. 並列先行** | 「タスクを直列で消化」 | Workflow・定期実行・バックグラウンドで手を増やす | A/B 不在で並列化すると、欠陥と迷子を高速で量産する |

**推奨：A（検証先行）。** 理由は鉄則に従う。高自律の上限は検証強度で決まる以上、検証が binding constraint。ここを上げない限り B も C も安全に効かない。委任した瞬間に不可視の欠陥が溜まる。

さらに本repo特有の事実——スキル＝markdown中心で「検証済み」の定義が曖昧。**製品で「何を満たせば人手レビュー無しで信頼してよいか」を先に言語化する**のが、高自律を実際に解禁する最短路。A はそれを強制する。
