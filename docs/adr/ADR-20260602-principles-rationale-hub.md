# ADR-20260602: principles.md（旧 workflow-design.md）の責務 = Explanation 根拠ハブ／原則の射程は全ライフサイクル横断（ADR-20260531-2 を Supersede）

## Context

本 ADR は ADR-20260531-2（`workflow-design.md` を Explanation 根拠ハブへ縮退）を Supersede し、その責務定義を引き継いだ上で、**スコープに関する文面の曖昧さを解消**する。単体で成立するよう、責務定義の全体を再記述する。

### 背景（責務再定義の経緯）

`docs/workflow-design.md`（#221 Phase 3 で `docs/principles.md` にリネーム）は、性質の異なる3層を一身に抱えていた。

- **構造層**: 全体フロー概観、ドメイン構造「決めること」、Delivery 各ゲート記述、検証ゲート構成定義
- **操作層**: spec.md 構成例、デリバリーアイテム粒度ガイドライン、ADR 記録の判断基準、テスト方針、ドキュメンテーション戦略、ドキュメント間結合度原則、スキルとフェーズの対応、作業単位の階層・Sprint 運用
- **根拠層**: 基本原則、なぜ事前にドメイン構造が必要か（70%見通し）、変更パターン A/B/C、責任分担マトリクス、アンチパターン

構造層・操作層には別の SoT が存在する（構造は `docs/development/event-storming.md` / domain-model、操作は各 SKILL.md・`CLAUDE.md`・`docs/references/`）。これらを principles.md が二重に抱えることが構造ドリフトの温床だった。一方、根拠層には固有のホームが無い。よって principles.md を**根拠層のみを保持する Explanation 根拠ハブ**へ縮退させた。

### Supersede の理由（解消する曖昧さ）

ADR-20260531-2 は次の2決定を併記していた。

- **Decision #4**: 原則・責任分担・アンチパターンは**全ライフサイクルに効く横断原則であり、フェーズに縛らない**。
- **Decision #5**: **スコープは build スライス（Discovery→Delivery）と明示**。

この2つは「**広さ**（どのフェーズを扱うか・フェーズ列挙）」と「**原則の射程**（原則がどこまで効くか）」という別 facet を指していたが、文面上 #5 が「principles.md は build に閉じた内容しか持てない」という**内容の制約**として誤読される余地があった。実際、Issue #233 のリファインメント時にこの誤読が「検証ギャップ・再帰性等の横断原則は principles.md に載らない（別ホームへ委譲すべき）」という誤った結論を生みかけた。

さらに現状の principles.md は、#232 系の作業（ADR-20260601）で追加された「**承認ゲート軸**」の節で「**担い手はフェーズではなく判断の性質で決まる**」と既に明記しており、フェーズ非依存＝横断的な原則を実際に保持している。すなわち「build 限定」という文面と doc の実態が内部矛盾している。

本 ADR はこの曖昧さを解消し、「**広さの再記述禁止**」と「**原則の射程は横断**」を分離して明示する。

## Decision

`docs/principles.md`（旧 `workflow-design.md`）の責務を以下に（再）定義する。

1. **文書類型 = Diátaxis の Explanation（根拠ハブ）**。Reference（event-storming / domain-model）、How-to（各 SKILL.md）、原子的決定（ADR）と役割分担し、本 repo に欠けていた「方法論の根拠・哲学」ジャンルを埋める。
2. **保持する内容は根拠層のみ**（＝各フェーズの「なぜ」）。代表例: 基本原則、70%見通し、情報設計の原則、変更パターン、責任分担マトリクス（承認ゲート軸を含む）、テスト方針、レビューの役割分担、アンチパターン。**この列挙は網羅でなく代表例であり、最新の節構成は principles.md 本体を正とする**（ADR を目次化してドリフトさせない）。構造層・操作層は各 SoT へ移転する。
3. **結合方向は一方向**: 根拠（hub）→ 参照 → 構造（event-storming / domain-model / big-picture）／操作（各 SKILL.md / CLAUDE.md）。構造・操作の唯一の SoT はそれぞれの専用文書であり、根拠ハブは再記述せずリンクで委譲する。
4. **編成は原則別（フェーズ別でない）**。原則・責任分担・アンチパターンは全ライフサイクルに効く横断原則であり、フェーズに縛らない。
5. **「広さ」を再記述しない（≠ 扱う原則を build に限定する、ではない）**。ライフサイクル全体のフェーズ列挙・広さの SoT は `docs/big-picture.md` が持つため、本書は**広さを再記述せず**リンクで委譲する。これは「**扱うフェーズの広さ**」への制約であって、「**原則の射程**」への制約ではない。Decision #4 のとおり、原則の射程は全ライフサイクル横断でよく、build に閉じない。AI×人間協働の構造的歪み（検証ギャップ・再帰性／自動化バイアス等）のような横断的根拠は、build を越える射程を持っても Explanation ジャンルとして本書が抱える。
   - **判定の目安**: 「どのフェーズが存在し各々何をするか（広さ・列挙）」→ big-picture へ委譲。「なぜそうするか（根拠・哲学）」が複数フェーズに効く → principles.md に置いてよい。

## Consequences

**得られた利益**:

- 構造・操作・根拠の責務が分離され、各層が単一の SoT を持つ。構造ドリフトの温床（同一事実の二重保持）が解消される。
- 「広さ」と「原則の射程」の混同が解消され、横断原則（検証ギャップ・再帰性等）を principles.md に置く根拠が明確になる。承認ゲート軸（フェーズ非依存）と従来のスコープ文面との内部矛盾も解消する。
- 本 repo に欠けていた Explanation ジャンルが埋まり、Diátaxis 4類型（Explanation / Reference / How-to / 原子的決定）の役割分担が明示される。

**受容したトレードオフ**:

- 根拠層に固有ホームが無いため、完全削除（案B）ではなく縮退（案A）に留める。根拠ハブという新ジャンルの維持コストを受容する。

**将来の留保事項**:

- 未実装フェーズ（リリース・運用・観測）の**方法論・手順**は、支えるスキルも Design-Level モデルも無いため書き起こさない。ただしこれは「方法論を書かない」であって「原則の射程を build に限定する」ではない。広さ（フェーズ列挙）は big-picture に委ねる。
