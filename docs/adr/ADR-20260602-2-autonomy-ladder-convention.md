# ADR-20260602-2: 自律度モデル L0–L3 の規約（提案→承認二段・L0/L3 縮退形）

## Status

Accepted

## Context

自律度モデル（L0–L3）の導入が親トラッカー Issue #232 で進行している。先行する子1（#237）の整合方針 ADR（[ADR-20260601](ADR-20260601-autonomy-approval-gate-alignment.md)）が、自律度を Command 属性などとしてドメインモデルへ持ち込まず、`docs/principles.md` の責任分担マトリクスへ **承認ゲート軸** を追加して L2/L3 の差（承認ゲートの有無）を表現する方針を Accepted として確定した。同 ADR は承認ゲート軸を「有り／なし」の二値軸として固定する一方、L0–L3 各レベルの正式定義・判定基準は本規約 ADR（#238）および判定ルーブリック（#239）に留保している。

本 ADR は、その留保された **L0–L3 各レベルの定義**・**「提案→承認」二段方針**・**L0/L3 縮退形** を規約として確定する。Issue #232 の確定設計判断③（自律度モデルを ADR 化する）・④（成果物は整合方針 ADR／規約 ADR の2本に分離）に対応する規約 ADR にあたる。

### 背景: 自律度は静的な梯子でなく検証能力の成熟で動く動態（A4）

L0–L3 を固定的な序列として扱うと、運用実態を取り違える。自律度は management-by-exception のドリフトを起こす。すなわち、検証精度（ガードレール・eval・自動検証）の向上に連れて、同じ判断クラスでも承認ゲートを要する L2 から自律実行の L3 へと担い手が移動しうる。そしてその移動は無償ではなく、「どこからを例外として人へ戻すか」の線引きを設計する上流の仕事（ガードレール／eval の整備）を同時に増やす。

したがって本規約は L0–L3 を静的な梯子としてではなく、**検証能力の成熟に応じて L2↔L3 を移動しうる動態** として位置づける。この動態性は規約そのもの（どう定義するか）ではなく、規約をなぜこの形にするかの背景（Why）であり、判定基準の確定（#239）が独立した上流仕事として要る理由を説明する。

## Decision

自律度モデルの規約として以下を確定する。

1. **L0–L3 の4レベルを定義する**。L0（人のみ）／L1（人主導・AI補助）／L2（AI主導・人承認）／L3（AI自律）の4レベルを、担い手と承認ゲートの有無で定義する。担い手はフェーズではなく判断の性質で決まる。承認ゲートは L2 のみ「有り」、L0/L1/L3 は「なし」となる（二値軸、ADR-20260601）。各レベルの正式な定義テーブル（自律度レベル・概要・承認ゲート）は SoT である `docs/principles.md` 承認ゲート軸テーブル（下記項5）を正とし、二重管理を避けるため本 ADR では再掲しない。

2. **「提案→承認」二段方針**。L2 の承認ゲートは、AI が提案を起票し（提案）、人間がそれを確定する（承認）の二段で表現する。この二段は技術的意思決定集約のライフサイクル（Proposed → Accepted、[ADR-20260511](ADR-20260511-technical-decision-aggregate-foundation.md)）と同型であり、既存パターンの再利用で表現できる。承認ゲートは「有り／なし」の二値軸（ADR-20260601）であり、L2 のみが「有り」となる。

3. **L0/L3 縮退形**。L0 と L3 はともに承認ゲートが「なし」だが、二段（提案→承認）のどちら側が成立しないかで区別される。
   - **L0 縮退形（提案が成立しない）**: AI による提案がそもそも介在しないため、二段の起点（提案）が欠ける。
   - **L3 縮退形（承認が成立しない）**: AI が提案から確定まで自律実行するため、二段の終点（人間の承認）が欠ける。

   同じ「なし」でも縮退の向きが逆である（L0＝起点欠落／L3＝終点欠落）。

   入出力の例:
   - 入力: 自律度 L2 → 出力: 承認ゲート＝有り（AI が提案 Proposed、人が承認 Accepted の二段が成立）
   - 入力: 自律度 L3 → 出力: 承認ゲート＝なし（AI が提案から確定まで自律実行、承認段が縮退）

4. **判定基準は判定ルーブリック（#239）へ委譲する**。ある判断がどのレベルに該当するか（各レベルを分ける具体的条件・閾値）は本 ADR では確定せず、判定ルーブリック（#239）の責務とする。本 ADR は各レベルの定義と承認ゲート対応・縮退形までを規約化する。

5. **正式定義の SoT は `docs/principles.md` 承認ゲート軸テーブルに置く**。ADR はある時点の決定を記録するフロー型文書、`docs/principles.md` は生きた規約を保持するストック型文書という性質差に基づき、L0–L3 正式定義の単一情報源（SoT）は `docs/principles.md` の承認ゲート軸テーブルとする。本 ADR は「L0–L3 をこう定義・規約化することを決定した」という決定と根拠（上記 Context の動態性を含む）を記録し、定義本体の生きた参照先は `docs/principles.md` とする。これにより同一定義が ADR と principles.md の二箇所に分散して二重管理になることを防ぐ。

## Consequences

**得られた利益**:

- L0–L3 の定義・二段方針・縮退形が規約として固定され、判定ルーブリック（#239）・記法拡張（#240）が参照すべき確定した土台ができる。
- 承認ゲートの二段表現（提案→承認）が技術的意思決定集約のライフサイクル（Proposed→Accepted、ADR-20260511）と同型であることが明示され、既存パターンの再利用で表現できる。
- 定義の SoT を principles.md（ストック）に一元化し、ADR（フロー）は決定と根拠の記録に徹することで、同一定義の二重管理を避けられる。

**受容したトレードオフ**:

- 各レベルの判定基準（どこからが L2 か L3 か）は本 ADR では確定せず #239 に委ねるため、本 ADR 単独では「ある判断がどのレベルか」を一意に判定できない。定義（本 ADR）と判定基準（#239）の責務が分離する。
- L0–L3 定義の SoT を principles.md 側に置く方針は、`docs/principles.md` L6 の「個別の技術的意思決定は `docs/adr/` が持つ／本書は再記述せず一方向参照」という従来の結合方向とは逆向き（定義のストックを principles.md 側に置く）になる。この扱いは承認ゲート軸セクションに注記して整合させる。

**将来の留保事項**:

- 各レベルの判定基準（自律度を決める要因と ADR 粒度判定の発想上の対応を含む）は判定ルーブリック（#239）で確定する。
- ゲート二段表現の記法拡張（Command:提案→Event:提案済→Policy:承認→Event:承認済 等の記法）は #240 で扱う（本規約の中身が固まった後に着手）。
- 自律度の動態性（検証精度向上に伴う L2↔L3 の移動）を運用でどう観測・記録するかは、dogfooding の運用観測を踏まえて別途検討する。
