# ADR-20260701: 学習シグナルの復元不能性基準と distill 出力形の2系統分離

## Status

Accepted

## Context

growth の capture/distill を直近の窓（2026-06-29〜30、36 セッション・119 観察）に適用するドッグフーディングを実施したところ、産出された候補（distill 出力 13 件）のナレッジ価値が著しく低く、capture/distill が価値ある知識を構造的に取りこぼしていることが判明した（#432）。

評価の判定基準は「リポジトリ（コード・git・ADR・spec）やハーネスから後で復元できない知識か」（復元不能性）である。設計判断の結論は ADR/spec に残るが、ユーザーの選好表明・却下理由・会話でしか交わされなかった文脈は、捕まえなければ消える。これが本来 capture が狙うべき的である。

証拠分析（119 観察の悉皆分類＋セッションログ 7/41 本のサンプル走査）で、二重の失敗が定量的に裏付けられた。

1. **capture 段のシグナルが摩擦に偏る**: 現行 4 シグナル（訂正・ツール拒否・反復試行・期待違反）は摩擦・エラー事象に偏って発火する。119 観察のうち復元可能ノイズが 65.5%（78 件、うち origin=tool-result が 78 件にほぼ一致）を占め、その大半はハーネスガード発火（"File has not been read yet"・worktree 破棄ガード）・ツールスキーマ検証・リトライ・出力 truncate・API 一時障害といった、ハーネスが既に強制済みで後から決定的に再導出可能な事象だった。一方、最も価値の高い会話知（目標表明・意思決定基準・設計意図・却下理由）は、そもそも予測誤差の形を持たず 4 シグナルの検出射程外にあり、capture が記録すらしなかった（偽陰性）。

2. **distill 段の平板化が価値を破壊する**: 拾えた数少ない判断知（選好・却下理由・設計判断）も、「実行可能な振る舞い差分」へ変換する過程で固有の文脈・設計理由を削ぎ落とされ、平板な普遍ルールへ畳み込まれた。13 候補中、判断系は 3 件（すべて user-utterance 由来）で、その 3 件すべてが汎用 behavior-diff へ平板化されていた。さらに behavior-diff 化できない一回性の設計境界（例:「プランを追跡対象に変えることは無い／追跡可否は利用者に委ねられている」）は、distill の棄却ステップ（原理1「実行不能な内省は捨てる」＋§3「トリガー×振る舞い差分の両方が読めるもののみ合格」）で完全に消滅した。

この症状は capture/distill の実装不具合ではなく、設計前提への反証である。「予測誤差シグナルを貯めれば学びになる」（原理3「客観痕跡 > 主観内省」周辺と 4 シグナルの分類体系）は、少なくとも現分類体系では成り立っていない。価値の軸は単一ではなく、判断知（単発で消える＝復元不能性が軸）と摩擦知（単発では無価値だが横断集計で「配布したルールが機能していない」が見える＝再発が軸、#417）の 2 つに分かれており、現設計はこの 2 軸を 1 本の摩擦偏重シグナルに畳んでいた。

本 ADR は、粒度判定 4 項目（後戻りコスト高／複数モジュール波及／採用理由揮発／ツール自動強制不可）すべてに該当する構造的判断であり、DESIGN.md §7 で「ADR 化候補」として保留されていた原理3・分類体系・出力契約の改訂方針を確定する。

## Decision

学習シグナルの設計を、復元不能性を一次基準とする 2 軸へ再編する。以下の 6 点を決定する。

### 1. 価値の 2 軸化（知識型ごとにゲートを一致させる）

学習シグナルの価値基準を、知識型ごとに直交する 2 軸へ分離する。フィルタは対象の価値軸と一致していなければ精度・再現のいずれかを失うため、1 本のゲートで両型を測らない。

| 知識型 | 価値の軸（ゲート） |
|---|---|
| 判断知（選好・却下理由・設計判断） | 復元不能性 |
| 摩擦知（環境摩擦・手続違反） | ハーネス非強制ルールの再発（#417） |

### 2. capture のシグナル再定義（4 シグナルは補助化、判断知検出子を追加）

- 復元不能性を判断知のゲートへ昇格する。
- 現行 4 シグナル（訂正・ツール拒否・反復試行・期待違反）は廃止せず、「摩擦サブセットの補助検出器」へ降格する（user 訂正系は復元不能性と重なり依然有効なため、対称的に残す）。
- 予測誤差の形を持たない判断知（目標表明・選好・却下理由・設計判断）を拾う判断知検出子を新規に追加する。

### 3. capture フィルタ（直交 2 ゲート）

- ハーネス強制摩擦のみを既定除外する（File-not-read ガード・worktree ガード・ツールスキーマ検証・API 一時障害など、origin=tool-result かつハーネス発のガード/deny/エラーとして機械的に判別可能な telemetry）。
- ハーネス非強制のルール再発（ガードを持たない CLAUDE.md ルール等が反復違反される＝#417 の真の的）は保持する。これによりフィルタ反転と #417 を両立させる。

### 4. distill 出力形の 2 系統分離

- `candidates.md` に `type` フィールドを設け、`behavior-diff`（摩擦系）と `decision-record`（判断系）を同居させる。provenance・upsert・ライフサイクルは共有する。
- 摩擦系は現行の behavior-diff＋N 再発カウント＋強制化（プリフライト/deny/ガード）の形を維持する。
- 判断系の `decision-record` スキーマは `decision`（何を決めたか）/ `rejected-alternatives`（却下案）/ `rationale`（却下・採択理由）/ `context`（どの設計局面か）を持つ。原理1「実行不能な内省は捨てる」の例外口として、behavior-diff 要求と N 再発カウントを免除する（一回性の設計境界をカウントでなく決定の記録として残す）。

### 5. promote 検証の型適応（流路は不変）

- promote の検証を型で分岐する。摩擦系は現行どおり予測・反証（原理2）で検証する。判断系は予測的反証でなく「復元不能で・まだ有効で・配布価値があるか」を検証する（反証条件＝既にリポに記録済み＝復元可能／後に覆された／carry-forward 価値のない一回性）。
- 流路は両型とも `candidates.md → promote → Issue（揉む場）→ 既存ワークフロー（L2 ゲート）` を維持する。decision-record を learnings.md へ直送しない（揉む場・L2 ゲート・Distribute 段の翻訳規約をスキップしないため）。
- decision-record の scope は大半が project-local（growth 自身の設計判断は universal でなく閉じた空間＝本リポの ADR/docs へ向かう）。

### 6. 原理3 の精緻化

原理3「客観痕跡 > 主観内省」を「客観痕跡は摩擦知には必要十分だが判断知には不十分であり、復元不能性を直交基準として追加する」へ精緻化する。痕跡を一切残さない proactive な設計意図・基準（偽陰性の中核）は、純粋な痕跡ベース検出器には構造的に不可視であるため、痕跡の有無とは独立した復元不能性軸で拾う。

## Consequences

- **得られる利益**: store の信号対雑音比が向上する。復元可能 telemetry を capture 段で落とし、判断知を文脈ごと（decision-record）保存し、予測誤差の形を持たない偽陰性を判断知検出子で回収する。判断知が平板化・ドロップで失われる二重損失を閉じる。
- **受容したコスト**: capture に「判断知か否か」「ハーネス強制摩擦か否か」という解釈的・機械的判定が加わり、純摩擦検出より複雑になる（ただしハーネス強制摩擦の判別は決定的）。`candidates.md` のスキーマ拡張が promote（型適応検証）・`personal-store-spec.md`・`learning-promotion-spec.md`（#383、Phase 2 の learnings.md 翻訳規約）へ波及する。
- **#417 との関係**: 摩擦系の N 再発カウントを保持するため、ADR-20260629（distill の台帳突合・再発知見化）と両立する。Supersede は不要。本 ADR の直交 2 ゲートは、ADR-20260629 が確立した「既知ルール再発の知見化」を判断知側に拡張するものであり、摩擦知側の扱いを変えない。
- **本 ADR に含めない範囲（後続実装 Issue 側で詰める）**:
  - 判断知検出子の具体的な検出基準（どの発話形を判断知と判定するか）。
  - `decision-record` の正確なフィールド定義と `candidates.md` 上の書式。
  - promote の型適応検証の合否境界（判断系の反証可能性の運用基準）。
  - `learning-promotion-spec.md`（#383）の decision-record 対応拡張（Phase 2）。
  - DESIGN.md 本文（原理3・§3 シグナル分類体系・distill 出力契約）および capture/distill/promote の SKILL.md・spec の実際の改訂。
- **将来の留保事項**: 判断知検出子の precision/recall が想定を外す（判断知の取りこぼし継続、または会話の過剰捕捉）場合、検出子の補助化度合い・復元不能性ゲートの判定境界を再調整する。decision-record と behavior-diff の同居が candidates ライフサイクルを複雑化しすぎる場合、ファイル分離を再検討する。
