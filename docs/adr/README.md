# ADR（Architecture Decision Records）運用ルール

本ディレクトリは横断的・後戻りコスト高な技術的意思決定（アーキテクチャ・設計方針）を ADR として蓄積・参照・廃止する運用基盤。`docs/development/event-storming.md` の「技術的意思決定」集約の実体であり、同集約の状態遷移・コマンド・イベント定義と整合する。

## 配置

- `docs/adr/` 配下にフラット構造で配置する。サブディレクトリは作らない
- 本 README と各 ADR ファイルが同階層に並ぶ（雛形は `manage-adr` スキル配下にある。「テンプレート」節を参照）

## 採番方式

ファイル名規則: `ADR-YYYYMMDD[-N]-<slug>.md`

- `YYYYMMDD`: ADR 起票日（採番日＝起票日）
- `[-N]`: 同日2件目以降のみ付与。`-2` から開始（1件目は `-N` なし）
- `<slug>`: 短い英数字ハイフン区切り（例: `adr-operation-foundation`、`technical-decision-aggregate-foundation`）

採番例:

| 起票日 | 件数 | ファイル名 |
|---|---|---|
| 2026-05-11 | 1件目 | `ADR-20260511-<slug>.md` |
| 2026-05-11 | 2件目 | `ADR-20260511-2-<slug>.md` |
| 2026-05-11 | 3件目 | `ADR-20260511-3-<slug>.md` |

採番衝突時（同日に並行で起票する場合）の解消手順:

1. `ls docs/adr/ADR-YYYYMMDD-*.md` で対象日の既存 ADR ファイル一覧を取得する
2. 既存ファイルから最大番号を特定する（`-N` なしを `1` として扱い、`-2`、`-3` を順に番号化）
3. 最大番号+1 で発番する（例: 既存が `-N` なしと `-2` の2件なら、3件目は `-3`）

注: ADR 識別子は集約インスタンス識別子（`docs/development/event-storming.md`「技術的意思決定」集約）と一致する文字列。本 README の採番方式は #130 / #169 で確定した運用方針を正とする。

## ライフサイクル

状態は承認軸と有効性軸の2軸に分離して管理する（ADR-20260711-3 決定1）。各軸の値は各 ADR ファイル冒頭の front-matter（`status` / `validity` / `superseded-by`）が唯一の権威であり、日本語のユビキタス言語で確定する。

- **承認軸**（`status`・不変の履歴事実）: `提案中` / `承認済み` / `却下`
    - `提案中`: 起案され、承認者の判断を待っている状態
    - `承認済み`: 承認された履歴事実。以後不変
    - `却下`: 提案が承認に至らず終了。一度も運用されない終端
- **有効性軸**（`validity`・可変の現在の効力）: `有効` / `上書き済み` / `廃止済み`
    - `有効`: 現在運用中
    - `上書き済み`: 後継 ADR に置き換えられた（`superseded-by` に後継識別子を保持）
    - `廃止済み`: 後継なしで放棄された

- `提案中` をスキップして直接 `承認済み` に到達する経路は持たない。即承認のケースは「提案 → 承認」を連続実行として扱う
- 遷移ごとの `status` / `validity` / `superseded-by` の組み合わせ（遷移表・front-matter スキーマ必須ルール）は正本である ADR-20260711-3 決定2 を参照。遷移表は本 README・雛形には再掲しない
- 上記の値域列挙（各軸の値とその定義）は本 README が実体として持つ。正本（ADR-20260711-3 決定1）と重複しているが、生きたルールの出典構造そのものの再設計は #515 に委ねており、それまで本 README 側に残置する
- 旧運用（英文4状態 `Proposed` / `Accepted` / `Deprecated` / `Superseded` を単一 `Status` 欄で管理）は ADR ファイルの運用としては廃止した。`docs/development/event-storming.md`「技術的意思決定」集約の状態機械自体の書き換えは別 Issue（#463 フェーズ3）に委ねられており未反映。ADR ファイルの実運用は本 README・front-matter を正とする
- 2軸分離の背景・設計判断の詳細は ADR-20260711-3 を参照

## 粒度判定基準

ADR 化要否の判定は以下の4項目チェックリストで行う。

1. 後戻りコストが高い
2. 複数モジュール・複数開発者に波及する
3. 採用理由が時間経過で揮発しやすい
4. ツールで自動強制できない

判定ルール:

- **3点以上**: ADR 化推奨
- **2点以下**: ADR 化しない
- **判断に迷ったら ADR 化しない**: 原則として ADR 化しない方向に倒す。後から必要性が判明したら遡及して ADR 化する

ADR 過多は検索性を損ね、運用基盤の存在意義を失わせるため、判定境界では「書かない」を優先する。

### 起票のタイミングとエスカレーション

ADR 化要否は上記4項目で判定するが、「いつ起票するか」は別の問題である。最初から ADR にしようとせず、まず PR の説明に判断を書く。後から他の機能でも同じ問いが出てきた時点で ADR に昇格させる。

```
特定機能の決定 → まずPRの説明に書く
      ↓
他の機能でも同じ問いが出てきた
      ↓
ADRに昇格（docs/adr/ に起票。採番方式に従う）
```

ADR 化を検討すべき決定が発生したことを示す実務シグナル（いずれかに該当したら粒度判定基準で要否を判定する）:

- 複数の選択肢を検討して片方を却下した
- トレードオフを受け入れた
- 制約や前提条件がある

## 命名規約のADR化基準

命名規約のうち ADR 化対象とするかどうかは、構造的か書式的かで振り分ける。

| 種別 | 特徴 | 例 | 配置 |
|---|---|---|---|
| 構造的命名規約 | 横断・後戻りコスト高 | ファイル配置規約、集約名、ドメイン用語統一 | ADR 化（`docs/adr/`） |
| 書式的命名規約 | 局所・ツール強制可能 | 変数命名、import 順、空白・改行 | Linter／フォーマッタ規約配置（ADR 外） |

ツールで自動強制できる規約は ADR 外。粒度判定基準の項目4（ツールで自動強制できない）に照らして振り分ける。

## 廃止・上書き手順

有効性軸（`validity`）が可変性のゲートを兼ねる。`有効` な ADR は生きた文書として本文を直接編集してよいが、`上書き済み` / `廃止済み` に退役した ADR は凍結された歴史的成果物として編集しない（ADR-20260711-3 決定3）。

**凍結原則の射程**: 決定3 が禁じるのは**決定内容の編集**である。根拠は2つある。第一に、同決定の節タイトルは「編集機構（3段構え）と判定フロー」であり、当該文は core / 非core / 些末 を振り分ける判定フローのゲートとして置かれている。第二に、`manage-adr` スキルの編集判定基準も退役 ADR を「本フローの対象外」と表現しており、同じ射程で運用されている。

したがって、front-matter（状態）にも `## Decision`（内容）にも触れず、状態の重複表示を消すだけの是正は、決定の記録を書き換えないため決定3 の射程外である。退役 ADR に対しても実施してよい。除去しようとする表示が固有の情報を持つ場合、それを `## 関連ADR` へ移してから除去する操作（＝移設）も、情報の総体が保存される限り同じく射程外である。一方、決定内容に触れる編集は退役 ADR に加えてはならない。

### 編集要否の判定フロー（decision tree）

`有効` な ADR への変更は、次のフローで core / 非core / 些末 を判定する。境界アンカーは「その変更に再承認が要るか」。

```
1. その編集で「決定の結論・方向・前提」が変わるか？
   YES（結論の反転・前提の失効・後継への置換）
      → core = 再承認が要る
      → 新規 ADR を起票し、旧 ADR の validity を「上書き済み」に
        （旧 ADR が多決定で生存決定が残る場合は下記「分割」へ）
   NO → 2 へ
2. 決定内容に触れるか？（パラメータ値・適用範囲・条件・理由付け）
   YES（骨子は保つが決定関連情報を更新）
      → 非core = 既承認の枠内
      → 直接編集 ＋ 本文「## 変更履歴」に1行
   NO（typo・整形・リンク・語句・体裁のみ、決定内容ゼロ）
      → 些末 → 直接編集、履歴は git のみ
```

**安全則: 判定に迷ったら core 寄り（新規 ADR）に倒す。** core を非core と誤判定すると、再承認イベントを飛ばして core 変更を権威記録に焼き込む不可逆な不整合が残る。逆に非core を core と誤って新規 ADR にしても過剰なだけで可逆（後で統合できる）。可逆性の非対称から core 寄りが安全である。

### 3段構え編集機構の対応表

| 変更の性質 | 対応 | 変更履歴の在り処 |
|---|---|---|
| 些末（typo・整形・リンク修正、決定内容ゼロ） | 直接編集 | git のみ |
| 非core かつ決定に関わる（パラメータ・適用範囲＋理由） | 直接編集＋`## 変更履歴` に1行 | ファイル内 |
| core（再承認が要る） | 有効性を `上書き済み` にし、新規 ADR を起票（旧 ADR が多決定で生存決定が残る場合は下記「分割」へ） | 別 ADR ファイル＋相互参照 |

原則: ADR ファイルは、あらゆる consumption 面（Wiki・Confluence・artifact・抽出ファイル）に対して自己完結する。git は repo の実装詳細であり、決定関連情報の SSoT にはしない。

### 上書き（後継ADRによる置換）

1. 新 ADR を起票する（採番方式に従う。front-matter は `status: 提案中` で開始）
2. 承認後、新 ADR の front-matter を `status: 承認済み` / `validity: 有効` に更新する
3. 新 ADR の `## 関連ADR` に `Supersedes: <旧ADR識別子>` を記載する
4. 旧 ADR の front-matter `validity` を `上書き済み` に書き換え、`superseded-by: <新ADR識別子>` を記載する
5. 旧 ADR の `## 関連ADR` に `Superseded by: <新ADR識別子>` を記載する

旧 ADR の `status`（承認軸）は変更しない。後継ADR識別子は文字列参照のみで、集約インスタンスへの直接参照は持たない（自己参照・循環参照の構造的不整合を防ぐ）。

上書きに伴う状態の変化は手順4の front-matter の書き換えのみで表現する。**旧 ADR の本文に `## Status` 節を新設しない**（節を持たない ADR は節を持たないまま退役する）。状態は front-matter が唯一の権威であり、本文へ状態を書き足すことは重複を生むだけで、以後の遷移で取り残される drift 源になる。

### 分割（多決定ADRの部分上書き）

複数の決定を束ねた ADR の一部だけが core 反転し、残りの決定が生存する場合に使う。原 ADR 全体をそのまま `上書き済み` にすると生存決定を誤って退役させるため、上書きの前に生存決定を後継へ救出する。

反転を伴わずに、束ね ADR を予防的に分解する場合（衛生的分割）にも同じ手順を用いる。この場合は退役する決定がなく、全決定が生存決定として後継へ移る。

1. 生存決定の後継 ADR を起票する。一体で反転するクラスタごとに1本以上とし、`status: 提案中` で開始して承認後に `status: 承認済み` / `validity: 有効` へ更新する
2. 後継の `## Decision` に生存決定の本文を**逐語**で転記する。トップレベル項目の先頭番号は、後継での単独可読性のため振り直してよい。番号付き決定の前に置くリード文のような非決定文は、逐語の対象ではないため後継の文脈に合わせて書いてよい
3. 後継の残りの節を書く。`## Context` は新規記述とする。`## Consequences` は生存決定に対応する記述を保持し、退役する決定に由来して無効となる記述のみ除く。`## 関連ADR` は後継が保持する決定集合に照らして書き下ろす（原 ADR の `Related:` を機械的に引き継がない）
4. 手順2〜3 で原 ADR から改変した箇所（先頭番号の振り直し、リード文等の非決定文の書き換え、`## Consequences` の調整）を、後継の `## Context` に理由とともに記載する。決定件数の言い換えのような機械的な追随も開示に含める（「以下の4点を決定する」→「以下の2点を決定する」の類は、後継だけを読む読者には決定が落ちた痕跡と区別がつかないため）
5. 後継の `## 関連ADR` に `Supersedes: <原ADR識別子>` を記載する
6. 原 ADR の front-matter `validity` を `上書き済み` に書き換え、`superseded-by` に**全後継をカンマ区切りで列挙**する（例: `superseded-by: ADR-A, ADR-B`）
7. 原 ADR の `## 関連ADR` に `Superseded by: <後継識別子>` を全後継について記載する。表記（後継ごとにバレットを立てるか、単一行へ読点区切りで並べるか）はファイル内の既存様式に合わせてよく、要求されるのは全件が挙がっていることのみ。既存の `Related:` 行は退役時点のスナップショットとして残置し、分割の一環としては編集しない（この節は本手順が書き込む対象であり、既存行が巻き込まれうる。分割以外の文脈での編集可否は、退役 ADR 一般の凍結原則の射程に従う）

締めに次の4点を確認する。

- **網羅性（手順1〜2）**: 原 ADR の `## Decision` のトップレベル決定を全件列挙し、各決定が「手順2 でいずれかの後継へ逐語転記された」か「core 反転中で退役する」かの**いずれか一方**に分類されていること。未分類の決定と、両方に該当する決定（退役対象でありながら後継へ逐語転記されている）が、いずれも0件であること。分割は 1→N で生存決定が複数後継へ分散するため、後継を1本ずつ検査するだけでは全体の網羅を誰も見ないまま終わる。転記し忘れた決定は下記の逐語性検査でも切り出し対象に入らず、差分として現れない。逆に反転中の決定が後継へ紛れ込むと、反転が黙って無効化される（反転中の決定はそれ自身の後継が改訂後の内容として担うのであって、原文の逐語転記先を持たない）
- **逐語性（手順2）**: 後継ごとに、網羅性で「逐語転記された」に分類した決定に対応する箇所のみを原 ADR から切り出し、後継の `## Decision` の**トップレベル決定本文**と突き合わせる（両辺とも決定本文に揃える。原 ADR 側は退役決定を含み、後継側はリード文等の非決定文を含みうるため、節どうしをそのまま比較すると差分が出るのが正常になり検査にならない）。切り出し元を後継の内容ではなく網羅性の分類から定めることで、後継にのみ存在する決定——退役決定の混入や、分割中に書き足された新規の決定——が差分として現れる。トップレベル項目の先頭番号を正規化したうえで差分がないこと。切り出し結果が非空であることと行数が一致することもあわせて確認する（正規化や切り出しの失敗が空入力として差分ゼロに化けるため）
- **開示（手順2〜4）**: 手順4 の開示対象3種それぞれについて、改変が生じていれば後継の `## Context` に対応する記載があることを確認する。(a) 先頭番号の振り直し、(b) リード文等の非決定文の書き換え（決定件数の言い換えを含む）は、原 ADR と後継を突き合わせて改変の有無を判定する。(c) `## Consequences` は差分を列挙し、各差分が開示に対応していること。(a) は逐語性が正規化で潰し、(b) は逐語性が両辺から外した情報であり、いずれも開示側でしか拾えない。逐語検証は `## Decision` の決定本文にのみ適用し、`## Consequences` には適用しない（意図的な再構成が入るため差分ゼロを要求できない）
- **相互参照（手順5〜7）**: `bash scripts/lint-adr.sh docs/adr` が exit 0 であること。`superseded-by` のリスト値と各後継の `Supersedes:` 宣言が双方向で照合される。加えて、原 ADR 本文の `Superseded by:` に全後継が挙がっていることを確認する。lint の本文走査は `Supersedes:` のみを対象とするため、原 ADR 側の記載漏れは exit 0 のまま素通りする（front-matter には残るが、本文だけを見る消費面に後継が現れなくなる）。あわせて原 ADR の `## 関連ADR` を分割の前後で突き合わせ、分割に由来する変更が `Superseded by:` の追加のみであることを確認する。分割以外を根拠とする変更（例: 決定4 に基づく `Amends:` / `Amended by:` 行の除去）が同時に入る場合は、その根拠を示したうえで、`Related:` 等それ以外の既存行が不変であることを確認する（同節は手順7 が書き込む対象であり、既存行が巻き込まれうる）

分割は判定フロー（core / 非core / 些末）の第4分類ではない。決定のファイル間再配置という構造操作であり、分割中に決定内容は変えない。決定内容の変更が要る場合は、分割とは別に core / 非core 編集として分離する。

### 廃止（後継なしの放棄）

1. 対象 ADR の front-matter `validity` を `廃止済み` に書き換える（`superseded-by` は空のまま）
2. `## Consequences` 末尾など本文に廃止理由・廃止日時を追記する
3. 後継 ADR は存在しないため `## 関連ADR` の変更は不要

対象 ADR の `status`（承認軸）は変更しない。

### 却下（起票後、未承認のまま終了）

1. 対象 ADR の front-matter `status` を `却下` に書き換える（`validity` は空のまま）
2. 却下理由を本文に追記する

### Amends / Amended by（廃止・新規起票では使わない）

旧運用（別 ADR 起票で一部の決定〔facet〕のみを改訂し、旧 ADR を承認済みのまま維持する仕組み）は廃止した（ADR-20260711-3 決定4）。既存ファイルに残る `Amends:` / `Amended by:` の相互参照は、凍結された歴史的記録として保持・遡及編集しない扱いを採らず、live な関係を生存語彙（`Supersedes:` / `Superseded by:` / `Related:` および `## 変更履歴`）へ移行し、退役済みの関係は後継の supersede 連鎖へ吸収したうえで、corpus から除去する。新規起票では使わない。パラメータ・適用範囲のみの改訂は上記「非core = 直接編集＋変更履歴」で扱い、決定の骨子を覆す改訂は「core = 新規 ADR」で扱う。

### index の再生成・検証（手動実行）

`docs/adr/index.md` は `validity: 有効` の ADR のみを列挙する導出ビュー（生成物）であり、人手編集しない。

- 生成: `bash scripts/gen-adr-index.sh docs/adr`
- 検証（front-matter スキーマ・index 同期・相互参照双方向性の3レイヤ）: `bash scripts/lint-adr.sh docs/adr`

## テンプレート

新規 ADR の起票は `manage-adr` スキル経由を主経路とする。同スキルが採番・front-matter の写入・相互参照の書き込みを一貫して行い、操作後に lint-adr で自己検証する。

雛形の実体は [`plugins/dev-workflow/skills/manage-adr/references/template.md`](../../plugins/dev-workflow/skills/manage-adr/references/template.md) にある。スキルを導入していない場合はこれをコピーして使用する。MADR 準拠の必須項目（Title / Context / Decision / Consequences）と本プロジェクト追加項目（関連ADR）、front-matter（`status` / `validity` / `superseded-by`）を含む。

MADR の Status 相当は front-matter の `status`（承認軸）/ `validity`（有効性軸）が担う。状態は front-matter が唯一の権威であり、**本文に `## Status` 節を置かない**。新規起票時も、承認・上書き・廃止・却下のいずれの遷移時も同様である。

## 設計判断の意味的な逆引き

スキル/エージェント/参照ファイル群には「モデル（LLM）の制約への対処」が暗黙的に含まれている（自己評価バイアス、コンテキスト膨張、早期収束、指示の確率性、ハルシネーション等）。設計要素別に該当 ADR を横断検索したい場合は、人手キュレーションの索引を本 README には持たず、各 ADR 本文（Context / Decision）を直接参照する（ADR-20260711-3 決定7。旧索引は手動 drift 源のため廃止した）。設計要素横断のギャップ地図が必要になった場合は、別 Issue で改めて検討する。
