# ADR-20260629: プランファイルの所在解決を worktree／メイン横断で一意化する

## Status

Accepted

## Context

`docs/plans/*.md`（実装プランファイル）は `docs/plans/.gitignore` により既定で未追跡のスクラッチ artifact として扱われる。追跡するか否かは利用者の選択に委ねられている。

一方 `implementation` スキルは作業着手時に新規 git worktree を作成して作業ディレクトリを移すことがある。worktree は別の作業ディレクトリであり、未追跡ファイルは伝播しない。このため次の問題が起こる:

- `plan-issue` をメイン側で実行してプランを作成した後、`implementation` を別 worktree で実行すると、未追跡プランがメイン側に取り残されて worktree から見えず、`implementation` が黙って簡易パスへ縮退する（プラン由来のタスク分解・AC↔テストケース対応表を取りこぼす）。
- `plan-issue` は既存プランを「特定」して読む／編集する判断点（ステップ5の連番衝突回避、分割計画マスターの有無による初回/N回目判定、N回目モードの Read/Glob、人間レビュー却下後リカバリの編集対象パス）をいずれも現作業ディレクトリ固定の Glob で行う。プランが別 worktree にあるとこれらが一斉に mis-detect し、誤って初回モードで分割計画を丸ごと再生成する等の重複・取り違えを招く。

設計判断の検討では複数案を比較し、一部を却下した（粒度判定基準の実務シグナル「複数の選択肢を検討して片方を却下した」に該当）。

粒度判定基準（`docs/adr/README.md`）の4項目チェックリスト評価:

1. **後戻りコストが高い** → 該当。2スキルが共有する所在解決契約であり、後から方式を変えると両スキルの記述・`allowed-tools` が連動して揺れる。
2. **複数モジュール・複数開発者に波及する** → 該当。`implementation` と `plan-issue`（SKILL.md・split-plan-procedure.md）の計5箇所に波及する。
3. **採用理由が時間経過で揮発しやすい** → 該当。「なぜメイン固定や hook コピーを採らなかったか」は明示しないと失われ、後続で蒸し返されやすい。
4. **ツールで自動強制できない** → 該当。スキル本文の規約であり lint 等で強制できない。

4/4 該当のため ADR 化する。本決定は起票時点で既に2スキルへ波及しているため、「まず PR 説明に書き再出時に昇格」ではなく即時 ADR 化する。

## Decision

プランファイルの所在解決を、`plugins/dev-workflow/references/plan-location-resolution.md` を単一出典とする共有規約に統一し、`implementation`（探索）と `plan-issue`（既存プランの特定／編集）の双方がこれを参照する。

- **解決順序（現ディレクトリ優先・メインフォールバック）**: 現作業ディレクトリの `docs/plans` を先に Glob し、見つからなければメイン worktree の `docs/plans` を Glob する。メイン worktree ルートは `git worktree list --porcelain` の先頭エントリで解決する（正典。代替の `git rev-parse --git-common-dir` の親は参考併記）。両スキルの `allowed-tools` に `Bash(git worktree list*)` を最小追加する。
- **gitignore 状態を判定しない**: 解決はトポロジ（現ディレクトリ → メイン worktree フォールバック）のみに依存する。プランが追跡されている場合は git の branch 配信が現作業ディレクトリ（worktree）へプランを届けるため①で見つかり、未追跡でメインに取り残された場合は②で拾う。これにより追跡／未追跡の両レジームを判定なしでカバーする。
- **書き込み先ルール**: 新規作成は現作業ディレクトリへ。既存特定後の再生成・編集・N回目は特定された解決ディレクトリへ in-place で行い、現作業ディレクトリへコピーしない（正典の二重化を防ぐ）。連番計算の基準ディレクトリは解決ディレクトリとする。
- **不変条件の保持**: 本規約は探索／書き込みするディレクトリのみを規定する。各スキルの連番衝突回避機構（整数比較による最大連番抽出・旧 plan 非上書き）や初回/N回目モード判定のロジックは変更しない。
- **却下した代替案**:
  - **「必ずメイン側へ固定する案」を却下**。プランが追跡されている場合、メイン固定はプランを branch に乗せず worktree の外へ置き、利用者の追跡意図を壊す。
  - **「worktree 作成時の hook によるプランコピー」を却下**。コピーは正典を2つに分裂させ、追跡時には `git status` 汚染・divergence を生む。

## Consequences

- `implementation` と `plan-issue` が「プランの在処」で食い違わなくなり、worktree とメインのどちらにプランがあっても（追跡／未追跡を問わず）取り違えなく作業できる。
- worktree が1つだけの通常環境では `git worktree list` 先頭エントリが現作業ディレクトリ自身に解決されるため、改修前と同一の挙動に帰着する（後方互換）。
- 両スキルに `git worktree list` 実行権限が増える。破壊系（`git worktree add/remove`）は含めず読み取り専用に留める。
- **受容した既知制約**: `plan-issue` を worktree-X で実行・`implementation` を別 worktree-Y で実行・かつプランが未追跡の場合、プランは worktree-X にのみ存在し解決順序の①②がともに空振りしてプランを特定できない。これは仕様上の未解決ケースとして共有 reference の「既知制約」節に明記し、回避策（プランを追跡対象にする／同一 worktree で両スキルを実行する／plan-issue をメインで実行する）を併記する。将来この経路を救う必要が生じた場合は別途検討する。
- `git worktree list` が失敗する異常環境では現作業ディレクトリのみで解決する縮退とする（通常環境では発生しない前提）。
