# ADR-20260421: コーディングエージェント固有挙動をドメインモデルに持ち込まずアクター内部実装として扱う

## Status

Accepted

## Context

開発コンテキストの DDD ドメインモデル化（`docs/development/event-storming.md` / `domain-model.md`）の議論で、コーディングエージェント固有の挙動詳細（ループ上限、打ち切り条件、リトライ回数、コスト/時間上限など）をドメインモデルに含めるかどうかの判定が必要となった。

Issue #139（`event-storming.md: エスカレーション集約の独立化と用語汎化（F2再検討）`、CLOSED 2026-04-21）の議論で、ループ上限（Issue #133 N5）のドメインモデル化可否を検討。`event-storming.md:330` の既存方針「作成主体（人間/コーディングエージェント）によらず」と整合させる必要があった。

## Decision

開発コンテキストのドメインモデルでは、コーディングエージェント固有の挙動詳細をドメインモデルに持ち込まず、アクター内部実装として抽象化する。

1. **対象範囲**: コーディングエージェント固有の制御パラメータ全般
    - ループ上限（N回失敗でエスカレーション等）
    - 打ち切り条件・リトライ回数
    - コスト/時間上限
    - 「判断依頼の発行タイミング」などアクター内部の判定
2. **モデル化の原則**:
    - 人間アクターとエージェントアクターは同じコマンド・同じドメインイベントを共有する
    - 人間の「無意識思考→諦め」もエージェントの「アルゴリズム判定」もアクター内部プロセスとして等価
    - ドメインイベント・コマンド・ポリシー設計時、エージェント固有の制御パラメータを持ち込まない
3. **置き場所**:
    - エージェント固有の打ち切り条件・ループ制御は `dev-loop` 等のスキル実装側契約に置く
    - アクター内部の判定はアクター責務としてモデル外に追い出す

## Consequences

**得られた利益**:

- ドメインモデルが実装詳細から分離され、安定する
- 人間/エージェント両アクターを同じ抽象で扱えるため、アクター置換時のモデル変更が不要
- `event-storming.md:330`「作成主体によらず」原則と整合

**受容したトレードオフ**:

- エージェント固有の制御パラメータは別レイヤー（スキル実装側）に分散するため、横断的な可視性が低下
- 「ドメインモデルとスキル実装契約」の責務分担を理解していないと、どこに何が定義されているか追跡しづらい

**将来の留保事項**:

- スキル実装側契約（dev-loop の Phase ループ上限・3周収束失敗時の挙動等、Issue #84 OPEN）の設計は別途進化
- Issue #86（各スキル/エージェントにモデル依存仮説（設計根拠）を記録、OPEN）でアクター内部実装の根拠記録方式を検討
