# ADR-20260407: dev-loop の正規パス/簡易パス入力分岐

## Context

dev-loop は Issue・計画ファイル・PRレビュー指摘・会話コンテキストの4種類を入力として受け取る。すべての入力に対して plan-issue 由来の完了条件フォーマットを強制すると、軽微な修正（PRレビュー指摘・会話コンテキスト由来の typo 修正等）に対して過剰な準備工程を課す。一方、計画ファイルに基づく実装（正規パス）では、人間レビュアーが承認した完了条件を活用することで品質保証が強化される（ADR-20260406）。

このため、入力種別に応じた処理分岐と、それぞれのパスでの品質保証レベル差を明示する必要がある。

関連: Issue #108（`dev-loop: 正規パス/簡易パスの入力分岐実装とworkflow-design.md更新`、CLOSED 2026-04-07）、PR #116。

## Decision

dev-loop に「正規パス」と「簡易パス」の2系統を明示的に導入する。

1. **正規パス**: plan-issue が生成した計画ファイル（`docs/plans/issue-{番号}.md`）に検証すべき振る舞いが具体的に定義されている場合
    - 作業リストは計画ファイルのタスク分解セクションから導出（自力導出はスキップ）
    - レビュー契約は計画ファイルの検証方針（主）と Issue AC（補完）から合成
    - 矛盾検出時はエスカレーション
2. **簡易パス**: 上記以外（Issue 直接、PRレビュー指摘、会話コンテキスト、計画ファイルが薄い場合）
    - 作業リストは dev-loop 内で自力導出
    - レビュー契約はインプットの受け入れ条件（あれば）から導出、なければ作業リストから「何が完了していれば正しいか」を導出
3. **品質保証レベル差の明示**:
    - 正規パスは人間レビュアーが完了条件まで承認済みのため code-reviewer 判定の信頼性が高い
    - 簡易パスは実装者が自力導出した契約のため自己評価バイアスが残る。重要度の高い変更は plan-issue 経由の正規パスを推奨
4. **依存最小化のインターフェース契約**: dev-loop は plan-issue の出力フォーマット全体ではなく、「検証すべき振る舞いセクションの存在」だけを判定の手がかりとする。フォーマット細部の変更が dev-loop に波及しない

## Consequences

**得られた利益**:

- 軽微な修正（typo・コメント修正等）に過剰なPlanning コストを課さない
- 正規パスでは plan-issue で検証された完了条件を再利用でき、レビュー品質が向上
- パス別の品質保証レベル差を明示することで、実装者・レビュアーが期待される厳密さを判断できる

**受容したトレードオフ**:

- パスが2系統に分岐するため、dev-loop の Phase 0 ロジック（入力種別判定）が複雑化
- 簡易パスでの自己評価バイアスは残るが、軽微な変更でのコスト効率を優先

**将来の留保事項**:

- 簡易パスでの品質問題（Issue #84「3周収束失敗時の挙動」OPEN）の運用観察に応じて、簡易パスでも plan-issue 経由を強制するか判定が必要
