# ADR-20260628: エージェント定義の粒度モデル — ロール型（能力／知識ドメイン粒度）を正とする

## Context

### 背景

dev-workflow のサブエージェント定義を「どの粒度でモデル化するか」の方針が未明文化だった（#387）。発端は #368（`plan-issue` が plan 本文をメイン context に往復させない設計）の検討中、FAIL 修正の担い手を「plan エージェントが生成と改訂の双方を所有するか／別エージェントへ分けるか」で議論したこと。この問いは plan-issue に閉じず、エージェント定義の粒度という横断論点に一般化される。新規エージェント追加時に「工程ごとに増やす」か「能力・ロール単位で束ねる」かの判断基準が存在せず、混在のまま増殖するとドメイン知識の重複（例: `plan` と `test-designer` がともに AC を解釈）や、本来一つの責務が複数エージェントに割れる歪みを招く。

### 現状6エージェントの混在実態

`plugins/dev-workflow/agents/` の6エージェントを本文まで精査した結果、frontmatter の `description` は全て特定スキルの特定工程に固定されているが、**本文の能力は工程に依存していない**ものが含まれる。

- `refactorer`: 入力はベースブランチ＋差分のみ。フロー固有契約ゼロ。任意の差分に5観点を当てる純粋なロール能力。
- `code-reviewer`: 「レビュー契約」は **optional input** であり正体の一部ではない。6観点レビューは工程非依存。
- `plan` / `plan-reviewer` / `test-designer` / `test-spec-validator`: フロー固有の成果物・契約（AC↔テストケース対応表、骨格、テスト仕様、前段成果物）を前提とする。

つまり再利用可能な能力（`code-reviewer` / `refactorer`）が工程ラベルを着ている状態であり、Issue が指摘した「両型の混在」は事実だった。

### 粒度モデルの3類型とトレードオフ（AC1）

| 観点 | ユースケース／工程型 | 能力／ロール型 | 中間（能力／知識ドメイン粒度） |
|---|---|---|---|
| 定義 | スキルの特定工程に束ねる | 任意成果物に対する再利用能力 | 工程より粗く職種より細かい束ね方 |
| 決定論的フロー制御 | 強（工程に固定され誤用しにくい） | 弱（汎用ゆえ脱線・誤入力リスク） | 強（契約はスキルが渡し検証する） |
| 知識の再利用性 | 低（工程ごとに重複） | 高 | 高 |
| エージェント増殖 | 工程ごとに増える | 抑制 | 抑制 |
| ドメイン知識の重複 | 起きやすい | 起きにくい | 起きにくい |
| 職種写像への滑り | 無 | 有（ADR-20260513 と衝突） | 無（能力で切り職種で切らない） |
| 自己文書性（配置の明示） | 高（定義に工程が書かれる） | 低（スキル側に配線が委ねられる） | 中（"used by" 注記で緩和） |

工程型・ロール型は二者択一ではなく、**フロー固有の契約をエージェントの正体に焼き込むか、入力として渡すか**という抽象度の引き方の違いである。契約を入力に持ち上げれば、6エージェントは全て「名前付き成果物に対する具体的変換」というロール能力に還元できる（工程型に内在的に縛られたエージェントは現 roster に1つも無い）。中間粒度はこの「ロール型＋抽象度規律」として両型の欠点を回避する。

### ADR化要否判定（AC4）

`docs/adr/README.md` の粒度判定基準（4項目）で判定:

1. 後戻りコストが高い → ✅（誤った粒度はリネーム・フロー再配線で高コスト）
2. 複数モジュール・複数開発者に波及する → ✅（全スキル・将来の全エージェントに影響）
3. 採用理由が時間経過で揮発しやすい → ✅（「契約は入力／能力が正体」の機微は文書化しないと失われる）
4. ツールで自動強制できない → ✅（linter で粒度は強制不可）

**4/4 → ADR 化推奨。** よって本決定を ADR 化する。

## Decision

dev-workflow のエージェント定義の粒度モデルを以下の通り定める。

1. **ロール型（能力／知識ドメイン粒度）を正とする。** エージェントの正体は「名前付き成果物に対する具体的な変換」（例: diff→review、code→refactor、requirements→plan、requirements→test仕様、test仕様→validation、plan→review）として定義する。特定スキルの工程を正体にしない。

2. **フロー固有の契約は入力に持ち上げ、オーケストレータ（スキル）が渡す。** AC↔テストケース対応表・レビュー契約・テスト仕様フォーマット・前段成果物の存在等は、エージェント定義に焼き込まず入力として受け取る。**スキル＝フローと契約の所有者／エージェント＝再利用可能な能力**という責務分離を保つ。これは ADR-20260421（エージェント固有の制御パラメータをドメインモデルに焼き込まず入力／アクター内部に隔離する）の、エージェント定義への適用である。

3. **抽象度を以下の3境界で縛る。**
   - **下限（粗すぎ禁止）**: 「何でもレビューする artifact-reviewer」のような曖昧抽象に潰さない。能力は1つの成果物・1つの変換に対して具体的であること（`plan-reviewer` と `code-reviewer` は対象成果物・必要力量が異なる別能力として保つ）。
   - **工程ラベル禁止**: 「Phase 2 のレビュアー」を正体にしない。工程はスキルが配線する利用箇所であり、エージェント正体ではない。
   - **職種写像禁止**: PM／QA／UX 等の人間職種で切らない（ADR-20260513 の弱シグナルを維持）。能力名（`test-designer` 等）で切る。

4. **転用可能な能力を持たず「特定スキルの手順列だけ」のエージェントは作らない。** そうしたものに出会ったら、工程型の逃げ道を設けるのではなく、**役割を再考し責務分解で転用可能な能力を抽出する設計シグナル**として扱う。分解の結果が手順列のみに留まるなら、それはエージェントではなくスキル本体のフロー制御として書く。

### 既存ADRとの整合

- ADR-20260421: 支持。フロー固有契約を入力に隔離する本決定は同 ADR の原則の適用。
- ADR-20260513: 整合。職種写像禁止（境界3）が同 ADR の弱シグナルを維持する。
- ADR-20260525-2: 整合。フラット名前空間での集約に対し、能力で自己記述する命名を後押しする。

いずれとも矛盾しないため上書き（後継 ADR による置換）手続きは不要（Related のみ）。

## Consequences

### 得られる利益

- 決定論的フロー制御がスキル（オーケストレータ）に一元化され、工程束縛によるフロー制御のワーカーへの漏れが解消される。
- 能力の再利用余地が開き、エージェント増殖とドメイン知識の重複が抑制される。

### 受容するコスト

- スキルが契約を毎回明示的に渡す必要が生じ、スキルコードが冗長化する（ただしフローが属すべき場所での冗長化であり、設計上は正しい移動）。
- エージェント定義から「どの工程で使うか」の自己文書性が低下する（"used by" 注記で緩和）。
- 現状の4つの工程型は各1箇所しか呼ばれておらず実利用の再利用実績は低い。本決定が今すぐ買うのは実再利用より概念的統一と将来の再利用余地であり、冗長化コストの先払いである。
- 汎用ロール能力が誤入力で静かに低品質化するリスク。エージェントは必要入力を宣言し、スキルが渡す責任を負う入力契約規律で緩和する。

### 既存6エージェントの分類と移行要否（AC3）

本 Issue では振る舞いを変更しない（粒度モデルの方針決定のみ）。移行の実装は別 Issue とする。

| エージェント | 現状 | 新方針での正体 | 移行要否 |
|---|---|---|---|
| `refactorer` | 工程ラベル（Phase 1e）／本文は純ロール | code→refactor | **要**（description の工程固定を外し利用箇所を "used by" 注記へ降格。振る舞い不変・メタデータのみ） |
| `code-reviewer` | 工程ラベル（Phase 2）／契約は既に optional | diff→review | **要**（同上） |
| `plan-reviewer` | 工程型／plan成果物前提 | plan→review | **要**（契約・前提を入力として明示化し正体を能力へ再記述） |
| `test-designer` | 工程型／Phase 1a 固有 | requirements→test仕様 | **要**（同上） |
| `test-spec-validator` | 工程型／前段成果物前提 | test仕様→validation | **要**（同上） |
| `plan` | 工程型／AC↔TC接続契約・骨格を内包 | requirements→plan（計画手法は入力） | **要**（契約・骨格を入力へ持ち上げ。#368/ADR-20260627-2 の生成・改訂分離と整合させる） |

6エージェント全てが移行対象（いずれも振る舞いは不変、定義のメタデータ・契約受け渡しの再設計）。移行は段階的に **#398** で追跡・実施する。

### 将来再検討する条件

- モデル更新でフロー脱線・誤入力耐性が大きく向上し、入力契約規律のコストが不要化した場合。
- ロール能力の粒度が現実の再利用で粗すぎ／細かすぎと判明した場合（境界1の見直し）。
