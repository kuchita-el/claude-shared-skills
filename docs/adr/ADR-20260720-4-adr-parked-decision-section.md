# ADR-20260720-4-adr-parked-decision-section: ADR スキーマに「保留した決定」セクション（意図的な非決定のスナップショット）を導入する

## Context

ホットスポット H4「ADR のライフサイクル管理」由来の運用再設計トラック（Issue #463）の後続。本 ADR は #496（旧 Amend 関係の生存語彙移行）の設計議論から派生した Issue #527 の決定を記録する。

#496 で、旧 Amend 関係のうち「対象側が facet を**パーク（未決）**し、後続 ADR がそれを充足した」ものは上書きではなく `Related:` で表現すると決めた。この「充足か上書きか」の判定は、**対象 ADR が『パークした』ことを記録しているか否か**に依存する。パーク記録が無ければ、後続の充足が「決定の上書き」と誤読され、core／非core 判定を誤る。パークの明示は、後続の充足が空欄埋めか決定の反転かを後世の読み手・分類者が判別できる証跡である。

しかし現行スキーマ（ADR-20260711-3）には**未決を表す第一級のセクションが無い**。実運用ではパークは `## Consequences` 内の「将来の留保事項」「留保事項・後続」「名前付きの未決」等アドホックな散文に散在し、セクション名・書式が統一されず、機械的に位置特定できない。加えて `Related:` と未決は `lint-adr.sh` の**検査対象外**（強制は `Supersedes:` 双方向のみ）で、dangling 参照を機械検出できない。

**意味論の選択（本 ADR の核心）**: パークを「未解決事項（open questions／TODO）」として持つと、解決時に対象 ADR を**さかのぼって状態更新する義務**が生じ、更新漏れが drift を生む。さらに退役（`上書き済み`／`廃止済み`）した ADR は凍結原則（ADR-20260711-3 決定3）で編集できず、生きた TODO 追跡と両立しない。したがってパークは「この ADR が意図的に決めなかったこと」の**起票時点のスナップショット**として持ち、解決の記録は後継側へ外部化する。

## Decision

### 1. 「保留した決定」セクションをスナップショットとして新設する

ADR スキーマに第一級セクション `## 保留した決定` を導入する。

- **意味論**: 起票時点で本 ADR が意図的に決めなかった facet の**不変記録**。`open`／`resolved`／`rejected` 等の状態フィールドを持たせず、解決時も本 ADR を更新しない。
- **位置**: `## Consequences` の後、`## 関連ADR` の前。
- **省略可**: パークする facet が無ければ**本節ごと省略**する（`## 関連ADR` と異なり「該当なし」は書かない）。corpus の大半はパーク無しであり、空節のノイズを出さない。
- **項目構造**: バレットで「保留した facet の説明（想定継承先: `#<Issue番号>` ／ `<ADR-slug>`、起票時点で分かれば。**非拘束のポインタ**であり、追随更新の義務を負わない）」。

### 2. 充足・棄却の規約

- **元 ADR は「閉じる」操作をしない**。これがパークの開閉ルールの実体である。スナップショットゆえ、解決しても本 ADR の `保留した決定` は書き換えない。
- **充足は後継 ADR 側に記録する**: 後継 B が本 ADR A の保留 facet を決めたら、B の `## 関連ADR` に `- Related: <A の full slug>（A が保留した〔facet〕を充足。上書きでない）` を書く（#496／#526 で確立した記法）。A は無編集・`validity` 不変。
- **分類**: 読み手／分類者は「A の `保留した決定` に facet がある（意図的に未決）＋ B の『充足。上書きでない』」を根拠に**充足（非core・Related）**と判定する。facet が A の `## Decision` にあれば**上書き（core・Supersedes）**である。この対比が判別の証跡となる。
- **棄却・放置**: 後に不要と判断されても A は編集しない。棄却は後継 ADR／Issue 側に記録するか、何も起きなければ「A が決めなかった facet」のまま残る。更新義務が無いため rot しない。

### 3. 非 Supersede 関係の参照妥当性 lint（要件定義。実装は #474）

`lint-adr.sh` に、非 Supersede 関係の**参照妥当性検査**を追加する要件を定義する。実装は #474 が担う。

- **検査内容**: front-matter を持つ ADR の `## 関連ADR` 内の各 `Related: <slug>`、および `## 保留した決定` 欄が ADR を指す `<slug>` について、`<slug>.md` が ADR_DIR に実在すること（dangling／typo 検出）。参照は full slug 完全一致で解決する。
- **不検査**: `Related:` の双方向性、パークの open／resolved 状態、Issue 番号参照（lint はオフラインで実行され `gh` を使えないため、`#<番号>` の実在は検査しない）。
- **`Related:` 双方向を強制しない**（＝一方向 `Related:` は合法）。双方向強制は、後継が `Related` を書くたび元 ADR へ追記＝さかのぼり編集を要し、退役 ADR では不可能で、スナップショット意味論・凍結原則に反する。#496 の分類は「A のパーク欄（不変）＋ B の一方向 `Related:`」で足り、双方向は分類に不要である。双方向は慣用（推奨）に留める。
- **#474 への申し送り**: 検査を有効化する前に、現行 corpus に dangling な `Related:` 参照が無いことを確認する（あれば lint が red 化する）。新レイヤとして足すかレイヤ3を拡張するかは #474 の実装裁量とする。

## Consequences

- **得られる利益**: #496 の充足／上書き分類の証跡が不変のスナップショットとして残り、rot しない。退役 ADR のパーク欄も凍結したまま矛盾しない。散在していた未決表現が第一級セクションへ集約され、機械的に位置特定できる。dangling 参照が機械検出可能になる（#474 実装後）。
- **受容したトレードオフ**: `Related:` 双方向を強制しないため、片側だけの `Related:`（A は `Related: B` と書くが B が A に言及しない）という drift は lint で検出しない。これはスナップショット意味論・凍結原則との両立を優先した結果で、双方向は慣用に留める。既存 ADR へのパーク欄の網羅的バックフィルは行わない。本 ADR 自身が `Related: ADR-20260711-3-...` を一方向で持つ（711-3 は無編集）ことは、この一方向 `Related:` 合法の最初の実例である。
- **記録先を新規 ADR とした理由**: 本決定（パーク形式化＋参照妥当性 lint）は ADR-20260711-3 の 2 軸モデルを反転せず独立に追加するものであり、パーク欄と当該 lint を撤去しても 2 軸モデルは無傷で、独立に反転しうる。ADR-20260711-3 決定3 は「新規 ADR は独立に反転しうる core を1本に束ねない」と定めるため、711-3 への決定追加ではなく別 ADR（`Related: 711-3`）とした。

## 保留した決定

- 参照妥当性 lint の実装（レイヤ配置・コード・corpus 事前確認）は本 ADR では決めない（想定継承先: #474）。
- 既存 ADR への保留欄の網羅的バックフィルの要否・方法は本 ADR では決めない（想定継承先: 形式確定後に必要なら別途起票）。
