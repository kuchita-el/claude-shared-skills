# ADR-20260607: dev-loop / plan-issue の単位は現状維持とし、参照機構②/③・検証ゲート主担い・ポリシー実現方式を確定する

## Context

ADR-20260531 は実装メカニクスの superpowers 委譲境界を決定し、ADR-20260606-3 はその依存方式（経路(b) ソフト/任意依存）とフォールバック方針 C を確定した。ただし ADR-20260606-3 項4 は「メカニクスごとの参照機構②/③割当、および撤去/保持の線引きは #264（単位妥当性判定）で確定する」として明示的にパークしていた。

子B（#263、`docs/references/skill-realization-mapping.md`）が開発コンテキストの全コマンド/ポリシーに担い手を1対1で割り当て、接続契約シーム S1〜S7 を抽出した。その際、本表だけでは決めきれない2論点を #264 へ申し送った:

- **L1: 検証ゲートを実行する（#11）の主担い**。実行体は CI/test runner（ハーネス）だが「検証してから完了と言う」規律は superpowers `verification-before-completion`。主担いをハーネス（暫定）とするか superpowers とするか。
- **L2: ポリシー（P5〜P10）の実現方式と主担い**。イベント駆動方式（ハーネス自動反応）か オーケストレーション方式（明示駆動）か。方式により全ポリシーの主担いが変わる。

本 ADR は、子Bマッピングの担い手境界に対し現行スキル単位（dev-loop = 実装+レビュー+PR、plan-issue = 計画）が整合するかを判定し（#264 の主目的）、上記 L1/L2 と参照機構②/③割当を確定する。判定基準は「担い手境界がスキル内の責務分割線と一致するか」とし、担い手1つ=スキル1つの厳格な1:1対応は求めない。

関連: Epic #261、依存 #263（CLOSED）、feasibility gate #262（CLOSED）。

## Decision

### 1. 単位判定: dev-loop / plan-issue とも現状維持（再分割しない）

子Bマッピングの担い手境界は、「どのスキルか」を分ける横線ではなく、**各フェーズ内の「superpowers コア vs dev-workflow 接続契約シェル」を分ける縦線**である。dev-loop の各フェーズ（実装 #7,#8 / 検証 #11 / レビュー #13-15 / PR #12）はいずれも superpowers 委譲先＋dev-workflow 接続契約の二層構造になっている。

- **plan-issue = 現状維持**。担い手境界（#1 `writing-plans` 委譲＋シーム S1/S2 接続契約＋#2「実装プランを確定する」人間ゲート）が単位境界と整合する。Issue→計画 の単位は一貫している。
- **dev-loop = 現状維持**。担い手境界は縦線であり、フェーズAとフェーズBを切り分ける横の分割線ではないため、複数スキルへの再分割は不要。Issue→PR ループは orchestration 責務として一貫した単位である（複数の superpowers スキルを呼ぶことは、基準上 1:1 を要求しないため再分割理由にならない）。
- **plan-issue↔dev-loop 境界**も、#2「実装プランを確定する（人間）」が自然なシームとなり整合する。

担い手境界はスキル境界ではなくスキル内部の「メカニクス／接続契約」境界に宿る。したがって変わるのは単位ではなく内部実装であり、内部の重い自前機構（test-spec-validator / リトライ統治 / 振動検知 / refactorer）の撤去は方針 C・子D/子E のスコープとする（本 ADR は単位を現状維持と確定するに留める）。

### 2. L1: 検証ゲート（#11）の主担い = superpowers `verification-before-completion`

dev-loop 文脈の検証ゲート（Phase 1c）は本番 CI ではなく、「コマンドを実行し出力を確認してから完了と言う」規律そのものである。これは superpowers `verification-before-completion` が提供する規律と一致する。よって主担いを superpowers に確定する。本番ビルドの CI 自動実行（ポリシー P3「ビルド成果物公開」等）は別物であり、引き続きハーネスが担う。

### 3. L2: ポリシー実現方式 = 委譲先 superpowers 実行スキルによるオーケストレーション

ポリシー P5〜P10 は、dev-workflow が独自のポリシーエンジン／イベントバスを持って実現するのではなく、**委譲先 superpowers 実行スキルの内部オーケストレーション＋シーム接続契約**として実現する。ドメインモデルの「ハーネス／イベント駆動」表現は概念モデル（ツール非依存、ADR-20260531 項3）として維持し、realization 層ではこの委譲オーケストレーションへ写像する。現行 dev-loop の設計（Phase 1b↔1c↔3 の自前ループ統治）からの乖離は意図的に許容する。

| ポリシー | 実現の担い手（realization） | 内容 |
|---|---|---|
| P5 検証実行起動 / P6 実装完了反映 / P7 検証失敗時タスク再開 | superpowers 実行スキル（`executing-plans` / `test-driven-development` / `verification-before-completion`） | 検証ループは委譲先スキルの内部オーケストレーションで実現。dev-workflow は検証方針（シーム S5）のみ供給。**dev-loop の自前リトライ／振動統治は撤去**（方針 C）し、最小インライン（基本 TDD＋セルフレビュー）のみフォールバック保持 |
| P8 判断依頼発行反映 / P9 判断依頼解消反映 | dev-workflow 接続契約（シーム S2） | 判断依頼／エスカレーションの堀としてインライン実現。superpowers に該当なし |
| P10 実装開始反映 | メインループのスキル起動（参照機構②） | #2「実装プランを確定する」人間ゲート通過後、メインループが次スキル（`executing-plans`）を起動するハンドオフ |
| P2 変更要求対応 / P3 ビルド成果物公開 | ハーネス | 本 Epic の委譲再構成スコープ外の自動駆動（CI 等）。現状維持 |

### 4. 参照機構②/③ 割当

割当規則: **オーケストレータ（メインループで動作する dev-loop / plan-issue）が直接呼ぶ process／規律スキル → ②（メインループの Skill ツール呼び出し）**。**ディスパッチされたサブエージェント内で走るメカニクス → ③（サブエージェント定義 `skills:` への preload）**。

| シーム | 対象コマンド | superpowers 委譲先 | 参照機構 | 理由 |
|---|---|---|:---:|---|
| S1 計画起動 | プランニングを開始する | `writing-plans` | ③ | plan-issue は Plan サブエージェントへ計画生成を委譲。その定義に preload する |
| S3 実装起動 | 実装を開始する | `executing-plans` / `subagent-driven-development` | ② | メインループが実装フェーズを起動する process スキル。**#265 で精緻化**: 主候補＝`subagent-driven-development`・フォールバック＝`executing-plans`（選択基準＝サブエージェント利用可否） |
| S4 TDD 実行 | タスクを完了する | `test-driven-development` | ② | 現行はメインループで TDD を実行。子Dが subagent 駆動実装を採用する場合のみ③（実装サブエージェントへ preload）。**#265 で解消**: 子D＝dev-loop は TDD（S4）を実装サブエージェント定義へ preload せずメインループから `Skill` 呼び出し（②）する方式を採用したため条件③は発生せず、S4＝②に最終固定（S3 の `subagent-driven-development` 主候補採用とは独立。下記 Consequences 参照） |
| S5 検証ゲート | 検証ゲートを実行する | `verification-before-completion` | ② | メインループ Phase 1c の規律として直接呼ぶ（L1 と整合） |
| S6 PR 化 | PRを作成する | `finishing-a-development-branch` | ② | メインループ Phase 4 |
| S7 レビュー契約 | レビュー依頼をする / 変更を要求する / レビューを承認する | `requesting-code-review` | ② | メインループ Phase 2 がレビュー依頼を起動（レビュー実行は内部でサブエージェント化される） |
| （作業環境） | （実装着手時） | `using-git-worktrees` | ② | 作業開始時にメインループが呼ぶ |

シーム S2（判断依頼）は dev-workflow 固有で superpowers 委譲がないため②/③の対象外。接続契約としてインライン保持する。

## Consequences

**得られた利益**:

- 単位を現状維持と確定したことで、子D/子E は「スキルの作り直し」ではなく「現行単位の内部メカニクス入れ替え」に集中できる。再分割に伴う Issue ライフサイクル・呼び出し規約・ドキュメント参照の連鎖改変を回避できる。
- L1/L2 と②/③割当を確定したことで、ADR-20260606-3 項4 の留保が解消し、子D（dev-loop 改変）/子E（plan-issue 改変）が配線先を一意に参照できる。
- ポリシー実現を委譲先 superpowers のオーケストレーションへ写像したことで、dev-loop の自前ループ統治（リトライ・振動検知）を方針 C のとおり撤去でき、保守を superpowers へ移せる。

**受容したトレードオフ**:

- realization 層がドメインモデルの「ハーネス／イベント駆動」表現と乖離する。これはモデルのツール非依存性（ADR-20260531 項3）を守るための意図的な写像であり、乖離自体は許容する。
- ②/③ の S4（TDD）割当は子Dの内部実装方針（メインループ実行 vs subagent 駆動実装）に依存する条件付き確定であり、子Dで最終固定する。**→ #265（PR #276、マージ済）でクローズ**: 子D＝dev-loop は TDD（S4）を実装サブエージェント定義の `skills:` へ preload（③）せず、メインループから `Skill` 呼び出し（②）する方式を採用したため、条件③は発生せず **S4＝② に最終固定**。S3 で `subagent-driven-development` を主候補にしても③へ転じない橋渡しは2点: (1) dev-loop は**自前の実装サブエージェントを定義せず** superpowers の subagent ディスパッチャへ委譲するため、TDD を preload する対象のサブエージェント定義がそもそも存在しない（preload 先が無い）。(2) ③の主利得（実装隔離・並列ディスパッチ）は複数タスクを捌くマルチタスク実行器で初めて活き、dev-loop の 1 Issue→1 PR 単位は非該当（③再検討は単位再定義時に限る。#265 / 後退時 #267）。よって S3 の subagent 駆動実装採用と S4＝② は独立に両立する。本先送り（「子Dで最終固定する」）は解決済み。
- superpowers 非導入時は L1/L2/②③ の委譲パスが効かず、最小インライン（基本 TDD＋セルフレビュー）に縮退する（ADR-20260606-3 のトレードオフを継承）。

**将来の留保事項**:

- 単位現状維持の妥当性は、子D/子E の内部メカニクス入れ替え後のドッグフード（#267）で品質後退がないことをもって最終確認する。後退が出た場合は本判定の再検討（単位再分割の再評価）を要する。
