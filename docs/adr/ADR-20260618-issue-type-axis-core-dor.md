# ADR-20260618: Issue種別軸の導入とコアDoR改訂方針

## Status

Accepted

## Context

Issue管理スキル（`create-issue` / `refine-issue`）は単一の共有DoR（`dor-default.md`）で運用しており、Issueの**種別差**を扱わない。サイズ軸（Small/Medium/Large）のみで充足項目を段階化するため、ドキュメント・ADR起票・バグ修正・機能追加といった性質の異なるIssueにも一律のチェック項目が適用される。

加えて、Issueに**実現方法（HOW）が混入することを防ぐ仕組みがない**。本来Issueは「何を／なぜ（WHAT/WHY）」を定義する単位であり、実装方式はプランニング・実装フェーズで決まるべきだが、現行DoRにはこの抽象度を担保する品質項目が存在しない。

これらを是正する「種別軸の導入」と「コアDoRの改訂」は、`create-issue` / `refine-issue` / `dor-default.md` / プラグイン共有 `references/`（ADR-20260604）という複数スキル・共有資源に波及する。作成側（shift-leftで前倒し充足）と精査側（作成後の精査）が同一DoRを参照する構造上（ADR-20260604）、片側だけを変えると判定基準がドリフトするため、改訂方針は両側を一括して固定する必要がある。後戻りコストが高く、横断的であるため、実装に先立ち方針をADRとして記録する。

本ADRは方針と採否理由を確定する。各決定の確定仕様（`dor-default.md` の最終チェック項目文言・種別マトリクス等）は後続Issue（#290 コアDoR改訂 / #291 種別軸 / #292 高度ガードレール）で詰める。

### 粒度判定（`docs/adr/README.md` 4項目チェックリスト）

| # | 項目 | 該当 | 理由 |
|---|---|---|---|
| 1 | 後戻りコストが高い | ✅ | `create-issue` / `refine-issue` / `dor-default.md` / 共有 `references/` に波及。誤った方針で実装後の是正コストが大きい |
| 2 | 複数モジュール・複数開発者に波及する | ✅ | 2スキルと共有DoR、後続3Issue（#290-#292）の実装前提となる |
| 3 | 採用理由が時間経過で揮発しやすい | ✅ | 「なぜサイズ軸に種別軸を足したか」「なぜブロッカーを不在ゲートから把握化したか」は後で揮発する判断 |
| 4 | ツールで自動強制できない | ✅ | DoR判定はモデル／人の解釈に依存し、Linter等で機械強制できない |

4項目すべて該当（4点）。判定ルール「3点以上でADR化推奨」に合致する。

## Decision

以下の3点を方針として確定する。確定仕様は後続Issueへ委ねる。

### (a) 種別軸×サイズ軸の二軸化

DoRの判定軸を、現行のサイズ軸（Small/Medium/Large）単独から、**Issue種別軸とサイズ軸の二軸**へ拡張する。種別（例: バグ修正・機能追加・リファクタリング・ドキュメント・ADR起票 等）に応じて適用するチェック項目を切り替え、サイズで段階を重ねる構成とする。

- サイズ軸は廃止せず維持する（追加的拡張であり、既存のサイズ段階化を削除しない）。
- 種別の具体的な分類・各種別に紐づくチェック項目集合は #291 で確定する。

### (b) コアDoR改訂3点

種別軸導入の土台として、全種別共通のコアDoRを次の3点で改訂する。

1. **ブロッカーの「把握」化**: 現行の「ブロッカーがない」（不在をReadyゲートとする）を、「ブロッカーが把握されている」へ改める。未解決の依存・前提が存在しても、それが特定・追跡されていれば作業着手は可能であるとの認識に合わせる。
2. **ゴール・AC重複整理**: 「ゴールがわかる（完了判断基準）」と「受入条件が検証可能な形で定義されている」は完了判断の点で重複する。両者の役割を整理し、重複を排した項目構成にする。
3. **「適正な抽象度」品質項目の新設**: IssueにHOW（実現方法）が混入することを防ぐ品質項目を新設する。Issueは WHAT/WHY に留め、実装方式の確定はプランニング・実装フェーズへ委ねる抽象度を担保する。

各項目の最終文言・判定基準は #290 で確定する。

### (c) プロジェクト独自規約による上書き互換

(a)(b) のコア改訂は、プロジェクト固有DoR（`{project}/.claude/dor/definition.md`）による上書き・拡張と互換に保つ。ADR-20260604 が確立した読み込み優先順位（プロジェクト固有 → プラグイン共有 `dor-default.md`）を踏襲し、種別軸・コア改訂後もプロジェクト側が項目を追加・上書きできる構造を維持する。高度なガードレール（プロジェクト独自の追加規約）の取り扱いは #292 で詰める。

### 不採用案

- **サイズ軸を種別軸へ置換する案**: 種別ごとにサイズ段階を内包させ単軸化する案。既存のサイズ段階化を破棄し後戻りコストが大きいため不採用。二軸の追加的拡張を採る。
- **HOW混入防止をスキル側のロジックのみで担保する案**: DoR項目化せず `create-issue` / `refine-issue` の手続きに埋め込む案。判定基準が共有DoRに現れず作成側・精査側でドリフトするため不採用。コアDoR項目として明示する。

## Consequences

- 種別差に応じたDoR適用が可能になり、性質の異なるIssue（ドキュメント・ADR等）に過不足のないReady判定ができる。
- HOW混入防止がDoR項目として明示され、作成側（`create-issue`）と精査側（`refine-issue`）が同一基準で抽象度を担保できる。
- 二軸化により、`dor-default.md` のチェック項目定義とサイズ判定ロジックの構造が複雑化する。種別×サイズの組み合わせ表現方法は #291 の設計課題として残る。
- (b)-1 のブロッカー把握化により Ready ゲートが緩む。把握済みだが未解決の依存を抱えたまま着手し、途中で停止するケースを許容する受容コストを伴う（緩和の運用詳細・歯止めは #290）。
- 後続Issue（#290-#292）は本ADRの方針を実装前提として固定でき、実装中の方針再交渉を避けられる。
- 本ADRは方針確定に留め、確定仕様は後続Issueに委ねる。後続Issueでの仕様詳細化の結果、本方針の前提が崩れる場合は本ADRを再検討する。

### 既存ADRとの矛盾確認

DoR・Issue管理に触れる既存ADRを確認した結果、Supersededに該当するものはない。

- **ADR-20260604（DoR共有資源のプラグインルート集約）**: DoR定義の**配置**と読み込み優先順位を扱う。本ADRはDoRの**内容（軸・項目）**を扱い、配置・優先順位は踏襲する（(c) で明示）。矛盾せず、Related とする。
- **ADR-20260601（自律度・承認ゲート整合）**: 本文で「ブロッカー」に言及するが、これは後続Issueのブロッカー解除という文脈語であり、DoR項目「ブロッカー」とは無関係。矛盾なし。

既存ADRの決定を無効化・上書きするものではないため、Superseded／Amended 手続きは不要。
