# ADR-20260606-3: superpowers 委譲をソフト/任意依存とし、フォールバック方針 C で実現する

## Context

ADR-20260531 は実装メカニクスの superpowers 委譲を決定したが、依存可否（配布形態・ライセンス）を「将来の留保事項」としてパークしていた。Epic #261 の feasibility gate である Issue #262 でこの留保を検証・確定したため、当該 facet を本 ADR で改訂する。

確認した事実・前提:

- **ライセンス**: superpowers v5.1.0 は MIT（一次情報＝`LICENSE` 本体・`.claude-plugin/plugin.json` の `"license": "MIT"` で検証済）。移植・参照とも法的に可。
- **ハーネスの依存機構**: Claude Code は `plugin.json` の `dependencies` によるプラグイン間ハード依存宣言（自動インストール・semver 制約・未充足時の無効化）を備える（`plugin-dependencies.md` で確認）。
- **方針の選択**: superpowers を使うかどうかはユーザーに委ねる（強制的な同梱・自動導入はしない）方針を採る。
- **未明示だった点**: ADR-20260531 は委譲を決めたが、superpowers 非導入時の dev-workflow の挙動（フォールバック）を明示していなかった。任意依存とするなら、非導入時にも dev-workflow が動作する必要がある。

## Decision

ADR-20260531 の「superpowers へ委譲する」基本方針は維持しつつ、その**依存方式と非導入時フォールバック**の facet を以下に確定する。

1. **採用経路: (b) ハーネス参照のソフト/任意依存**。superpowers のスキルを Skill ツールで参照呼び出しする。`plugin.json` の `dependencies` によるハード依存宣言は採らない（superpowers の利用はユーザー任意）。
2. **vendoring (a) は不採用**。superpowers スキルを dev-workflow にコピーする方式は、コピー後の更新追従コストが ADR-20260531 の主便益「保守競争からの離脱」に反するため採らない（MIT につき法的には可能だが、堀の趣旨に反する）。
3. **フォールバック方針 C**。ADR-20260531 が「セレモニー」と判定した重い自前機構（独立 test-spec 検証・リトライ統治・振動検知）は撤去する。一方で**最小インライン（基本 TDD＋セルフレビュー）をフォールバックとして保持**し、superpowers 導入時は委譲スキルで強化する。これにより非導入でも基本機能は動作し、重い機構の保守は手放す。
4. **参照機構の既定**。一律化せず、メカニクスごとに最適な呼び出し方を選ぶ。選択肢は ② メインループの Skill ツール呼び出し／③ サブエージェント定義 `skills:` フィールドへの preload の2択。個別メカニクスへの ②/③ 割当は単位妥当性判定（#264）で確定する。実現マッピング（#263）はその入力となる接続契約シームを列挙するに留める。

## Consequences

**得られた利益**:

- superpowers を強制せず、軽量導入とユーザーの選択を尊重できる。
- 重い自前機構の保守を手放しつつ、superpowers 非導入時でも基本機能（最小 TDD＋セルフレビュー）が動作する。
- ライセンス上の障害がない（MIT）。委譲先の選定が法的に確定した。

**受容したトレードオフ**:

- 任意依存のため、品質は superpowers 導入有無で2段階になる（非導入時は最小インラインに縮退し、委譲による強化が効かない）。
- 完全な保守離脱ではなく**部分削減**にとどまる。dev-workflow は最小インラインのフォールバックを保守し続ける。
- ハード依存宣言の堅牢性（自動インストール・バージョン固定・未充足時の明示エラー）は得られない。superpowers のバージョン差異・破壊的変更はソフト参照側で吸収する必要がある。

**留保事項・後続**:

- メカニクスごとの ②/③ 割当、および撤去/保持の線引きは #264（単位妥当性判定）で確定する。#263（実現マッピング）はその入力となる接続契約シームの列挙までを担う。
- ローカル開発（`setup-local.sh` = `--plugin-dir ./plugins/dev-workflow`）は現状維持。superpowers が読み込まれないため委譲パスはローカルで実行できないが、これを許容する。委譲パスの検証は superpowers 導入済の実環境（#267 ドッグフード）で行う。
