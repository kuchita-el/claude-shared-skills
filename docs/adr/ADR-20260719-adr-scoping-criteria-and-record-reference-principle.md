# ADR-20260719: ADR 化判断の3基準を決定として確定し、記録は可変文書を現在の参照先として指さない

## Context

本リポジトリの ADR 運用のうち、状態モデル・遷移スキーマ・編集機構は ADR-20260711-3 が決定として記録を持つ。一方、「ある決定を ADR として記録すべきか」「いつ起票するか」「命名規約をどこまで ADR の対象とするか」という ADR 化要否の3基準は、決定としての記録を持たないまま運用ドキュメント（`docs/adr/README.md`）に散文として蓄積されてきた。

#515 は、運用ルールの「生きた正本」を ADR 側から手順書（スキル）側へ反転させる作業である。ADR は決定の記録であって現在生きているルールの正本ではない、という原則に立つ。運用ルールの大半は、実務手順の所在をスキル側へ移し、ADR 側には決定の記録だけを残せば反転が完了する。

しかし上記3基準だけは、この反転が成立しなかった。3基準は運用ドキュメントが唯一の出所であり、これを決定として記録した ADR が corpus に存在しない。実務手順を移設すると、判定項目・スコア境界の由来（なぜ4項目なのか、なぜ境界で「書かない」に倒すのか）が corpus から失われ、後年に再検討する足場が無くなる。よって本 ADR で3基準を決定として記録し、反転の受け皿を作る。

反転作業の中で、これとは別種の構造的な誤りが見つかった。凍結された記録であるはずの ADR が、後から書き換わりうる可変文書を「現在の参照先」として現在形で指している箇所である。指し先の内容が変われば、記録の意味は書き手の意図と無関係に変質する。さらに退役した ADR は凍結原則（ADR-20260711-3 決定3）により決定内容を編集できないため、変質が起きても修復手段がない。この誤りは3基準の問題とは独立しており、以後の全 ADR 起票に効く横断規範を要する。決定4はこれを扱う。

## Decision

### 1. 粒度判定基準

ADR 化要否は、次の4項目チェックリストで判定する。各項目に該当すれば1点とする。

1. 後戻りコストが高い
2. 複数モジュール・複数開発者に波及する
3. 採用理由が時間経過で揮発しやすい
4. ツールで自動強制できない

判定ルールは次のとおりとする。

- **3点以上**: ADR 化を推奨する
- **2点以下**: ADR 化しない（非該当）
- **判断に迷ったら ADR 化しない**: 原則として ADR 化しない方向に倒す。後から必要性が判明したら、その時点で遡及して ADR 化する

境界で「書かない」を優先するのは、ADR 過多が検索性を損ね、蓄積した決定を参照する運用そのものの存在意義を失わせるためである。

### 2. 起票のタイミングとエスカレーション

ADR 化要否の判定と「いつ起票するか」は別の問題として扱う。最初から ADR にしようとせず、特定機能の決定はまず PR の説明に書く。後から他の機能でも同じ問いが出てきた時点で ADR へ昇格させる。

ADR 化を検討すべき決定が発生したことを示す実務シグナルは次の3つとし、いずれかに該当したら決定1の粒度判定基準で要否を判定する。

- 複数の選択肢を検討して片方を却下した
- トレードオフを受け入れた
- 制約や前提条件がある

### 3. 命名規約の ADR 化基準

命名規約は、構造的か書式的かで ADR 化対象を振り分ける。

| 種別 | 特徴 | 例 | 配置 |
|---|---|---|---|
| 構造的命名規約 | 横断・後戻りコスト高 | ファイル配置規約、集約名、ドメイン用語統一 | ADR 化（`docs/adr/`） |
| 書式的命名規約 | 局所・ツール強制可能 | 変数命名、import 順、空白・改行 | Linter／フォーマッタ規約として配置（ADR 外） |

ツールで自動強制できる規約は ADR 外とする。判断に迷う場合は、決定1の項目4（ツールで自動強制できない）に照らして振り分ける。

### 4. 記録の参照原則

記録（ADR）は、可変文書を現在の参照先として指さない。指す必要がある場合は、時点を固定した過去形の記述にするか、参照先の実質を自己完結的に述べる。

**適用範囲**: 本原則は以後の ADR 起票および `有効` な ADR の編集に効くほか、既存 corpus にも及ぶ。本 ADR を起票した同一の変更セットは、`有効` な ADR 群および運用文書が `docs/adr/README.md` を現在の参照先として指していた箇所を是正の対象に含める。ただし退役した ADR の `## Decision` は、次の適用限界のとおり対象から除く。

**適用限界**: 既に退役（`validity: 上書き済み` / `廃止済み`）した ADR の `## Decision` に本原則違反が残る場合、凍結原則（ADR-20260711-3 決定3）が優先し、是正しない。退役という事実自体がその記述に時点を供給するため、記録としての読み方は保たれる。この適用限界は本原則の構成要素であり、無条件の禁止を立てたうえで後から逃がす例外ではない。

本決定は決定1〜3の系ではなく、独立した横断規範として立てる。#515 が3基準の参照の張り替え先を運用ドキュメント外へ取れなかった理由そのものが本原則の違反構造にあり、また本原則は3基準と無関係に、以後の全 ADR 起票に対して効くためである。

## Consequences

**得られた利益**:

- ADR 化要否の3基準が決定として corpus に入り、実務手順の置き場所とは独立に根拠が保存される。本 ADR 起票時点（2026-07-19）で、これらの基準を実務手順として運用する実体は `manage-adr` スキルの参照ファイルへ移されており、本 ADR は決定の記録に徹する。以後、実務手順の所在や表現が変わっても、決定の骨子は本 ADR が保持する。
- 決定4により、記録が指し先の変化で意味を変質させる経路が塞がれる。とくに退役後は修復手段が無いため、起票時点での自己完結性が事後の可搬性を決める。

**受容したトレードオフ**:

- 決定1〜3の実質は実務手順側と同内容を持つ。本 ADR は決定4の自己完結性要求に従い、参照ではなく実質の再記述を選んだ。両者の間には表現の乖離が起こりうるが、記録側は決定の骨子（4項目・境界・振り分け軸）のみを保持し、運用上の記法や補足は実務手順側が担うため、乖離は骨子の外に限られる。

**適用限界の実例（`ADR-20260531-2` の未編集判断）**:

`ADR-20260531-2`（`validity: 上書き済み`、後継 `ADR-20260602-principles-rationale-hub`）は、その `## Decision`（同ファイル L23-32）の L29 で、操作層の SoT の一つとして `adr/README` を挙げている。字面のうえでは決定4の違反形（記録が可変文書を現在の参照先として指す）に見えるが、本 Issue ではこれを未編集とする判断を採った。論拠は次の4点である。

- **(a)** L29 は `## Decision` の内部にあり、決定内容そのものである。ADR-20260711-3 L74 は射程の一般規定を「この凍結原則の射程は決定内容の編集である」と置いたうえで、退役 ADR にも実施してよい類型を2つ具体的に認可している（front-matter（状態）にも `## Decision`（内容）にも触れず、状態の重複表示を消すだけの是正／固有情報を `## 関連ADR` へ移してから除去する移設）。この2類型は認可された具体例であって射程外の網羅列挙ではなく、可否を決めているのは一般規定のほう、すなわち決定内容に触れるか否かである。L29 はこの一般規定に照らして射程内であり、加えて同 L74 は「逆に、決定内容に触れる編集は退役 ADR に加えてはならない」と明示的に禁じている。
- **(b)** 同 L74 の末尾は、3段構えの判定フローを「`有効` な ADR への決定内容の編集を振り分けるもの」と限定している。したがって `validity: 上書き済み` の `ADR-20260531-2` に対し、「非core だから直接編集してよい」という分類を適用することはできない。
- **(c)** 退役した ADR の決定本文は、その退役時点の記録として読まれる。L29 の記述は 2026-05-31 時点の状態の記録として正しく、その後に SoT が移動した事実は当該 ADR の退役という事実が既に伝えている。すなわち、`ADR-20260711-3` の `## 変更履歴` L178 / L180 が可変文書への言及に対して採っている「時点明示」に相当する処理が、退役という事実によって構造的に既に効いている。
- **(d)** したがって本件は「記録が可変文書を現在の参照先として指している」未処理案件には当たらない。決定4の適用限界に照らし、凍結原則を優先して未編集とする判断である。

**適用有無の判定基準（`ADR-20260711-3` の README 言及が処理と未処理に分かれた理由）**:

本 ADR を起票した変更セットは、`ADR-20260711-3` が `docs/adr/README.md` に言及する箇所のうち `## 変更履歴` の L178 / L180 を改訂し、`## Decision` の L141 と `## Consequences` の L165 は改訂しなかった。同一ファイル内で処理が分かれるため、両者を分けた基準を次のとおり記録する。

- **基準**: その記述の真偽が、参照先の現在の内容に依存して決まるかどうか。依存するなら時点固定または自己完結化を要し、依存しないなら記録として既に閉じている。
- **改訂した2箇所**: L178 は凍結原則の射程について「射程自体は README …で既に同内容が運用されており」と述べ、L180 は「操作層の手順は README …が対応する」と述べていた。いずれも README が現にその規定を保持していることを現在形で主張しており、README を導線へ縮退させれば主張が偽になる。よって「当時の」を補い、時点を固定した過去の事実の記述へ改めた。
- **改訂しなかった2箇所**: L141 は決定7 が「廃止する」と述べた対象の同定（README 内蔵の索引を廃止する）であり、L165 は当時どこへ委譲したかの予定の記録である。いずれも README が現在何を保持しているかを主張していないため、指し先が空になっても記述は真のまま保たれる。
- **決定4 との関係**: 本基準は決定4 の禁止範囲を狭めるものではない。決定4 は「可変文書を現在の参照先として指さない」を無条件に課しており、本基準はある記述がその「現在の参照先として指す」に当たるかを個別に見分ける手順にすぎない。判定に迷う場合は、参照先を空にした状態でその記述を読み直し、偽になるなら適用対象、真のままなら対象外として振り分ける。本 ADR 起票時点（2026-07-19）で、この振り分けを以後の編集へ適用する実務手順は `manage-adr` スキルの編集判定リファレンスにも置かれた。本節の記述は、当該変更セットで処理が分かれた理由の記録である。

**射程外是正による dangling の解消**:

3基準を運用ドキュメントから移設すると、凍結 ADR 3本（`ADR-20260511-technical-decision-aggregate-foundation`、`ADR-20260531-2-workflow-design-rationale-hub`、`ADR-20260711-store-state-model-captures-stateless`）が持つ運用ドキュメントへの参照が指し先を失う。当初はこれを、凍結原則により修復手段のない dangling として想定していた。

しかし該当3箇所はいずれも各 ADR の `## Decision` の外（`## Context` および状態記述行）にあり、決定内容に触れず情報の総体が保存される是正であるため、`ADR-20260711-3` L74 の射程規定に照らして凍結原則の射程外である。よって本 Issue はこの3箇所を射程外是正の対象とし、同一の変更セットで是正する。当初想定されていた「修復手段のない dangling」は、これにより概ね解消される。

`plugins/growth/` 配下からの参照は #532 で修復されるため、恒久的 dangling には当たらない。
