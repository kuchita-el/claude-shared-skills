# ADR-20260621: completion-judgment.md の原則を素の会話に届ける手段として SessionStart フックを採用する

## Status

Accepted

## Context

#161 で `plugins/dev-workflow/references/completion-judgment.md` を新設し、完了/クローズ判定の早期収束防止原則を1ソース化した。`refine-issue` はこの原則を参照し、#268 では `dev-loop` Phase4 と `plan-issue` への組み込みを予定している。

しかし、これらはすべて**スキル起動時にのみ原則が読み込まれる**構造である。スキルを起動していない「素の会話」状態（#129 の元事例）では、原則は配布物の外に漏れ出ない。

#325 spike では、この構造的限界を越える手段として以下の3カテゴリを調査した:

**カテゴリA: フック方式**

Claude Code のプラグインフックは `hooks/hooks.json` に定義するだけで、プラグインを有効化したユーザーのフックに自動マージされる（ユーザー側の追加 hook 設定は不要）。

| フック | 発火タイミング | 発火頻度 | 主な副作用 |
|---|---|---|---|
| `SessionStart` | セッション開始時（startup/clear/compact） | 低 | 1セッションにつき1回のシステムメッセージ追加 |
| `Stop` | メインエージェント停止判断時 | 高（毎ターン） | 毎ターンの LLM コスト |
| `UserPromptSubmit` | 全プロンプト送信時 | 最高 | 全プロンプトに対するコスト・遅延 |

**カテゴリB: 配布物外の対処**

グローバル `~/.claude/CLAUDE.md` への原則記載指示・CLAUDE.md テンプレート提供など。ユーザーが手作業を行った場合のみ全会話に恒久的に効く。強制力はないが副作用もない。

**カテゴリC: ドキュメント側補完**

README/利用ガイドへの警告。ユーザーがドキュメントを読んだ場合のみ認知される。介入としては最も軟弱。

**ADR-20260606-3 との整合**

ADR-20260606-3 は「superpowers を強制せず、ユーザーの選択を尊重する」哲学を確定した。本件の「プラグインのフック自動マージ」はこの哲学と緊張関係にある。ただし、フックは「プラグインを有効化する」というユーザーの明示的なアクションの延長であり、プラグインが提供する能力としての注入は opt-in の範囲とみなせる。

## Decision

**SessionStart フックで `completion-judgment.md` の原則を systemMessage として素の会話に注入する。**

具体的には:

1. `plugins/dev-workflow/hooks/hooks.json` を新設し、`SessionStart` フックで `completion-judgment.md` の原則（または要約）をシステムメッセージとして注入する
2. `Stop` / `UserPromptSubmit` は発火頻度・副作用の観点で本用途に不適として不採用とする
3. カテゴリB（README 案内）は補完として実施する（SessionStart フックが効かない環境向けのフォールバック）

**不採用とした選択肢と理由:**

- `Stop` / `UserPromptSubmit`: 発火頻度が高くトークンコストが大きい。完了判定チェックを毎ターン・毎プロンプトで行うことは UX の劣化を招く
- カテゴリC（ドキュメント補完のみ）: 到達範囲が最も限定的であり、構造的限界を「越える」とは言い難い。補完として維持するが主手段にはしない

## Consequences

**得られた利益:**

- dev-workflow プラグインを有効化したユーザーの全セッションに完了判定原則が届く
- スキル起動を介さない「素の会話」での早期クローズ提案を抑止できる可能性が生まれる
- SessionStart は低頻度のため、副作用（トークンコスト・UX 劣化）を最小化できる

**受容したトレードオフ:**

- セッション開始時点でのみ注入されるため、会話が長くなると原則の効力が薄れる場合がある（モデルのコンテキスト内の希釈）
- プラグインを有効化した全ユーザーの全セッションにシステムメッセージが追加される（明示的に選択した機能ではなく、プラグイン有効化の副産物として適用される点でADR-20260606-3の哲学との摩擦がある）
- `completion-judgment.md` の内容が変わった場合、フックのシステムメッセージも更新が必要（またはファイル参照で自動追従する実装にする）

**留保事項:**

- 注入するシステムメッセージの内容・量（全文 vs 要約）は実装 Issue で設計する
- 発火頻度と UX への影響は実装後の dogfood で観察し、必要なら `Stop` との組み合わせやオプトアウト機構を検討する
- カテゴリB の補完（README 案内）は本 ADR とは独立した軽微な変更として実施可
