# ADR-20260627: スキル実行時のコンテキスト予算規約をプラグイン共有 references で単一出典化する

## Context

スキル起動後のメインセッション context 消費量が、起動側モデルの圧縮設定に依存して挙動が割れる。圧縮せずに済むタスクが別モデルでは圧縮を要し、スキルの途中で圧縮が走るとスキル指示・中間状態が失われて挙動が不安定になる。

原因はメイン context に載る5要素のうち (b) references からメインで Read した補助ファイル、(c) サブエージェント返却レポート、(d) gh / git の生出力、(e) 生成中間成果物の扱いに関する規約逸脱に収斂する。これらは「パスを渡し処理エージェントが自前 Read する／生出力をファイル化する／返却はサマリのみ」とすればメインから排除できる。この型は `refine-issue` 全件モードに理想形として存在するが、1スキルのローカル記述にとどまり、複数スキル（`create-issue`・`refine-issue` 1件モード・`plan-issue` の plan 本文往復）が参照できる単一出典が存在しない。

CLAUDE.md「スキル設計の token 規律」は既に存在するが、`description` 長・SKILL.md 行数・`allowed-tools` 最小化という**作成時の静的フットプリント**が中心で、実行中にメイン context を通過するデータのフローを規約化していない。

同じ問い（実行時に何をメインへ載せないか）は既に複数スキルで再発しており、ADR-20260719 決定2「起票のタイミングとエスカレーション」の昇格条件（他機能でも同じ問いが出てきた）を満たす。粒度判定基準の4項目に照らすと、後戻りコスト（後続改修群が基準に依存）△〜○、複数モジュール・複数開発者への波及（全スキル横断）○、採用理由の揮発（モデルの圧縮挙動依存）○、ツールで自動強制できない○ で 3〜4 点に達し、ADR 化推奨域にある。

## Decision

スキル実行時のコンテキスト予算規約を、プラグイン共有 references `plugins/dev-workflow/references/context-budget.md` として単一出典化する。

- 規約の**内容**（パス渡し原則・生ツール出力のファイル化・サブエージェント返却のサマリ化・滞留カテゴリ規則）は `context-budget.md` が保持し、本 ADR は**単一出典化の決定**と既存資産との関係を記録する。これは DoR（ADR-20260604 / `dor-default.md`）・完了判定原則（ADR-20260621 / `completion-judgment.md`）と同型の「内容＝reference／決定＝ADR」分割である。
- 配置はプラグインルート `references/` 集約原則（ADR-20260604）に従う。複数スキルが共有する参照資源であるため特定スキル配下に置かない。
- token 軸を2軸に分離する。**作成時静的**軸は CLAUDE.md「スキル設計の token 規律」が継続所管し、**実行時フロー**軸は本規約が所管する。CLAUDE.md にはフロー軸への委譲ポインタのみ置き、規約本文を二重化しない。
- 起動後ピーク context の上限は**スカラー（行数／トークン）で定めず、「滞留カテゴリ規則」で導出**する。メインへ滞留してよいのは SKILL.md 本体と構造化サマリのみとし、上限は既存の SKILL.md 行数規律（約 170 行）＋有界サマリから自動的に導かれる。

### 不採用案

- **案①: CLAUDE.md「スキル設計の token 規律」へ規約本文をインライン追記** — 規約本文が常時ロードされ、context 節約という目的と自己矛盾する。on-demand ロードされる reference を優先し不採用。
- **案②: ピーク context にハードな行数／トークン上限を新設** — 実行中の測定が困難で、モデル依存のトークン化と閾値のドリフトを招く。加えてこの上限はツールで自動強制できない閾値であり、遵守もドリフトの検知も書き手の裁量に委ねられる。滞留カテゴリ規則による導出を優先し不採用。

## Consequences

- 実行時コンテキスト予算の判断基準が単一出典化され、後続の個別スキル改修（参照ファイルのパス渡し統一・Issue 取得のファイル化＋comments 絞り・サブエージェント返却契約の明文化・SKILL.md 本体の縮退）が共通基準を参照して設計・レビューできる。
- token 規律が作成時／実行時の2軸に整理され、二重管理が回避される。CLAUDE.md は静的軸＋委譲ポインタへ縮退する。
- 上限をスカラーで持たないため、モデル更新（context 窓拡張）時も閾値見直しが不要。滞留カテゴリ規則は質的不変条件として残る。
- 本規約は、コンテキスト膨張・トークン効率対策に関する既存の点 ADR（ADR-20260525・ADR-20260525-2・ADR-20260604）を束ねる傘決定の位置づけになる（設計要素別の逆引きは各 ADR 本文を参照する。ADR-20260711-3 決定7）。
- `${CLAUDE_PLUGIN_ROOT}` / `${CLAUDE_SKILL_DIR}` の解決は Claude Code 側の仕様に依存する。将来この置換挙動が変わった場合は本 ADR を再検討する。
