# ADR-20260711-4: capture 見出しキーの同定規則（実行時刻＋run 内序数サフィックス）

## Context

growth の生観察 store（`captures.md`）は各観察を `## <timestamp>` 見出しで記録する。この見出しキーは2つの役割を兼任する。

- **順序キー（カーソル）**: distill はカーソルより新しい観測を処理し、処理後カーソルを最新見出しへ前進させる（`personal-store-spec.md`「distill 処理源選択と処理済みカーソル」・ADR-20260711-2）。要求は **capture 時刻の単調増加**。
- **同定キー（provenance）**: 候補（`candidates.md`）から由来観測への一意参照。distill の upsert キー・provenance 除外・再発回数 N の導出がこのキーに乗る（`personal-store-spec.md`「provenance 規約」）。要求は **一意性**。

現行実装は見出しを `date -u +"%Y-%m-%dT%H:%M:%SZ"`（capture **実行時刻**・秒精度）で採る。1 run で複数観察を検知すると全観察が同一実行時刻となり、見出しキーが run 内で衝突する。`personal-store-spec.md` provenance 規約は「timestamp が秒単位で一意」を前提としていたが、この前提は 1 run 複数観察で破れる。衝突は次を壊す:

- **upsert 上書き**: 別物の2観察が同一キーだと、後の仮説が前の仮説を置換して失う（silent loss）。
- **provenance 除外の巻き添え**: 巻き戻し再導出で、同一キーの一方が live 候補を持つと他方も除外され、二度と distill されない。
- **N 過小カウント**: 同一シグナルの2再発が1キーに潰れ、再発計測が劣化する。

加えて `personal-store-spec.md` L66 は見出しを「観測時刻」と定義する一方、実装は「実行時刻」を入れており、定義と実装がずれていた。本論点は #459（ADR-20260711 / -2）で store 状態モデルが provenance ＋処理済み highwater カーソルへ移行し、カーソルが見出しキーと同一キー空間で「capture 時刻の単調増加」に依拠するため、キー規則の確定はカーソルモデルの健全性の土台でもある。

関連 Issue: #434（本決定の起票元）、#459（カーソルモデル）、#451（単一 captures.md スケール、本決定を土台とする）、#470（origin 分類 spike、本 Issue から分離）。

## Decision

見出しキーの同定規則を **capture 実行時刻＋run 内序数サフィックス**に確定する。

1. **クロックは実行時刻を維持**する（`date -u`）。イベント時刻へは寄せない。実測で jsonl 行のイベント時刻は行順と時刻順が一致せず（sidechain ゼロの素セッションでも逆転を観測）、非単調であるため、カーソルが依拠する「capture 時刻の単調増加」（ADR-20260711-2）を破るため。
2. **1 run で2件以上記録する場合のみ**、各見出しに run 内序数サフィックス `-NN`（2桁ゼロ埋め・`-01` 始まり・記録順）を付す。1件のみの run はサフィックスを付さない（既存 store・spec 例との後方互換、差分最小）。
3. **カーソル比較は辞書順・同一キー空間を不変に保つ**。固定幅サフィックスは `T < T-01 < T-02` を辞書順で保証し、順序を壊さない。ADR-20260711-2 のカーソル highwater モデル本体は変更しない（本 ADR はその一意性 facet を補完する並立決定であり、上書きではない）。
4. **L66 定義を実装へ揃える**。見出しの定義を「観測時刻」から「capture 実行時刻（＋run 内序数）」へ改め、定義と実装のずれを解消する。

### 却下した代替案

- **イベント時刻 ms 化**（jsonl 行 timestamp 採用、`date -u` 廃止）: 一意性を ms 精度に賭ける案。却下理由は2点。(a) イベント時刻は非単調（実測: 行順≠時刻順の逆転あり）で、カーソルの単調増加前提を破る。(b) ms 精度でも衝突が残存する（実測: 64 timestamp 中 8 が重複）。
- **UUIDv7 / ULID**（時刻順序を内包する一意 ID）: 順序と一意性を単一キーで満たす点は優れるが、統合コストで却下。(a) 生成手段が環境に無い（`uuidgen` util-linux 2.39 は v7 非対応、Python 3.12 は `uuid.uuid7()` 非搭載、インラインインタプリタは規約で禁止）ためヘルパー保守が要る。(b) 不透明見出しが grep 前提の個人 markdown store の可読性を損なう。(c) ISO 8601 モデル（spec 各所）の全面改修を要する。(d) 1 run 数件の識別に大エントロピーは過剰。→ Phase 3 並行 capture 段で再訪する（下記留保）。
- **安定 ID による順序と同定の分離**（provenance を opaque ID、カーソル順序を実行時刻に分離）: 2キー化で spec の単一キー空間モデルを崩す。スコープ過大につき却下。

## Consequences

**得られる利益**:

- provenance 参照・upsert・candidate-status 前進・N 計測の同定対象が一意に定まり、1 run 複数観察でも衝突しない。
- カーソルが依拠する「capture 時刻の単調増加・同一キー空間」を維持（むしろ run 内同順位の曖昧さを解消して強化）する。
- 見出し定義と実装のずれを解消する。
- ISO 8601 の可読性を維持し、生成は既存 `date -u` ＋番号付けで足りるため実装変更が最小。

**受容したトレードオフ・留保事項**:

- run 内序数は **run 内一意**のみを保証する。hook 自発化（Phase 3）で同一 store への並行 capture が生じると、run をまたいだ大域一意性は保証されない。その段階では時刻順序を内包する一意 ID（UUIDv7 / ULID 等）への移行を再訪する（`personal-store-spec.md`「provenance 規約」の将来留保に明記）。
- サフィックスの有無で見出し形式が分岐する（1件 run は bare、複数 run は `-NN`）。同一実行時刻に別 run が跨る場合の bare と `-NN` の並びは、単独利用者・逐次 capture では現実的にほぼ起きない残余エッジとして受容する。
