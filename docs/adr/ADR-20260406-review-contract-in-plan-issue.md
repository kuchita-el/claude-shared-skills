# ADR-20260406: レビュー契約の所在を plan-issue に移行する

## Context

旧設計では dev-loop の Phase 0（入力理解）で実装者自身がレビュー契約（検証可能な完了条件）を自力導出していた。Anthropic「長期実行型エージェントハーネス設計」記事および運用観察を踏まえた検討で、以下の問題が識別された。

- 実装者自身が自分のレビュー基準を作成すると自己評価バイアスが生じる
- Plan レビュー時に人間が完了条件も含めて承認していないため、レビュー契約の妥当性が後段で検証されない
- code-reviewer サブエージェントがレビュー契約に基づき判定する設計（PR #70 で導入）のため、契約の品質が直接レビュー品質に影響する

関連: Issue #82（`plan-issue: 検証可能な完了条件の標準化とレビュー契約のplan-issue側への移行`、CLOSED 2026-04-06）。

## Decision

レビュー契約（検証可能な完了条件）の作成と承認を、dev-loop の Phase 0 ではなく plan-issue 段階に移行する。

1. **作成タイミング**: plan-issue がプランを生成する際、検証可能な完了条件セクションを必須項目として含める
2. **承認タイミング**: 人間レビュアーが Plan を承認する時点で、完了条件もあわせて承認する
3. **dev-loop 側の取り扱い**:
    - 正規パス（plan-issue 経由）: レビュー契約は Plan から取得し、dev-loop 内では自力導出しない
    - 簡易パス（Issue 直接・PRレビュー指摘・会話コンテキスト）: 従来通り dev-loop 内で自力導出する
4. **インターフェース契約の最小化**: dev-loop は plan-issue の出力フォーマット全体ではなく、「完了条件セクションの存在」のみをインターフェース契約として依存する。これによりフォーマット細部の変更が dev-loop に波及しない

## Consequences

**得られた利益**:

- 実装前の段階で完了条件の妥当性が人間レビュアーにより検証される
- code-reviewer は plan-issue 承認済みの契約に基づいて判定するため、判定基準の信頼性が向上
- 自己評価バイアスが除去される

**受容したトレードオフ**:

- plan-issue 段階で完了条件まで精密に設計する必要があり、プラン作成コストが増加
- 簡易パスでは依然として自力導出が残るため、品質保証レベルがパス別に異なる（dev-loop が明示）
- Plan の品質が低い場合、レビュー契約の品質も低下する直列依存リスク

**将来の留保事項**:

- 完了条件の標準化（検証可能な書式）は plan-issue 内のサブテンプレートとして継続改善
- 簡易パスでの品質保証レベル明示の運用観察（Issue #84 OPEN）に応じて、簡易パスでも plan-issue 経由を強制する判定が必要になるかは未定
