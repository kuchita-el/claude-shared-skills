# ADR-20260711-3: ADR 運用モデルの再設計（承認軸と有効性軸の2軸分離・導出ビュー index・編集機構）

## Status

Accepted

## Context

ホットスポット H4「ADR のライフサイクル管理」由来の運用モデルは、ADR-20260511 で正式化され、`docs/development/event-storming.md`「技術的意思決定」集約（Issue #130）に状態機械として定義された。この運用が 34 本規模に達した現在、外部監査（Issue #462）を起点とする再点検で構造的な欠陥が判明した。

現行モデルは 1 つの `Status` 欄に **2 つの軸** を混在させている。

- **承認ワークフロー軸**: この ADR が承認されたか（提案 → 承認 → 却下）。これは承認レビューの歴史事実であり、本来不変。
- **有効性軸**: この決定が今も効いているか（有効 / 後継に置換 / 放棄）。これは時間とともに変わる現在の効力。

`Accepted → Superseded` のように有効性起因で `Status` を書き換える運用は、有効性軸の情報を承認軸のフィールドに焼き込んでおり、「承認された歴史事実」を「今の効力」で上書きしてしまう。結果として、記憶を持たないエージェントが「今どれが有効な ADR か」を単一の出典から安価に判別できない（Accepted / Superseded / Deprecated が混在した一覧しか得られない）。

直近の ADR-20260711（captures の無状態化＝記録から状態を分離する）は「記録から有効性・状態を分離する」同型の実例であり、本 ADR はその原理を ADR 運用自身に適用する位置づけである。

本 ADR は Issue #463 の検討（フェーズ1: 対象モデル確定、フェーズ2: 機構設計）の決定を正式化し、ADR-20260511 の状態モデル・廃止扱いを再設計する。ADR-20260511 が定めた採番方式・配置は変更せず本 ADR に引き継ぐ。

関連ホットスポット: H4（運用モデルの再設計）

## Decision

### 1. Status を承認軸と有効性軸の2軸に分離する

1 つの `Status` 欄に混在していた 2 軸を、別フィールドに分離する。値は日本語のユビキタス言語で確定する。

- **承認軸（不変・承認の歴史記録）**: `提案中` / `承認済み` / `却下`
    - `提案中`: 起案され、承認者の判断を待つ
    - `承認済み`: 承認された履歴事実。以後不変
    - `却下`: 提案が承認に至らず終了。一度も運用されない終端
- **有効性軸（可変・現在の効力）**: `有効` / `上書き済み` / `廃止済み`
    - `有効`: 現在運用中
    - `上書き済み`: 後継 ADR に置き換えられた（後継識別子を保持）
    - `廃止済み`: 後継なしで放棄された

`却下`（承認軸: 一度も承認されなかった終端）と `上書き済み` / `廃止済み`（有効性軸: 承認後に効力を失った状態）を別語とし、2 軸が語彙レベルで衝突しないようにする。これは ADR-20260511 のライフサイクル状態（`Proposed / Accepted / Deprecated / Superseded` の英文4状態）と、`docs/development/event-storming.md` の「状態名を英文表記で確定」という決定の両方を置き換える。

### 2. front-matter を唯一の権威とし、有効 ADR index は導出ビューとする

状態系は各 ADR ファイル冒頭の front-matter に置く。キーは英語（構造的フィールド名）、値は上記の日本語ユビキタス言語とする。

```yaml
---
status: 承認済み          # 承認軸・不変（提案中 / 承認済み / 却下）
validity: 有効            # 有効性軸・可変（有効 / 上書き済み / 廃止済み）
superseded-by: ADR-...   # validity=上書き済み のとき必須
---
```

front-matter を単一の権威とし、有効 ADR index（`docs/adr/index.md`）はそこから機械生成する純粋な導出ビューとする。index は `validity: 有効` の ADR のみを列挙し、ファイル全体が生成物（build artifact）であって人手編集しない。これにより「本文を読めば確実（確実性）」と「一覧で探せる（検索性）」を drift させずに両立する。

front-matter スキーマの必須ルール（drift-lint が強制する）:

| 遷移 | status | validity | superseded-by |
|---|---|---|---|
| 起票 | 提案中 | （無し） | （無し） |
| 承認 | 承認済み | 有効 | （無し） |
| 上書き | 承認済み（不変） | 上書き済み | 必須 |
| 廃止 | 承認済み（不変） | 廃止済み | （無し） |
| 却下 | 却下 | （無し） | （無し） |

キーを英語としたのは、キーが状態概念そのものではなく構造的なフィールド名であり、英語キーでも概念の二重管理による drift を生まないためである（ユビキタス言語の本体は値側にある）。冒頭で `status` と `validity` が隣接することで「承認済み」を「有効」と誤読する罠も防ぐ。生成入力を本文見出しの正規表現抽出ではなく front-matter の key-value 解析としたのは、後者のほうが堅牢なためである。

### 3. 編集機構（3段構え）と判定フロー

有効性が可変性のゲートを兼ねる。`有効` の ADR は生きた文書として本文を直接編集してよい。`上書き済み` / `廃止済み` に退役した ADR は凍結された歴史的成果物として編集しない。

`有効` な ADR への変更は、次の decision tree で core / 非core / 些末 を判定する。境界アンカーは「その変更に再承認が要るか」。

```
1. その編集で「決定の結論・方向・前提」が変わるか？
   YES（結論の反転・前提の失効・後継への置換）
      → core = 再承認が要る
      → 新規 ADR を起票し、旧 ADR の validity を「上書き済み」に
   NO → 2 へ
2. 決定内容に触れるか？（パラメータ値・適用範囲・条件・理由付け）
   YES（骨子は保つが決定関連情報を更新）
      → 非core = 既承認の枠内
      → 直接編集 ＋ 本文「## 変更履歴」に1行
   NO（typo・整形・リンク・語句・体裁のみ、決定内容ゼロ）
      → 些末 → 直接編集、履歴は git のみ
```

| 変更の性質 | 対応 | 変更履歴の在り処 |
|---|---|---|
| 些末（typo・整形・リンク修正、決定内容ゼロ） | 直接編集 | git のみ |
| 非core かつ決定に関わる（パラメータ・適用範囲＋理由） | 直接編集＋`## 変更履歴` に1行 | ファイル内 |
| core（再承認が要る） | 有効性を `上書き済み` にし、新規 ADR を起票 | 別 ADR ファイル＋相互参照 |

**安全則: 判定に迷ったら core 寄り（新規 ADR）に倒す。** core を非core と誤判定すると、再承認イベントを飛ばして core 変更を権威記録に焼き込む不可逆な不整合が残る。逆に非core を core と誤って新規 ADR にしても過剰なだけで可逆（後で統合できる）。可逆性の非対称から core 寄りが安全である。

原則: ADR ファイルは、あらゆる consumption 面（Wiki・Confluence・artifact・抽出ファイル）に対して自己完結する。git は repo の実装詳細であり、決定関連情報の SSoT にはしない。決定粒度の変更履歴（git の commit 粒度とは別物）はファイル内 `## 変更履歴` に持つ。

### 4. Amended 機構を廃止する

ADR-20260511 系の運用にあった Amended（別 ADR 起票で facet のみ改訂し旧 ADR を Accepted 維持）機構は廃止する。パラメータ・適用範囲の改訂は上記「非core = 直接編集＋`## 変更履歴`」で扱い、決定の骨子を覆す改訂は core = 新規 ADR で扱う。

既存の `Amends:` / `Amended by:` 相互参照（実データ7本）は front-matter に載せず、本文 `## 関連ADR` に歴史的相互参照として凍結保持する（遡及編集しない）。新スキーマに `amends` キーは設けない。

### 5. drift-lint（3レイヤ）と合流先

有効性の同期は、CI 不在下でも「index を再生成して一致するか」を検査する drift-lint で担保する。検査契約は3レイヤ:

- **レイヤ1（front-matter スキーマ検証）**: 上記スキーマ必須ルールを満たすこと。
- **レイヤ2（index 同期）**: `docs/adr/index.md` を再生成し、コミット済みと差分ゼロであること。
- **レイヤ3（相互参照双方向性）**: `A.superseded-by=B` ⟺ B 本文 `## 関連ADR` に `Supersedes: A` が存在すること。既存 Amended 関係は凍結扱いで検査対象外。

合流先は `scripts/lint-adr.sh` として実体化し、`.claude/settings.json` の PreToolUse フック（既存 `validate-skills.sh` と並置、`git commit` 前ゲート）から呼ぶ。人間も手動実行できる二面構成とする。CI 設定ファイル（`.github`）は基盤不在のため作らない。

### 6. ライフサイクル操作は tooling に委譲する

ADR の各遷移（起票 → 承認 → 上書き / 廃止 / 却下）で front-matter を正しく書く操作は tooling（スキル / プラグイン）に委譲し、drift-lint を最終ガードとして不整合を検出する二重化とする。人間・エージェントが README 手順を手作業で追う運用に依存しない。

### 7. モデル制約由来の設計判断インデックス（15件）を廃止する

`docs/adr/README.md` 内蔵の「モデル制約由来の設計判断インデックス」は、設計要素別の人手キュレーション意味索引であり、モデル更新時の再評価トリガーとして設計されていた。その再評価フローは実運用で回っておらず、今回の再設計の主眼（手動 drift 源を排する）と衝突する手動 drift 源であるため廃止する。意味逆引きは各 ADR 本文で代替し、ギャップ地図は必要なら別 Issue で扱う。

### 8. 採番方式・配置（ADR-20260511 から不変で引き継ぐ）

ADR-20260511 が定めた次の2項目は変更せず、本 ADR に引き継ぐ（自己完結原則により restate）。

- **採番方式**: `ADR-YYYYMMDD[-N]`。同日1件目は `-N` なし、2件目以降は `-2` から付与。採番衝突時は対象日の ADR 一覧から最大番号+1。採番日＝起票日。
- **配置**: `docs/adr/` 配下にフラット構造。サブディレクトリは作らない。

## Consequences

**得られる利益**:

- 記憶を持たないエージェントが「今どれが有効な ADR か」を単一の導出ビュー（`docs/adr/index.md`）から安価に判別でき、承認の歴史記録（承認軸）が不変に保たれる。
- 有効性の同期を drift-lint で機械的に担保できるため、CI 不在の制約が本質的問題でなくなる。
- 本 ADR 自体が、新モデルの編集判定フロー（core 変更＝新規 ADR で旧 ADR を上書き）を自分自身に適用した実例になっている（ADR-20260511 を全体 Supersede）。

**受容したトレードオフ**:

- ADR-20260511 の採番方式・配置は不変だが、Amended 機構を廃止したため部分改訂ができず、全体 Supersede ＋ restate の形を取る。不変部分の再掲という冗長を、自己完結原則の対価として受容する。
- front-matter への移行が完了するまで、既存 ADR の `## Status` 見出し形式と新スキーマが混在する。この過渡状態は本 ADR では解消せず、移行（フェーズ3）に委ねる。

**将来の留保事項（本 ADR のスコープ外＝ downstream）**:

- 機構の実装（`scripts/lint-adr.sh` と PreToolUse フック合流、`docs/adr/index.md` 生成、`docs/adr/README.md` 廃止手順・`template.md` の改訂、ライフサイクル操作スキル）は実装 Issue に委ねる。
- 移行（全 ADR への front-matter 付与、`ADR-20260531-2` の Superseded 実例の validity 移設、既存 `Amends:` / `Amended by:` 関係7本の grandfather、英語状態名から日本語ユビキタス言語への移行、`docs/development/event-storming.md`「技術的意思決定」集約の状態機械の作り替え〔英文4状態→2軸・`却下` 終端追加〕、index 初期生成）は Issue #463 フェーズ3に委ねる。
- 本 ADR および ADR-20260511 の front-matter 化もフェーズ3の一括移行に含める（本 ADR は現行 `## Status` 形式で起票している）。
