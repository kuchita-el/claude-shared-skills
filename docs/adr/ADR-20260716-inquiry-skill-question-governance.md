# ADR-20260716: inquiry スキルによる問いの統治と spike の下流非伝播

## Context

### 起票時点の構造的空白

Issue 種別軸（ADR-20260618）は DoR 層で導入され、`create-issue` / `refine-issue` / `dor-default.md` / `issue-type-profiles.md` に配線された。しかし波及先の列挙は当該4資源に留まり、下流（`plan-issue` / `implementation`）への伝播は検討対象に入っていない。調査の結果、下流伝播に関する賛成・反対・保留・スコープ外のいずれの記録も ADR・Issue・docs に存在せず、意図的なスコープ外化ですらない純粋な空白であることが確認された。

その結果、以下の非対称が生じている。

- `issue-type-profiles.md` は spike の完了を「決定が下る／文書ができる」と定義し、「ADR 起票もこの種別で扱う」と明記する。しかし**その完了を生み出すスキルが存在しない**。DoR 層で定義した完了概念が実行系で宙に浮いている
- `implementation` の `description` は「コードの説明・調査のみ」「設計議論・技術比較検討」「ドキュメントのみの変更」を明示除外する。除外はあるが、**では誰が扱うのかの受け皿指定がない**
- `plan-issue` は `gh issue view --json ...,labels,...` で labels を取得しているが、種別分岐に使用していない。配線点は開いているが結線されていない

### 実運用で観測された症状

大きめの検討課題を与えると、課題を細分化した後に、細分化した課題を解くこと自体が目的化し、元の課題に対する解が出ない。この症状は spike ラベルの付いた Issue に限らず発生する。

内訳として以下が挙げられた。

- 決定的な課題（これを決定すれば推移的に他も決まる課題）があるとき、その課題と他の課題の優先度の重み付けができていない
- 統合ステップがない
- 分解の完全性を検証しない
- 分解が細かすぎる
- 問いではなく作業に分解される

### 診断: 実装分解と調査分解の型の不一致

上記5症状は独立ではなく、単一の根から派生する。実装タスクと調査タスクでは分解の型が異なる。

| | 実装タスクの分解 | 調査の問いの分解 |
|---|---|---|
| 構造 | 依存順序を持つが本質はリスト | 含意グラフ（問いが問いを決める） |
| 全ノードの扱い | 全部やる必要がある | 全部解く必要はない |
| 完了 | 全タスク消化＝機能完成（構成的） | 決定的な問いが解ければ残りは推移的に決まる／不要化する |
| 最適戦略 | 網羅 | 枝刈り |

`superpowers:writing-plans` は「全タスクを列挙して全部やる」メソドロジーであり、`plan-issue` はこれに計画骨格を委譲している。実装においてはタスク消化の集合が機能完成と一致するため正しく機能し、統合ステップを別途持つ必要がない（PR が統合を兼ねる）。一方、調査に同じ分解を適用すると、subquestion 群への解答の集合は元の問いへの解答と一致しないにもかかわらず、パイプラインは網羅型の分解メソドロジーしか持たない。したがって**構造的に**「全 subquestion の消化」が目的化する。観測された5症状はこの型の不一致の現れである。

### カバレッジ調査の結果

既存スキル・エージェントの実定義を調査した結果、「解く」手段と「問いの統治」機構でカバレッジが大きく異なることが判明した。

「解く」手段は概ね充足している。

| 調査種別 | 担い手 |
|---|---|
| コードベース内の事実調査 | `Explore`（組み込みエージェント） |
| Web・文献調査 | `deep-research`（CLI バンドルの workflow。SKILL.md は存在しない） |
| 依存・互換性調査 | `dependency-check` |
| ドメイン理解・モデリング | `event-storming` / `domain-modeling` |
| 設計案の生成・比較 | `brainstorming`（ただし終端が `writing-plans` 固定で、比較検討で止まる出口がない） |
| 履歴・経緯調査 | 空白（`Explore` の定義は open-ended analysis を明示除外する） |
| 実験・PoC | 空白 |
| 論点整理・ADR 起票 | 実行側が空白（DoR 定義のみ存在） |

一方、「問いの統治」機構には決定的な空白がある。

| 機構 | カバレッジ |
|---|---|
| 1. 根の問いの固定・保持 | `deep-research`（全プロンプトへ verbatim 埋め込み）／`dependency-check`（各 Phase に「解決する問題」を明記） |
| 2. 問いへの分解 | `deep-research`（フラット5並列）／`dependency-check`（固定 DAG） |
| 3. 問い同士の含意構造 | **全スキルで空白**。`dispatching-parallel-agents` に至っては問いの独立性を前提条件として要求し、依存のある問いを対象外に落とす |
| 4. 決定的な問いの特定・枝刈り | **実質空白**。唯一 `dependency-check` が打ち切り条件を持つが、人手で事前に並べた固定 DAG 上のハードコードであり、動的に生成された問い集合には効かない。`deep-research` のランク付けは claim（答え）が対象であり、問いが対象ではない |
| 5. 根の問いへの統合 | `deep-research`（synthesis）／`dependency-check`（Phase 6） |

すなわち、欠けているのは「解く」手段ではなく**機構3と機構4**である。既存スキルはいずれもフラット並列分解＋全実行であり、構造的に「全部解くのが目的化」する。

### 決定的な構造的事実: 調査はプランと実行を分離できない

調査は適応的である。問いを解いた結果によって次に解くべき問いが変わり、含意グラフ自体が書き換わる。したがって事前に完全なプランを書くことは原理的にできない（プランを書く段階で既に調査が必要になる）。`plan-issue` → `implementation` の2段構えは、実装の不確実性が低くプラン可能であるという性質に依存した構造であり、調査へは移植できない。

### ADR 起票タイミングについて

ADR-20260719 決定2「起票のタイミングとエスカレーション」は、最初から ADR 化せず PR 説明に書き、他の機能でも同じ問いが出た時点で昇格させることを定める。本 ADR はこの段階を経ずに直接起票している。この規約の前提は「特定機能の決定」であるところ、本決定は起票時点で既に横断的であり（`plan-issue` / `implementation` / `create-issue` / `refine-issue` の4スキルに跨る）、昇格条件が起票時点で満たされていると判定した。規約のバイパスではなく前提不該当である。

## Decision

### 決定1: inquiry スキルを新設する

「問いの統治」を担うスキルを新設する。持つのは機構1〜5（根の問いの固定・問いへの分解・含意グラフ・決定的な問いの特定と枝刈り・根の問いへの統合）と、終端の成果物化のみとする。

コアフローは以下の適応的ループとする。

```
0. 根の問いを受領・固定（以降、全工程で verbatim 保持）
1. 問いへの分解（作業ではなく問い）
2. 含意グラフ構築（Qi → Qj: Qi が決まれば Qj が決まる／消える）
3. 決定的な問い（pivot）の特定
   ── 人間ゲート1: 分解は根の問いを覆っているか／pivot 判定は妥当か
4. pivot を解く（委譲）
5. 再評価・枝刈り（各未決問いを3値判定）
   ── (c) 未決が残れば 3 へ戻る
   ── 人間ゲート2: ループ脱出時、「前提消滅」と判定した全問いをまとめて確認
6. 根の問いへの統合
   ── 人間ゲート3: 決定の確認
7. 成果物化
```

ステップ3〜5の適応的ループが本体である。これがあるためプランと実行を分離しない。

人間ゲートは実行順に3つ置く。ゲート2の配置はループ脱出時とし、ループ内のステップ5には置かない。「前提消滅」の判定はループごとに発生しうるが、ステップ5に毎回ゲートを置くと確認回数がループ回数に比例し、繰り返しの割り込みによる確認疲れで形骸化する。ループ脱出時に一括確認すれば確認は1回で済み、かつ統合（ステップ6）より前であるため、誤判定を戻しても統合をやり直す必要がない。代償として、誤って「前提消滅」と判定した問いが数ループのあいだ閉じたまま残り、その間の調査が無駄になりうることを受容する。枝刈りは「解かない」ことであって誤った結論を出すことではないため、ループ脱出時に戻せれば「調べるべきことを調べずに終わる」は防げる。

### 決定2: 「解く」は委譲する

各問いの解決手段は自前で持たず、既存スキル・エージェントへ委譲する。委譲は「問い＋必要な調査の種類」を渡すインターフェースとし、特定のツール名を直接指定しない。これは委譲先の可換性を担保するためである。空白領域（履歴・経緯調査、実験・PoC）は当面汎用エージェント／`implementation` へ委譲し、力不足が判明した場合は委譲テーブルの当該行のみを専用エージェント／専用スキルへ差し替える。inquiry 本体は不変に保つ。

この委譲境界は ADR-20260531（実装メカニクスは superpowers へ委譲し、dev-workflow は堀と接続契約を担う）と同型である。inquiry においては調査メカニクスが委譲対象であり、問いの統治が堀にあたる。

### 決定3: spike を下流へ伝播させず inquiry へ分岐する

Issue 種別軸を `plan-issue` / `implementation` へ伝播させない。spike は inquiry へ分岐する。`plan-issue` が labels を取得済みでありながら未結線である状態は、「結線しない」という決定として確定させる。

### 決定4: docs 種別は本 ADR の対象外とする

docs 種別の実行者不在は本 ADR で扱わない。実運用上の症状として観測されているのは docs の品質（造語・冗長・一文の長さ）であって実行者不在ではないため、inquiry に docs を含める根拠が現時点で存在しない。

### 却下した代替案

**案A: `plan-issue` / `implementation` へ種別分岐を追加する。** 却下理由は Context「調査はプランと実行を分離できない」。2段構えに乗らないものを2段構えへ押し込む解であり、観測された症状の根（型の不一致）を制度化してしまう。加えて両スキルとも `CLAUDE.md`「スキル設計の token 規律」の 170 行目安の付近にあり、網羅型と枝刈り型という2つの型を同居させると分岐が深くなる。`workflow-patterns.md`「パターン判定結果から `implementation` パスを直接結線せず、`implementation` 側は計画ファイルの有無とインプット種別で判定すること」（ADR-20260407 由来）とも擦る。

**案C: 共有 reference のみを新設し、既存スキル群が参照する。** 却下理由は、規律の置き場所ができるだけで spike Issue の実行者不在が全く解消しないこと。ただし本決定は案Cの要素を吸収する（統治の型定義は reference へ置き、スキルが参照する形を採りうる）。

## Consequences

### 受容したコスト

- パイプラインが2本になる。inquiry と `plan-issue` / `implementation` の境界整理が必要になる
- `brainstorming` と機能が一部重複する。差分は終端であり、`brainstorming` は `writing-plans` へ固定的に接続するため比較検討で止まる出口を持たない。inquiry はこの出口を持つ
- 履歴・経緯調査と実験・PoC の空白は委譲による暫定対応であり、根本的には埋まっていない。委譲インターフェース（決定2）が差し替えの可換性を担保することで、この負債を局所化する
- docs 種別の実行者不在は未解消のまま残る（決定4）。実害が観測された時点で再検討する
- 「前提消滅」の確認をループ脱出時の一括（人間ゲート2）としたため、誤って「前提消滅」と判定した問いが数ループのあいだ閉じたまま残り、その間の調査が無駄になりうる。ループ脱出時に戻せるため誤った結論には至らないが、費やした調査は回収できない。ループ内の毎回確認と引き換えに受容したコストである

### 得られた利益

- 案Aが解こうとした「種別軸の下流伝播」という課題は、「伝播させない」という形で決着する。DoR 層で止まっていた非対称は、下流へ配線するのではなく分岐先を用意することで解消する
- 「解く」手段の既存資産を再利用でき、inquiry 自体は薄く保てる

### 将来再検討する条件

- docs 種別の実行者不在について実害が観測された場合
- 履歴・経緯調査または実験・PoC の委譲先が力不足と判明した場合（委譲テーブルの差し替えで対応し、本 ADR の決定自体は変更しない）
- inquiry と `brainstorming` の重複が運用上の混乱を生んだ場合

### 関連する決定

枝刈りを可能にするための `completion-judgment.md` の判定値域拡張は ADR-20260716-2 で扱う。誤枝刈りを防ぐ検証機構のうち、**危険度に応じて機構を割り当てるという設計則**（「推移的に決定済み」は自動検証、「前提消滅」は人間ゲート）は ADR-20260716-2 決定4 で確定する。これは決定2（値域拡張）を安全に成立させる条件であり、決定2と分離できないため ADR 側に置く。一方、その具体値（検証の票数、ゲートの文言、台帳の記述形式）はスキル定義および references に置く。

台帳の置き場所は粒度判定基準で 2〜3 点となり、「判断に迷ったら ADR 化しない」原則により ADR 化せず、スキル定義および references に記述する。
