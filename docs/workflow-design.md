# AI×人間 開発ワークフロー設計書
## ── コーディングエージェントを基盤とした開発ライフサイクル

---

## 設計思想

### 前提

**対象範囲:** 本設計書は「AIエージェントと人間の協働」における作業フローと品質ゲートを定義する。チーム内の合意形成プロセス（レビュー承認フロー、複数ステークホルダー間の意思決定等）はチーム規模・組織構造に依存するため、本設計書のスコープ外とする。

**継続的に更新するドキュメント:** 以下のドキュメントはDiscoveryで骨格を作り、Deliveryを通じてボトムアップに育てる。Discovery完了時に完備されている必要はない。

- **CLAUDE.md:** プロジェクト開始時に最低限の記述（言語・フレームワーク・ディレクトリ構造）を置き、レビューで発見したパターンを都度追記する
- **ADR:** 技術スタック選定等の初期判断があれば記録する。以降はPRの設計判断が複数機能で参照されるようになった時点で昇格させる

### 基本原則

本ワークフローは3つの原則に基づく。

**原則①：計画と実装を分離する**
AIにいきなりコードを書かせない。仕様（Spec）を先に作り、人間がそれを承認してから実装に入る。仕様が「信頼できる唯一の情報源（Single Source of Truth）」となり、AIの実装を検証する基準になる。

**原則②：フェーズゲートで品質を積み上げる**
実装中の数百のマイクロ判断を逐一レビューするのではなく、フェーズの境界（ゲート）で集中的にレビューする。これにより、レビューの認知負荷を下げつつ品質を維持できる。

**原則③：AIにAIを監視させ、人間は判断に集中する**
書いたAI（セッション）とは別のAI（セッション）にレビューさせる。人間はAIが苦手な「文脈を踏まえた判断」に集中する。

### 全体フロー概観

```
┌─────────────────────────────────────────────────────────────┐
│                     開発ライフサイクル                          │
│                                                             │
│  ┌─ Discovery ─────────────────────────────────────────┐    │
│  │  問題定義 → ドメイン構造 → デリバリーアイテム定義 → 設計判断の記録  │    │
│  │  [人間主導, AI補助]  [人間承認]                        │    │
│  └───────────────────────────────────────────────────┘    │
│         │                                                   │
│         ▼  Discovery完了の定義を満たしたら                      │
│  ┌─ Delivery ──────────────────────────────────────────┐    │
│  │  技術設計+タスク分解 → 実装+レビュー+PR                  │    │
│  │  → 自動レビュー → 人間レビュー → マージ                 │    │
│  │  [AI自走: plan-issue, dev-loop]  [人間がゲートキーパー]  │    │
│  └───────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

## Discoveryフェーズ

**目的：「何を作るか」を明確にし、Deliveryで迷わない状態を作る。**

Discoveryの各活動は人間主導で行う。AIはドメインの正解を自律的に出せないため、ドメインエキスパートの知識が不可欠である。AIはたたき台の生成、構造化、網羅性チェック等の補助に使う。

### 問題定義

プロジェクトに1つ。「誰のどんな問題を解くか」を合意する。

**決めること：**

- 誰がどんな問題を抱えているか（ユーザーセグメント + 課題）
- なぜ今解決に値するか（ビジネス的な動機）
- 成功をどう測るか（KPI / Success Metrics）
- スコープ外の明示（「これはやらない」）

**成果物：** `docs/problem-statement.md`

**人間が判断すべきこと：** 問題の切り口が正しいか、KPIが事業目標と整合しているか。

### ドメイン構造

ドメインの全体像と内部構造を合意する。デリバリーアイテムの実装に先立って作成し、AIが局所最適な設計判断を下すリスクを防ぐ。

**なぜ事前に必要か：**

デリバリーアイテムを個別に切り出してAIに実装させると、AIは各デリバリーアイテムの局所的な情報だけで設計判断を下す。たとえば契約管理ドメインで「契約を新規作成できる」だけを渡すと、AIはその時点で必要な構造だけでモデルを作る。次の「契約を変更できる」で `amendment` の概念が登場し、さらに「契約を解約できる」「自動更新する」とデリバリーアイテムが進むたびにモデルの拡張・再設計が発生し、最悪の場合、数アイテム後に「ドメインモデルをゼロから作り直したい」事態に陥る。

これを防ぐために、デリバリーアイテムの実装に先立ってドメインの全体像を「70%の見通し」として確保する。完璧な事前設計ではなく、デリバリーアイテムの実装を通じて発見した知見はドメイン構造にフィードバックする。

**決めること：**

- ドメインイベント、コマンド、集約、状態遷移（業務の全体像）
- 集約の境界
- データ型、ワークフロー型の定義
- バウンデッドコンテキストとコンテキスト間の依存方向（複数ドメインの場合）

**作成プロセス：**

| ステップ | やること | スキル | 成果物 |
|---------|---------|--------|--------|
| 1. Big Picture | システム全体のイベントフロー・コンテキストマップの洗い出し | `/event-storming` | `docs/big-picture.md` |
| 2. Design Level | 単一ドメインの深掘り（イベント→コマンド→集約→状態遷移） | `/event-storming {ドメイン名}` | `docs/{domain}/event-storming.md` |
| 3. ドメインモデリング | event-stormingの結果からデータ型・ワークフロー型を定義 | `/domain-modeling {ドメイン名}` | `docs/{domain}/domain-modeling.md` |

**人間が判断すべきこと：** 状態遷移が現実のビジネスプロセスと合っているか、集約の境界が適切か、型がドメインの制約を正しく表現しているか、将来の拡張の見通し。

### デリバリーアイテム定義

ドメイン構造を踏まえて、何をどの順で届けるかを合意する。デリバリーアイテム単位で切り出し、それぞれのスコープと受け入れ基準を定義する。

**決めること：**

- 各デリバリーアイテムのスコープ（ドメイン構造のどの部分を実装するか）
- 受け入れ基準（テスト方針のデフォルト観点を考慮）
- 優先順位と提供順序

**成果物：** ユースケース仕様（`docs/{domain}/use-cases/{name}/spec.md`）+ Issue（チケット）

ユースケース仕様（spec.md）を作成した後、ACを転記してIssueを起票する。ユースケース仕様はストック情報（蓄積・繰り返し参照される）として残り、Issueは作業駆動用のフロー情報として使い分ける。

**ユースケース仕様（spec.md）の構成：**

```markdown
# 契約の新規作成

## ドメインマッピング
- 集約: Contract
- 状態遷移: [*] → Draft → PendingApproval → Active
- コマンド: CreateContract, SubmitContract, ApproveContract

## 受け入れ基準
- Draft状態の契約を作成できる
- Draft契約を承認申請に提出できる
- 承認者が契約をActiveにできる

## 今回は実装しないが、ドメイン構造で考慮済みのもの
- Amendment（集約として定義済みだがロジックは後続アイテム）
- 自動更新（状態遷移として定義済みだが後続アイテム）
- 解約（状態遷移として定義済みだが後続アイテム）

## 外部リソース
- デザイン: https://figma.com/file/xxx
- 詳細仕様: https://xxx.atlassian.net/wiki/...
```

「今回は実装しないがドメイン構造で考慮済み」のセクションにより、AIは将来の拡張の見通しを理解した上でコードを書ける。

**デリバリーアイテムの粒度ガイドライン：**

判断基準は「1つのPRとして、レビュアーが30分以内に意図を理解してレビューできるか」である。行数は言語・フレームワーク・テスト戦略で大きく変わるため、ハードな基準にしない。

適切な粒度の例（タスク4〜6個）：

| デリバリーアイテムの例 | タスク数目安 |
|------------|-----------|
| パスワードリセット機能 | 4〜5 |
| 一覧画面に価格帯フィルターを追加 | 3〜4 |
| 注文履歴のCSVエクスポート | 3〜4 |
| チームメンバーの招待機能 | 5〜6 |

大きすぎる例（分割すべき）：「ユーザー認証システムの構築」→ サインアップ / ログイン+セッション管理 / パスワードリセット / OAuth連携 / 二要素認証 に分割。

小さすぎる例（まとめるべき）：「カラムを1つ追加」「日付フォーマット統一」「ボタンの色変更」→ 関連する変更とまとめるか軽量プロセスで処理。

判断チェックリスト：
- 受け入れ基準が3〜7個で書けるか（少なすぎ＝小さすぎ、多すぎ＝大きすぎ）
- レビュアーが30分以内に変更の意図と影響を理解できるか
- マージした時点で何らかの価値が完結しているか
- AIが1セッションで全体像を把握できるか

**人間が判断すべきこと：** ビジネス価値の優先順位、デリバリーアイテムの粒度が適切か。

### 設計判断の記録（ADR）

技術的な意思決定をADR（Architecture Decision Record）として `docs/adr/` に記録する。

**書くべき判断基準：** 以下のどれかに当てはまるとき。
- 複数の選択肢を検討して片方を却下した
- トレードオフを受け入れた
- 制約や前提条件がある

**エスカレーションパターン：** 最初からADRにしようとせず、まずPRの説明に書く。後から他の機能でも同じ問いが出てきたらADRに昇格させる。

```
特定機能の決定 → まずPRの説明に書く
      ↓
他の機能でも同じ問いが出てきた
      ↓
ADRに昇格（docs/adr/{番号}-{タイトル}.md）
```

**成果物：** `docs/adr/{番号}-{タイトル}.md`

### Discovery完了の定義

以下を満たしたらDeliveryに移行する。

- [ ] Problem StatementとSuccess Metricsが合意されている
- [ ] 主要ドメインのドメイン構造が合意されている（event-storming.md + domain-modeling.md）
- [ ] 最初のSprintで着手するデリバリーアイテムのユースケース仕様が作成され、IssueがACまで書かれている

## Deliveryフェーズ

**目的：Discoveryの成果物を起点に、エージェントを使って継続的に価値を届ける。**

### Deliveryの全体フロー

```
Issue → 技術設計+タスク分解（plan-issue）
  → 実装+セルフレビュー+PR作成（dev-loop）
  → 自動レビュー（CI）
  → 人間レビュー
  → マージ
```

### 技術設計＋タスク分解

| 項目 | 内容 |
|------|------|
| 実行者 | AI（`plan-issue` スキル） |
| 入力 | Issue + ドメイン構造ドキュメント + テスト方針（CLAUDE.md）+ 既存コードベース |
| 出力 | `docs/plans/issue-{番号}.md`（実装設計とタスク分解を1ファイルに統合） |
| ゲート | 人間が判断依頼の確認事項・検証方針の妥当性・タスク粒度を承認 |

`/plan-issue {Issue番号}` で起動。スキルの内部手順は `skills/plan-issue/SKILL.md` を参照。

**検証方針の設計：** 検証方針セクションで、ACでは自明でない振る舞いをBDD的に記述する。この記述はIssue ACとは異なり、選択した技術アプローチに依存する設計レベルの振る舞い仕様である。dev-loopのレビュー契約およびtest-designerのテスト設計の入力となる。各エントリに検証レベル（ユニット/統合/E2E/目視確認）を付与する。

記述対象の決定基準:

- コードベースとIssueだけでは妥当性を判断しにくく、かつ誤った場合の修正コストが高い振る舞いを対象とする
- 仕様変更（ドメイン構造やユースケース仕様の変更）に伴う技術設計への影響のうち、振る舞いとして表現できるものを記述する
- ACをそのままBDDに書き直しただけのエントリは含めない

**判断依頼の設計：** 高リスクの設計判断とプランに関する確認事項を統合し、人間に判断を求める。各項目は判断待ち（実装ブロック）または前提確認（仮定を明記して進行可能）に分類する。

**人間のレビュー観点：**

- 既存アーキテクチャとの一貫性
- ドメイン構造で定義された集約・状態遷移に従っているか
- 検証方針の妥当性（重要な振る舞いが漏れなく識別されているか）
- テストレベルの選択の適切さ
- タスク粒度・順序・依存関係の適切さ

### 実装＋セルフレビュー＋PR作成

| 項目 | 内容 |
|------|------|
| 実行者 | AI（`dev-loop` スキル） |
| 入力 | Issue番号 or 承認済みの実装プラン（`docs/plans/issue-{番号}.md`）。計画ファイルの検証方針セクション有無により正規パス/簡易パスに分岐 |
| 出力 | 実装コード + テストコード + PR |
| ゲート | 検証ゲート（型チェック + lint + テスト全パス）→ セルフレビュー → PR作成 |

`/dev-loop {Issue番号}` で起動。内部フェーズ・サブエージェント構成の詳細は `skills/dev-loop/SKILL.md` を参照。

**設計のポイント（原則との対応）：**

- **テスト先行（原則①）**: テスト設計と実装を別エージェントに分離し、テストが「仕様のドキュメント」として機能する。plan-issueの検証方針セクション（人間承認済み）がある場合、test-designerはそれを入力としてコードレベルのテストケースに変換する
- **検証ゲート（原則②）**: セルフレビューに入る前に機械的な品質チェック（型チェック + lint + テスト全パス）を通す
- **コンテキスト分離（原則③）**: セルフレビューは実装とは別コンテキストのサブエージェントが実行し、自己評価バイアスを排除する
- **エスカレーション**: 判断ポイントが発生した場合、Issueにコメントを投稿してセッションを終了する

**正規パス/簡易パスの入力分岐：**

- **正規パス（plan-issue由来）**: Issue番号入力時に `docs/plans/issue-{番号}.md` を自動探索、または計画ファイルパスを直接入力。計画ファイルに検証方針セクション（`## 検証方針` + `### 検証すべき振る舞い`、人間承認済み）が存在する場合に適用。検証方針セクション（主）+ Issue AC（補完）からレビュー契約を導出する。矛盾検出時はエスカレーション
- **簡易パス**: 計画ファイルがない場合、または検証方針セクションがない場合。PRレビュー指摘・会話コンテキスト起点の場合も常に簡易パス。従来通りPhase 0でACから自力導出

**TDD実行方針：** dev-loopはタスク単位でTDDサイクル（テスト作成 → 実装 → リファクタ）を回す。各タスクの完了時にコミットし、1タスク＝1コミットの粒度を保つ。これにより、問題発生時のロールバックが容易になる。

### 自動AIレビュー（CI連動）

| 項目 | 内容 |
|------|------|
| 実行者 | AIレビューボット（CodeRabbit / PR-Agent等） |
| 入力 | PRの差分 |
| 出力 | PR上のインラインコメント + サマリー |
| ゲート | 自動（重大な問題があればブロック） |

AIレビューの役割分担：

| レビュー観点 | 担当 | 検出方法 |
|------------|------|---------|
| 構文エラー・型エラー | CI（自動） | コンパイラ, リンター |
| コーディング規約違反 | CI（自動） | リンター, フォーマッター |
| 既知の脆弱性パターン | AIボット | SAST + LLMパターンマッチ |
| テストカバレッジ不足 | AIボット | カバレッジレポート + LLM分析 |
| 命名の一貫性 | AIボット | コードベース全体との比較 |
| ロジックの正しさ | AIボット + 人間 | LLM推論 + 人間の判断 |
| 仕様との整合性 | 人間 | ユースケース仕様 / ACとの突合 |
| アーキテクチャ適合性 | 人間 | 設計原則との照合 |
| ビジネス価値の判断 | 人間 | ドメイン知識 |

### 人間レビュー

| 項目 | 内容 |
|------|------|
| 実行者 | 人間（チームメンバー） |
| 入力 | AIレビュー済みのPR + AIサマリー |
| 出力 | Approve / Request Changes |
| ゲート | 最終的なマージ判断 |

**人間がレビューすべき観点（AIに任せられないもの）：**

- この変更は本当に必要か？（要件の妥当性）
- チームの暗黙知と整合しているか？
- 将来の拡張性に問題はないか？
- ユーザー体験への影響は？
- 他のチームへの影響は？

AIレビューボットが生成するPRサマリーとウォークスルーを起点に読むことで、大きな差分でも変更の意図と影響範囲を素早く把握できる。人間はAIが「OK」と判断した部分は軽く確認し、AIが指摘した部分と、AIが判断できない高次の観点に集中する。

### マージ

CIがすべて通り、AIレビューに重大な指摘がなく、人間レビュアーがApproveしたらマージ可能。

### 変更パターン別の運用フロー

すべての変更にフルプロセスを適用するとオーバーヘッドが過大になる。変更の性質に応じて、適用するプロセスのレベルを使い分ける。

```
変更の要求が来た
    │
    ├─ ドメインの概念（状態遷移、エンティティ構造）が変わる？
    │     │
    │     YES → パターンC: ドメイン構造更新 → ユースケース仕様作成/更新 → 実装
    │
    ├─ 既存デリバリーアイテムの振る舞い（受け入れ基準）が変わる？
    │     │
    │     YES → パターンA: ユースケース仕様更新 → 実装
    │
    └─ 上記いずれでもない（内部的な修正、バグ修正、軽微な調整）
          │
          └─ パターンB: 直接実装（Issue/チケットで管理すれば十分）
```

**パターンA：既存デリバリーアイテムの拡張（ユースケース仕様を更新する）**

例：契約一覧に「契約金額の範囲」で検索できるフィルターを追加する場合。既存のユースケース仕様に受け入れ基準を追記し、実装プランを更新してから実装する。ドメイン構造の更新は不要（契約のライフサイクルやエンティティ構造は変わらないため）。

ユースケース仕様更新時は変更履歴をドキュメント内に残す：

```markdown
## 変更履歴
- v1.0: 初回リリース（契約名、ステータスで検索）
- v1.1: 契約金額の範囲フィルターを追加 ← 今回
```

**パターンB：軽微な修正（ユースケース仕様の更新不要）**

例：一覧のデフォルトソート順変更、バグ修正、リファクタリングなど。受け入れ基準が変わらない変更にはSpecプロセスを適用しない。既存テストの期待値修正と実装変更をコミットすれば十分。

判断基準：受け入れ基準が変わるならユースケース仕様を更新、内部挙動だけが変わるならユースケース仕様の更新不要。

**パターンC：ドメインモデルに影響する変更（ドメイン構造も更新する）**

例：契約に多段階の承認ワークフローを追加する場合。状態遷移が変わり、新しい集約（Approval）が追加される。

更新の順序が重要：ドメイン構造更新（`event-storming` スキルの再実行、人間が承認）→ ユースケース仕様作成/更新 → 実装。

**日常の開発では大半がパターンAとBであり、パターンCは月に数回程度。** ユースケース仕様の運用が重くなりすぎないためには「受け入れ基準が変わるときだけユースケース仕様を更新する」くらいのルールが現実的である。

## テスト方針

Discovery（AC作成）とDelivery（テスト設計・TDD実行）の両フェーズで参照する横断的ガイドライン。プロジェクト固有の追加方針（使用フレームワーク、カバレッジ基準、モック方針等）はCLAUDE.mdに定義する。

**デフォルトで考慮する観点：**

- 正常系: 主要なユースケースパスの網羅
- 異常系: 不正入力、権限不足、外部サービス障害等
- 境界値: 上限・下限、空・null・ゼロ、型の境界
- 状態遷移: ドメイン構造で定義された遷移パスと禁止遷移の検証

**観点の具体化：** デフォルト観点はplan-issueの検証方針セクションで、選択した技術アプローチに応じたBDD的記述として具体化される。test-designerはこの検証方針セクションをコードレベルのテストケースに変換する。

## プロジェクト管理

### 作業単位の階層

| 単位 | 定義 | 期間の目安 |
|------|------|-----------|
| テーマ | 関連するデリバリーアイテムの束。1つのビジネス目標や技術的目標に対応する | 数週間〜数ヶ月（複数Sprint） |
| デリバリーアイテム | ユースケース仕様からACを転記したIssueで管理する作業単位。マージ時点で何らかの価値が完結している | 数日〜1 Sprint |
| タスク | デリバリーアイテムを技術的に分解した実装単位。plan-issueで生成する | 数時間（1コミット） |

各単位は機能的な作業にも技術的な作業（Enabler）にも適用できる:

| 単位 | 機能的な例 | 技術的な例（Enabler） |
|------|-----------|---------------------|
| テーマ | ログイン体験の改善 | Next.jsメジャーバージョン移行 |
| デリバリーアイテム | パスワードリセット機能 | App Router移行 |
| タスク | リセットリンク送信APIの実装 | 認証ページのApp Router移行 |

### <a id="sprint-ops"></a>Sprint運用

```
Sprint開始
├─ ユースケース仕様 → Issue起票（ACを転記）
├─ Issue → plan-issue → dev-loop → PR
├─ 生成コードを人間がレビュー（ACとの差分・副作用確認）
├─ PRにIssue番号を紐付けてマージ
└─ 振り返り → 新Issue起票 / 必要ならADR更新 / ドメイン構造へのフィードバック
```

テーマ・Sprint・ロードマップの管理にはプロジェクト管理ツール（GitHub Projects、Jira等）を使用する。ツール選定はプロジェクトに委ねる。ツール別の運用モデルケースは [Sprint運用のモデルケース](references/sprint-management-examples.md) を参照。

### ディレクトリ構成

```
my-project/
├── CLAUDE.md                              # エージェント向けコンテキスト・規約
├── .claude/
│   ├── skills/                            # 共有スキル（シンボリックリンク）
│   └── agents/                            # プロジェクト固有のサブエージェント
├── docs/
│   ├── problem-statement.md               # 問題定義・Success Metrics
│   ├── big-picture.md                     # システム全体のイベントフロー・コンテキストマップ
│   ├── adr/                               # 設計判断の記録
│   │   └── {番号}-{タイトル}.md
│   ├── plans/                             # 実装プラン（技術設計＋タスク分解）
│   │   └── issue-{番号}.md
│   └── {domain-name}/                     # ドメイン単位でまとめる
│       ├── event-storming.md              # イベント・コマンド・集約・状態遷移
│       ├── domain-modeling.md             # データ型・ワークフロー型定義
│       └── use-cases/                      # ユースケース単位
│           └── {name}/
│               └── spec.md               # ドメインマッピング・AC・外部リソースリンク
├── src/
└── tests/
```

### ストック情報とフロー情報

ドキュメントを「ストック」と「フロー」に区別する。

| 分類 | 性質 | 例 |
|------|------|-----|
| ストック | 蓄積され、繰り返し参照される | problem-statement.md, event-storming.md, domain-modeling.md, ユースケース仕様（spec.md）, ADR |
| フロー | 作業を駆動し、完了後は参照頻度が下がる | Issue, PR, 実装プラン（docs/plans/） |

**原則：ストックとフローは異なる整理原理に従う。** ストック情報はソフトウェアアーキテクチャの構造（ドメイン > ユースケース）で整理する。フロー情報はプロジェクト管理の構造（テーマ > デリバリーアイテム > タスク）で整理する。両者はデリバリーアイテム（Issue）がどのユースケース仕様を参照するか、で接続される。

**原則：フェーズではなくドメインで分類する。** Discovery/Deliveryは時間軸（フェーズ）であり、ディレクトリ構造（空間軸）に反映すべきではない。`docs/discovery/` のようなフェーズ名のディレクトリは作らない。ドメイン構造もユースケース仕様もDiscoveryで作成されるが、Deliveryで継続的に参照・更新される。

### ドキュメント間の結合度

**原則：ドメイン内は密結合OK、ドメイン間は疎結合にする。**

各ドメインのドキュメント（event-storming.md、domain-modeling.md、ユースケース仕様）は同一ドメイン内で密に参照し合ってよい。一方、あるドメインのドキュメントが別ドメインの内部ドキュメントを直接参照してはならない。

ドメイン間の接点は `docs/big-picture.md`（コンテキストマップ）に集約する。各ドメインのドキュメントがドメイン間の関係を知る必要がある場合は、big-picture.mdを参照する。

```
docs/
├── big-picture.md              ← ドメイン間の唯一の接点
├── contract/
│   ├── event-storming.md  ──┐
│   ├── domain-modeling.md ──┤ ドメイン内: 密結合OK
│   └── use-cases/         ──┘
└── billing/
    ├── event-storming.md  ──┐
    ├── domain-modeling.md ──┤ ドメイン内: 密結合OK
    └── use-cases/         ──┘

contract/ ←✕直接参照✕→ billing/     ← ドメイン間: big-picture.md経由
```

この原則により、ドメイン数が増えてもドキュメント間の整合性コストがドメイン数の二乗に比例して増大することを防ぐ。

### <a id="external-tools"></a>外部ツールとの連携

DiscoveryのドキュメントをConfluence等の外部ツールに置くこと自体は問題ない。ただしコーディングエージェントは外部ツールを参照できないため、**エージェントが必要とする情報はリポジトリに転記する**。

```
外部ツール（ソースオブトゥルース）     リポジトリ（エージェント向け）
──────────────────────────────      ──────────────────────────
問題定義（詳細）          →転記→    docs/problem-statement.md
ドメインモデル（詳細）    →転記→    docs/{domain}/
ADR（詳細）               →転記→    docs/adr/
ユーザーストーリー        →起票→    Issue（ACまで記載）
デザインカンプ            →リンク→  ユースケース仕様の外部リソース欄
```

### CLAUDE.md

CLAUDE.mdはエージェントの行動を制約する最も重要なファイルである。プロジェクトのコーディング規約、テストルール、アーキテクチャ制約、ブランチ戦略等を記述する。内容はプロジェクトごとに異なるため、本設計書ではテンプレートを規定しない。

### スキルとフェーズの対応

主要なワークフローフェーズは共有スキルとして実装されている。

| 設計書のフェーズ | 対応スキル | 状態 | 入力 | 出力 |
|---|---|---|---|---|
| ドメイン構造（イベント・集約） | `event-storming` | 実装済み | ドメイン名 | `docs/{domain}/event-storming.md`, `docs/big-picture.md` |
| ドメイン構造（型・ワークフロー） | `domain-modeling` | 未実装（#57） | ドメイン名 | `docs/{domain}/domain-modeling.md` |
| 技術設計+タスク分解 | `plan-issue` | 実装済み | Issue番号 | `docs/plans/issue-{番号}.md`（判断依頼・検証方針を含む） |
| 実装+レビュー+PR | `dev-loop` | 実装済み | Issue番号 or 計画ファイル | 実装コード + テスト + PR |

各スキルの内部フェーズ・サブエージェント構成の詳細は、各 `skills/{skill-name}/SKILL.md` を参照。

## アンチパターンと対策

| アンチパターン | なぜ危険か | 対策 |
|-------------|----------|------|
| ドメイン構造なしでデリバリーアイテムを積み上げる | データモデルが断片的になり、数アイテム後に破壊的な作り直しが発生 | デリバリーアイテムの実装に先立ってドメイン構造を作成し、全体像を確保する |
| ドメイン構造で完璧を目指す | ウォーターフォール化し、着手が遅れる | 「70%の見通し」で十分。残り30%はデリバリーアイテムの実装を通じて発見・反映する |
| すべての変更にフルプロセスを適用 | 軽微な修正でもオーバーヘッドが発生し、チームが疲弊する | パターンA/B/Cの判断フローに従い、受け入れ基準が変わるときだけユースケース仕様を更新 |
| 仕様なしでいきなり実装させる | AIの解釈が要件とずれ、大規模な手戻りが発生 | 必ず技術設計+タスク分解を経てから実装に入る |
| 巨大なタスクを1つのセッションで | コンテキスト枯渇で品質が劣化 | 1タスク＝1コミットの粒度でサブエージェントに委任 |
| デリバリーアイテムの粒度が大きすぎる | PRが肥大化しレビュー品質が低下、AIレビューの精度も落ちる | 受け入れ基準3〜7個、レビュアーが30分で理解できるサイズに分割 |
| AIレビューだけで人間レビューを省略 | ビジネスロジックの誤りやアーキテクチャ劣化を見逃す | AIレビューは「フィルター」、人間レビューは「判断」 |
| テスト生成をAIに丸投げ | ハッピーパスのみのテストになりがち | テスト観点（異常系、境界値）を明示的に指示 |
| CLAUDE.mdを更新しない | 同じ間違いが繰り返される | レビューで発見したパターンを都度追記 |
| AIの出力を無条件に信頼 | 微妙なロジックエラーが本番に混入 | セルフレビュー＋人間レビューの二段構え |
| ストック情報をIssueだけに書く | Issueがcloseされると埋もれ、後続アイテムで過去のスコープが不明になる | ユースケース仕様にストック情報を残し、IssueはACの転記+作業駆動用として使う |
