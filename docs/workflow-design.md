# AI×人間 開発ワークフロー設計書
## ── Claude Code を基盤とした Spec-Driven Development の実装ガイド

---

## 1. 設計思想

### 1.1 基本原則

本ワークフローは3つの原則に基づく。

**原則①：計画と実装を分離する**
AIにいきなりコードを書かせない。仕様書（Spec）を先に作らせ、人間がそれを承認してから実装に入る。仕様書が「信頼できる唯一の情報源（Single Source of Truth）」となり、AIの実装を検証する基準になる。

**原則②：フェーズゲートで品質を積み上げる**
実装中の数百のマイクロ判断を逐一レビューするのではなく、フェーズの境界（ゲート）で集中的にレビューする。これにより、レビューの認知負荷を下げつつ品質を維持できる。

**原則③：AIにAIを監視させ、人間は判断に集中する**
書いたAI（セッション）とは別のAI（セッション）にレビューさせる。人間はAIが苦手な「文脈を踏まえた判断」に集中する。

### 1.2 全体フロー概観

```
┌─────────────────────────────────────────────────────────┐
│                    開発ライフサイクル                       │
│                                                         │
│  ① 要件定義 ──→ ②③ 技術設計+タスク分解                    │
│     [AI作成]      [AI作成]                               │
│     [人間承認]     [人間承認]                              │
│         │             │                                  │
│         ▼             ▼                                  │
│  ④⑤⑥ 実装+セルフレビュー+PR作成                           │
│     [AI自走: dev-loop]                                   │
│     [検証ゲート+人間最終承認]                               │
│         │                                                │
│         ▼                                                │
│  ⑦ 自動レビュー ──→ ⑧ 人間レビュー ──→ ⑨ マージ           │
│     [AIボット]      [人間判断]        [自動/人間]          │
└─────────────────────────────────────────────────────────┘

  凡例:  [AI作成]=AIが生成  [人間承認]=人間がゲートキーパー
        [AI自走]=AIが自律実行  [CI検証]=自動パイプライン
```

## 2. プロジェクト基盤の構築

### 2.1 ディレクトリ構成

```
my-project/
├── CLAUDE.md                    # プロジェクトの「憲法」
├── .claude/
│   ├── skills/                  # 共有スキル（シンボリックリンク）
│   │   ├── event-storming/      # ドメインSpec作成（イベント・集約・状態遷移）
│   │   ├── domain-modeling/     # ドメインSpec作成（データ型・ワークフロー型定義）
│   │   ├── plan-issue/          # 技術設計＋タスク分解
│   │   ├── dev-loop/            # 実装＋セルフレビュー＋PR作成
│   │   └── refine-issue/        # Issue精査
│   └── agents/                  # プロジェクト固有のサブエージェント（必要に応じて）
├── docs/
│   ├── big-picture.md                 # システム全体のイベントフロー・コンテキストマップ
│   ├── plans/                         # 実装プラン（技術設計＋タスク分解）
│   │   └── issue-{番号}.md
│   └── {domain-name}/                 # ドメイン単位でまとめる
│       ├── event-storming.md          # イベント・コマンド・集約・状態遷移（ドメインレベルSpec）
│       ├── domain-modeling.md         # データ型・ワークフロー型定義（ドメインレベルSpec）
│       └── specs/                     # featureレベルSpec（ユーザーストーリー単位）
│           └── {feature-name}/
│               └── requirements.md    # 要件定義書
├── src/
├── tests/
└── .github/
    └── workflows/
        └── ai-review.yml       # AI自動レビューのCI設定
```

### 2.2 CLAUDE.md の設計

CLAUDE.md はAIの行動を制約する最も重要なファイルである。以下にテンプレートを示す。

```markdown
# プロジェクト: {プロジェクト名}

## コーディング規約
- 言語: TypeScript (strict mode)
- フレームワーク: Next.js 15 / App Router
- テスト: Vitest + Testing Library
- スタイル: ESLint + Prettier（設定済み）
- コミットメッセージ: Conventional Commits形式

## テストルール
- すべてのPRには対応するテストが必須
- テストカバレッジは既存より下げない
- テスト実行: `npm test`（タイムアウト30秒）
- モックは外部依存（DB、API、ファイルシステム）にのみ使用
- 内部モジュール間のモックは禁止
- テストファイルは `*.test.ts` として対象ファイルと同ディレクトリに配置
- テスト名は振る舞いを日本語で記述してよい

## ブランチ戦略
- 機能ブランチ: `feature/{issue-number}-{short-description}`
- コミット前に `npm run lint && npm test` を必ず実行

## アーキテクチャ制約
- ビジネスロジックはドメイン層に配置（`src/domain/`）
- 外部サービスとの通信はアダプタ層に隔離（`src/adapters/`）
- 新しい依存ライブラリの追加は事前に人間の承認が必要

## 禁止事項
- `any` 型の使用禁止
- `console.log` の本番コードへの混入禁止
- 既存のAPIコントラクトを破壊する変更の無断実行禁止
- 新しいデザインパターンの無断導入禁止（既存パターンに従う）
```

### 2.3 スキルとフェーズの対応

主要なワークフローフェーズは共有スキルとして実装されている。

| 設計書のフェーズ | 対応スキル | 入力 | 出力 |
|---|---|---|---|
| ドメインSpec作成（イベント・集約） | `event-storming` | ドメイン名 | `docs/{domain}/event-storming.md`, `docs/big-picture.md` |
| ドメインSpec作成（型・ワークフロー） | `domain-modeling` | ドメイン名 | `docs/{domain}/domain-modeling.md` |
| ① 要件定義 | `requirements-writer` | feature名 | `docs/{domain}/specs/{feature}/requirements.md` |
| ②③ 技術設計+タスク分解 | `plan-issue` | Issue番号 | `docs/plans/issue-{番号}.md` |
| ④⑤⑥ 実装+レビュー+PR | `dev-loop` | Issue番号 or 計画ファイル | 実装コード + テスト + PR |

`dev-loop` は内部に以下のサブエージェントを持つ:
- **test-writer**: テストを先に作成（実装バイアスの排除）
- **code-reviewer**: 実装とは別コンテキストでセルフレビュー（セキュリティ、テスト十分性、規約準拠等を統合的に検証）

プロジェクト固有の追加エージェントが必要な場合は `.claude/agents/` に配置する。

## 3. Specの2層構造とfeatureの粒度

### 3.1 なぜ2層構造が必要か

featureをユーザーストーリー単位で切り出してAIに実装させると、AIは各featureの局所的な情報だけで設計判断を下す。たとえば契約管理ドメインで「契約を新規作成できる」だけを渡すと、AIはその時点で必要な構造だけでモデルを作る。次の「契約を変更できる」で `amendment` の概念が登場し、さらに「契約を解約できる」「自動更新する」とfeatureが進むたびにモデルの拡張・再設計が発生し、最悪の場合、数feature後に「ドメインモデルをゼロから作り直したい」事態に陥る。

これを防ぐために、featureレベルのSpecの「上位」にドメインレベルのSpecを置く。

```
docs/
├── big-picture.md                   # システム全体のイベントフロー（先に作る）
└── contract/                        ← ドメイン単位でまとめる
    ├── event-storming.md            # 契約ドメインの全体像（先に作る）
    ├── domain-modeling.md           # データ型・ワークフロー型定義
    │
    └── specs/                       ← featureレベル（順次作る）
        ├── create-contract/
        ├── amend-contract/
        ├── terminate-contract/
        └── auto-renew-contract/
```

**ドメインSpec** は「ドメイン全体のライフサイクル」を俯瞰する文書で、個別featureの実装に先立って作成する。**featureSpec** は従来通りユーザーストーリー単位で、ドメインSpecを参照しながら書く。

### 3.2 ドメインSpecの作成プロセス

ドメインSpecの作成は人間主導で行う。AIが自律的にドメインの正解を出せる領域ではないため、ドメインエキスパートの知識が不可欠である。

**ステップ1：イベントストーミング（人間主導、AI補助）**

`event-storming` スキルを使い、ドメインの全体像を対話的に洗い出す。

- **Big Picture**（システム全体）: `/event-storming` で起動。ドメインイベントの洗い出し → イベントフロー整理 → バウンデッドコンテキスト発見 → コンテキスト間の依存方向を整理する。出力先: `docs/big-picture.md`
- **Design Level**（単一ドメイン深掘り）: `/event-storming {ドメイン名}` で起動。ドメインイベント → コマンド → アクター・ポリシー → 集約 → 状態遷移を段階的に分析する。出力先: `docs/{domain-name}/event-storming.md`

人間がここで判断すべきことは、状態遷移が現実のビジネスプロセスと合っているか、集約の境界が適切か、将来の拡張の見通しである。

**ステップ2：ドメインモデリング（人間主導、AI補助）**

`domain-modeling` スキルを使い、event-stormingで洗い出した集約・コマンド・イベントを元に、データ型とワークフローを擬似言語で定義する。

- `/domain-modeling {ドメイン名}` で起動。event-stormingの出力（`docs/{domain-name}/event-storming.md`）が存在すれば自動的に入力として活用する
- Simple Types（値オブジェクト・列挙型）→ Complex Types（レコード・直和型）→ ワークフロー型 → エラー型の順に段階的に定義する
- 出力先: `docs/{domain-name}/domain-modeling.md`

人間がここで判断すべきことは、型がドメインの制約を正しく表現しているか、不正な状態が型レベルで排除されているかである。

**ステップ3：featureの切り出し（ドメインSpecを参照しながら）**

ドメインSpecが存在する状態でfeatureを切ると、各featureのSpecに「このfeatureで実装するスコープ」と「実装しないが考慮済みのもの」を明記できる。

```markdown
# Feature: 契約の新規作成

## 参照
- ドメインSpec: docs/contract/event-storming.md
- 型定義: docs/contract/domain-modeling.md

## このfeatureで実装するスコープ
- 状態遷移: [*] → Draft → PendingApproval → Active
- 集約: Contract（の基本属性のみ）
- API: POST /contracts, PATCH /contracts/:id/submit,
       PATCH /contracts/:id/approve

## このfeatureでは実装しないが、ドメインSpecで考慮済みのもの
- Amendment（集約として定義済みだがロジックは後続feature）
- 自動更新（状態遷移として定義済みだが後続feature）
- 解約（状態遷移として定義済みだが後続feature）
```

「このfeatureでは実装しないがドメインSpecで考慮済み」のセクションにより、AIは将来の拡張の見通しを理解した上でコードを書ける。

### 3.3 ドメインSpecの運用方針

ドメインSpecは「完璧な事前設計」ではなく「70%の見通し」を目指す。ウォーターフォール的な全量設計とは明確に異なり、featureの実装を通じて発見した知見はドメインSpecにフィードバックする。作成には半日〜1日を要するが、3つ目以降のfeatureでの手戻り回避を考えると十分に投資対効果が高い。

### 3.4 featureの粒度ガイドライン

featureの粒度はユーザーストーリーに近い。判断基準は「1つのPRとして、レビュアーが30分以内に意図を理解してレビューできるか」であり、PRの差分が300〜500行程度に収まるサイズが目安である。

**適切な粒度の例（タスク4〜6個、PR差分300〜500行）：**

| featureの例 | タスク数目安 |
|------------|-----------|
| パスワードリセット機能 | 4〜5 |
| 一覧画面に価格帯フィルターを追加 | 3〜4 |
| 注文履歴のCSVエクスポート | 3〜4 |
| チームメンバーの招待機能 | 5〜6 |
| Webhook設定画面 | 5〜7 |

**大きすぎる例（分割すべき）：**

「ユーザー認証システムの構築」は1つのfeatureではなく、以下のように分割する：

| 分割後のfeature | タスク数目安 |
|---------------|-----------|
| メール/パスワードによるサインアップ | 4〜5 |
| ログイン/ログアウト + セッション管理 | 3〜4 |
| パスワードリセット | 4〜5 |
| OAuth連携（Google/GitHub） | 各3〜4 |
| 二要素認証 | 4〜5 |

**小さすぎる例（まとめるべき）：**

「ユーザーモデルにカラムを1つ追加」「APIレスポンスの日付フォーマット統一」「ボタンの色変更」などは、単独のfeatureとするオーバーヘッドのほうが大きい。関連する変更とまとめるか、軽量プロセスで処理する。

**判断チェックリスト：**

- 受け入れ基準が3〜7個で書けるか（少なすぎ＝小さすぎ、多すぎ＝大きすぎ）
- 差分が500行を大幅に超えないか（AIレビューの精度にも影響する）
- マージした時点で何らかの価値が完結しているか
- AIが1セッションで全体像を把握できるか

### 3.5 変更パターン別のSpec運用フロー

すべての変更にフルプロセスを適用するとオーバーヘッドが過大になる。変更の性質に応じて、適用するプロセスのレベルを使い分ける。

```
変更の要求が来た
    │
    ├─ ドメインの概念（状態遷移、エンティティ構造）が変わる？
    │     │
    │     YES → パターンC: ドメインSpec更新 → featureSpec作成/更新 → 実装
    │
    ├─ 既存featureの振る舞い（受け入れ基準）が変わる？
    │     │
    │     YES → パターンA: featureSpec更新 → 実装
    │
    └─ 上記いずれでもない（内部的な修正、バグ修正、軽微な調整）
          │
          └─ パターンB: 直接実装（issue/チケットで管理すれば十分）
```

**パターンA：既存featureの拡張（featureSpecを更新する）**

例：契約一覧に「契約金額の範囲」で検索できるフィルターを追加する場合。既存の requirements.md に受け入れ基準を追記し、実装プランを更新してから実装する。ドメインSpec の更新は不要（契約のライフサイクルやエンティティ構造は変わらないため）。

Spec更新時は変更履歴をドキュメント内に残す：

```markdown
## 変更履歴
- v1.0: 初回リリース（契約名、ステータスで検索）
- v1.1: 契約金額の範囲フィルターを追加 ← 今回
```

**パターンB：軽微な修正（Spec更新不要）**

例：一覧のデフォルトソート順変更、バグ修正、リファクタリングなど。受け入れ基準が変わらない変更にはSpecプロセスを適用しない。既存テストの期待値修正と実装変更をコミットすれば十分。

判断基準：受け入れ基準が変わるならSpecを更新、内部挙動だけが変わるならSpec不要。

**パターンC：ドメインモデルに影響する変更（ドメインSpecも更新する）**

例：契約に多段階の承認ワークフローを追加する場合。状態遷移が変わり、新しい集約（Approval）が追加される。

更新の順序が重要：ドメインSpec更新（`event-storming` スキルの再実行、人間が承認） → featureSpec作成/更新 → 実装。

**日常の開発では大半がパターンAとBであり、パターンCは月に数回程度。** Specの運用が重くなりすぎないためには「受け入れ基準が変わるときだけSpecを更新する」くらいのルールが現実的である。

## 4. 各フェーズの詳細設計

### フェーズ①：要件定義

| 項目 | 内容 |
|------|------|
| 前提 | ドメインSpec（`docs/{domain}/`）が作成・承認済みであること |
| 実行者 | AI（`requirements-writer` スキル） |
| 入力 | 機能の概要説明（人間が口頭/テキストで提供）+ ドメインSpec |
| 出力 | `docs/{domain}/specs/{feature}/requirements.md` |
| ゲート | 人間が要件の正しさ・完全性を承認 |

**スキルによる実行:**

`/requirements-writer {feature名}` で起動。以下を自動で実施する:

1. ユーザーから機能の概要を聞き取る
2. 該当ドメインのドメインSpec（`docs/{domain}/event-storming.md`）を読み、全体像を把握する
3. 既存のコードベースを探索し、関連するモジュールを特定する
4. 要件定義書を作成する
   出力先: `docs/{domain}/specs/{feature}/requirements.md`
   - ドメインSpecで定義された状態遷移・集約のうち、このfeatureでカバーする範囲を明示すること
   - このfeatureではスコープ外だが、ドメインSpecで考慮済みの項目も明記すること
5. 要件定義書の内容をサマリーとして表示し、人間にレビューを依頼する

**AIへの指示のポイント：**

- 「まだコードは書くな」を明示する
- 受け入れ基準は具体的なGiven/When/Then形式で出力させる
- 曖昧な点は仮定を置かず質問させる（`ask_user_question`ツールの活用）
- エッジケースを網羅的に列挙させる

**人間のレビュー観点：**

- ビジネス要件との整合性
- 受け入れ基準の網羅性と具体性
- スコープが適切か（大きすぎないか）
- セキュリティ・パフォーマンス要件の妥当性


### フェーズ②③：技術設計＋タスク分解

| 項目 | 内容 |
|------|------|
| 実行者 | AI（`plan-issue` スキル） |
| 入力 | Issue + ドメインSpec + 既存コードベース |
| 出力 | `docs/plans/issue-{番号}.md`（技術設計とタスク分解を1ファイルに統合） |
| ゲート | 人間がアーキテクチャとの整合性・タスク粒度を承認 |

**スキルによる実行:**

`/plan-issue {Issue番号}` で起動。以下を自動で実施する:

1. Issue本文とドメインSpec（`docs/{domain}/event-storming.md`）を読み込む
2. 既存のコードベースを探索し、関連するモジュールを特定する
3. 技術設計（変更対象ファイル、データフロー、APIエンドポイント、テスト戦略等）を作成する
4. タスク分解（タスク順序、依存関係、完了条件）を行う
5. セルフレビューと改善ループを回した上で、人間にレビューを依頼する

**人間のレビュー観点：**

- 既存アーキテクチャとの一貫性
- ドメインSpecで定義された集約・状態遷移に従っているか
- テスト戦略の妥当性
- タスク粒度・順序・依存関係の適切さ
- 影響範囲の見積もりの正確性


### フェーズ④⑤⑥：実装＋セルフレビュー＋PR作成

| 項目 | 内容 |
|------|------|
| 実行者 | AI（`dev-loop` スキル） |
| 入力 | Issue番号 or 承認済みの実装プラン（`docs/plans/issue-{番号}.md`） |
| 出力 | 実装コード + テストコード + GitHub PR |
| ゲート | 検証ゲート（型チェック + lint + テスト全パス）→ セルフレビュー → PR作成 |

**スキルによる実行:**

`/dev-loop {Issue番号}` で起動。以下のフェーズを自動で実施する:

```
Phase 0: インプットの理解
  Issue本文・実装プランの読み込み
  ↓
Phase 1a: テスト生成
  test-writerサブエージェントがテストを先に作成
  ↓
Phase 1b: 実装
  テストを通すことを目標に実装
  ↓
Phase 1c: 検証ゲート
  型チェック + lint + テスト全パス確認
  ├─ FAIL → Phase 1bに戻る
  └─ PASS ↓
Phase 2: セルフレビュー
  code-reviewerサブエージェント（実装とは別コンテキスト）
  ↓
Phase 3: 判定
  ├─ 指摘なし → Phase 4
  ├─ 指摘あり・3周目未満 → Phase 1bに戻る
  └─ 3周目 → Phase 4（未解消ありとして報告）
  ↓
Phase 4: PR作成 + 完了報告
```

**設計のポイント：**

- **テスト先行**: test-writerと実装者を別エージェントにすることで、テストが「仕様のドキュメント」として機能する
- **検証ゲート**: セルフレビューに入る前に機械的な品質チェックを通す
- **コンテキスト分離**: code-reviewerは実装とは別コンテキストでレビューし、バイアスを排除する
- **エスカレーション**: 判断ポイントが発生した場合、Issueにコメントを投稿してセッションを終了する


### フェーズ⑦：自動AIレビュー（CI連動）

| 項目 | 内容 |
|------|------|
| 実行者 | AIレビューボット（CodeRabbit / PR-Agent等） |
| 入力 | PRの差分 |
| 出力 | PR上のインラインコメント + サマリー |
| ゲート | 自動（重大な問題があればブロック） |

**GitHub Actions設定例（`.github/workflows/ai-review.yml`）：**

```yaml
name: AI Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  # 1. 標準的なCI（リント、型チェック、テスト）
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck
      - run: npm test -- --coverage

  # 2. AIによる自動レビュー（例: PR-Agent）
  ai-review:
    runs-on: ubuntu-latest
    needs: ci  # CIが通ってからレビュー
    steps:
      - uses: Codium-ai/pr-agent@main
        env:
          OPENAI_KEY: ${{ secrets.OPENAI_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

  # 3. Claude Codeによる深層レビュー（オプション）
  deep-review:
    runs-on: ubuntu-latest
    needs: ci
    steps:
      - uses: actions/checkout@v4
      - name: Security Review
        run: |
          claude -p "このPRの差分を読んで、セキュリティの観点で
          レビューしてください。問題があればGitHub上にコメントしてください。" \
          --allowedTools Read,Grep,Glob,Bash
```

**AIレビューの役割分担：**

| レビュー観点 | 担当 | 検出方法 |
|------------|------|---------|
| 構文エラー・型エラー | CI（自動） | TypeScript compiler, ESLint |
| コーディング規約違反 | CI（自動） | ESLint, Prettier |
| 既知の脆弱性パターン | AIボット | SAST + LLMパターンマッチ |
| テストカバレッジ不足 | AIボット | カバレッジレポート + LLM分析 |
| 命名の一貫性 | AIボット | コードベース全体との比較 |
| ロジックの正しさ | AIボット + 人間 | LLM推論 + 人間の判断 |
| 仕様との整合性 | 人間 | 要件定義書との突合 |
| アーキテクチャ適合性 | 人間 | 設計原則との照合 |
| ビジネス価値の判断 | 人間 | ドメイン知識 |


### フェーズ⑧：人間レビュー

| 項目 | 内容 |
|------|------|
| 実行者 | 人間（チームメンバー） |
| 入力 | AIレビュー済みのPR + AIサマリー |
| 出力 | Approve / Request Changes |
| ゲート | 最終的なマージ判断 |

**人間がレビューすべき観点（AIに任せられないもの）：**

- この変更は本当に必要か？（要件の妥当性）
- チームの暗黙知と整合しているか？
- 将来の拡張性に問題はないか？
- ユーザー体験への影響は？
- 他のチームへの影響は？

**AIサマリーの活用法：**
AIレビューボットが生成するPRサマリーとウォークスルーを起点に読むことで、大きな差分でも変更の意図と影響範囲を素早く把握できる。人間はAIが「OK」と判断した部分は軽く確認し、AIが指摘した部分と、AIが判断できない高次の観点に集中する。


### フェーズ⑨：マージ

CIがすべて通り、AIレビューに重大な指摘がなく、人間レビュアーがApproveしたらマージ可能。

## 6. テスト戦略の詳細

### 5.1 テストレベル別の設計方針

**ユニットテスト（AIが主導）：**
- ドメインロジックの全パスをカバー
- 正常系・異常系・境界値を網羅
- プロパティベーステストで入力空間を広く探索
- テスト名は振る舞いを明確に表現

**統合テスト（AI+人間の協働）：**
- Testcontainersで実際のDBを使用
- APIエンドポイントのリクエスト/レスポンスを検証
- スナップショットテストでAPIコントラクトの回帰を検出
- エラーハンドリングの統合的な動作を確認

**テストの品質検証（AIが主導）：**
- ミューテーションテストでテストの実効性を測定
- ビジネスクリティカルなモジュールはミューテーションスコア90%以上を要求
- ユーティリティ系は70%以上を許容

### 5.2 AIにテストを書かせるプロンプトのテンプレート

```markdown
以下の観点で {対象モジュール} のテストを書いてください。

## ユニットテスト
- 正常系: {具体的なシナリオを列挙}
- 異常系: {具体的なシナリオを列挙}
- 境界値: {具体的な境界条件を列挙}

## 統合テスト
- {外部依存との連携シナリオを列挙}

## テストの書き方ルール
- 既存テストのパターン（命名、ヘルパー関数）に従う
- Arrange-Act-Assert パターンを使う
- 各テストは独立して実行可能にする
- モックは外部依存にのみ使用する
- テスト名は「{操作}した場合に{期待結果}となる」の形式
```

## 7. 導入ステップ

### Step 1: 基盤を整える（1〜2日）

- CLAUDE.md を作成し、プロジェクトの規約を記述
- プロジェクトに必要なスキル（`event-storming`, `dev-loop`, `plan-issue` 等）をセットアップする
- 主要ドメインのドメインSpec（`docs/{domain}/event-storming.md`）を `event-storming` スキルで作成する（人間主導、AI補助）

### Step 2: 小さく試す（1〜2週間）

- 小規模な機能（1〜3タスク程度）で全フェーズを一度通す
- 各フェーズゲートでの人間のレビュー負荷を計測
- CLAUDE.md にプロジェクト固有のルールを追記していく

### Step 3: CI連動を追加（1週間）

- GitHub ActionsにAIレビューボット（CodeRabbitまたはPR-Agent）を追加
- 自動テスト・lint・型チェックのパイプラインを整備
- PRテンプレートを整備

### Step 4: チームに展開（2〜4週間）

- チームメンバーにワークフローを共有
- フェーズゲートでのレビュー基準をチームで合意
- 定期的にCLAUDE.mdを更新し、学びを蓄積

### Step 5: 計測と改善（継続的）

- PR作成からマージまでのリードタイム
- AIレビューの指摘のうち、人間が「有用」と判断した割合
- マージ後の本番バグ発生率
- テストのミューテーションスコアの推移

## 8. アンチパターンと対策

| アンチパターン | なぜ危険か | 対策 |
|-------------|----------|------|
| ドメインSpecなしでfeatureを積み上げる | データモデルが断片的になり、数feature後に破壊的な作り直しが発生 | featureの実装に先立ってドメインSpecを作成し、全体像を確保する |
| ドメインSpecで完璧を目指す | ウォーターフォール化し、着手が遅れる | 「70%の見通し」で十分。残り30%はfeature実装を通じて発見・反映する |
| すべての変更にフルSpecプロセスを適用 | 軽微な修正でもオーバーヘッドが発生し、チームが疲弊する | パターンA/B/Cの判断フローに従い、受け入れ基準が変わるときだけSpecを更新 |
| 仕様なしでいきなり実装させる | AIの解釈が要件とずれ、大規模な手戻りが発生 | 必ずフェーズ①〜②③を経てから実装に入る |
| 巨大なタスクを1つのセッションで | コンテキスト枯渇で品質が劣化 | 1タスク＝1コミットの粒度でサブエージェントに委任 |
| featureの粒度が大きすぎる | PRが肥大化しレビュー品質が低下、AIレビューの精度も落ちる | 受け入れ基準3〜7個、差分300〜500行を目安に分割 |
| AIレビューだけで人間レビューを省略 | ビジネスロジックの誤りやアーキテクチャ劣化を見逃す | AIレビューは「フィルター」、人間レビューは「判断」 |
| テスト生成をAIに丸投げ | ハッピーパスのみのテストになりがち | テスト観点（異常系、境界値）を明示的に指示 |
| CLAUDE.mdを更新しない | 同じ間違いが繰り返される | レビューで発見したパターンを都度追記 |
| AIの出力を無条件に信頼 | 微妙なロジックエラーが本番に混入 | セルフレビュー＋人間レビューの二段構え |
