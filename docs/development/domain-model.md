# 開発コンテキスト ドメインモデル

開発コンテキストのDesign Level（`docs/development/event-storming.md`）を、関数型ドメインモデルとして形式化したもの。記法は `docs/references/domain-model-notation.md` に従う。

## スコープ

- **対象**: 実装プラン作成開始からPRマージまで
- **対応するDL**: `docs/development/event-storming.md`
- **集約**: 実装プラン、実装、PRレビュー

## 概観（AND / OR 表記）

```
開発フロー = 実装プラン作成 AND 実装 AND PRレビュー

実装プラン状態   = 作成中 OR エスカレーション中 OR 確定
実装状態        = タスク消化中 OR 検証中 OR 完了
PRレビュー状態  = PR作成済み OR レビュー中 OR 変更要求対応中 OR マージ可能 OR マージ済み

// 集約間の主なイベントの流れ
実装プラン --[実装プランが確定した]--> 実装
実装       --[検証ゲートを通過した]--> PRレビュー
PRレビュー --[変更が要求された]-----> 実装プラン（P2 新規作成）
```

## 共通値オブジェクト

複数の集約で参照される型。

```fsharp
// 日時
type 日時 = 日時 of string  // ISO 8601

// チケット（バックログ管理由来）
type チケットID = チケットID of string

type チケット種別 =
  | ユーザーストーリー
  | バグ
  | タスク

type チケット = {
  ID: チケットID
  件名: string
  種別: チケット種別
}

// 開発者（実装プラン確定、PR作成・マージ等で参照される）
type 開発者ID = 開発者ID of string

type 開発者 = {
  ID: 開発者ID
  名前: string
}
```

## 実装プラン集約

### 責務

技術設計・タスク分解の管理。チケット入力から確定プラン出力までを担う。エスカレーションによる中断・再開と、変更要求起因の新規作成を含む。

### 固有値オブジェクト

```fsharp
// 識別子
type プランID = プランID of string
type タスクID = タスクID of string

// テキスト値
type 質問内容 = 質問内容 of string
type 回答内容 = 回答内容 of string
type 技術方針内容 = 技術方針内容 of string

// タスク（サブエンティティ）
type TDDフェーズ =
  | Red       // テスト先行作成
  | Green     // プロダクションコード実装
  | Refactor  // リファクタリング

type タスク = {
  ID: タスクID
  内容: string
  順序: int
  TDDフェーズ: TDDフェーズ
}
```

### 入力イベント（他集約から）

| イベント | 発火元 | 役割 |
|---|---|---|
| `外部_トリアージされた` | 外部（バックログ管理） | チケットを受け取って実装プラン作成を開始する |
| `変更が要求された` | PRレビュー集約 | P2ポリシーのトリガー（新規実装プラン作成） |
| `外部_仕様の回答が返された` | 外部（企画） | P1ポリシーのトリガー（エスカレーション中→作成中） |
| `外部_PRDが修正された` | 外部（企画） | 仕様変更による再計画の契機 |

### 発火するイベント

```fsharp
type 実装プランが確定した = {
  プランID: プランID
  チケットID: チケットID
  確定者: 開発者ID
  確定日時: 日時
}

type 仕様の不明点がエスカレーションされた = {
  プランID: プランID
  チケットID: チケットID
  質問: 質問内容
  エスカレーション日時: 日時
}

type 仕様の回答が届いた = {
  プランID: プランID
  チケットID: チケットID
  回答: 回答内容
  回答日時: 日時
}

type 横断的な技術方針が決定された = {
  方針内容: 技術方針内容
  決定者: 開発者ID  // テックリードも開発者として扱う
  決定日時: 日時
}
```

### 状態型

```fsharp
type 作成中プラン = {
  ID: プランID
  チケットID: チケットID
  タスク: List<タスク>
  作成者: 開発者ID
}

type エスカレーション中プラン = {
  ID: プランID
  チケットID: チケットID
  タスク: List<タスク>
  作成者: 開発者ID
  質問: 質問内容
  エスカレーション日時: 日時
}

type 確定プラン = {
  ID: プランID
  チケットID: チケットID
  タスク: List<タスク>
  作成者: 開発者ID
  確定日時: 日時
}

type 実装プラン =
  | 作成中 of 作成中プラン
  | エスカレーション中 of エスカレーション中プラン
  | 確定 of 確定プラン
```

### コマンド

```fsharp
// 失敗理由
type プラン作成失敗理由 =
  | 仕様不明点あり of 質問内容
  | 技術方針未決 of 相談事項: string

// コマンドシグネチャ
type 実装プランを作成する =
  チケット -> 開発者 -> Result<実装プランが確定した, プラン作成失敗理由>

type 仕様の不明点をエスカレーションする =
  作成中プラン -> 質問内容 -> 仕様の不明点がエスカレーションされた

type 横断的な技術方針を決定する =
  技術方針内容 -> 開発者 -> 横断的な技術方針が決定された
```

### 状態遷移関数

```fsharp
type エスカレーションする =
  作成中プラン -> 質問内容 -> エスカレーション中プラン

type 回答を受け取る =
  エスカレーション中プラン -> 回答内容 -> 作成中プラン

type 確定する =
  作成中プラン -> 開発者 -> Result<確定プラン, プラン作成失敗理由>
```

### ポリシー

```fsharp
// P1: プラン再開ポリシー
// トリガー: 外部_仕様の回答が返された（企画から）
// アクション: エスカレーション中プランを作成中に戻す
type プラン再開ポリシー =
  外部_仕様の回答が返された -> 回答を受け取る

// P2: 変更要求対応ポリシー
// トリガー: 変更が要求された（PRレビュー集約）
// アクション: 新規の実装プランインスタンスを作成する（巻き戻しではない）
type 変更要求対応ポリシー =
  変更が要求された -> 実装プランを作成する
```

### 型で表せなかった不変条件

| # | 不変条件 | 担保方法 |
|---|---|---|
| 1 | 実装プランの確定判断は開発者のみ | コマンド `確定する` の入力に `開発者` を要求 |
| 2 | 変更要求時は新規プランインスタンスを作成（巻き戻しでない） | ポリシー `変更要求対応ポリシー` が `実装プランを作成する` を呼ぶ設計。型だけでは強制不可 |

## 実装集約

### 責務

TDDサイクルによるコード変更の管理。タスク消化と検証ゲート実行を担う。

### 固有値オブジェクト

```fsharp
type 実行コンテキスト =
  | ローカル
  | CI

type 失敗種別 =
  | 型エラー
  | Lint違反
  | テスト失敗
```

### 入力イベント（他集約から）

| イベント | 発火元 | 役割 |
|---|---|---|
| `実装プランが確定した` | 実装プラン集約 | 実装集約のインスタンスを生成するトリガー |

### 発火するイベント

```fsharp
type 検証ゲートを通過した = {
  プランID: プランID
  通過日時: 日時
  実行コンテキスト: 実行コンテキスト
}

type 検証ゲートが失敗した = {
  プランID: プランID
  失敗日時: 日時
  実行コンテキスト: 実行コンテキスト
  失敗種別: 失敗種別
}
```

### 状態型

```fsharp
type タスク消化中状態 = {
  プランID: プランID
  タスク進捗: List<タスク>
}

type 検証中状態 = {
  プランID: プランID
  実行コンテキスト: 実行コンテキスト
  開始日時: 日時
}

type 完了状態 = {
  プランID: プランID
  通過日時: 日時
}

type 実装 =
  | タスク消化中 of タスク消化中状態
  | 検証中 of 検証中状態
  | 完了 of 完了状態
```

### コマンド

```fsharp
// 検証ゲート実行は成功または失敗のいずれかのイベントを発火
type 検証ゲートを実行する =
  検証中状態 -> Result<検証ゲートを通過した, 検証ゲートが失敗した>
```

### 状態遷移関数

```fsharp
// 確定プランから実装を開始
type 実装を開始する =
  確定プラン -> タスク消化中状態

// 全タスク完了で検証に移る
type 検証を開始する =
  タスク消化中状態 -> 実行コンテキスト -> 検証中状態

// 検証通過で完了
type 完了させる =
  検証中状態 -> 検証ゲートを通過した -> 完了状態

// 検証失敗でタスク消化中に戻る
type 検証失敗で巻き戻す =
  検証中状態 -> 検証ゲートが失敗した -> タスク消化中状態

// タスク内部遷移（集約の状態遷移ではない）
type タスクをRedに進める = タスク -> タスク
type タスクをGreenに進める = タスク -> タスク
type タスクをRefactorに進める = タスク -> タスク
```

### ポリシー

（なし。実装集約は外部ポリシーを持たない）

### 型で表せなかった不変条件

| # | 不変条件 | 担保方法 |
|---|---|---|
| 1 | 検証ゲートは全タスク完了後のみ実行 | `検証を開始する` の入力を `タスク消化中状態` に限定。タスク進捗完了の判定は関数内部で実施 |
| 2 | CI実行の検証ゲートは自動トリガー | 外部システム（CI）の責務。ドメインモデルは `実行コンテキスト` で識別するのみ |

## PRレビュー集約

### 責務

レビュー・マージのゲート管理。PR作成から承認・マージまでを担う。

### 固有値オブジェクト

```fsharp
type PRID = PRID of string
type レビュアーID = レビュアーID of string

type レビュアー = {
  ID: レビュアーID
  名前: string
}

type 変更要求コメント = 変更要求コメント of string

// 制約付き値
type 必要Approve数 = 必要Approve数 of int
type 必要Approve数を作る = int -> Result<必要Approve数, string>
// ルール: 入力が1以上なら成功、0以下なら "必要Approve数は1以上" を返す
```

### 入力イベント（他集約から）

| イベント | 発火元 | 役割 |
|---|---|---|
| `実装プランが確定した` | 実装プラン集約 | PR作成の前提となる確定プランの生成を示す |
| `検証ゲートを通過した` | 実装集約 | レビュー依頼の事前条件 |

### 発火するイベント

```fsharp
type PRが作成された = {
  PRID: PRID
  プランID: プランID
  作成者: 開発者ID
  作成日時: 日時
}

type レビュー依頼がされた = {
  PRID: PRID
  依頼者: 開発者ID
  依頼先: List<レビュアーID>
  必要Approve数: 必要Approve数
  依頼日時: 日時
}

type 変更が要求された = {
  PRID: PRID
  要求者: レビュアーID
  コメント: 変更要求コメント
  要求日時: 日時
}

type レビューが承認された = {
  PRID: PRID
  承認者: レビュアーID
  承認日時: 日時
}

type 必要Approve数に達した = {
  PRID: PRID
  到達日時: 日時
  承認者: List<レビュアーID>
}

type PRがマージされた = {
  PRID: PRID
  プランID: プランID
  マージ者: 開発者ID
  マージ日時: 日時
}
```

### 状態型

```fsharp
type PR作成済み状態 = {
  PRID: PRID
  プランID: プランID
  作成者: 開発者ID
}

type レビュー中状態 = {
  PRID: PRID
  プランID: プランID
  必要Approve数: 必要Approve数
  承認済みレビュアー: List<レビュアーID>
  依頼先: List<レビュアーID>
}

type 変更要求対応中状態 = {
  PRID: PRID
  プランID: プランID
  変更要求: 変更が要求された
}

type マージ可能状態 = {
  PRID: PRID
  プランID: プランID
  承認済みレビュアー: List<レビュアーID>
}

type マージ済み状態 = {
  PRID: PRID
  プランID: プランID
  マージ日時: 日時
}

type PRレビュー =
  | PR作成済み of PR作成済み状態
  | レビュー中 of レビュー中状態
  | 変更要求対応中 of 変更要求対応中状態
  | マージ可能 of マージ可能状態
  | マージ済み of マージ済み状態

// 事前条件を型で表すエイリアス
type 検証ゲート通過済みPR = PR作成済み状態
  // 生成には「検証ゲートを通過した」イベントの受信が前提
type レビュー中PR = レビュー中状態
type マージ可能PR = マージ可能状態
```

### コマンド

```fsharp
// 失敗理由
type マージ失敗理由 =
  | Approve数不足
  | コンフリクトあり

// コマンドシグネチャ
type PRを作成する =
  確定プラン -> 開発者 -> PRが作成された

type レビュー依頼をする =
  検証ゲート通過済みPR -> List<レビュアー> -> 必要Approve数 -> レビュー依頼がされた

type 変更を要求する =
  レビュー中PR -> レビュアー -> 変更要求コメント -> 変更が要求された

type レビューを承認する =
  レビュー中PR -> レビュアー -> レビューが承認された

type PRをマージする =
  マージ可能PR -> 開発者 -> Result<PRがマージされた, マージ失敗理由>
```

### 状態遷移関数

```fsharp
// PR作成 → 検証ゲート通過 → レビュー依頼 → レビュー中
type レビュー依頼で遷移する =
  検証ゲート通過済みPR -> List<レビュアー> -> 必要Approve数 -> レビュー中状態

// 変更要求を受ける
type 変更要求で遷移する =
  レビュー中状態 -> レビュアー -> 変更要求コメント -> 変更要求対応中状態

// 変更要求対応後の再依頼
type 再レビュー依頼する =
  変更要求対応中状態 -> List<レビュアー> -> 必要Approve数 -> レビュー中状態

// 個別承認の蓄積（状態内遷移）
type 承認を蓄積する =
  レビュー中状態 -> レビュアー -> レビュー中状態

// 必要Approve数到達でマージ可能へ
type 必要Approve数に到達したか判定する =
  レビュー中状態 -> Option<マージ可能状態>

// マージ
type マージする =
  マージ可能状態 -> 開発者 -> Result<マージ済み状態, マージ失敗理由>
```

### ポリシー

（なし。このコンテキスト内での自動トリガーは持たない）

### 型で表せなかった不変条件

| # | 不変条件 | 担保方法 |
|---|---|---|
| 1 | レビュー依頼は検証ゲート通過後のみ | 型 `検証ゲート通過済みPR` で表現済み |
| 2 | 必要Approve数は1以上 | `必要Approve数を作る` 生成関数で検査 |
| 3 | 必要Approve数到達時のみマージ可能 | `マージ可能状態` への遷移は `必要Approve数に到達したか判定する` を通る |
| 4 | 同一レビュアーの二重承認を防ぐ | `承認を蓄積する` 実行時に重複チェック。型では表現困難 |

## ワークフロー

主要ユースケースを関数合成として示す。

### UC1: チケットからPRマージまでの正常系

```fsharp
type チケットからマージまで =
  チケット
    -> 実装プランが確定した
    -> 検証ゲートを通過した
    -> PRが作成された
    -> レビュー依頼がされた
    -> 必要Approve数に達した
    -> PRがマージされた
```

### UC2: エスカレーション→再開フロー

```fsharp
type エスカレーションと再開 =
  作成中プラン
    -> 仕様の不明点がエスカレーションされた
    -> 外部_仕様の回答を返す           // 企画コンテキスト
    -> 外部_仕様の回答が返された
    -> 実装プランが確定した
```

### UC3: 変更要求→再プラン作成フロー

```fsharp
type 変更要求と再作成 =
  レビュー中状態
    -> 変更が要求された
    -> 実装プランが確定した             // 新規インスタンス（P2ポリシー）
    -> 検証ゲートを通過した
    -> レビュー依頼がされた
    -> PRがマージされた
```

## コンテキスト境界

### 外部イベント（他コンテキスト由来）

```fsharp
// バックログ管理コンテキスト由来
type 外部_トリアージされた = {
  チケットID: チケットID
  優先度: string
  担当者: 開発者ID
  トリアージ日時: 日時
}

// 企画コンテキスト由来
type 外部_PRDが修正された = {
  PRDID: string
  関連チケットID: チケットID
  修正内容: string
  修正日時: 日時
}

type 外部_仕様の回答が返された = {
  プランID: プランID
  回答: 回答内容
  回答者: string  // 企画担当
  回答日時: 日時
}
```

### 外部コマンド（他コンテキストへの依頼）

```fsharp
// 企画への仕様問い合わせ
type 外部_仕様の回答を返す =
  質問内容 -> 外部_仕様の回答が返された

// リリースコンテキストへの引き渡し
type 外部_ビルド成果物公開を依頼する =
  PRがマージされた -> unit  // 後続はリリースコンテキストの責務
```

### 開始イベント

開発コンテキストは `外部_トリアージされた` を入力として実装プラン作成を開始する。

```fsharp
type 開発を開始する =
  外部_トリアージされた -> チケット -> 実装プランを作成する
```

## モデル化で露出した論点

型定義を試みる過程で明らかになった、DL文書では曖昧だった論点を記録する。

| # | 論点 | 影響 | 対応案 |
|---|---|---|---|
| M1 | **必要Approve数の決定タイミング** がDLに明記されていない。レビュー依頼時にレビュイーが都度決定するのか、チケット種別ごとに既定値があるのか | `レビュー依頼をする` の入力に `必要Approve数` を要求したが、生成元が不明確 | DLに「必要Approve数はレビュー依頼時点でレビュイーが決定する」と明記するか、集約の初期化タイミングを明示する |
| M2 | **実装集約の開始トリガー** がDLで明示されていない。「実装プランが確定した」をトリガーとして実装集約が生成される前提でモデル化したが、これが正しいか確認が必要 | `実装を開始する : 確定プラン -> タスク消化中状態` を追加 | DLに状態遷移「プラン確定 → 実装開始」を明示するか、起動条件を記述 |
| M3 | **エスカレーション時のタスク保持** がDL文書では明示されていない。エスカレーション中もタスク一覧を保持する前提でモデル化したが、明示されていない | `エスカレーション中プラン` にタスクを含めた | DLに「エスカレーション中もプランの作業中データは保持される」と明記 |
| M4 | **変更要求時の新規プラン作成** において、旧プランの扱い（破棄 / 履歴保持 / 参照関係）が未定義 | 型では新旧プランの関係性を表現できなかった | DLに「旧プランは履歴として参照可能か、破棄するか」を明記 |
| M5 | **複数PRの並行レビュー** のモデルが未検討。1つの実装プランから複数PRが作成されるケースの型化 | `PRを作成する : 確定プラン -> ...` は1:1を想定している | 1プラン:NPRの関係があるか、DLで明確化 |
| M6 | **CI失敗時のリトライフロー** が集約の状態遷移に含まれていない。失敗後に修正して再実行する際のインスタンス同一性 | `検証失敗で巻き戻す : 検証中状態 -> タスク消化中状態` でインスタンスを巻き戻す前提 | 同一インスタンス巻き戻しでよいか、変更要求対応（新規作成）と挙動が異なる理由を明記 |
| M7 | **技術方針イベントの集約所属** が曖昧。「横断的な技術方針が決定された」は実装プラン集約に属すとされているが、複数プランに影響する横断イベント | 現状は実装プラン集約のイベントとして扱ったが、集約のライフサイクルと合わない | 独立集約（例: 技術方針リポジトリ）として切り出すか、参照関係のみ持つか検討 |
