# 依存パッケージ更新 影響分析パイプライン

指定パッケージの更新について、互換性の問題を特定し、ボトルネックがあればアップグレードパスを構築し、コード影響を分析する。

## 引数

- `$ARGUMENTS`: パッケージ名 または パッケージ名@バージョン
  - `react` — 最新バージョンへの更新を分析
  - `vite@8.0.0` — 特定バージョンへの更新を分析

PR番号は引数として受け付けない。ユーザーがPRの文脈を添えて依頼した場合は、PRから対象パッケージと目標バージョンを特定して通常フローに合流する。PR番号だけが渡された場合はユーザーに意図を確認する。

## 手順

各Phaseには打ち切り条件がある。条件を満たした場合、後続Phaseをスキップできる。

### Phase 1: 環境把握

**解決する問題**: どのパッケージマネージャで作業するか

プロジェクトルートのロックファイルからPMを自動検出し、対応するreferenceファイルを読み込む。現時点では `references/npm.md`（npm / pnpm / yarn / bun対応）のみ。

検出できない場合はエラーメッセージを出力して終了する。

### Phase 2: 差分把握

**解決する問題**: 現在と目標の差分は何か

1. **引数のパース**: `パッケージ名@バージョン` 形式なら目標バージョンを抽出。パッケージ名のみならレジストリから最新バージョンを取得
2. **現在バージョンの確認**: referenceファイルのコマンドに従いインストール済みバージョンを取得
3. **変更種別の判定**: セマンティックバージョニングに基づき判定
   - **メジャー**: 破壊的変更の可能性 → Phase 3で詳細分析
   - **マイナー**: 新機能追加。通常は互換性あり
   - **パッチ**: バグ修正のみ

> **打ち切り**: パッチ更新の場合、Phase 3〜5をスキップしてPhase 6で簡潔なレポートを出力する。

### Phase 3: 互換性確認

**解決する問題**: 上げたら何が壊れるか

以下の3つの観点で互換性の問題を洗い出す。

#### 3a. Breaking Changes分析

1. レジストリ情報から `homepage` または `repository.url` を取得
2. WebFetchでリリースノート・CHANGELOG・マイグレーションガイドを確認
3. 「BREAKING」「breaking change」「removed」「deprecated」等のキーワードから破壊的変更を抽出
4. 各変更をAPI削除・シグネチャ変更・デフォルト値変更等に分類

> semverが厳密に守られているとは限らないため、マイナー更新でもBreaking Changesの確認を行う。

#### 3b. peerDeps互換性チェック

1. レジストリから更新先バージョンの `peerDependencies` を取得
2. `package.json` の依存バージョンと照合
3. peerDepsの要求範囲を満たさないパッケージを特定

#### 3c. 逆peerDeps特定

1. referenceファイルの「依存元特定」コマンドで逆依存パッケージを取得
2. 逆依存パッケージが更新先バージョンをサポートしているか確認

> **打ち切り**: 互換性に問題がなければPhase 4をスキップする。

### Phase 4: アップグレードパス構築

**解決する問題**: 上げるために何を先に上げるべきか

Phase 3b/3cで依存パッケージの互換性の問題（peerDeps非互換・逆peerDeps影響）が見つかった場合に実行する。Breaking Changes（3a）のみの場合はコード変更で対処するため、このPhaseは不要。PMの解決エンジンを活用してボトルネックを特定し、依存順にソートしたアップグレード手順を構築する。

1. **ボトルネック特定**: referenceファイルの「ボトルネック分析」コマンド（dry-run install等）を実行し、依存解決エラーを取得する
2. **依存順の整理**: エラー情報を基に、先にアップグレードすべきパッケージを特定する
3. **アップグレード手順の構築**: 依存順にソートした手順テーブルを作成する

自前での再帰的peerDeps探索は行わない。PMの出力を基に判断する。ボトルネックが解消可能かどうかもPMの出力から判断する。

### Phase 5: コード影響分析

**解決する問題**: 自分のコードのどこを直すか

Phase 3でBreaking Changesが見つかった場合に実行する。

1. **import/require箇所の検索**: Grepで対象パッケージのimport/require文を検索
2. **使用APIの特定**: import先のモジュール・関数・型を抽出
3. **Breaking Changesとの突合**: Phase 3aで抽出した破壊的変更に該当する使用箇所を特定
4. **影響ファイル一覧の作成**: ファイルパスと該当行、必要な変更の概要を整理

> **打ち切り**: Breaking Changesがない場合は不要。

### Phase 6: 出力

以下の形式で分析結果を提示する。該当しないセクションは省略する。

```markdown
## {パッケージ名}

| 項目 | 結果 |
|---|---|
| 現在バージョン | {current_version} |
| 目標バージョン | {target_version} |
| 変更種別 | メジャー / マイナー / パッチ |
| Breaking Changes | {件数}件 |
| 互換性の問題 | {件数}件 |
| コード影響箇所 | {件数}ファイル |
| 総合リスク | 高 / 中 / 低 |

### Breaking Changes

| 変更内容 | 種類 | 影響箇所 |
|---|---|---|
| {変更の説明} | API削除 / シグネチャ変更 / デフォルト値変更 | {ファイル数}ファイル |

### 互換性の問題

| パッケージ | 種類 | 要求 | 現在 | 状態 |
|---|---|---|---|---|
| {pkg} | peerDep / 逆peerDep | {required} | {current} | 非互換 |

### アップグレードパス

対象: {パッケージ名} {current} → {target}

| 順序 | パッケージ | 現在 | 更新先 | 理由 |
|------|-----------|------|--------|------|
| 1 | {pkg} | {current} | {target} | {理由} |

⚠️ 注意事項:
- {アップグレードに伴う注意点}

### コード影響箇所

| ファイル | 行 | 使用API | 必要な変更 |
|---|---|---|---|
| {file_path} | {line} | {api_name} | {変更内容} |

### 推奨アクション

**判定**: アップデート推奨 / 段階的アップデート推奨 / 慎重に検討

1. [最初にやるべきこと]
2. [次にやるべきこと]
```

推奨アクションの判定基準:

- **アップデート推奨**: Breaking Changesなし、互換性の問題なし
- **段階的アップデート推奨**: 影響はあるが対応可能。アップグレードパスに従った具体的手順を提示
- **慎重に検討**: 広範な影響あり。代替案やタイミングの提案を含める

## 注意事項

- **PMコマンドの実行許可**: PM固有のコマンドは `allowed-tools` に含めていない。`settings.json` の `allowedTools` でプロジェクトごとに許可設定すること（例: `Bash(npm *)`, `Bash(pnpm *)`, `Bash(yarn *)`, `Bash(bun *)`）
- **WebFetchの制限**: リリースノートのページが大きすぎる場合は取得に失敗することがある。その場合はCHANGELOGやマイグレーションガイドを試す
- **モノレポ対応**: モノレポの場合は対象ワークスペースの `package.json` を基準に分析する。ルートの `package.json` だけでなく、影響を受けるワークスペースも確認する
