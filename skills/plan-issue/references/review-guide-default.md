# プランレビュー基準（デフォルト）

実装プランの品質をレビューするための基準。9つの観点でプランを評価する。

## レビュー観点

| # | 観点 | チェック項目 |
|---|---|---|
| 1 | 実現可能性 | 各ステップで変更対象のファイル/モジュールが実在するか。存在しないAPIや未導入ライブラリを前提にしていないか |
| 2 | スコープ適切性 | Issueの要件に対して過不足がないか。Issue外の改善を混ぜていないか |
| 3 | 依存関係 | ステップ間の順序制約が明示されているか。外部サービスやCI/CDへの影響が考慮されているか |
| 4 | ステップ粒度 | 各ステップが1コミット程度の粒度か。「〜を適切に実装する」のような曖昧な記述がないか |
| 5 | 検証方法 | 完了を確認する手段（テスト、動作確認コマンド等）が各ステップに対して明示されているか |
| 6 | リスク | 破壊的変更、マイグレーション、後方互換性への影響が識別されているか |
| 7 | テストによる仕様の網羅性 | Issueの受け入れ条件・仕様上のルールが検証方針に漏れなく反映されているか。テストが「仕様のドキュメント」として機能する粒度で記載されているか |
| 8 | タスクの自己完結性 | 各タスクがサブエージェント1セッションで完結できるか。参照コンテキストが広すぎる場合、事前調査ステップが計画に含まれているか |
| 9 | アーキテクチャ準拠 | DBスキーマ変更・API変更・新規依存ライブラリが既存のアーキテクチャ方針と整合しているか。技術設計の詳細セクションが必要十分に記載されているか |

## 各観点の詳細

### 1. 実現可能性

- 変更対象ファイルがGrep/Globで実在確認されているか
- 前提としている機能・ツール・仕組みが既に利用可能か
- 新たに必要なものがある場合、そのステップが計画に含まれているか

### 2. スコープ適切性

- Issueの受け入れ条件と計画の成果物が1:1で対応しているか
- 「ついでに」的な改善・リファクタリングが混入していないか
- 必要な変更が漏れていないか（受け入れ条件の各項目がカバーされているか）

### 3. 依存関係

- タスク間の実行順序に矛盾がないか（依存先のタスクが先に実行される構成か）
- 外部サービス・CI/CDパイプラインへの影響が識別されているか
- 並行作業可能なタスクが明示されているか

### 4. ステップ粒度

- 各タスクが1コミット（数十分〜数時間）で完了できる規模か
- 実装内容が具体的に記述されているか（「適切に」「必要に応じて」等の曖昧表現がないか）
- 変更するファイルが明示されているか

### 5. 検証方法

- 各タスクに完了条件が明示されているか
- テスト実行、コマンド実行、目視確認等の具体的な検証手段が記載されているか
- 検証手段が再現可能か（誰が実行しても同じ結果を得られるか）

### 6. リスク

- 既存機能への影響が識別されているか
- 破壊的変更がある場合、影響範囲と対処方針が記載されているか
- 後方互換性を損なう変更がある場合、移行手順が考慮されているか

### 7. テストによる仕様の網羅性

- Issueの受け入れ条件が検証方針に反映されているか
- 正常系だけでなく異常系・エッジケースも考慮されているか
- テスト記述が「仕様のドキュメント」として読めるレベルの粒度か

### 8. タスクの自己完結性

- 各タスクの説明だけで、実装に必要な情報が揃っているか
- 広範囲のコードベース理解を前提としていないか
- コンテキストが大きい場合、事前調査ステップ（Grep/Globで対象を特定する等）が含まれているか

### 9. アーキテクチャ準拠

- DBスキーマ変更がある場合、既存のテーブル設計・命名規則と整合しているか
- マイグレーション方針（破壊的変更の有無、ロールバック手順）が明記されているか
- API変更がある場合、既存のエンドポイント設計・命名規則・レスポンス形式と整合しているか
- 新規依存ライブラリがある場合、選定理由と代替案との比較が記載されているか
- 既存の技術スタックや方針に反する選択をしていないか（している場合はその根拠が示されているか）

## レビュー手順

1. プランの各セクション（技術設計、タスク分解）を上記9観点で確認する
2. 問題がある場合、具体的な指摘と修正案を記録する
