# Domain Modeling フロー

ドメインのデータ型・状態・コマンド・イベント・ワークフローを記法ガイド準拠の擬似コードで形式化する。各フェーズはユーザーとの対話を前提とし、Autoモードでは AI 判断で一括実行する。

記法は `${CLAUDE_SKILL_DIR}/references/domain-model-notation.md`（以下「記法ガイド」）に従う。フィールド名と型名の同名禁止（`A: B` で A と B を同じにしない）等の細則は記法ガイドを参照。

## 出力先

- `docs/{domain}/domain-model.md` — 集約単位で型・コマンド・イベント・ポリシー・ワークフローを構造化した文書（記法ガイド「成果物構成」セクションの文書構造に準拠）

## Phase 0: 入力の読み込み

ドメイン名 `{domain}` をもとに事前文書を読み込み、以降のフェーズの前提を揃える。

**進め方:**

1. `docs/{domain}/event-storming.md` が存在する場合、Read で読み込み、集約・コマンド・イベント・ポリシー・状態遷移・用語集を頭に入れる
2. 存在しない場合の分岐:
   - **Interactive モード**: ドメイン概要・主要集約を最小ヒアリングする
     - 「このドメインの目的を1〜2文で説明してください」
     - 「主要な集約（一貫性境界となるエンティティ）を列挙してください」
   - **Auto モード**: `--auto <file>` で渡された入力ファイルの内容のみを前提にし、追加質問は行わない
3. `docs/big-picture.md` が存在する場合、対象ドメインのコンテキスト境界・関連イベントを確認する。以降は境界を尊重する

**ファシリテーション原則:**

- ユーザーが使った用語をそのまま型名として採用する。勝手に英訳・一般化しない
- event-storming.md のホットスポット（未解決事項）は、本フェーズで型化を試み、型では表せない場合は「型では表せなかった不変条件」として残す

## Phase 1: 値の定義

ID・制約付き値・列挙型など、複数の集約・コマンド・イベントから参照される基礎データ型を定義する。

**ガイド質問:**

- このドメインで識別子（ID）として登場するものは何ですか？（例: 注文ID、顧客ID）
- 範囲・形式に制約のある値はありますか？（例: 1以上の整数、メール形式の文字列、日時）
- 限られた選択肢から選ぶ値はありますか？（例: 通貨種別、ステータス区分）

**進め方:**

1. event-storming.md のイベント・コマンドのフィールド名から候補を抽出する
2. 各値について次の3分類のどれかを決める:
   - **単純な値**: `プランID = 文字列` のように右辺の原始型で定義
   - **制約付き値**: 生成ルールと失敗理由をセットで記述（記法ガイド「制約を持つ値」参照）
   - **列挙型**: OR で選択肢を列挙
3. ユーザーに確認: 「次の値を共通値として定義しますがよろしいですか？」
4. **共通値の判定**: 2つ以上の集約で参照される型のみ共通に置く。判断に迷う場合はその集約固有とする（記法ガイド「共通値の判定基準」参照）

**出力サンプル:**

```
プランID = 文字列
チケットID = 文字列
日時 = 日時

必要Approve数 = 1以上の整数

必要Approve数を作る:
    契機: 外部指示
    入力: 整数
    成功時: 必要Approve数
    失敗時:
        - ゼロ以下
```

## Phase 2: 状態の定義

各集約の状態型を OR で定義し、無効な状態組合せを排除する。

**ガイド質問:**

- この集約は時間とともにどのような状態を取りますか？（event-storming.md の状態遷移図を参照）
- 状態ごとに必要な属性は何ですか？（ある状態にしか存在しない属性に注意）
- 「ありえない状態」は何ですか？（例: 判断依頼中なのに確定日時がある）

**進め方:**

1. 集約ごとに状態の OR を定義する
2. 各状態を独立したレコード（AND の束ね）として書く。状態固有の属性のみを含める
3. ありえない状態を型レベルで排除できているかをユーザーと確認する

**ファシリテーション Tips:**

- 状態間で属性が重複しても OK。むしろ重複を許容して状態ごとに必要な属性のみを持たせる方が、無効状態を排除できる
- 「すべての状態を1つのレコードで表す」（フラグや nullable で状態を区別する）アンチパターンを避ける

**出力サンプル:**

```
実装プラン =
    作成中
    OR 判断依頼中
    OR 確定

作成中プラン =
    ID: プランID
    AND チケットID
    AND タスクの一覧

判断依頼中プラン =
    ID: プランID
    AND チケットID
    AND 質問内容: 文字列

確定プラン =
    ID: プランID
    AND チケットID
    AND タスクの一覧
    AND 確定日時
```

## Phase 3: 集約の定義

集約ごとに「状態型 + コマンド + ポリシー」を束ねる。1つの集約のライフサイクルを集約セクション内で完結して読めるように構成する。コマンドは集約の状態変化を引き起こす契機を統一的に表現し、状態遷移用の独立サブセクションは設けない。

**ガイド質問:**

- 各コマンドはどのような契機（外部指示 / イベント受信 / ポリシー / スケジュール）で発火しますか？
- 各コマンドはどのような入力を受け取り、成功時にどのイベント・状態を返しますか？
- コマンドが失敗するケースはありますか？（`失敗時:` 配下に箇条書きで列挙）
- このイベントを受信したときに自動発火するコマンドはありますか？（ポリシー）

**進め方:**

1. 集約ごとにセクションを作り、Phase 2 の状態型を配置する
2. 全コマンドを「契機 / 入力 / 成功時（イベント・状態） / 失敗時（箇条書き）」で記述する
3. `契機:` フィールドは必須。値は4種列挙値（`外部指示` / `イベント受信(...)` / `ポリシー(...)` / `スケジュール`）から選ぶ
4. 入力イベント（他集約から）と発火するイベント（自集約から）を分けて配置する
   - **発火イベントの定義は発火元集約のセクションに1箇所のみ**置く
   - 受け手集約は名前と参照先のみ記載（再定義しない）
5. ポリシーは **コマンドを発行する側の集約（受け手集約）** に配置する
6. 型では表せなかった不変条件（外部リソース依存・ドメインルール）を末尾に記載する

**集約セクションの構造:**

```
## 集約: 実装プラン

### 責務
チケットを起点とした実装プランのライフサイクル管理。

### 固有値
（この集約のみで使用する値）

### 入力イベント（他集約から）
- 変更が要求された（→ 変更管理集約）

### 発火するイベント
実装プランが確定した =
    プランID
    AND チケットID
    AND 確定者: 開発者ID
    AND 確定日時

### 状態型
実装プラン =
    作成中 OR 判断依頼中 OR 確定
（詳細は Phase 2 で定義済み）

### コマンド
実装プランを作成する:
    契機: 外部指示
    入力: チケット AND 開発者
    成功時: 実装プランが確定した / 作成中プラン
    失敗時:
        - 仕様不明点あり (質問内容を持つ)
        - 技術方針未決 (相談事項を持つ)

判断依頼を発行する:
    契機: 外部指示
    入力: 作成中プラン AND 質問内容: 文字列
    成功時: 判断依頼が発行された / 判断依頼中プラン

確定する:
    契機: 外部指示
    入力: 作成中プラン AND 開発者
    成功時: 実装プランが確定した / 確定プラン
    失敗時:
        - Approve数不足
        - 必須タスク未完了

### ポリシー
変更要求対応ポリシー:
    きっかけ: 変更が要求された
    実行するコマンド: 実装プランを作成する

### 型で表せなかった不変条件
- Approve数の充足判定は外部リソース（レビュー集約の状態）に依存する
```

## Phase 4: ワークフローの定義

外部（他コンテキスト・UI・スケジューラ等）に公開するユースケースの契約を定義する。集約内部で完結するコマンドはここに書かない。

**ガイド質問:**

- 外部から起動されるユースケースは何ですか？（UI 操作、API、スケジューラ）
- そのユースケースは複数の集約をまたぎますか？（またがない場合はコマンドのみで十分）
- 各ステップはどの集約の責任ですか？

**進め方:**

1. 集約横断のユースケースを抽出する
2. ワークフロー名は動詞句「〜する」を既定とする（業務用語として定着した名詞句は例外）
3. `入力` `成功時` `失敗時` でワークフロー全体の契約を示す。失敗ケースは `失敗時:` 配下に箇条書きで列挙する
4. `ステップ` で内部の順序付き処理を列挙する。ステップ名は `〜ステップ` 接尾辞、括弧内に責任集約を記載
5. 順序が複雑な場合は Mermaid flowchart を併用する

**出力サンプル:**

```
実装プランを確定する:
    入力: チケット AND 開発者
    成功時: 実装プランが確定した
    失敗時:
        - 仕様不明点あり
        - 技術方針未決

    ステップ:
        1. プラン作成ステップ（実装プラン集約）
        2. プラン確定通知ステップ（通知集約）
```

## Phase 5: コンテキスト境界の定義

他コンテキスト由来の外部イベント・他コンテキストへの外部コマンドを `外部_` プレフィックスで明示する。

**進め方:**

1. event-storming.md / big-picture.md から、本コンテキスト外で発生するイベント・本コンテキストから他コンテキストへ送る依頼を抽出する
2. 外部イベントは型を `外部_` プレフィックスで定義する
3. 外部コマンドは `観測するイベント:` ラベルで結果イベントを記述する（同期的な `出力:` は使わない）

**出力サンプル:**

```
外部_バグチケットが作成された =
    チケットID
    AND 起票元コンテキスト: 文字列
    AND 発見日時

外部_バグチケット作成を依頼する:
    契機: 外部指示
    入力: 不具合報告
    観測するイベント: 外部_バグチケットが作成された
```

## Phase 6: ドキュメント生成

Phase 0-5 の結果を記法ガイド「成果物構成」セクションの文書構造に従って `docs/{domain}/domain-model.md` に出力する。

**文書構造:**

```
1. スコープ
2. 概観
3. 共通値
4. 集約 A
   - 責務 / 固有値 / 入力イベント / 発火するイベント
   - 状態型 / コマンド / ポリシー / 型で表せなかった不変条件
5. 集約 B（同じ構造）
...
N. ワークフロー
N+1. コンテキスト境界
N+2. DL文書との対応表
N+3. モデル化で露出した論点
```

**進め方:**

1. `docs/{domain}/` ディレクトリの存在を確認し、なければ `mkdir -p` で作成する
2. 各セクションを記法ガイドの規約に従って記述する
3. **DL文書との対応表**: event-storming.md のイベント・コマンド・集約と本文書の型の対応を表にする。型化で名前が変わった場合は理由も記載
4. **モデル化で露出した論点**: 型定義を試みて見つかった DL 文書の不備・不明点・記法ガイドで規定されていない記述方針等を残す
5. 生成後、ファイルパスを報告し、ユーザーに過不足を確認する（Interactive モード）。Auto モードは生成完了のみ報告する

**生成時のチェック:**

- フィールド名と型名の同名禁止（記法ガイド）に違反していないか
- 発火イベントの定義が発火元集約に1箇所のみ置かれているか（受け手集約での再定義禁止）
- ポリシーが受け手集約のセクションに置かれているか
- 共通値が2つ以上の集約で参照されているか（単独参照は集約固有に降格）
- 失敗理由がコマンドの `失敗時:` 配下の箇条書きで列挙されているか（独立型定義 `〜失敗理由 =` は廃止）
- 全コマンドに `契機:` フィールドが記述され、値が4種列挙値（外部指示 / イベント受信(...) / ポリシー(...) / スケジュール）のいずれかに該当するか
