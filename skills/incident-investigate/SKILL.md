# インシデント詳細調査（Incident Investigation）

トリアージ後に根本原因を特定し、修正方針を立てるための詳細調査を行います。

## 引数

- `$ARGUMENTS`: エラーメッセージまたはスタックトレース（テキスト）

## 手順

### Step 1: エラーメッセージ詳細解析

`$ARGUMENTS` で受け取ったエラーメッセージを詳細に解析する。

#### 1.1 ファイルパス・行番号・シンボル名の抽出

エラーメッセージからファイルパス・行番号・シンボル名（関数名・クラス名等）を抽出する。プロジェクト外のファイルはスキップし、プロジェクト内のフレームのみ対象とする。

複数のスタックフレームが存在する場合は、すべて抽出してリスト化する。

#### 1.2 エラー種別の判定

以下のカテゴリに分類する。検出キーワードはプロジェクトで使用しているフレームワーク・ライブラリのエラーパターンに応じて判断する:

| カテゴリ               | 検出キーワードの例                                                   |
| ---------------------- | -------------------------------------------------------------------- |
| ドメインエラー         | `DomainError`, `NotFoundError`, `ValidationError`, `BusinessRule` 等 |
| データベースエラー     | `connection refused`, `ECONNREFUSED`, `database`, `query failed` 等  |
| APIフレームワークエラー | HTTPステータスコード、フレームワーク固有のエラーコード 等            |
| 型/コンパイルエラー    | コンパイラ固有のエラーコード、型不一致、未定義プロパティ 等               |
| ランタイムエラー       | `ReferenceError`, `TypeError`, `SyntaxError`, `RangeError`           |
| その他                 | 上記に該当しないもの                                                 |

### Step 2: 発生箇所の特定

Step 1で抽出したファイルパスと行番号を使って、該当コードを確認する。

#### 2.1 ファイルパスが抽出できた場合

1. `Glob` でファイルの存在を確認する（パスが相対パスの場合、プロジェクトルートからの探索も試みる）
2. `Read` で該当ファイルの該当行の前後を表示する（前後5行程度）
3. シンボル名が判明している場合はシンボル全体を取得する

#### 2.2 ファイルパスが抽出できなかった場合

1. エラーメッセージ内のシンボル名やキーワードを特定する
2. `Grep` でコードベース内を検索する
3. 候補が見つかった場合は、そのファイル・シンボルの情報を取得する

### Step 3: 呼び出し元の追跡（1レベル）

発生箇所のシンボルについて、呼び出し元を1レベルまで追跡する。

1. `Grep` で発生箇所のシンボルを参照しているコードを検索する
2. 参照元のファイルパスとシンボル名をリスト化する
3. 参照が多数ある場合は、スタックトレースに含まれるもの、または主要な呼び出し元に絞る（最大5件）

### Step 4: 最近の変更との関連確認

発生箇所のファイルについて、直近の変更履歴を確認する。

1. `git log --oneline -10 -- {ファイルパス}` で発生箇所の直近の変更を表示する
2. 最近の変更がある場合は、リグレッションの可能性を提示する
3. 関連するコミットがあれば、その変更内容の概要を記載する

### Step 5: 結果出力

以下のMarkdownフォーマットで結果を出力する:

````markdown
## インシデント調査結果

### 入力

```
[受け取ったエラーメッセージをそのまま表示]
```

### エラー種別

[判定したエラー種別]（[根拠となるキーワード]）

### 発生箇所

`[ファイルパス]:[行番号]` - `[シンボル名]`

### 関連コード

| ファイル         | 行       | 内容               |
| ---------------- | -------- | ------------------ |
| `[ファイルパス]` | [行番号] | `[該当行のコード]` |

### 呼び出し元

1. `[ファイルパス]:[行番号]` - `[シンボル名]`
2. `[ファイルパス]:[行番号]` - `[シンボル名]`

### 最近の変更
````

## エッジケース処理

### ファイルパスが含まれないエラーメッセージ

- エラーメッセージ内のキーワード（クラス名、メソッド名、エラーメッセージ文字列）でコードベースを検索する
- 候補が見つからない場合は「発生箇所を特定できませんでした」と出力し、手動調査のヒント（検索キーワード候補）を提示する

### ファイルがプロジェクト内に存在しない場合

- 外部ライブラリのエラーの可能性を提示する
- スタックトレースの次のフレーム（プロジェクト内のコード）を発生箇所として扱う

### 複数のスタックフレームがある場合

- プロジェクト内のフレームをすべてリスト化する
- 最初のプロジェクト内フレームを主要な発生箇所として扱う
- 残りのフレームは「呼び出し元」セクションに含める

### git logが実行できない場合

- 「最近の変更」セクションをスキップし、その旨を注記する

## 補足

- **読み取り専用**: このスキルはコードの読み取りと調査のみを行い、ファイルの編集は行わない
- **目的**: 根本原因の特定と修正方針の検討材料を提供すること。トリアージ（優先度判断）は `/incident-triage` で行う
