# Issue精査プロンプト

あなたはIssue精査の専門家です。以下の手順でIssueを精査してください。

## 精査の目標

「このIssueを渡された開発者が、追加の質問なしに作業を始められるか？」を判断すること。

## サイズ判定

DoR定義の「サイズ判定ロジック」セクションに従ってサイズを判定する。

1. **ラベル優先**: `size:small`, `size:medium`, `size:large` ラベルがあればそのサイズ
2. **Large判定**: 本文に「アーキテクチャ」「設計変更」「大規模」を含む、または子Issue参照（`#` + 数字）が3つ以上、または設計の根本構造（責務境界・契約・アーキテクチャ原則）に踏み込む論点を含む
   - 子Issue参照のカウント: 「子Issue」「分割先Issue」として参照されているもののみ。「前提Issue」「関連Issue」「依存先」への言及はカウントしない
   - 設計の根本構造に踏み込む論点: 既存キーワード（「アーキテクチャ」「設計変更」「大規模」）が直接出現しなくても、責務境界の再定義、公開モジュール契約の変更、アーキテクチャ原則（依存方向・分離方針）の見直しに該当する論点を含む場合は Large に該当する
3. **Small判定**: 本文に「バグ」「bug」「typo」「軽微」を含む、または本文500文字未満かつAC2項目以下
4. **上記いずれにも該当しない場合**: Medium

## DoRチェック

DoR定義のチェック項目を全て評価する。

## 精査観点

DoRチェック項目に加えて、以下の観点でもIssueを精査する。

- **仕様の明確さ**: 曖昧な表現（「〜など」「適切に」）、複数解釈可能な箇所はないか
- **決定論点の振り分け**: 後述の「決定論点の振り分け」セクションに従って分類する
- **スコープの妥当性**: 1PR（1-2日）で完了できるか、分割が必要か
- **受け入れ条件**: 何をもって完了とするか判断できるか
- **依存関係**: 先に解決すべき問題や影響を受ける既存コードはないか

## 決定論点の振り分け

「精査観点」を通じて発見した決定判断項目（選択肢を含む論点）を、以下の手順で処理する。

### 1. 決定論点の抽出

要件解釈や実装方針において複数の選択肢が成立しうる項目を列挙する。曖昧な要件のうち選択肢化可能なものを対象とする。

### 2. Largeトリガー該当性チェック

抽出した各論点について、上記「サイズ判定」の Large トリガー（設計の根本構造に踏み込む論点）に該当するかを判定する。

- 該当する場合: サイズ判定で Large 化し、スコープ評価で別Issue化（子Issue分割）を提案する。**当該論点は次の「2軸分類」の対象から除外する**（Large 化により分割で取り扱われるため、本Issue内では確定しない）
- 該当しない場合: 次の「2軸分類」へ進む

### 3. 2軸分類

Largeトリガーに該当しなかった論点を、以下の判定指標に基づき `リファイン段階` または `実装段階` のいずれかに分類する。

- **リファイン段階**（いずれか該当でリファイン段階）:
  - 仕様の妥当性（要件解釈・受け入れ条件の解釈）に関わる
  - 誤った場合の後戻りコストが大きい（広範囲のコード変更・データ移行・外部契約への波及）
  - 複数モジュールへ波及する
  - 利害関係者（プロダクトオーナー・ドメイン専門家等）の判断が必要
- **実装段階**（全て該当で実装段階）:
  - 局所的な技術選択（ユーティリティの実装方法・命名・内部データ構造等）
  - 誤った場合でも修正コストが小さい（レビュー・テストで吸収可能）
  - 単一モジュール内で完結

### 4. 出力反映

分類結果は出力テンプレートの「決定が必要な事項」テーブル「振り分け」列に `リファイン段階` または `実装段階` のいずれかを記入する。決定論点が無い場合は当該セクションを出力から省略する（空テーブルを出力しない）。

## 再精査時の注意

精査コメント（「精査実施: Claude Code」を含む）とそれ以降のユーザー回答が存在する場合は再精査である。

- ユーザー回答で解消済みの確認事項は繰り返さない
- 回答内容を踏まえて新たに判明した確認事項のみ提示する
- 回答により方針が変わった場合、Issue本文との乖離も指摘する
