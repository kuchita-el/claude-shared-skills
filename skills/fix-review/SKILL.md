# PRレビュー指摘フィックスパイプライン

PRレビューの指摘を修正し、テスト・lint・差分確認・コミットまでを一貫して実行します。再レビューで新たな指摘があれば自動で修正ループを継続します。

## 引数

- なし（会話コンテキストから指摘を取得する）

## 前提条件

- PRが作成済みであること
- 会話にレビュー指摘が含まれていること（レビューツールの実行結果、PRコメント等）

## ワークフロー概要

```
外側ループ（最大3周）:
  Phase 0: 指摘の整理
  Phase 1: 指摘ごとの修正（内側ループ）
    Step 1: コード修正
    Step 2: テスト実行
    Step 3: Lint実行
    Step 4: 差分確認 + ステージング
    Step 5: コミット
  Phase 2: 再レビュー
  Phase 3: 判定 → 指摘なし→Phase 4 / 指摘あり→Phase 0に戻る / 3周目→Phase 5
Phase 4: 完了報告
Phase 5: 打ち切り報告
```

## 詳細手順

### Phase 0: 指摘の整理

1. 会話コンテキストからレビュー指摘を一覧化する
2. 各指摘を以下に分類する:
   - **修正必須**: バグ、ロジックエラー、セキュリティ問題、規約違反
   - **提案**: リファクタリング案、命名改善、パフォーマンス提案
3. 分類結果をユーザーに提示し、対応する指摘を確認する

### Phase 1: 指摘ごとの修正（内側ループ）

確認済みの指摘を1件ずつ修正する。関連する指摘はまとめて修正してよい。

#### Step 1: コード修正

- EditツールまたはWriteツールで修正を行う
- 修正前に影響範囲を確認する

#### Step 2: テスト実行

プロジェクトのテストコマンドを実行する（プロジェクトの設定ファイルを確認して適切なコマンドを使用）。

- **成功**: Step 3へ進む
- **失敗**: 以下のリトライ手順を実行（最大3回）
  1. 失敗原因を分析する
  2. コードを修正する
  3. テストを再実行する
  4. 3回失敗した場合: ユーザーに失敗内容を提示し、手動確認に切り替える

#### Step 3: Lint実行

プロジェクトのlintコマンドを実行する（プロジェクトの設定ファイルを確認して適切なコマンドを使用）。

- **成功**: Step 4へ進む
- **失敗**: 以下の手順を実行
  1. lint自動修正コマンドを実行する（例: `lint:fix` scriptがあれば実行）
  2. `git status` で自動修正された差分を確認する
  3. 自動修正で直らないエラーは手動で修正する
  4. 再度lintを実行する
  5. 上記を最大3回繰り返す
  6. 3回失敗した場合: ユーザーに失敗内容を提示し、手動確認に切り替える

#### Step 4: 差分確認 + ステージング

```bash
git status
git diff
```

- 変更内容が意図通りか確認する
- 対象ファイルのみをステージングする（`git add -A` は使わない）

```bash
git add {対象ファイル1} {対象ファイル2} ...
```

#### Step 5: コミット

全ステップがパスした場合のみコミットする。

```bash
git commit -m "fix: レビュー指摘対応 - {変更要約}" -m "対応指摘: {指摘概要}"
```

- 関連する複数の指摘をまとめた場合は、対応指摘を箇条書きで列挙する

Phase 1の全指摘の修正が完了したら、Phase 2へ進む。

### Phase 2: 再レビュー

修正後のコードを再レビューする。以下の優先順位でレビュー手段を選択する:

1. **プロジェクトのレビュースキル**: `.claude/skills/` 配下にレビュー用スキル（`review-pr` 等）があれば Skill ツールで実行する
2. **自力レビュー**: レビュースキルが見つからない場合、修正した全ファイルを Read で読み直し、以下の観点でセルフレビューを行う
   - 修正が指摘の意図を正しく反映しているか
   - 新たなバグやロジックエラーが混入していないか
   - コーディング規約・既存パターンとの一貫性

### Phase 3: 判定

再レビュー結果に基づいて判定する:

- **指摘なし**: Phase 4（完了報告）へ進む
- **指摘あり かつ 3周目未満**: Phase 0に戻り、新たな指摘を整理する
- **3周目**: Phase 5（打ち切り報告）へ進む

### Phase 4: 完了報告

PR番号を取得し、完了コメントを投稿する:

```bash
gh pr view --json number --jq '.number'
```

1. Write ツールで一時ファイルを作成する:

```bash
# /tmp/fix-review-complete-{timestamp}.md に以下を書き出す
```

```markdown
## レビュー指摘対応完了

全ての指摘に対応しました。

### 対応サマリ

| 指摘    | 対応内容    | コミット      |
| ------- | ----------- | ------------- |
| {指摘1} | {対応内容1} | {commit hash} |
| {指摘2} | {対応内容2} | {commit hash} |

---

_自動投稿: Claude Code (`/fix-review`)_
```

2. PRコメントを投稿する:

```bash
gh pr comment {pr_number} --body-file /tmp/fix-review-complete-{timestamp}.md
```

### Phase 5: 打ち切り報告

3周のループで解消できなかった指摘がある場合:

1. Write ツールで一時ファイルを作成する:

```markdown
## レビュー指摘対応（一部未解消）

3回の修正ループで解消できなかった指摘があります。手動確認をお願いします。

### 対応済み

| 指摘    | 対応内容    | コミット      |
| ------- | ----------- | ------------- |
| {指摘1} | {対応内容1} | {commit hash} |

### 未解消

| 指摘    | 未解消理由 |
| ------- | ---------- |
| {指摘N} | {理由}     |

---

_自動投稿: Claude Code (`/fix-review`)_
```

2. PRコメントを投稿する:

```bash
gh pr comment {pr_number} --body-file /tmp/fix-review-incomplete-{timestamp}.md
```

## 注意事項

- **テスト・lintコマンドの実行許可**: テスト・lintコマンドは `allowed-tools` に含まれていないため、実行時に都度ユーザーの許可が必要です。頻繁に使用する場合は `settings.json` の `allowedTools` でプロジェクトごとに許可設定することを推奨します
- **全ステップパスまでコミットしない**: テストまたはlintが失敗している状態でコミットしない
- **lint:fix後のステージング忘れ防止**: lint:fix実行後は必ず `git status` で差分を確認し、ステージングする
- **ファイル単位のステージング**: `git add -A` や `git add .` は使わず、対象ファイルを明示的に指定する
