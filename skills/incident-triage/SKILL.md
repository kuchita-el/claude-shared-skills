# インシデントトリアージ（Incident Triage）

エラーの重大度・影響範囲を素早く判断し、対応の優先度を決める材料を提示します。

## 引数

- `$ARGUMENTS`: エラーメッセージまたはスタックトレース（テキスト）

## 手順

### Step 1: エラーメッセージの初期解析

`$ARGUMENTS` で受け取ったエラーメッセージを解析する。

#### 1.1 ファイルパス・シンボル名の抽出

エラーメッセージからファイルパスとシンボル名（関数名・クラス名等）を抽出し、ドメイン/モジュールを特定する。詳細なコード読解はしない。

エラーメッセージからファイルパス・行番号・シンボル名（関数名・クラス名等）を抽出する。プロジェクト外のファイルはスキップし、プロジェクト内のフレームのみ対象とする。

#### 1.2 エラー種別の判定

以下のカテゴリに分類する。検出キーワードはプロジェクトで使用しているフレームワーク・ライブラリのエラーパターンに応じて判断する:

| カテゴリ           | 検出キーワードの例                                                   |
| ------------------ | -------------------------------------------------------------------- |
| ドメインエラー     | `DomainError`, `NotFoundError`, `ValidationError`, `BusinessRule` 等 |
| データベースエラー | `connection refused`, `ECONNREFUSED`, `database`, `query failed` 等  |
| APIフレームワークエラー | HTTPステータスコード、フレームワーク固有のエラーコード 等        |
| 型/コンパイルエラー | コンパイラ固有のエラーコード、型不一致、未定義プロパティ 等               |
| ランタイムエラー   | `ReferenceError`, `TypeError`, `SyntaxError`, `RangeError`           |
| その他             | 上記に該当しないもの                                                 |

#### 1.3 影響モジュール/ドメインの特定

抽出したファイルパスのディレクトリ構造から、影響を受けるモジュール/ドメインを特定する。プロジェクトのディレクトリ構成を参考に判断する。

### Step 2: 影響範囲の推定

発生箇所のシンボルから呼び出し元を辿り、影響するエンドポイントを特定する。

#### 2.1 ファイルパス・シンボル名が抽出できた場合

1. Grepでシンボルの参照箇所を検索する
2. 呼び出し元を辿り、最終的にどのAPIエンドポイント/ルーターに到達するかを特定する
3. 「このエラーは○○と○○のAPI機能に影響する」を出力する

#### 2.2 ファイルパスのみ抽出できた場合（シンボル名なし）

1. 該当ファイルを読み込み、エラーメッセージ内のキーワードと照合し、関連するシンボルを特定する
2. 以降は2.1の手順2-3に従う

#### 2.3 ファイルパスが抽出できなかった場合

「影響範囲の自動推定ができませんでした」と表示し、手動確認を推奨する。Step 3のGitHub検索に進む。

### Step 3: 過去の類似インシデント検索

エラーメッセージから検索キーワードを生成し、過去の類似インシデントを検索する。

#### 3.1 検索キーワードの生成

以下の優先順位でキーワードを生成する:

1. エラーメッセージ内の固有名詞（クラス名、メソッド名、エラーコード）
2. 発生箇所のファイル名
3. エラー種別名（上記が汎用的すぎる場合のフォールバック）

#### 3.2 GitHub Issue検索

```
gh issue list --search "{キーワード}" --state closed --limit 5 --json number,title,url
```

- クローズ済みIssueを検索（解決策がある可能性が高いため）
- 最大5件まで表示

#### 3.3 GitHub PR検索

```
gh pr list --search "{キーワード}" --state merged --limit 5 --json number,title,url
```

- マージ済みPRを検索
- 最大5件まで表示
- 関連するPRが見つかった場合、`gh pr view {番号} --json files` で変更ファイルを取得する

#### 3.4 類似度判定

検索結果に対して以下の基準で類似度を判定する。「一致」は検索結果のタイトルまたは本文に該当文字列が含まれることを指す。

| 類似度 | 基準                                                                     |
| ------ | ------------------------------------------------------------------------ |
| 高     | エラーメッセージのキーワード + 発生箇所のファイルパスの両方が一致        |
| 中     | キーワードまたはファイルパスのいずれかが一致                             |
| 低     | 関連キーワード（エラー種別名、影響モジュール名など間接的な用語）のみ一致 |

#### 3.5 エッジケース

- **検索結果が0件**: 「過去の類似インシデントは見つかりませんでした」と表示
- **GitHub APIエラー**: 検索をスキップし、「GitHub検索でエラーが発生したため、過去のインシデント検索をスキップしました」と注記（スキル全体は中断しない）
- **汎用的すぎるキーワード**: ファイル名検索を優先し、それも不明ならエラー種別名を使用

### Step 4: 重大度の判定

Step 1-3の結果を総合し、重大度を判定する。最終的な判断は人間が行うため、判定根拠を明示する。

#### 4.1 重大度基準

| 重大度   | 基準                                                                |
| -------- | ------------------------------------------------------------------- |
| Critical | 複数エンドポイントに影響 + 過去の類似インシデントなし（未知の問題） |
| High     | 単一エンドポイントに影響、または過去に同種の問題が再発              |
| Medium   | 影響範囲が限定的 + 過去に解決実績あり                               |
| Low      | 開発時のみ発生（型/コンパイルエラー等）、またはユーザー影響なし        |

#### 4.2 判定根拠の出力

以下の情報を判定根拠として出力し、人間が重大度の妥当性を検証できるようにする:

- 影響を受けるエンドポイントの数と一覧（Step 2の結果）
- 過去の類似インシデントの有無と解決状況（Step 3の結果）
- エラー種別とユーザー影響の有無（Step 1の結果）

### Step 5: 暫定対応案の生成

Step 1-4の結果を基に、暫定対応案を生成する。

#### 5.1 エラー種別に基づく初動アクションの選定

Step 1.2で判定したエラー種別に応じて、初動アクションの候補を選定する:

| エラー種別             | 初動アクション候補                                                         |
| ---------------------- | -------------------------------------------------------------------------- |
| データベースエラー     | DB接続状態の確認、コネクションプールの状態確認、マイグレーション状態の確認 |
| APIフレームワークエラー | リクエストパラメータの確認、認証状態の確認、ルーター定義の確認            |
| ドメインエラー         | 入力値のバリデーション確認、ビジネスルールの前提条件の確認                 |
| 型/コンパイルエラー    | 型定義の整合性確認、依存パッケージのバージョン確認                         |
| ランタイムエラー       | 変数の初期化状態の確認、null/undefinedチェックの追加、境界値の確認         |
| その他                 | エラーメッセージの詳細調査、ログの確認                                     |

影響範囲（Step 2）の結果も考慮し、影響が広範な場合はより保守的なアクション（サービス再起動の回避等）を優先する。

#### 5.2 過去知見の活用

Step 3で見つかった類似インシデントの解決策を参照し、今回のケースに適用可能か判定する:

- **類似度「高」の過去インシデント**: 解決策をそのまま暫定対応案の第一候補とする
- **類似度「中」の過去インシデント**: 解決策を参考情報として提示し、適用可否を人間が判断する
- **類似度「低」または該当なし**: 5.1のエラー種別ベースのアクションのみを提示する

#### 5.3 重大度に応じた緊急度ラベルの付与

Step 4で判定した重大度に基づき、対応の緊急度ラベルを付与する:

| 重大度   | 緊急度ラベル | 対応期限の目安   |
| -------- | ------------ | ---------------- |
| Critical | 今すぐ対応   | 即時             |
| High     | 本日中       | 当日中           |
| Medium   | 次スプリント | 次スプリント以内 |
| Low      | バックログ   | 優先度に応じて   |

#### 5.4 暫定対応 vs 恒久対応の判定

以下の4つの基準で、暫定対応と恒久対応の必要性を判定する:

| 判定基準         | 評価観点                                                 |
| ---------------- | -------------------------------------------------------- |
| 再発リスク       | 同じ条件で再発する可能性があるか                         |
| 影響範囲         | 影響が広範か限定的か（Step 2の結果）                     |
| 過去の再発履歴   | Step 3で同種の問題が複数回報告されているか               |
| 根本原因の明確さ | エラーメッセージとスタックトレースから原因が特定できるか |

判定結果は以下の3パターン:

1. **暫定対応のみで完了可能**: 再発リスクが低く、影響範囲が限定的で、根本原因が明確な場合
2. **暫定対応 + 恒久対応Issue作成を推奨**: 暫定対応で一旦回復するが、再発リスクや影響範囲を考慮すると恒久対応が必要な場合
3. **恒久対応を優先**: 暫定対応では問題が解決しない、または暫定対応自体がリスクを伴う場合

#### 5.5 対応案の構造化

5.1〜5.4の結果を、Step 6の出力テンプレート「推奨アクション」セクションで使用する形式に整理する:

- 暫定対応アクションを優先順位付きで列挙し、それぞれに根拠を付与する
- 恒久対応の要否判定結果と4つの判定基準の状況をまとめる
- 恒久対応が必要な場合、次ステップ（`/create-issue`, `/incident-investigate`）を案内する

### Step 6: トリアージ結果の出力

以下のMarkdownフォーマットで結果を出力する:

```markdown

- **発生箇所**: `[ファイルパス]` - `[シンボル名]`（抽出できた場合）

### 重大度: [Critical/High/Medium/Low]

- 影響エンドポイント数: [N件]（[エンドポイント名の一覧]）
```

## エッジケース処理

### ファイルパスが含まれないエラーメッセージ

- エラーメッセージ内のキーワード（クラス名、メソッド名、エラーメッセージ文字列）でStep 3のGitHub検索を行う
- 影響範囲の推定（Step 2）はスキップし、「影響範囲の自動推定ができませんでした」と注記する

### ファイルがプロジェクト内に存在しない場合

- 外部ライブラリのエラーの可能性を提示する
- スタックトレースの次のフレーム（プロジェクト内のコード）を発生箇所として扱う

### 複数のスタックフレームがある場合

- 最初のプロジェクト内フレームを主要な発生箇所として扱う
- Step 2の影響範囲推定は主要な発生箇所に対して実施する

## 補足

- **読み取り専用**: このスキルはコードの読み取りと調査のみを行い、ファイルの編集は行わない
- **目的**: 対応優先度の判断材料を素早く提示すること。根本原因の特定は `/incident-investigate` で行う
- **重大度は提案**: 最終的な重大度の判断は人間が行う。スキルは判定根拠を明示し、人間が検証・調整できるようにする
