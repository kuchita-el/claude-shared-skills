# 汎用開発ループ

Issue・計画ファイル・PRレビュー指摘・会話コンテキストを起点に、実装→セルフレビュー→修正→PR作成までを一貫して実行する。

## 引数

- `$ARGUMENTS`: Issue番号/URL、計画ファイルパス、PR番号/URL、または省略（会話コンテキストを使用）

## ワークフロー

```
Phase 0: インプットの理解（再開検知・エスカレーション履歴確認を含む）
Phase 1a: テスト設計・検証（test-designer → test-spec-validator）
Phase 1b: TDD小サイクル + コード修正
Phase 1c: 検証ゲート → 失敗→Phase 1b / 全パスまたは全スキップ→Phase 1d
Phase 1d: ステージング・コミット
Phase 2: セルフレビュー
Phase 3: 判定 → ブロッカーなし→Phase 4 / ブロッカーあり・3周目未満→Phase 1b / 3周目→Phase 4
Phase 4: PR作成 + 完了報告

※ 全フェーズ共通: 判断ポイント発生時はエスカレーション（後述）
```

### Phase 0: インプットの理解

引数と会話コンテキストからインプット種別を自動判定する:

| 種別 | 判定条件 | 読み込み方法 |
|---|---|---|
| Issue | `#数字`、Issue URL、数字のみ | `gh issue view` |
| 計画ファイル | ファイルパス（`.md` 等） | `Read` |
| PRレビュー指摘 | PR番号/URL or 現在のブランチにPRが存在 | `gh pr view` + 会話コンテキスト |
| 会話コンテキスト | 上記いずれにも該当しない | 直前の会話から要件・指摘を抽出 |

読み込んだ内容から以下を確認・判定する。

#### エスカレーション履歴の確認

インプットがIssueの場合、Issueコメントから `<!-- dev-loop:escalation -->` マーカーを検索する。未回答のエスカレーション（マーカー後にコメントがない）があれば報告して終了する。回答済みのエスカレーション内容は作業リストの入力コンテキストに含める。

#### 実行環境の検知

現在のブランチ名から以下を判定する。

- **worktree環境**: worktree内で実行されている場合、ブランチ切り替えに制約がある（他worktreeで使用中のブランチにはcheckoutできない等）。制約に遭遇したら、現在のブランチ上で作業を進め、必要に応じて手動マージ手順を報告する
- **別セッション再開**: 作業ブランチ上にいる場合、コミット履歴と差分から現在の実装状況を把握し、未完了の作業を継続する

#### ブランチの準備

新規作業の場合（既存の作業ブランチ上にいない場合）、以下の手順でブランチを作成する:

1. CLAUDE.md、CONTRIBUTING.md等からベースブランチ（develop、main等）を確認する
2. 指定がない場合はデフォルトブランチ（main/master等）をベースにする
3. 最新を取得（`git fetch`）し、ベースブランチから新しいブランチを作成する

#### 作業リストとレビュー契約の作成

仕様に曖昧さや設計判断が必要な箇所があればエスカレーションする。問題なければ以下の2つを作成する。

**作業リスト（実装者向け）:** 実装/修正すべき事項の一覧。

**レビュー契約（レビュアー向け）:** インプットの受け入れ条件・完了条件から導出した、具体的なチェック項目のリスト。レビュー契約はPhase 2でcode-reviewerに渡され、機能要件の検証基準として使用する。

レビュー契約の導出ルール:
- インプットに受け入れ条件がある場合（Issue/計画ファイル）、各条件を検証可能なチェック項目に変換する
- 受け入れ条件がない場合、作業リストの各項目から「何が完了していれば正しいか」を導出する
- 各項目は「〜が存在する」「〜が返される」「〜が呼ばれる」等、差分から検証可能な形式で記述する

### Phase 1: 実装/修正/検証

#### Phase 1a: テスト設計・検証

入力ソースからテスト仕様を設計し、独立検証を経てPhase 1bに引き渡すフェーズ。テストコードの生成はPhase 1bのTDD小サイクルで行う。

作業リストの内容がテスト設計を明らかに必要としない場合（ドキュメントのみの変更、設定ファイルの変更、typo修正等）はPhase 1a全体をスキップしてPhase 1bに進む。判断がつかない場合はtest-designerに委任する。

1. **入力ソースの準備（3層構造）**: Phase 0で取得したIssue本文（層1）、計画ファイル（層2）を確認する。層3（追加ドキュメント）はユーザー指定があればそのファイルパスを、なければ計画ファイル内の参照（「参照ドキュメント」セクション等のファイルパス・URL）からファイルパスを抽出する。層3の内容はサブエージェントが必要に応じて読み込むため、メインでは読み込まない
2. **test-designerの起動**: このスキルの `agents/test-designer.md` を読み込み、層1〜2の内容と層3のファイルパス一覧を含めてプロンプトを構築する。Agent toolが使える場合はtest-designerサブエージェントに委任する。Agent toolが使えない場合や起動に失敗した場合はインラインで直接実行する（インライン実行時は層3のファイルパスを自分でReadする）
3. **結果に応じた分岐**:
   - テスト不要 → Phase 1bに進む
   - 矛盾検出 → エスカレーション
   - テスト仕様 → ステップ4へ
4. **テスト仕様の妥当性検証**: このスキルの `agents/test-spec-validator.md` を読み込み、テスト仕様と層1〜2の内容・層3のファイルパス一覧を渡してプロンプトを構築する。Agent toolが使える場合はtest-spec-validatorサブエージェントに委任する。Agent toolが使えない場合や起動に失敗した場合はインラインで直接実行する（インライン実行時は層3のファイルパスを自分でReadする）。差し戻し時は過去の検証結果を差し戻し履歴として渡す。結果に応じて分岐する:
   - 合格 → テスト仕様をPhase 1bに引き渡して終了
   - 不合格 → test-designerに検証結果をフィードバックとして渡し、ステップ2から再実行する。差し戻し上限は3回。3回を超えても合格しない場合はエスカレーションする
   - 仕様不明瞭 → エスカレーション（元の要件の曖昧さが原因であり、テスト仕様の改善では解決できない）

#### Phase 1b: TDD小サイクル + コード修正

Phase 1aからテスト仕様が引き渡された場合、テスト仕様を1つずつTDD小サイクルで消化する。テスト仕様がない場合（Phase 1aでテスト不要判定、またはPhase 1aスキップ時）は作業リストに基づいて従来通りコードを修正する。

Phase 1cまたはPhase 3からの再入時はTDD小サイクルを再実行しない。テストコードは既に存在するため、失敗したテスト・lint・型チェックの修正やレビュー指摘への対応のみを行う。再入時に同じ修正→失敗を繰り返している場合（振動パターン）はエスカレーションする。

**TDD小サイクル（テスト仕様がある場合の初回実行時）:**

テスト仕様のテストケース（T1, T2, T3, ...）を順番に以下のサイクルで消化する:

1. テスト仕様1つ（例: T1）のテストコードを実装する
2. テストを実行し、失敗を確認する（Red）
   - テストが最初からパスした場合: 「警告: T{n} はコード実装前にパスしました。テストの妥当性を確認してください」と出力し、そのテスト仕様はGreen済みとして次のテスト仕様に進む
3. テストが通るようにコードを実装する（Green）
4. テストを実行し、パスを確認する
   - 失敗した場合: コード修正を再試行する。テスト仕様単位で最大3回リトライし、3回を超えたら「警告: T{n} は3回リトライしても通りませんでした。次のテスト仕様に進みます」と出力してスキップする。スキップしたテスト仕様のIDを記録する
   - 既存テスト（消化済みのテスト仕様のテスト）がリグレッションで失敗した場合: リグレッションの修正を優先する
5. 次のテスト仕様に進み、ステップ1に戻る

テスト実行コマンドが `allowedTools` で許可されていない場合: Red/Green確認（ステップ2・4）をスキップする。テストコード実装（ステップ1）とコード実装（ステップ3）は1テスト仕様ずつ逐次実行し、テスト実行による確認はPhase 1cの検証ゲートに委ねる。ただしRed確認のスキップはTDDの効果を大きく損なうため、`settings.json` の `allowedTools` でテスト実行コマンドを許可することを推奨する（例: `Bash(npm test*)`, `Bash(pytest*)`）。

全テスト仕様の消化完了後、作業リストにテスト以外の実装項目（ドキュメント更新、設定変更等）が残っている場合はそれらを対応してからPhase 1cに進む。スキップされたテスト仕様がある場合はPhase 1cの検証ゲートで検出される。

#### Phase 1c: 検証ゲート

セルフレビューの前に、型チェック・lint・テストで機械的に検出できる問題を解消する。コマンドが見つからない場合や `allowedTools` で許可されていない場合はスキップしてよい。失敗したらPhase 1bに戻る。5回繰り返しても通らなければエスカレーションする。Phase 1bでの修正が別の検証を壊すパターン（lint修正→テスト失敗→テスト修正→lint失敗等）を検知した場合もエスカレーションする。

#### Phase 1d: ステージング・コミット

対象ファイルのみをステージング（`git add -A` は使わない）してコミットする。テストコードと対応する実装コードは同じコミットにまとめ、意味のある単位でコミットする。コミットスタイルはプロジェクトの既存コミットや `CLAUDE.md` の規約に合わせる。

Phase 1内でエスカレーションが発動する場合は、エスカレーション手順の後にWIPコミット（`WIP: {作業内容の要約}`）を行ってからセッションを終了する。次のセッションで作業を継続できるようにするため。

### Phase 2: セルフレビュー

人間のレビュアーに渡す前に、自分自身で差分をレビューして品質を高めるフェーズ。セルフレビューがこのスキルの主要な付加価値であり、スキップしてはならない。

このスキルの `agents/code-reviewer.md` を読み込み、そこに記載されたレビュー手順に従う。Agent toolが使える場合はcode-reviewerサブエージェントに委任し、以下のコンテキストを渡す:

- **ベースブランチ名**: 差分取得用
- **要件情報**: Issue本文/計画ファイルの内容
- **レビュー契約**: Phase 0で作成した完了チェックリスト

Agent toolが使えない場合（サブエージェントとして実行されている場合等）や起動に失敗した場合は、同じ手順をインラインで直接実行する。

レビュー結果に仕様の曖昧さで判断できない指摘が含まれる場合はエスカレーションする。

### Phase 3: 判定

レビュー結果の**ブロッカーの有無**に基づいて判定する。改善提案のみの場合は修正ループに戻さない。

- **ブロッカーなし**（指摘なし、または改善提案のみ） → Phase 4へ
- **ブロッカーあり かつ 3周目未満** → Phase 1bに戻る（ブロッカーの指摘を作業リストとして扱う）
- **3周目** → Phase 4へ（未解消ありとして報告）
- **同じ指摘が繰り返される**（振動パターン） → エスカレーション

改善提案はPhase 4のPR本文に記載し、人間のレビュアーに判断を委ねる。

### Phase 4: PR作成 + 完了報告

PRが未作成の場合は `gh pr create` で作成する。PRが既存の場合は変更をプッシュする。

PR本文にセルフレビュー結果のサマリを含める:

```markdown
## セルフレビュー実施済み

レビュー: {N}周実施、ブロッカー{M}件を修正済み

### 修正した指摘
| 重大度 | 指摘 | 対応内容 | コミット |
|---|---|---|---|
| ブロッカー | ... | ... | ... |
```

未解消のブロッカーがある場合は「未解消」セクションを追加し、指摘内容と未解消理由を記載する。改善提案がある場合は「改善提案（未対応）」セクションに記載し、人間のレビュアーに判断を委ねる。

## エスカレーション機構

自律的に判断できない場面（仕様の曖昧さ、振動パターン、リトライ上限超過）で、質問を投稿してセッションを終了する仕組み。対話セッション中は `AskUserQuestion` で即時確認し、非対話時（自動実行等）は以下のルールで投稿先を決定する:

| インプット種別 | 投稿先 |
|---|---|
| Issue起点 | Issueコメント（`gh issue comment`） |
| PRレビュー指摘起点 | PRコメント（`gh pr comment`） |
| 会話コンテキスト起点（Issue/PRなし） | WIPコミットして終了（投稿先がないため） |

このスキルの `references/escalation-template.md` のテンプレートに沿ってコメントを作成し、投稿してセッションを終了する。

## 注意事項

- **型チェック・テスト・lintコマンドの実行許可**: プロジェクトごとに異なるため `allowed-tools` には含めていない。`settings.json` の `allowedTools` でプロジェクトごとに許可設定すること（例: `Bash(npx tsc*)`, `Bash(npm test*)`, `Bash(npm run lint*)`）
- **複数行コンテンツの受け渡し**: `allowed-tools` の Bash パターンは HEREDOC 等の複雑なコマンドにマッチしない。複数行コンテンツ（コミットメッセージ、PR本文等）は HEREDOC を使わない方法で渡すこと（例: Write ツールで一時ファイルに書き出して `-F` や `--body-file` で渡す、`-m` を複数回指定する等）
- **pushの失敗**: `git push` が失敗した場合（権限不足、保護ブランチ等）は、ユーザーに状況を報告して手動pushを依頼する
