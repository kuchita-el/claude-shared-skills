# 汎用開発ループ

Issue・計画ファイル・PRレビュー指摘・会話コンテキストを起点に、実装→セルフレビュー→修正→PR作成までを一貫して実行する。

## 引数

- `$ARGUMENTS`: Issue番号/URL、計画ファイルパス、PR番号/URL、または省略（会話コンテキストを使用）

## ワークフロー

```
Phase 0: インプットの理解
Phase 1: 実装/修正
Phase 2: セルフレビュー
Phase 3: 判定 → 指摘なし→Phase 4 / 指摘あり・3周目未満→Phase 1 / 3周目→Phase 4
Phase 4: PR作成 + 完了報告
```

### Phase 0: インプットの理解

引数と会話コンテキストからインプット種別を自動判定する:

| 種別 | 判定条件 | 読み込み方法 |
|---|---|---|
| Issue | `#数字`、Issue URL、数字のみ | `gh issue view` |
| 計画ファイル | ファイルパス（`.md` 等） | `Read` |
| PRレビュー指摘 | PR番号/URL or 現在のブランチにPRが存在 | `gh pr view` + 会話コンテキスト |
| 会話コンテキスト | 上記いずれにも該当しない | 直前の会話から要件・指摘を抽出 |

読み込んだ内容から実装/修正すべき事項を整理し、作業リストを作成する。

### Phase 1: 実装/修正

作業リストに基づきコード変更を実施する。

1. **コード修正**: 影響範囲を確認してから修正する
2. **テスト実行**（許可されている場合）: 失敗時は原因を分析して修正（最大3回リトライ、解決しなければユーザーに報告）
3. **Lint実行**（許可されている場合）: 自動修正を試み、それでも直らなければ手動修正（最大3回リトライ）
4. **差分確認・ステージング**: 変更が意図通りか確認し、対象ファイルのみをステージング（`git add -A` は使わない）
5. **コミット**: テスト・lintが許可済みの場合は全てパスしてからコミットする。`git log` で直近のコミットスタイルを確認し、`CLAUDE.md` にコミットメッセージ規約があればそれに従う

テスト・lintコマンドの探索優先順位（コマンドが見つからない場合は `AskUserQuestion` でユーザーに確認する）:
1. `CLAUDE.md` の記載
2. `package.json` の scripts
3. `Makefile` のターゲット

### Phase 2: セルフレビュー

`git diff <ベースブランチ>...HEAD` の差分ベースでレビューする（全ファイル再読しない）。ベースブランチは `main`、`master`、`develop` 等プロジェクトに応じて判定する。

他のレビュースキル（`/pr-review-toolkit:review-pr` 等）がインストールされている場合はそれを活用し、なければ変更ファイルの差分を読み直してセルフレビューする。

レビュー観点:
- 要件の充足（インプットで求められた事項が全て実装されているか）
- バグ・セキュリティ問題
- コーディング規約違反

### Phase 3: 判定

- **指摘なし** → Phase 4へ
- **指摘あり かつ 3周目未満** → Phase 1に戻る（指摘を作業リストとして扱う）
- **3周目** → Phase 4へ（未解消ありとして報告）

### Phase 4: PR作成 + 完了報告

**PRが未作成の場合**: `gh pr create` でPRを作成する。PR本文にセルフレビュー結果のサマリを含める。

**PRが既存の場合**: 変更をプッシュする（`git push` はプロジェクト設定で許可が必要。注意事項を参照）。

完了報告フォーマット（PR本文に含める）:

```markdown
## セルフレビュー実施済み

レビュー: {N}周実施、指摘{M}件を修正済み

### 修正した指摘
| 指摘 | 対応内容 | コミット |
|---|---|---|
| ... | ... | ... |
```

未解消の指摘がある場合:

```markdown
## セルフレビュー実施済み（一部未解消）

レビュー: 3周実施、指摘{M}件中{X}件を修正済み

### 修正した指摘
| 指摘 | 対応内容 | コミット |
|---|---|---|
| ... | ... | ... |

### 未解消
| 指摘 | 未解消理由 |
|---|---|
| ... | ... |
```

## 注意事項

- **テスト・lint・pushコマンドの実行許可**: プロジェクトごとに異なるため `allowed-tools` には含めていない。`settings.json` の `allowedTools` でプロジェクトごとに許可設定すること（例: `Bash(npm test*)`, `Bash(npm run lint*)`, `Bash(git push*)`）
