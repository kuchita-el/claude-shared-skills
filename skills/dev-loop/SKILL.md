# 汎用開発ループ

Issue・計画ファイル・PRレビュー指摘・会話コンテキストを起点に、実装→セルフレビュー→修正→PR作成までを一貫して実行する。

## 引数

- `$ARGUMENTS`: Issue番号/URL、計画ファイルパス、PR番号/URL、または省略（会話コンテキストを使用）

## ワークフロー

```
Phase 0: インプットの理解（再開検知・エスカレーション履歴確認を含む）
Phase 1a: テスト設計・検証・生成（test-designer → test-spec-validator → テストコード生成）
Phase 1b: コード修正
Phase 1c: 検証ゲート → 失敗→Phase 1b / 全パスまたは全スキップ→Phase 1d
Phase 1d: ステージング・コミット
Phase 2: セルフレビュー
Phase 3: 判定 → ブロッカーなし→Phase 4 / ブロッカーあり・3周目未満→Phase 1b / 3周目→Phase 4
Phase 4: PR作成 + 完了報告

※ 全フェーズ共通: 判断ポイント発生時はエスカレーション（後述）
```

### Phase 0: インプットの理解

引数と会話コンテキストからインプット種別を自動判定する:

| 種別 | 判定条件 | 読み込み方法 |
|---|---|---|
| Issue | `#数字`、Issue URL、数字のみ | `gh issue view` |
| 計画ファイル | ファイルパス（`.md` 等） | `Read` |
| PRレビュー指摘 | PR番号/URL or 現在のブランチにPRが存在 | `gh pr view` + 会話コンテキスト |
| 会話コンテキスト | 上記いずれにも該当しない | 直前の会話から要件・指摘を抽出 |

読み込んだ内容から以下を確認・判定する。

#### エスカレーション履歴の確認

インプットがIssueの場合、Issueコメントから `<!-- dev-loop:escalation -->` マーカーを検索する。未回答のエスカレーション（マーカー後にコメントがない）があれば報告して終了する。回答済みのエスカレーション内容は作業リストの入力コンテキストに含める。

#### 実行環境の検知

現在のブランチ名から以下を判定する。

- **worktree環境**: worktree内で実行されている場合、ブランチ切り替えに制約がある（他worktreeで使用中のブランチにはcheckoutできない等）。制約に遭遇したら、現在のブランチ上で作業を進め、必要に応じて手動マージ手順を報告する
- **別セッション再開**: 作業ブランチ上にいる場合、コミット履歴と差分から現在の実装状況を把握し、未完了の作業を継続する

#### 作業リストとレビュー契約の作成

仕様に曖昧さや設計判断が必要な箇所があればエスカレーションする。問題なければ以下の2つを作成する。

**作業リスト（実装者向け）:** 実装/修正すべき事項の一覧。

**レビュー契約（レビュアー向け）:** インプットの受け入れ条件・完了条件から導出した、具体的なチェック項目のリスト。レビュー契約はPhase 2でcode-reviewerに渡され、機能要件の検証基準として使用する。

レビュー契約の導出ルール:
- インプットに受け入れ条件がある場合（Issue/計画ファイル）、各条件を検証可能なチェック項目に変換する
- 受け入れ条件がない場合、作業リストの各項目から「何が完了していれば正しいか」を導出する
- 各項目は「〜が存在する」「〜が返される」「〜が呼ばれる」等、差分から検証可能な形式で記述する

### Phase 1: 実装/修正/検証

#### Phase 1a: テスト設計・検証・生成

入力ソースからテスト仕様を設計し、独立検証を経てテストコードを生成するフェーズ。

1. **入力ソースの収集（3層構造）**: Phase 0で取得したIssue本文（層1）、計画ファイル（層2）を確認する。層3（追加ドキュメント）はユーザー指定があればそれに従い、なければ計画ファイル内の参照（「参照ドキュメント」セクション等のファイルパス・URL）をもとにReadで収集する
2. **test-designerの起動**: このスキルの `agents/test-designer.md` を読み込み、収集した層1〜3の内容を含めてプロンプトを構築する。Agent toolが使える場合はtest-designerサブエージェントに委任する。Agent toolが使えない場合や起動に失敗した場合はインラインで直接実行する
3. **結果に応じた分岐**:
   - テスト不要 → Phase 1bに進む
   - 矛盾検出 → エスカレーション
   - テスト仕様 → ステップ4へ
4. **テスト仕様の妥当性検証**: このスキルの `agents/test-spec-validator.md` を読み込み、テスト仕様と入力ソース（層1〜3）を渡してプロンプトを構築する。Agent toolが使える場合はtest-spec-validatorサブエージェントに委任する。Agent toolが使えない場合や起動に失敗した場合はインラインで直接実行する。差し戻し時は過去の検証結果を差し戻し履歴として渡す。結果に応じて分岐する:
   - 合格 → ステップ5へ
   - 不合格 → test-designerに検証結果をフィードバックとして渡し、ステップ2から再実行する。差し戻し上限は3回。3回を超えても合格しない場合はエスカレーションする
   - 仕様不明瞭 → エスカレーション（元の要件の曖昧さが原因であり、テスト仕様の改善では解決できない）
5. **テストコードの生成**: テスト仕様をもとにテストコードを一括生成する
6. **初回パス時の妥当性検証（#72連携）**: Phase 1bの実装前にテストが実行された際（#72のRed確認等）、全てのテストがパスした場合はtest-spec-validatorでテスト仕様を再検証する。再検証の結果に応じた分岐はステップ4と同じ。テスト実行コマンドが `allowedTools` で許可されていない場合はこのステップをスキップする

#### Phase 1b: コード修正

影響範囲を確認してからコードを修正する。Phase 1cからの再入時に同じ修正→失敗を繰り返している場合（振動パターン）はエスカレーションする。

#### Phase 1c: 検証ゲート

セルフレビューの前に、型チェック・lint・テストで機械的に検出できる問題を解消する。コマンドが見つからない場合や `allowedTools` で許可されていない場合はスキップしてよい。失敗したらPhase 1bに戻る。10回繰り返しても通らなければエスカレーションする。

#### Phase 1d: ステージング・コミット

対象ファイルのみをステージング（`git add -A` は使わない）してコミットする。コミットスタイルはプロジェクトの既存コミットや `CLAUDE.md` の規約に合わせる。

Phase 1内でエスカレーションが発動する場合は、エスカレーション手順の後にWIPコミットを行ってからセッションを終了する。次のセッションで作業を継続できるようにするため。

### Phase 2: セルフレビュー

人間のレビュアーに渡す前に、自分自身で差分をレビューして品質を高めるフェーズ。セルフレビューがこのスキルの主要な付加価値であり、スキップしてはならない。

このスキルの `agents/code-reviewer.md` を読み込み、そこに記載されたレビュー手順に従う。Agent toolが使える場合はcode-reviewerサブエージェントに委任し、以下のコンテキストを渡す:

- **ベースブランチ名**: 差分取得用
- **要件情報**: Issue本文/計画ファイルの内容
- **レビュー契約**: Phase 0で作成した完了チェックリスト

Agent toolが使えない場合（サブエージェントとして実行されている場合等）や起動に失敗した場合は、同じ手順をインラインで直接実行する。

レビュー結果に仕様の曖昧さで判断できない指摘が含まれる場合はエスカレーションする。

### Phase 3: 判定

レビュー結果の**ブロッカーの有無**に基づいて判定する。改善提案のみの場合は修正ループに戻さない。

- **ブロッカーなし**（指摘なし、または改善提案のみ） → Phase 4へ
- **ブロッカーあり かつ 3周目未満** → Phase 1bに戻る（ブロッカーの指摘を作業リストとして扱う）
- **3周目** → Phase 4へ（未解消ありとして報告）
- **同じ指摘が繰り返される**（振動パターン） → エスカレーション

改善提案はPhase 4のPR本文に記載し、人間のレビュアーに判断を委ねる。

### Phase 4: PR作成 + 完了報告

PRが未作成の場合は `gh pr create` で作成する。PRが既存の場合は変更をプッシュする（`git push` はプロジェクト設定で許可が必要。注意事項を参照）。

PR本文にセルフレビュー結果のサマリを含める:

```markdown
## セルフレビュー実施済み

レビュー: {N}周実施、ブロッカー{M}件を修正済み

### 修正した指摘
| 重大度 | 指摘 | 対応内容 | コミット |
|---|---|---|---|
| ブロッカー | ... | ... | ... |
```

未解消のブロッカーがある場合は「未解消」セクションを追加し、指摘内容と未解消理由を記載する。改善提案がある場合は「改善提案（未対応）」セクションに記載し、人間のレビュアーに判断を委ねる。

## エスカレーション機構

自律的に判断できない場面（仕様の曖昧さ、振動パターン、リトライ上限超過）で、GitHub Issueへ質問コメントを投稿してセッションを終了する仕組み。対話セッション中は `AskUserQuestion` で即時確認し、非対話時（自動実行等）にIssueコメントを使う。

このスキルの `references/escalation-template.md` のテンプレートに沿ってコメントを作成し、Issueに投稿してセッションを終了する。

## 注意事項

- **型チェック・テスト・lint・pushコマンドの実行許可**: プロジェクトごとに異なるため `allowed-tools` には含めていない。`settings.json` の `allowedTools` でプロジェクトごとに許可設定すること（例: `Bash(npx tsc*)`, `Bash(npm test*)`, `Bash(npm run lint*)`, `Bash(git push*)`）
- **複数行コンテンツの受け渡し**: `allowed-tools` のBashパターンはヒアドキュメント等の複雑なコマンドにマッチしない。複数行のコミットメッセージやPR本文は Write ツールで一時ファイル（`/tmp/dev-loop-{timestamp}.md`）に書き出してから `-F` や `--body-file` で渡すこと
