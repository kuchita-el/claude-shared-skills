# 汎用開発ループ

Issue・計画ファイル・PRレビュー指摘・会話コンテキストを起点に、実装→セルフレビュー→修正→PR作成までを一貫して実行する。

## 引数

- `$ARGUMENTS`: Issue番号/URL、計画ファイルパス、PR番号/URL、または省略（会話コンテキストを使用）

## ワークフロー

```
Phase 0: インプットの理解（再開検知・エスカレーション履歴確認を含む）
Phase 1a: コード修正
Phase 1b: ステージング・コミット
Phase 1c: 検証ゲート → 失敗→Phase 1a / 全パス→Phase 2 / 未許可→警告してPhase 2
Phase 2: セルフレビュー
Phase 3: 判定 → 指摘なし→Phase 4 / 指摘あり・3周目未満→Phase 1a / 3周目→Phase 4
Phase 4: PR作成 + 完了報告

※ 全フェーズ共通: 判断ポイント発生時はエスカレーション（Issueコメント投稿→セッション終了）
```

### Phase 0: インプットの理解

引数と会話コンテキストからインプット種別を自動判定する:

| 種別 | 判定条件 | 読み込み方法 |
|---|---|---|
| Issue | `#数字`、Issue URL、数字のみ | `gh issue view` |
| 計画ファイル | ファイルパス（`.md` 等） | `Read` |
| PRレビュー指摘 | PR番号/URL or 現在のブランチにPRが存在 | `gh pr view` + 会話コンテキスト |
| 会話コンテキスト | 上記いずれにも該当しない | 直前の会話から要件・指摘を抽出 |

読み込んだ内容から以下のサブステップを順に実行する。

#### エスカレーション履歴の確認

インプットがIssueの場合、`gh issue view {number} --json comments` で取得済みのコメントから `<!-- dev-loop:escalation -->` マーカーを検索する。

- マーカー付きコメント（エスカレーション）を発見した場合、そのコメント以降の全コメントを「回答」として読み取る
- **未回答のエスカレーション**（マーカー付きコメントの後にコメントがない）が存在する場合、「未回答の質問があります」と報告してセッションを終了する
- 全エスカレーション→回答の内容を作業リストの入力コンテキストに含める

#### 別セッション再開の検知

`git status` で現在のブランチ名を確認し、`feature/{issue番号}-*` パターンに一致するか判定する。

- 一致する場合（= 前回セッションの作業ブランチ上にいる）、`git log main..HEAD` でコミット履歴、`git diff main...HEAD` で差分を確認して現在の実装状況を把握する
- 把握した内容を作業リストに反映（未完了の作業を特定して継続）

#### 仕様の曖昧さによるエスカレーション

インプットの理解過程で仕様の曖昧さや設計判断が必要な箇所を発見した場合、「エスカレーション機構」セクションの手順に従いエスカレーションを発動する。

#### 作業リストの作成

上記を踏まえ、実装/修正すべき事項を整理して作業リストを作成する。

### Phase 1: 実装/修正/検証

作業リストに基づきコード変更→コミット→検証を実施する。

#### Phase 1a: コード修正

1. **コード修正**: 影響範囲を確認してから修正する
2. **振動パターンの検出**: 同じ修正→失敗を繰り返している場合（テストやlintで同一エラーが再発する等）、「エスカレーション機構」セクションの手順に従いエスカレーションを発動する

#### Phase 1b: ステージング・コミット

1. **差分確認・ステージング**: 変更が意図通りか確認し、対象ファイルのみをステージング（`git add -A` は使わない）
2. **コミット**: `git log` で直近のコミットスタイルを確認し、`CLAUDE.md` にコミットメッセージ規約があればそれに従う。複数行のコミットメッセージは Write ツールで一時ファイルに書き出し `git commit -F <file>` で渡す

#### Phase 1c: 検証ゲート

機械的に排除可能な問題をレビュー前に解消するための検証ゲート。

**コマンド許可の前提確認**: 型チェック・lint・テストの各コマンドが `settings.json` の `allowedTools` で許可されているかを確認する。いずれも許可されていない場合は「検証ゲートのコマンドが許可されていません。検証をスキップしてレビューに進みます」と警告を出力し、Phase 2に続行する。

**第1層: 型チェック + lint**
- 型チェックコマンドとlintコマンドを両方実行する
- いずれかが失敗した場合、第2層をスキップしてPhase 1aに戻る

**第2層: テスト**
- テストコマンドを実行する
- 失敗した場合、Phase 1aに戻る
- 全パスした場合のみPhase 2に進む

**リトライ上限**: Phase 1a→1b→1cのループを10回繰り返しても全パスしない場合、「エスカレーション機構」セクションの手順に従いエスカレーションを発動する。

**コマンド探索優先順位**（型チェック・lint・テストの3種とも同一）:
1. `CLAUDE.md` の記載
2. `package.json` の scripts / `Makefile` のターゲット
3. `AskUserQuestion` でユーザーに確認

### Phase 2: セルフレビュー

`git diff <ベースブランチ>...HEAD` の差分ベースでレビューする（全ファイル再読しない）。ベースブランチは `main`、`master`、`develop` 等プロジェクトに応じて判定する。

他のレビュースキル（`/pr-review-toolkit:review-pr` 等）がインストールされている場合はそれを活用し、なければ変更ファイルの差分を読み直してセルフレビューする。

レビュー観点:
- 要件の充足（インプットで求められた事項が全て実装されているか）
- バグ・セキュリティ問題
- コーディング規約違反

レビュー中に仕様の曖昧さにより判断できない指摘がある場合、「エスカレーション機構」セクションの手順に従いエスカレーションを発動する。

### Phase 3: 判定

- **指摘なし** → Phase 4へ
- **指摘あり かつ 3周目未満** → Phase 1aに戻る（指摘を作業リストとして扱う）
- **3周目** → Phase 4へ（未解消ありとして報告）
- **振動パターン検出**（同じ指摘が繰り返される） → 「エスカレーション機構」セクションの手順に従いエスカレーションを発動する

### Phase 4: PR作成 + 完了報告

**PRが未作成の場合**: PR本文を Write ツールで一時ファイルに書き出し、`gh pr create --body-file <file>` でPRを作成する。PR本文にセルフレビュー結果のサマリを含める。

**PRが既存の場合**: 変更をプッシュする（`git push` はプロジェクト設定で許可が必要。注意事項を参照）。

完了報告フォーマット（PR本文に含める）:

```markdown
## セルフレビュー実施済み

レビュー: {N}周実施、指摘{M}件を修正済み

### 修正した指摘
| 指摘 | 対応内容 | コミット |
|---|---|---|
| ... | ... | ... |
```

未解消の指摘がある場合:

```markdown
## セルフレビュー実施済み（一部未解消）

レビュー: 3周実施、指摘{M}件中{X}件を修正済み

### 修正した指摘
| 指摘 | 対応内容 | コミット |
|---|---|---|
| ... | ... | ... |

### 未解消
| 指摘 | 未解消理由 |
|---|---|
| ... | ... |
```

## エスカレーション機構

判断ポイント発生時にGitHub Issueへ質問コメントを投稿し、セッションを終了する仕組み。

### AskUserQuestionとの関係

並存する。対話セッション中は従来通り `AskUserQuestion` で即時確認し、非対話時（自動実行等）はIssueコメントでエスカレーションする。

### エスカレーション手順

1. `skills/dev-loop/references/escalation-template.md` をReadツールで読み込み、テンプレートに沿ってコメント本文を作成する
2. Write ツールで `/tmp/dev-loop-escalation-{timestamp}.md` に書き出す
3. `gh issue comment {number} --body-file /tmp/dev-loop-escalation-{timestamp}.md` で投稿する
4. 投稿成功後、即座にセッションを終了する（以降のフェーズは実行しない）

投稿が失敗した場合はエラーメッセージをユーザーに報告して終了する（リトライ不要）。

## 注意事項

- **型チェック・テスト・lint・pushコマンドの実行許可**: プロジェクトごとに異なるため `allowed-tools` には含めていない。`settings.json` の `allowedTools` でプロジェクトごとに許可設定すること（例: `Bash(npx tsc*)`, `Bash(npm test*)`, `Bash(npm run lint*)`, `Bash(git push*)`）
- **複数行コンテンツの受け渡し**: `allowed-tools` のBashパターンはヒアドキュメント等の複雑なコマンドにマッチしない。複数行のコミットメッセージやPR本文は Write ツールで一時ファイル（`/tmp/dev-loop-{timestamp}.md`）に書き出してから `-F` や `--body-file` で渡すこと
