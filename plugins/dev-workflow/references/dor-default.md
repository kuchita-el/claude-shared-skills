# Definition of Ready (DoR)

Issueが作業開始可能かどうかを判定するための基準です。

これはデフォルト定義であり、最小限の共通項目のみを含みます。プロジェクト固有の要件（ACの必須化、技術設計の事前確定など）がある場合は `{project}/.claude/dor/definition.md` で拡張してください。

## チェック項目

全サイズ共通の項目です。

| 項目 | 説明 |
| --- | --- |
| 課題・目的が明確 | 何を解決/実現したいか、なぜ必要かがわかる |
| ゴールがわかる | Issueの目的が達成された状態を一文で要約できる（WHY/価値。検証手段は問わない。後述「ゴールと受入条件（AC）の役割分担」参照） |
| 受入条件が検証可能な形で定義されている | ゴールが表す状態を検証可能・客観的に分解した条件（AC）が1件以上記述され、各ACが「検証可能（verifiable）」かつ「客観的（objective）」である |
| 適正な抽象度 | IssueがWHAT/WHYに留まり、HOW（実現方法）が混入していない（solution-bearing種別に適用。後述「適正な抽象度 判定基準」参照） |
| スコープの輪郭がある | 大まかな対象範囲（IN/OUT）が示されている |
| ブロッカーが把握されている | 作業開始を阻む依存・前提が特定・追跡されている（既知 or なし が明示。後述「ブロッカー把握の判定基準」参照） |

「受入条件が検証可能な形で定義されている」は、AC自体の存在に加え、各ACの検証可能性・客観性まで含めて評価する。判定基準は後述「受入条件の検証可能性・客観性 判定基準」節を参照する。

### ゴールと受入条件（AC）の役割分担

「ゴール」と「受入条件（AC）」はいずれも完了に関わるが、役割が異なる。重複させず、それぞれの役割で記述する。

- **ゴール**: Issueの目的が達成された状態の一文要約（WHY/価値）。何のために・どうなれば価値が実現するかを述べ、検証手段は問わない。
- **受入条件（AC）**: そのゴール状態を検証可能・客観的に分解した合否条件。ゴールを「どう確かめるか」へ翻訳したもの。

判定指針: ゴールがACの単なる列挙・言い換えに堕していないかを確認する。ゴール欄に検証条件が箇条書きされACと同義になっている場合は重複とみなし、ゴールをWHY/価値の一文へ、検証条件をAC側へ振り分けるよう促す。逆に、ゴールが価値を・ACが検証条件を述べて役割が分離していれば、両者が完了に言及していても重複とはみなさない。

## 受入条件の検証可能性・客観性 判定基準

精査ではAC毎に以下の2軸でそれぞれ判定する。AC全体ではなく**1項目ずつ**評価し、不合格項目があれば改善提案の対象とする。

### 検証可能（verifiable）

以下のいずれかを満たす場合、検証可能と判定する。

- 観測可能な事象（UI表示・ログ・APIレスポンス・ファイル出力等）として表現されている
- 自動テスト（ユニット／統合／E2E）に置き換えられる入出力対が記述可能
- 合否を客観判定できる入力と期待出力のペアが具体例として示せる
- 時間・回数・容量等の定量閾値が含まれる
- 状態変化の前後がそれぞれ定義されている

逆に、以下のような表現は検証不可と判定する。

- 「動作が安定する」「自然に使える」「期待通りに動く」等、観測対象が特定できない
- 「考慮する」「対応する」「サポートする」等、完了の境界が示されない
- 「適切に処理する」「正しく動く」等、判定基準を別途定義しないと合否判定できない

### 客観的（objective）

以下を全て満たす場合、客観的と判定する。

- 主観的な形容詞（満足／快適／安定／適切／自然／使いやすい／違和感がない 等）を含まない
- 評価者によって判断が分かれない表現になっている
- 定量・定性の閾値や境界が明記されている、もしくは別途参照する基準が指定されている

逆に、以下のような表現は主観的と判定する。

- 「ユーザーが満足する」「快適に動作する」等、評価者依存の判断を含む
- 「過度に遅くない」「許容範囲内」等、境界が明示されない
- 「使いやすい」「直感的」等、観測指標と切り離された定性表現

## ブロッカー把握の判定基準

「ブロッカーが把握されている」は、ブロッカーの**不在**ではなく**把握**を求める。着手可否（now）と記述品質（ready）を分離し、未解決の依存があっても特定・追跡されていれば Ready とする。

- ✅ 把握済み（未解決可）: 作業開始を阻む依存・前提が特定され追跡されている。未解決でもよい（例: 「依存先 #123 が未マージだが、本文で前提として追跡明記」→ 充足）。
- ✅ 不在の明示: ブロッカーが無いことが明示されている。
- ❌ 未把握: ブロッカーの有無自体が本文から判断できない。依存の見落としの可能性があり、未充足とする。

歯止め: 把握化は「ブロッカー項目の無条件パス」ではない。有無が不明な状態は Not Ready のまま据え置く。把握されている（既知 or なし が明示）ことを満たして初めて充足とする。

## 適正な抽象度 判定基準

Issueは WHAT（何を）／WHY（なぜ）を定義する単位であり、HOW（実現方法）の確定はプランニング・実装フェーズへ委ねる。HOW が本文に混入していないかを判定する。

- ❌ HOW混入の例: 「X ライブラリで実装する」「このクラスを継承して〜」等、特定の実装方式・技術選定を完了条件や本文で固定している（解決方法が一意に縛られている）。
- ✅ WHAT/WHY に留まる例: 「〜できる状態にする（理由: 〜）」と達成状態と価値を述べ、実現方式は問わない。

適用範囲: 本項目は **solution-bearing種別（bug / feature / refactor / chore 等、解決手段＝コード・成果物を内包するIssue）** に適用する（厳しさは種別により異なり、feature が最も厳しく、refactor / chore は緩い。詳細は #291 の種別プロファイル）。**spike / docs 等、HOW の検討・文書化そのものが成果物となる種別には適用しない**（観測対象がHOWであり、HOW排除を求めるのは矛盾するため）。

適用対象種別の最終的な判定は Issue種別軸（#291）へ委ねる。ただし種別軸の導入前（#291 反映前）で形式的な種別検出が無い段階でも、精査・作成側はIssue本文の主旨から種別を判断し、調査・文書化を主目的とするIssue（spike / docs 相当）には本項目を適用しない。これにより、種別検出の配線が無い窓でも spike / docs 的Issueへの誤った不足指摘を避ける。

## サイズ判定ロジック

### 1. ラベル優先

以下のラベルがある場合、そのサイズとして判定:

- `size:small` → Small
- `size:medium` → Medium
- `size:large` → Large

### 2. 本文解析（ラベルがない場合）

| サイズ | 判定条件 |
| ------ | -------- |
| Small  | 「バグ」「bug」「typo」「軽微」を含む、または本文500文字未満かつAC2項目以下 |
| Large  | 「アーキテクチャ」「設計変更」「大規模」を含む、または子Issue参照（`#` + 数字）が3つ以上、または設計の根本構造（責務境界・契約・アーキテクチャ原則）に踏み込む論点を含む |
| Medium | 上記以外 |

#### 子Issue参照のカウント対象

「子Issue」「分割先Issue」として参照されているもののみカウント対象とする。「前提Issue」「関連Issue」「依存先」への言及はカウントしない。

## Ready 判定基準

- **Ready**: チェック項目を全て満たしている
- **Not Ready**: 1つ以上のチェック項目を満たしていない

適用条件付きの項目（「適正な抽象度」は solution-bearing種別に適用）は、適用対象外の種別では評価から除外し、Ready判定の必須条件に数えない。
