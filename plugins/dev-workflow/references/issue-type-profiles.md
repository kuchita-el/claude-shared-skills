# Issue種別プロファイル

Issueの**種別**に応じて、準備基準（DoR）とテンプレートを差別化するための定義です。

コアDoR（`dor-default.md`）が全種別共通の土台を定義するのに対し、本ファイルはその上に**種別差分**を重ねます。サイズ軸（Small/Medium/Large）とは独立した**二軸**であり、サイズ判定を置換しません（ADR-20260618 Decision (a)）。

これはデフォルト定義です。プロジェクト固有の種別プロファイルがある場合は `{project}/.claude/dor/type-profiles.md` で上書き・拡張してください（読み込み優先順位は DoR と同一）。

## 種別検出ロジック

サイズ判定ロジック（`dor-default.md`）と同型で、ラベル優先 → 本文推論 → 推論不能時の既定、の順で判定する。

### 1. ラベル優先

以下の `type:*` ラベルがある場合、その種別として判定する:

- `type:bug` → bug
- `type:feature` → feature
- `type:refactor` → refactor
- `type:spike` → spike
- `type:chore` → chore
- `type:docs` → docs

### 2. 本文推論（ラベルがない場合）

本文の**主旨**が以下のいずれかの種別に明確に該当する場合に、その種別と推論する。キーワードは判断の手がかり（例）であり、語の単純な包含で機械的に確定しない。各種別の「高確証シグナル」はその種別に固有で、他種別と取り違えにくい語に絞る。

| 種別 | 高確証シグナル（その種別に固有な手がかり） |
| --- | --- |
| bug | 不具合 / バグ / 再現手順 / エラーが出る / クラッシュ / 期待と異なる挙動 |
| spike | 調査 / 技術検証 / PoC / 技術選定 / 比較検討 / ADR起票 / 方針を決める |
| docs | ドキュメント / README / 文書化 / 利用ガイド作成 |
| refactor | リファクタリング / refactor / 内部構造の改善 / 構造改善（振る舞い不変） |
| chore | 依存更新 / バージョン上げ / CI設定 / ビルド設定 / 定型作業 |
| feature | 機能追加 / 新機能 / 〜できるようにする |

**汎用語による誤確定の禁止**: 「設定」「整理」「追加」「直す」等の汎用語は複数種別・featureと両立しやすいため、単独一致では種別を確定しない。特に **feature と具体種別（bug/refactor/chore等）が両立する場合**（例: 「ユーザー設定機能を追加したい」は feature だが「設定」が chore と、「追加」が feature と一致しうる）は、無理に具体種別へ寄せず §3 の既定（種別差分を非適用）に落とす。具体種別へ一意化してよいのは、その種別の高確証シグナルが明確で、feature を含む他種別と競合しない場合に限る。

> 設計意図: 誤分類は「非適用」より悪い。誤った種別へ確定すると、その種別の緩和（例: spike の観測可能AC緩和）や抽象度の厳しさ（例: feature の「高」が chore の「低」へ）を**能動的に誤適用**し、本来排除すべき HOW 混入の見逃し等を招く。一意に定まらないときは非適用（安全側）へ倒す。

### 3. 推論不能時の既定

`type:*` ラベルが無く、本文推論でも種別が一意に定まらない場合（汎用語のみの一致、または複数種別が競合する場合を含む）は、**種別差分を適用しない**（サイズ軸のみで動作する）。誤った種別固有セクションを強制ドラフトしたり、種別緩和を誤適用したりしないための安全側の既定とする。

## 種別プロファイル

各種別について {追加必須セクション・AC形/完了定義・適正な抽象度の厳しさ} の3要素を定義する。「適正な抽象度」の厳しさは、コアDoR（`dor-default.md`）の「適正な抽象度」項目を種別ごとにどの程度厳しく適用するかを示す。本ファイルは厳しさ値を提供する単一の源であり、HOW混入防止の高度ガードレール（#292）は `create-issue`（draft-first のHOW書換/退避）と `refine-issue`（refine-prompt.md「実現方法の混入（HOW混入）の指摘」）がこの厳しさ値を参照して配線する。

| 種別 | 追加必須セクション | AC形・完了定義 | 適正な抽象度の厳しさ |
| --- | --- | --- | --- |
| bug | 再現手順（環境・再現手順・期待挙動／実際の挙動） | 再現条件と期待挙動の対。原因・修正方法は未知でよい（完了＝期待挙動が再現環境で確認できる） | 中（特定の修正方式の固定は避けるが、原因調査の余地は残す） |
| feature | なし（コアDoR標準セクションで充足） | 観測可能な入出力対。ユーザー価値が実現した状態 | 高（HOW混入を最も厳しく排除し、WHAT/WHY に留める） |
| refactor | 対象範囲（対象モジュール／境界）・振る舞い不変の宣言 | 外部から見た振る舞いが不変であること（既存の挙動・テストが維持される）＋内部構造の改善目標 | 低（リファクタは本質的に HOW 寄り。実装方式への言及を許容する） |
| spike | 調査の問い（何を明らかにするか）・完了の成果物（決定 or 文書） | 観測可能な入出力対は不要。完了＝決定が下る／文書ができる。**観測可能AC不在を不足としない** | 適用外（HOW の検討自体が成果物。ADR起票もこの種別で扱う） |
| chore | なし | 作業完了の客観的確認（例: 依存バージョン更新＋CI通過）。軽量でよい | 低 |
| docs | 記述対象・読者・配置（どこに書くか） | 記述対象・読者・配置が明確で、文書が作成／更新された状態。観測可能な入出力対は不要 | 適用外（文書作成が成果物） |

### 種別緩和の要点

精査（`refine-issue`）・作成（`create-issue`）で種別プロファイルを適用する際の主な分岐:

- **bug**: 再現手順セクションの有無を必須として確認する。欠落していれば不足として指摘する。一方、原因・修正方法の記述は完了条件としない。
- **spike**: 「ACが観測可能な入出力対でない」ことを**不足として指摘しない**（完了＝決定/文書化のため）。調査の問いと成果物の定義の有無を確認する。
- **docs**: 観測可能AC不在を不足としない。記述対象・読者・配置の明確さを確認する。
- **refactor**: 振る舞い不変の宣言の有無を確認する。実装方式への言及は抽象度違反としない。

**HOW混入指摘の種別分岐**（タイトル・ゴール・ACへの指定的HOWの混入を `refine-issue` が助言指摘する際の分岐。`create-issue` の書換/退避の厳しさも同じ厳しさ値に連動する）:

- **指摘する** — 高（feature）/ 中（bug）: HOW混入を検出し助言する（bug は特定の修正方式の固定のみ。原因調査の余地は残す）。
- **完全抑制（1件も指摘しない）** — 低（refactor / chore）/ 適用外（spike / docs）: HOW的記述を「実現方法の混入」として指摘しない。refactor / chore は実装方式への言及を許容、spike / docs はHOW検討自体が成果物のため。
- いずれも**助言であり Ready 判定には影響しない**（非ブロッカー）。完全抑制は暫定方針であり、運用後に部分抑制へ再調整しうる。
- 本分岐は非ブロッカー助言の範囲であり、コアDoR「適正な抽象度」項目（`dor-default.md`、Ready判定に影響）の適用可否とは**別レイヤ**である。コア項目は solution-bearing 種別に継続適用され（refactor / chore は緩く評価）、本助言の完全抑制はそれを置換しない。

## プロジェクト固有プロファイルによる上書き

DoR（`dor-default.md` / `.claude/dor/definition.md`）と同一の読み込み優先順位に従う（ADR-20260604）:

1. プロジェクト固有: `{project}/.claude/dor/type-profiles.md`（存在すれば優先）
2. プラグイン共有デフォルト: `${CLAUDE_PLUGIN_ROOT}/references/issue-type-profiles.md`

プロジェクト固有ファイルが読み込めない場合は共有デフォルトにフォールバックする。種別プロファイル資源そのものが読めない場合は、種別差分を適用せずサイズ軸のみで動作する（ソフトデグレード）。
