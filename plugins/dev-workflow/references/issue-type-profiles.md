# Issue種別プロファイル

Issueの**種別**に応じて、準備基準（DoR）とテンプレートを差別化するための定義です。

コアDoR（`dor-default.md`）が全種別共通の土台を定義するのに対し、本ファイルはその上に**種別差分**を重ねます。サイズ軸（Small/Medium/Large）とは独立した**二軸**であり、サイズ判定を置換しません（ADR-20260618 Decision (a)）。

これはデフォルト定義です。プロジェクト固有の種別プロファイルがある場合は `{project}/.claude/dor/type-profiles.md` で上書き・拡張してください（読み込み優先順位は DoR と同一）。

## 種別検出ロジック

サイズ判定ロジック（`dor-default.md`）と同型で、ラベル優先 → 本文推論 → 推論不能時の既定、の順で判定する。

### 1. ラベル優先

以下の `type:*` ラベルがある場合、その種別として判定する:

- `type:bug` → bug
- `type:feature` → feature
- `type:refactor` → refactor
- `type:spike` → spike
- `type:chore` → chore
- `type:docs` → docs

### 2. 本文推論（ラベルがない場合）

本文に以下のキーワードが含まれる場合、対応する種別と推論する。複数一致する場合は、より具体的な種別（bug/spike/docs/refactor/chore）を feature より優先する。

| 種別 | 推論キーワード（例） |
| --- | --- |
| bug | バグ / bug / 不具合 / 再現 / エラー / 動かない / 直す |
| spike | 調査 / 検証 / PoC / 技術選定 / 比較検討 / ADR / 方針決定 |
| docs | ドキュメント / README / 文書化 / ガイド / 説明を追加 |
| refactor | リファクタリング / refactor / 整理 / 構造改善 / 内部構造 |
| chore | 依存更新 / 設定 / CI / ビルド / バージョン上げ / 雑務 |
| feature | 機能追加 / 新機能 / 実装したい |

### 3. 推論不能時の既定

`type:*` ラベルが無く、本文推論でも種別が一意に定まらない場合は、**種別差分を適用しない**（サイズ軸のみで動作する）。誤った種別固有セクションを強制ドラフトしたり、種別緩和を誤適用したりしないための安全側の既定とする。

## 種別プロファイル

各種別について {追加必須セクション・AC形/完了定義・適正な抽象度の厳しさ} の3要素を定義する。「適正な抽象度」の厳しさは、コアDoR（`dor-default.md`）の「適正な抽象度」項目を種別ごとにどの程度厳しく適用するかを示す（HOW混入防止の高度ガードレールは #292 で詰める。本ファイルは厳しさ値のみ提供する）。

| 種別 | 追加必須セクション | AC形・完了定義 | 適正な抽象度の厳しさ |
| --- | --- | --- | --- |
| bug | 再現手順（環境・再現手順・期待挙動／実際の挙動） | 再現条件と期待挙動の対。原因・修正方法は未知でよい（完了＝期待挙動が再現環境で確認できる） | 中（特定の修正方式の固定は避けるが、原因調査の余地は残す） |
| feature | なし（コアDoR標準セクションで充足） | 観測可能な入出力対。ユーザー価値が実現した状態 | 高（HOW混入を最も厳しく排除し、WHAT/WHY に留める） |
| refactor | 対象範囲（対象モジュール／境界）・振る舞い不変の宣言 | 外部から見た振る舞いが不変であること（既存の挙動・テストが維持される）＋内部構造の改善目標 | 低（リファクタは本質的に HOW 寄り。実装方式への言及を許容する） |
| spike | 調査の問い（何を明らかにするか）・完了の成果物（決定 or 文書） | 観測可能な入出力対は不要。完了＝決定が下る／文書ができる。**観測可能AC不在を不足としない** | 適用外（HOW の検討自体が成果物。ADR起票もこの種別で扱う） |
| chore | なし | 作業完了の客観的確認（例: 依存バージョン更新＋CI通過）。軽量でよい | 低 |
| docs | 記述対象・読者・配置（どこに書くか） | 記述対象・読者・配置が明確で、文書が作成／更新された状態。観測可能な入出力対は不要 | 適用外（文書作成が成果物） |

### 種別緩和の要点

精査（`refine-issue`）・作成（`create-issue`）で種別プロファイルを適用する際の主な分岐:

- **bug**: 再現手順セクションの有無を必須として確認する。欠落していれば不足として指摘する。一方、原因・修正方法の記述は完了条件としない。
- **spike**: 「ACが観測可能な入出力対でない」ことを**不足として指摘しない**（完了＝決定/文書化のため）。調査の問いと成果物の定義の有無を確認する。
- **docs**: 観測可能AC不在を不足としない。記述対象・読者・配置の明確さを確認する。
- **refactor**: 振る舞い不変の宣言の有無を確認する。実装方式への言及は抽象度違反としない。

## プロジェクト固有プロファイルによる上書き

DoR（`dor-default.md` / `.claude/dor/definition.md`）と同一の読み込み優先順位に従う（ADR-20260604）:

1. プロジェクト固有: `{project}/.claude/dor/type-profiles.md`（存在すれば優先）
2. プラグイン共有デフォルト: `${CLAUDE_PLUGIN_ROOT}/references/issue-type-profiles.md`

プロジェクト固有ファイルが読み込めない場合は共有デフォルトにフォールバックする。種別プロファイル資源そのものが読めない場合は、種別差分を適用せずサイズ軸のみで動作する（ソフトデグレード）。
