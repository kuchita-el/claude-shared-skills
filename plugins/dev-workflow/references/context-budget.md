# コンテキスト予算規約（スキル実行時）

スキル起動後、メインセッションの context に何を載せ、何を載せないかを判断するための単一の出典。各スキルはこの規約を参照し、自身の経路（参照ファイル読み込み・サブエージェント委譲・ツール出力処理）に適用する。

## 背景

スキル起動後のメイン context 消費量が、起動側モデルの圧縮設定に依存して挙動が割れる。圧縮せずに済むタスクが別モデルでは圧縮を要し、スキルの途中で圧縮が走るとスキル指示・中間状態が失われて挙動が不安定になる。メイン context を最小に保てば、この起動側モデル依存を縮減できる。

## 二軸の区別（二重管理の回避）

token に関わる規律は2軸ある。本規約は**実行時フロー軸**を扱い、作成時の静的フットプリント軸は CLAUDE.md「スキル設計の token 規律」が扱う。両者は補完関係であり、内容を重複させない。

| 軸 | 出典 | 縛る対象 |
|---|---|---|
| 作成時（静的） | CLAUDE.md「スキル設計の token 規律」 | `description` 長・SKILL.md 行数・`allowed-tools` 最小化（スキルという成果物の静的サイズ） |
| 実行時（フロー） | 本規約 | スキル実行中にメイン context を通過・滞留するデータ（下記5要素） |

## 適用対象

スキル実行中にメイン context に載りうる5要素:

- (a) SKILL.md 本体
- (b) references からメインで Read した補助ファイル
- (c) サブエージェント返却レポート
- (d) gh / git 等のツール生出力
- (e) 生成中間成果物

(a) は作成時軸（SKILL.md 行数規律）が縛る。本規約は (b)(c)(d)(e) をメインから排除するフローを定める。

## 原則

### 1. パス渡し原則

references・Issue・生成成果物等は、本文をメインへ Read せず、**パス／識別子のみ**を処理するエージェントへ渡す。本文が必要なエージェントが自前で Read する。(b)(e) に適用。

### 2. 生ツール出力のファイル化

gh / git 等の生 stdout をメインに残さない。後続処理が必要な出力はファイルへ書き出し、メインにはパスのみ残す。即時に使う最小限の値（番号・件数等）だけをメインで解釈する。(d) に適用。

### 3. サブエージェント返却のサマリ化

サブエージェントの返却は**構造化サマリ**に限定する。中間生成物の本文・ファイル全文・生出力を返却させない。複数件処理はサブエージェント側で集約し、メインへはサマリのみ返す。(c) に適用。

### 4. 滞留カテゴリ規則（ピーク上限の導出）

スキル実行後にメイン context へ**滞留してよいのは (a) SKILL.md 本体と構造化サマリのみ**とする。(b)(c)(d)(e) は net で滞留させない（一時的に経由しても、処理後はメインに残さない）。

この規則により、起動後ピーク context の上限は**新たな数値を定めずに導出される**: 上限 ≒ SKILL.md 本体（CLAUDE.md token 規律で約 170 行に制限済み）＋ 有界な構造化サマリ。スカラーの行数／トークン上限は設けない（実行中の測定困難・モデル依存のトークン化・閾値のドリフトを避けるため）。

## 後続改修が参照する基準節

本規約確定後の個別スキル改修は、以下の対応で本規約の各節を判断基準とする:

| 後続改修 | 基準節 |
|---|---|
| 参照ファイルのパス渡し統一 | 原則1（パス渡し原則） |
| Issue 取得のファイル化＋comments 絞り | 原則2（生ツール出力のファイル化） |
| サブエージェント返却契約の明文化 | 原則3（サブエージェント返却のサマリ化） |
| SKILL.md 本体の縮退 | 原則4（滞留カテゴリ規則）＋ CLAUDE.md「スキル設計の token 規律」 |

## 理想形の実例

`refine-issue` 全件モード（`skills/refine-issue/SKILL.md`）が本規約の体現例。メイン側はパス・Issue 番号のみをサブエージェントへ渡し、サブエージェントが必要ファイルを自前 Read し、精査結果を構造化データで返却する。新規・改修スキルはこの型を範とする。
