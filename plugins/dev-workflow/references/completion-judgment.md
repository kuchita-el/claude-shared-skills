# 完了判定の原則（早期収束の防止）

複数の判定項目を持つタスクで、一部項目の解決をもって全体を「完了」「Ready」と判定してしまう **早期収束** を防ぐための共通原則。

本リポジトリの各スキルが参照する単一の出典であり、各スキルは自身の文脈（精査・レビュー・完了報告等）に合わせてこの原則を具体化する。

## 適用層

`loop`（ループ完了）/ `issue`（Issue 完了）/ `refinement`（DoR 精査）/ `inquiry`（調査の問い）の 4 層に適用される。本原則は層中立であり、どの層に適用するかは呼び出し側（各スキル）が指定する。ただし `inquiry` 層のみ決着値の値域が拡張される（後述「調査の問いの層における決着値」）。

## 適用対象

複数の判定項目・チェック項目・指摘事項を持つタスク。例:

- `refine-issue`: DoR チェック項目
- `implementation`: レビュー契約のチェック項目、エスカレーション項目、完了報告

## 原則

### 1. 全項目を列挙して評価する（省略禁止）

判定項目はチェックリストとして全て列挙し、評価結果（合格/不合格）を1件ずつ明示する。一部項目の評価だけで全体の判定を導かない。「該当しそうなものだけ」「主要な項目だけ」で省略しない。

### 2. 全項目が解決するまで完了/Ready 判定をしない

1項目でも未充足（不合格、または保留）が残る場合、全体は「未完了」「Not Ready」とする。一部の項目が解決した時点で「完了」「Ready」「クローズ可能」と判定・提案しない。

### 3. 部分解決の別Issue切り出しは明示承認を得る

残った未解決項目を別Issueへ切り出して本タスクを区切る提案は許容するが、実行前にユーザーの明示承認を得る。承認なしに「残りは別Issueで」と判断して完了扱いにしない。

## 調査の問いの層における決着値

`inquiry` 層（調査の問い）に限り、各問いの決着値を以下の4値へ拡張する。他の3層（`loop` / `issue` / `refinement`）には適用しない。DoR 項目・レビュー指摘へ「推移的に決定済み」「前提消滅」を持ち込まない。

| 決着値 | 意味 |
|---|---|
| 解いた | 当該の問いを直接調査して答えを得た |
| 推移的に決定済み | 他の問い（pivot）の結論から答えが論理的に導出された |
| 前提消滅 | 他の問いの結論により、当該の問い自体が意味を失った |
| 未決 | いずれにも至っていない。完了判定を阻む唯一の値 |

原則1（全項目の列挙・省略禁止）はこの層でも保たれる。決着値が「解いた」以外であっても、全ての問いは台帳へ列挙され1件ずつ決着値を持つ。枝刈り（解かずに「推移的に決定済み」「前提消滅」で決着させること）は省略ではなく明示的な判定であり、原則1に抵触しない。この層での完了判定は「全問いが未決以外の値になり、かつ根の問いへの統合が出たこと」であり、「全問いを解いたこと」ではない。

**既定は安全側**: 枝刈りしてよいと確信できなければ枝刈りしない。迷う場合は「未決」へ倒す。誤枝刈り（調べるべきことを調べずに終わること）は、全問いを解いて元の問いを見失うことより悪い。
