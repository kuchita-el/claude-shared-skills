# プランファイルの所在解決

`docs/plans/` 配下のプランファイルを、worktree とメインが分裂した環境でも一意に特定・書き込みするための共通規約です。

`plan-issue`（プラン作成・既存プランの特定/編集）と `implementation`（プラン探索）が共有参照する単一の出典であり、両スキルはこの解決順序・書き込み先ルールに従う。基準が二重管理されてドリフトすることを避けるため、各スキル本文に解決ロジックを転記せず、本ファイルを参照する。

## 背景（なぜ所在解決が要るか）

`docs/plans/*.md` は `docs/plans/.gitignore` により既定で未追跡のスクラッチ artifact として扱われる（追跡するかは利用者の選択に委ねられる）。一方 `implementation` は作業着手時に新規 worktree を作成して作業ディレクトリを移すことがある。worktree は別の作業ディレクトリであり、未追跡ファイルは伝播しない。

このため「メイン側で作成したプランが worktree から見えない」状態が起こりうる。本規約は、現作業ディレクトリとメイン worktree の双方を決定論的な順序で探索することで、プランが追跡されているか否かを問わず（gitignore 状態を判定せずに）プランを特定する。

## 解決順序（既存プランの特定）(a)

Issue 番号（または slug）を入力に、次の順序で探索し、最初に見つかったディレクトリを **解決ディレクトリ** とする。

1. **現作業ディレクトリの `docs/plans`** を Glob で探索する。
   - 見つかれば → そのディレクトリを解決ディレクトリとする。
   - これは「同一ディレクトリで両スキルを実行する通常運用」と「プランが追跡されており git が worktree へ配信済みの運用」の双方をカバーする。
2. 見つからなければ、**メイン worktree の `docs/plans`** を探索する。メイン worktree ルートは下記「メイン worktree の解決」で得る。
   - 見つかれば → そのディレクトリを解決ディレクトリとする。
   - これは「メイン側で未追跡プランを作成し、別 worktree で implementation を実行した」ケースを救う（簡易パスへ黙って縮退しない）。
3. どちらにも無ければ → 既存プランなし。新規作成として扱う（書き込み先は下記参照）。

> **現ディレクトリ優先・メインはフォールバック専用** とすることで、両方に存在する場合の曖昧さを排除する。現ディレクトリは常に権威（同一 dir／git 配信済み worktree／plan-issue の書き込み先のいずれか）であり、メインは現ディレクトリが空のときだけ参照する。

## 書き込み先ルール (b)

- **新規作成**（解決順序で既存が見つからなかった）→ **現作業ディレクトリ** の `docs/plans` に書き込む。
  - プランが追跡される運用では、現ディレクトリ（＝作業ブランチの worktree）へ書くことで git の branch 配信に委ねられる。メイン側へ固定しない（利用者の追跡意図を壊さない）。
- **既存特定後の再生成・編集・N回目**（解決順序で既存が見つかった）→ **解決ディレクトリへ in-place** で書き込む。現作業ディレクトリへコピーしない。
  - 正典を2つに分裂させない（divergence・`git status` 汚染を防ぐ）ための不変条件。
  - 連番サフィックス（`issue-{番号}-2.md` 等）の計算も **解決ディレクトリ内** の既存ファイル群を基準に行う。現ディレクトリ基準にすると、メイン側の旧 plan を取りこぼして連番が逆行し、旧 plan を上書きするおそれがある。

## メイン worktree の解決

メイン worktree ルートは次のコマンドで取得する（正典）:

```bash
git worktree list --porcelain
```

出力先頭エントリの `worktree ` 行が示すパスがメイン worktree ルート。その配下の `docs/plans` を探索・書き込み対象とする。

- **代替**: `git rev-parse --git-common-dir` の親ディレクトリもメイン worktree ルートに解決できる。ただし共有規約としては上記 `git worktree list --porcelain` を正典に固定し、揺れを避ける（代替は参考）。
- 本コマンドの実行には各スキルの `allowed-tools` に `Bash(git worktree list*)` が必要。

### 後方互換（単一 worktree）

worktree が1つだけの通常環境では、`git worktree list` の先頭エントリは現作業ディレクトリ自身であり、メイン worktree フォールバックは現ディレクトリと同一パスに解決される。したがって解決順序・書き込み先は改修前と同一の挙動に帰着し、後方互換を保つ。

### エラーハンドリング（縮退）

`git worktree list` が失敗する異常環境（git 非インストール・非 git ディレクトリ等）では、メイン worktree フォールバックをスキップし、現作業ディレクトリのみで解決する縮退とする（通常環境では発生しない前提）。

## 既知制約

`plan-issue` を worktree-X で実行してプランを作成し、`implementation` を別の worktree-Y で実行し、かつプランが **未追跡**（gitignore 済み）の場合、プランは worktree-X にのみ存在し、worktree-Y の現ディレクトリにもメイン worktree にも無いため、解決順序の①②がともに空振りしてプランを特定できない。

本ケースは「エラー」ではなく仕様上の未解決ケースとして扱う。回避策:

- プランを追跡対象にする（`docs/plans/.gitignore` を外す）と git が branch 経由で配信し、本制約は解消する。
- `plan-issue` と `implementation` を同一 worktree で実行する。
- `plan-issue` をメイン worktree で実行する（フォールバックで救える状態にする）。

## 設計判断の要約（採用理由）

- **「必ずメイン側へ固定する案」は不採用**。プランが追跡されている場合、メイン固定はプランを branch に乗せず worktree の外へ置き、利用者の追跡意図を壊す。
- **「現ディレクトリ優先＋メイン worktree フォールバック」を採用**。追跡（git 配信）／未追跡（メイン取り残し）の両レジームを gitignore 状態の判定なしにカバーできる。
- **worktree 作成時の hook によるプランコピーは不採用**。コピーは正典を2つに分裂させ、追跡時には `git status` 汚染・divergence を生む。
- 本規約は **探索/書き込みするディレクトリのみ** を規定する。各スキルの連番衝突回避機構（整数比較による最大連番抽出・旧 plan 非上書き）や初回/N回目モード判定の不変条件は変更しない（解決ディレクトリ内にそのまま適用する）。

詳細な決定経緯は `docs/adr/ADR-20260629-plan-location-resolution.md` を参照。
