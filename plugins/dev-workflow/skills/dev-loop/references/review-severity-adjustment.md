# レビュー深刻度の調整ルール

dev-loop の Phase 2（セルフレビュー）で、レビュー結果を周辺コード文脈で再評価し、誤検知（差分の周辺で既に対処済みの指摘）の深刻度を調整するためのルール定義。

差分中心のレビューは、指摘箇所の上流・周辺で既に対処済みの問題まで指摘する（誤検知）ことがある。本ルールは指摘をフィルタリング（除外）せず、深刻度を格下げして補足情報を付加することで、Phase 3 の判定精度と人間レビュアーの判断を助ける。

## 深刻度スケール（2段）

dev-loop が Phase 3 の判定に用いる深刻度は以下の2段とする。

| 深刻度 | 意味 | Phase 3 での扱い |
|---|---|---|
| ブロッカー | マージすべきでない問題（バグ・セキュリティ・要件未充足・レビュー契約不合格） | 修正ループに戻す |
| 改善提案 | マージを妨げないが改善が望ましい指摘 | PR 本文に記載し人間に委ねる |

## レビュー実行経路のラベル → 2段の初期マッピング

Phase 2 のレビュー実行は経路により深刻度軸が異なる。調整の起点として、各経路の出力を以下で2段の初期値に変換する。この初期値は調整前の参考値であり、最終深刻度は後述の格下げ判定で確定する。

| レビュー実行経路 | 出力ラベル | 2段初期値 |
|---|---|---|
| メインループ自身のセルフレビュー（superpowers 非導入時。`agents/code-reviewer.md` の2段基準に準拠） | ブロッカー | ブロッカー |
| 同上 | 改善提案 | 改善提案 |
| `superpowers:requesting-code-review`（導入時） | Critical | ブロッカー |
| `superpowers:requesting-code-review`（導入時） | Important | ブロッカー |
| `superpowers:requesting-code-review`（導入時） | Minor | 改善提案 |

経路がここに無い深刻度ラベルを出力した場合、「マージを止めるべき指摘か否か」で判断し、止めるべきならブロッカー、そうでなければ改善提案に割り当てる（委譲先の出力スキーマは固定ではないため、未知ラベルでも本フォールバックで2段化される）。

## 格下げ判定

2段初期値が **ブロッカー** の指摘について、指摘箇所の周辺コードを調査し、既に対処済みの証拠があれば **改善提案** へ格下げする。

### 対処済みの証拠の例

指摘が問題視するリスクを、指摘箇所より上流または同一パスで無効化している実装。

- 入力バリデーション・サニタイズ（指摘が「未検証入力」を問題視する場合）
- 認証・認可チェック（指摘が「認可漏れ」を問題視する場合）
- null/境界ガード・早期 return（指摘が「null 参照・境界値」を問題視する場合）
- 例外ハンドリング・リソース解放（指摘が「未処理例外・リーク」を問題視する場合）

### 判定の原則

- 証拠は指摘が問題視するリスクを**実際に無効化している**こと。表層的な名前の一致では格下げしない（実際のコードパスを追跡して確認する）。
- 周辺コードに証拠が**見つからない指摘は維持**する（調整なし）。推測で格下げしない。
- 格上げ（改善提案 → ブロッカー）は本ルールの対象外とする。本ルールの主眼は誤検知（過剰なブロッカー指摘）の軽減にある。

## 出力フォーマット

本フォーマットは Phase 2 の調整作業の出力（ワーキング出力）であり、元の指摘を保持する（後述「不変条件」）。人間レビュアー向けの PR 本文への記載は、これを要約したテーブルとして別途行う（dev-loop SKILL.md Phase 4「深刻度調整（誤検知格下げ）」を参照。役割分担: 本フォーマット=作業出力／PR 本文=要約）。

各指摘について、調整方向と根拠を以下の形式で出力する。

```
**[{調整前} → {調整後}] {指摘タイトル}**
{元のレビュー指摘の内容（そのまま。除外・改変しない）}
**補足**: {格下げ時は根拠とした周辺コードのファイルパス:行を1つ以上含む。維持時は調整なしの旨}
```

格下げの例:

```
**[ブロッカー → 改善提案] 未バリデーションの入力使用**
src/handler.ts:30 で `userId` を直接クエリに使用しています。
**補足**: src/middleware/auth.ts:45 の認証ミドルウェアで既にバリデーション済みのため格下げ。実際のリスクは低いと考えられます。
```

維持の例:

```
**[ブロッカー → ブロッカー（維持）] SQL インジェクション**
src/repo.ts:80 で入力を文字列連結でクエリに埋め込んでいます。
**補足**: 周辺コードに対処の証拠が見つからないため維持。
```

## 不変条件

- 元のレビュー指摘の内容は常にそのまま出力に含める（除外・改変しない）。
- 調整するのは深刻度（2段）のみ。指摘の存在自体は消さない。
- 格下げ時は必ず根拠（周辺コードのファイルパス:行）を伴う。根拠を示せない場合は格下げせず維持する。
