# ADR の状態モデルと識別子規約（adr-model.md）

ADR の状態（承認軸・有効性軸）の値域と各値の定義、遷移ごとの front-matter 必須ルール、ファイルの配置と採番方式を定める。本ファイルがこれらの規約の実体であり、状態語彙・採番規則をここ以外で独自に再定義しない。

## 状態の2軸

状態は**承認軸**と**有効性軸**に分離して管理する。1つの状態欄に混在させると、「承認された歴史事実」と「現在効力を持つか」という性質の異なる2つの情報が互いを上書きしてしまうためである。値は日本語のユビキタス言語で確定する。

### 承認軸（`status`・不変の履歴事実）

| 値 | 定義 |
|---|---|
| `提案中` | 起案され、承認者の判断を待っている状態 |
| `承認済み` | 承認された履歴事実。以後不変 |
| `却下` | 提案が承認に至らず終了。一度も運用されない終端 |

### 有効性軸（`validity`・可変の現在の効力）

| 値 | 定義 |
|---|---|
| `有効` | 現在運用中 |
| `上書き済み` | 後継 ADR に置き換えられた（`superseded-by` に後継識別子を保持） |
| `廃止済み` | 後継なしで放棄された |

### 軸の関係

- `却下`（承認軸・一度も承認されなかった終端）と `上書き済み` / `廃止済み`（有効性軸・承認後に効力を失った状態）は別語である。2軸が語彙レベルで衝突しないよう、両者を混同して扱わない
- `提案中` をスキップして直接 `承認済み` に到達する経路は持たない。即承認のケースは「提案 → 承認」を連続実行として扱う
- 有効性軸は可変性のゲートを兼ねる。`有効` な ADR は生きた文書として本文を編集してよいが、`上書き済み` / `廃止済み` に退役した ADR は凍結された歴史的成果物として決定内容を編集しない

## front-matter（状態の唯一の権威）

状態系は各 ADR ファイル冒頭の front-matter に置く。キーは英語（構造的フィールド名）、値は上記の日本語ユビキタス言語とする。

```yaml
---
status: <提案中 | 承認済み | 却下>
validity: <有効 | 上書き済み | 廃止済み | （空）>
superseded-by: <後継ADRの識別子 | （空）>
---
```

- front-matter が状態の唯一の権威である。**本文に `## Status` 節を置かない**。新規起票時も、承認・上書き・廃止・却下のいずれの遷移時も同様である。本文へ状態を書き足すことは重複を生むだけで、以後の遷移で取り残される drift 源になる
- 「キー省略」と「キーあり値空」は同じ「空」として扱う。空にする軸は値を空にする（キー行自体は残してよい）
- 有効 ADR の一覧（index）は front-matter から機械生成する導出ビューであり、人手編集しない。これにより「本文を読めば確実（確実性）」と「一覧で探せる（検索性）」を drift させずに両立する

## 遷移ごとの必須ルール

各遷移後の `status` / `validity` / `superseded-by` は必ず次の組み合わせに一致させる。

| 遷移 | status | validity | superseded-by |
|---|---|---|---|
| 起票 | 提案中 | （無し） | （無し） |
| 承認 | 承認済み | 有効 | （無し） |
| 上書き | 承認済み（不変） | 上書き済み | 必須 |
| 廃止 | 承認済み（不変） | 廃止済み | （無し） |
| 却下 | 却下 | （無し） | （無し） |

- 上書き・廃止では `status` を変更しない（承認は歴史事実であり不変）
- `superseded-by` は単一後継を既定とするが、1本の ADR が複数の後継へ分割される場合はリスト値（カンマ区切り）を取りうる。この 1→N のケースでは、`上書き済み` の原 ADR が全後継を列挙し、各後継が本文の関連 ADR 節に逆参照を持つ（双方向で照合される）
- 後継 ADR は識別子の文字列参照のみで保持し、実体への直接参照は持たない（自己参照・循環参照の構造的不整合を防ぐ）

## 配置

- `docs/adr/` 配下にフラット構造で配置する。サブディレクトリは作らない

## 採番方式

ファイル名規則: `ADR-YYYYMMDD[-N]-<slug>.md`

- `YYYYMMDD`: ADR の起票日（採番日＝起票日）。外部コマンドを実行せず、実行時コンテキストの現在日付から取得する
- `[-N]`: 同日2件目以降のみ付与し、`-2` から開始する（1件目は `-N` なし）
- `<slug>`: 内容を表す短い英数字ハイフン区切りの文字列

**識別子**は番号部（`ADR-YYYYMMDD[-N]`）を指し、ファイル名から `.md` を除いた文字列（`ADR-YYYYMMDD[-N]-<slug>`）とは区別する。

同日の採番例:

| 起票日の件数 | ファイル名 |
|---|---|
| 1件目 | `ADR-YYYYMMDD-<slug>.md` |
| 2件目 | `ADR-YYYYMMDD-2-<slug>.md` |
| 3件目 | `ADR-YYYYMMDD-3-<slug>.md` |

### 採番衝突時の解消手順

同日に既存 ADR がある場合（並行して起票する場合を含む）は、次の手順で発番する。

1. 対象日のファイル（`ADR-YYYYMMDD*.md`）を配置ディレクトリから列挙し、既存の ADR 一覧を取得する
2. 既存ファイルから最大番号を特定する（`-N` なしを `1` として扱い、`-2`、`-3` を順に番号化）
3. 最大番号 + 1 で発番する

**入出力例**

- 入力: 起票日が 2031-04-09。同日付の既存 ADR が2件あり、それぞれ `-N` なし（`ADR-20310409-<slug>.md`）と `-2`（`ADR-20310409-2-<slug>.md`）
- 手順2: 既存の最大番号は `2`（`-N` なしを `1`、`-2` を `2` として番号化）
- 期待結果: 最大番号 + 1 の `-3` を発番し、ファイル名は `ADR-20310409-3-<slug>.md` とする
