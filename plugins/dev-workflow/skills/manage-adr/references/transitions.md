# 遷移リファレンス（transitions.md）

ADR の5遷移（起票・承認・上書き・廃止・却下）と分割の実行手順。front-matter の写入・採番の実行・相互参照の書き込みを、どの順に何へ書くかとして記す。状態の値域、遷移ごとの front-matter 必須ルール、採番方式そのものは SKILL.md が指す状態モデルの規約が実体を持つため、本ファイルではそれらを再定義せず、規約に従う適用手順のみを扱う。

## 目次

- 共通: front-matter の書き方
- 採番規則
- 起票（提案中）
- 承認（承認済み・有効）
- 上書き（上書き済み・双方向相互参照）
- 分割（多決定 ADR の部分上書き）
- 廃止（廃止済み）
- 却下（却下）
- 自己検証の対象ディレクトリ
- index の再生成・検証
- 入出力例（上書き遷移）

## 共通: front-matter の書き方

各 ADR ファイル冒頭に `---` で挟んだ front-matter を置く。キーは英語、値は日本語ユビキタス言語。各キーに書ける値と、遷移ごとにどの組み合わせを取るかは SKILL.md が指す状態モデルの規約に従う。

```yaml
---
status: <承認軸の値>
validity: <有効性軸の値 | （空）>
superseded-by: <後継 ADR の full slug | （空）>
---
```

- **値の説明・トレーリングコメントを書かない**。lint パーサ（`scripts/lint-adr.sh`）は `key: value` の行全体を値として取り込むため、`# 説明` を付けると値が壊れる。
- 「キー省略」と「キーあり値空」は同じ「空」として扱われる。空にする軸は値を空にする（キー行自体は残してよい）。
- H1 見出しは `# ADR-YYYYMMDD[-N]: <タイトル>` 形式にする（`gen-adr-index.sh` が `: ` 以降をタイトルとして抽出するため）。

各遷移後の `status`/`validity`/`superseded-by` の最終状態は、必ず SKILL.md が指す状態モデルの規約の必須ルール表に一致させること。

**遷移前の状態確認（実行ガード）**: 各遷移は期待する遷移元状態を前提とする — 承認は `status: 提案中` から、上書き・廃止は `承認済み`・`validity: 有効` な ADR に対して、却下は未承認の `status: 提案中` に対して行う。実行前に対象 ADR の現在の `status`/`validity` を確認し、期待と異なれば操作を中断して利用者に確認する。`承認済み` の `status` は不変であり、承認済み ADR へ却下操作を適用しない。lint は最終状態のスキーマ整合のみを検査し遷移元は見ないため、この不変条件は手順側で守る。

## 採番規則

識別子・ファイル名の形式、同日2件目以降の連番の付け方、採番衝突時の発番規則は SKILL.md が指す状態モデルの規約に従う。本節では実行手順としての補足のみを記す。

- 発番前に `Glob` で `<対象ディレクトリ>/ADR-YYYYMMDD*.md` を列挙し、同日の既存 ADR を確認してから採番する。対象ディレクトリは `docs/adr/` とし、本スキルの自己検証も同ディレクトリを対象とする。
- 相互参照（`superseded-by`／`Supersedes:`）に書き込む値は、番号部だけの識別子ではなく **full slug**（ファイル名から `.md` を除いた stem）とする。lint は参照先の存在を stem で解決するため、番号部のみを書くと参照先不在の違反になる。

## 起票（提案中）

新規 ADR ファイルを生成する。骨格は SKILL.md が指す雛形の構成（front-matter＋`## Context`／`## Decision`／`## Consequences`／`## 関連ADR` の見出し）に準拠しつつ、**見出し骨格＋空プレースホルダを新規に組み立てる**。決定内容の本文（Context／Decision／Consequences の中身）は上位／人間が埋める前提で、空プレースホルダのまま残す。

- front-matter: `status: 提案中`、`validity:`（空）、`superseded-by:`（空）
- ファイル名を採番規則で決定する。
- **テンプレートの HTML 注釈ブロック（`<!-- ... -->`）を生成物へ残さない**。とりわけ `## 関連ADR` 節のテンプレート注釈には表記例として `- Supersedes: ADR-YYYYMMDD[-N]-<slug>` の行が含まれ、lint-adr の本文走査は HTML コメントを認識せずこの例示行を実データの逆参照として拾い、存在しない後継先を指す相互参照違反（起票直後の自己検証が exit 1）を招く。`## 関連ADR` 節は注釈を除去し、先行・後継・関連が無ければ `該当なし` のプレーンな1行のみを置く。

## 承認（承認済み・有効）

対象 ADR の front-matter を編集する。

- `status: 承認済み`、`validity: 有効`。`superseded-by` は空のまま。
- `status` は以後不変（承認は歴史事実）。

## 上書き（上書き済み・双方向相互参照）

core 変更で新規 ADR（後継）が起票・承認済みであることを前提とする（後継が未作成なら起票を先行させる。上書き操作単体では後継ファイルを新規生成しない）。**旧 ADR と後継 ADR の2ファイルを両方更新する**。

1. **旧 ADR の front-matter**（後継への置換を記録）:
   - `status: 承認済み`（不変）、`validity: 上書き済み`、`superseded-by: <後継の full slug>`
2. **後継 ADR の本文 `## 関連ADR`**（逆参照を追加）:
   - `- Supersedes: <旧の full slug>` の行を1行追加する。
3. **旧 ADR の本文 `## 関連ADR`**（後継の明示）:
   - `- Superseded by: <後継の full slug>` の行を1行追加する。lint の本文走査は `Supersedes:` のみを対象とするため、この記載漏れは exit 0 のまま素通りする（front-matter には残るが、本文だけを見る消費面に後継が現れなくなる）。lint に頼らず、手順として実施したことを確認する。

手順1〜2 の双方向が揃わないと lint-adr レイヤ3（相互参照双方向性）が違反する。lint は次を厳格に要求するため厳守する:

- 後継本文の逆参照は `## 関連ADR` 節内の行で、`- Supersedes: <旧 full slug>`（行頭 `- `＝バレット、full slug 完全一致）。
- `superseded-by` の値と後継ファイル名の stem が一致し、`Supersedes:` の値と旧ファイル名の stem が一致すること（forward・reverse の両方向を検査）。

本節の手順は単一後継（1→1）の上書きを対象とする。原 ADR が複数の決定を束ねていて一部の決定だけが反転する場合（1→N の部分上書き）は、下記「分割（多決定 ADR の部分上書き）」の手順に従う。**分割の実行手順は本ファイルに収録済み**であり、残るのは分割を状態遷移モデルの中でどう位置づけるか（5遷移表への配置等）の統合で、これは未了であり別 Issue の管轄とする。

## 分割（多決定 ADR の部分上書き）

複数の決定を束ねた ADR の一部だけが core 反転し、残りの決定が生存する場合に使う。原 ADR 全体をそのまま `上書き済み` にすると生存決定を誤って退役させるため、上書きの前に生存決定を後継へ救出する。

反転を伴わずに、束ね ADR を予防的に分解する場合（衛生的分割）にも同じ手順を用いる。この場合は退役する決定がなく、全決定が生存決定として後継へ移る。

1. 生存決定の後継 ADR を起票する。一体で反転するクラスタごとに1本以上とし、`status: 提案中` で開始して承認後に `status: 承認済み` / `validity: 有効` へ更新する
2. 後継の `## Decision` に生存決定の本文を**逐語**で転記する。トップレベル項目の先頭番号は、後継での単独可読性のため振り直してよい。番号付き決定の前に置くリード文のような非決定文は、逐語の対象ではないため後継の文脈に合わせて書いてよい
3. 後継の残りの節を書く。`## Context` は新規記述とする。`## Consequences` は生存決定に対応する記述を保持し、退役する決定に由来して無効となる記述のみ除く。`## 関連ADR` は後継が保持する決定集合に照らして書き下ろす（原 ADR の `Related:` を機械的に引き継がない）
4. 手順2〜3 で原 ADR から改変した箇所（先頭番号の振り直し、リード文等の非決定文の書き換え、`## Consequences` の調整）を、後継の `## Context` に理由とともに記載する。決定件数の言い換えのような機械的な追随も開示に含める（「以下の4点を決定する」→「以下の2点を決定する」の類は、後継だけを読む読者には決定が落ちた痕跡と区別がつかないため）
5. 後継の `## 関連ADR` に `- Supersedes: <原 ADR の full slug>` を記載する
6. 原 ADR の front-matter `validity` を `上書き済み` に書き換え、`superseded-by` に**全後継の full slug をカンマ区切りで列挙**する（例: `superseded-by: <後継1 の full slug>, <後継2 の full slug>`）
7. 原 ADR の `## 関連ADR` に `Superseded by: <後継の full slug>` を全後継について記載する。表記（後継ごとにバレットを立てるか、単一行へ読点区切りで並べるか）はファイル内の既存様式に合わせてよく、要求されるのは全件が挙がっていることのみ。既存の `Related:` 行は退役時点のスナップショットとして残置し、分割の一環としては編集しない（この節は本手順が書き込む対象であり、既存行が巻き込まれうる。分割以外の文脈での編集可否は、退役 ADR 一般の凍結原則の射程に従う）

締めに次の4点を確認する。

- **網羅性（手順1〜2）**: 原 ADR の `## Decision` のトップレベル決定を全件列挙し、各決定が「手順2 でいずれかの後継へ逐語転記された」か「core 反転中で退役する」かの**いずれか一方**に分類されていること。未分類の決定と、両方に該当する決定（退役対象でありながら後継へ逐語転記されている）が、いずれも0件であること。分割は 1→N で生存決定が複数後継へ分散するため、後継を1本ずつ検査するだけでは全体の網羅を誰も見ないまま終わる。転記し忘れた決定は下記の逐語性検査でも切り出し対象に入らず、差分として現れない。逆に反転中の決定が後継へ紛れ込むと、反転が黙って無効化される（反転中の決定はそれ自身の後継が改訂後の内容として担うのであって、原文の逐語転記先を持たない）
- **逐語性（手順2）**: 後継ごとに、網羅性で「逐語転記された」に分類した決定に対応する箇所のみを原 ADR から切り出し、後継の `## Decision` の**トップレベル決定本文**と突き合わせる（両辺とも決定本文に揃える。原 ADR 側は退役決定を含み、後継側はリード文等の非決定文を含みうるため、節どうしをそのまま比較すると差分が出るのが正常になり検査にならない）。切り出し元を後継の内容ではなく網羅性の分類から定めることで、後継にのみ存在する決定——退役決定の混入や、分割中に書き足された新規の決定——が差分として現れる。トップレベル項目の先頭番号を正規化したうえで差分がないこと。切り出し結果が非空であることと行数が一致することもあわせて確認する（正規化や切り出しの失敗が空入力として差分ゼロに化けるため）
- **開示（手順2〜4）**: 手順4 の開示対象3種それぞれについて、改変が生じていれば後継の `## Context` に対応する記載があることを確認する。(a) 先頭番号の振り直し、(b) リード文等の非決定文の書き換え（決定件数の言い換えを含む）は、原 ADR と後継を突き合わせて改変の有無を判定する。(c) `## Consequences` は差分を列挙し、各差分が開示に対応していること。(a) は逐語性が正規化で潰し、(b) は逐語性が両辺から外した情報であり、いずれも開示側でしか拾えない。逐語検証は `## Decision` の決定本文にのみ適用し、`## Consequences` には適用しない（意図的な再構成が入るため差分ゼロを要求できない）
- **相互参照（手順5〜7）**: `bash scripts/lint-adr.sh <対象ディレクトリ>` が exit 0 であること。`superseded-by` のリスト値と各後継の `Supersedes:` 宣言が双方向で照合される。加えて、原 ADR 本文の `Superseded by:` に全後継が挙がっていることを確認する。lint の本文走査は `Supersedes:` のみを対象とするため、原 ADR 側の記載漏れは exit 0 のまま素通りする（front-matter には残るが、本文だけを見る消費面に後継が現れなくなる）。あわせて原 ADR の `## 関連ADR` を分割の前後で突き合わせ、分割に由来する変更が `Superseded by:` の追加のみであることを確認する。分割以外を根拠とする変更（例: 廃止された旧運用に由来する相互参照行の除去）が同時に入る場合は、その根拠を示したうえで、`Related:` 等それ以外の既存行が不変であることを確認する（同節は手順7 が書き込む対象であり、既存行が巻き込まれうる）

分割は編集判定フロー（core／非core／些末）の第4分類ではない。決定のファイル間再配置という構造操作であり、分割中に決定内容は変えない。決定内容の変更が要る場合は、分割とは別に core／非core 編集として分離する。

自己検証は `docs/adr` を対象に行う。分割は `validity` を変える操作であるため、index の再生成も伴う。隔離検証へ退避する場合は、原 ADR と**全後継**を同じディレクトリへ含める（下記「自己検証の対象ディレクトリ」の原則どおり、相互参照で指す相手を漏らさない）。

## 廃止（廃止済み）

後継なしで決定を放棄する。対象 ADR の front-matter と本文を編集する。

1. front-matter: `status: 承認済み`（不変）、`validity: 廃止済み`。**`superseded-by` は付与しない**（空のまま）。
2. `## Consequences` 末尾など本文に廃止理由・廃止日時を追記する。
3. 後継 ADR は存在しないため `## 関連ADR` の変更は不要。

## 却下（却下）

一度も承認に至らず終了する終端。対象 ADR の front-matter と本文を編集する。

1. front-matter: `status: 却下`。**`validity`・`superseded-by` は付与しない**（空のまま）。
2. 却下理由を本文に追記する。

- 廃止（`validity: 廃止済み`）と異なり `validity` を空に保つ。`status: 却下`＋`validity` 空はレイヤ1で合法（`status=承認済み` のときのみ `validity` 必須のため、`却下` では違反にならない）。

## 自己検証の対象ディレクトリ

SKILL.md の「各操作後の自己検証」は `docs/adr` を直接 lint する。採番・index 再生成・lint の `<対象ディレクトリ>` はいずれも `docs/adr` を指す。

**フォールバック（baseline red 時）**: 操作前から `docs/adr` が exit 1 で、本操作と無関係な違反が exit 0 到達を妨げる場合に限り、一時ディレクトリへ隔離コピーして検証してよい。この場合に**含めるファイルの規則**は下記。漏らすとレイヤ3 相互参照が false negative（正しい操作でも exit 1）になる。

**原則**: **変更/生成した ADR と、それが相互参照で指す相手をすべて含める** — forward（front-matter `superseded-by` が指す後継）と reverse（本文 `## 関連ADR` の `Supersedes:` が指す先行）の両方向。

- 上書きは**旧 ADR ＋後継 ADR の2ファイル**を必ず含める。旧単体だと旧の `superseded-by` が指す後継が不在となり forward が発火する（実測 exit 1）。旧＋後継で双方向が揃い exit 0 になる。
- 承認・上書き（旧 ADR）・廃止・却下の対象が過去に別 ADR を上書きしていて本文に `Supersedes: <先行>` を持つ場合、対象単体だと reverse（本文起点）が先行不在で発火する（実測 exit 1）。先行 ADR も含めれば exit 0 になる（廃止が現実的なケース）。
- 起票は通常は相互参照を持たないため、生成した ADR のみでよい。

コピー後、`validity` を変える操作（承認・上書き・分割・廃止）では隔離ディレクトリで `index.md` を再生成してから lint する（起票・却下は index 再生成不要）。

## index の再生成・検証

`index.md` は導出ビュー（生成物）であり人手編集しない。手編集しない扱いの根拠は SKILL.md が指す状態モデルの規約に置く。本節は再生成と検証の実行手順のみを記す。

- 生成: `bash scripts/gen-adr-index.sh <対象ディレクトリ>` の出力で `<対象ディレクトリ>/index.md` を再生成する
- 検証: `bash scripts/lint-adr.sh <対象ディレクトリ>` を実行する（front-matter スキーマ・index 同期・相互参照双方向性の3レイヤを検査し、違反0件で exit 0）

`validity` を変える遷移（承認・上書き・分割・廃止）の後は、検証の前に必ず生成を実行する。生成を飛ばすと index 同期レイヤが drift として発火する。自己検証では `<対象ディレクトリ>` を `docs/adr` とする（baseline red で隔離検証へ退避する場合のみ隔離ディレクトリを対象とする）。

## 入出力例（上書き遷移）

- **入力**: 旧 `ADR-20310409-foo.md`（`status: 承認済み`／`validity: 有効`）、後継 `ADR-20310415-2-bar`（起票・承認済みで full slug が確定している）
- **出力（旧側 front-matter）**: `status: 承認済み`（不変）／`validity: 上書き済み`／`superseded-by: ADR-20310415-2-bar`
- **出力（後継側 `## 関連ADR`）**: `- Supersedes: ADR-20310409-foo` を1行追加
