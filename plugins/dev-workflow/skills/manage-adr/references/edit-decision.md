# 編集判定フローリファレンス（edit-decision.md）

`validity: 有効` な既存 ADR への変更を、core／非core／些末 のいずれかに分類し、選ぶ操作へ分岐する。分類の判定基準（decision tree）と3段構え編集機構の対応は本ファイルが実体を持ち、スキル内の編集判定はすべて本ファイルの記述に従う。

本ファイルは「どの操作を選ぶか」の**分岐判定のみ**を担う。選んだ操作の front-matter 写入手順・採番・相互参照の書き方は本ファイルには置かず、SKILL.md 本体が遷移手順を一元的に参照する（参照は1段階まで＝本ファイルから他の reference ファイルへ委譲・相互参照しない）。

## 前提: 分類はスキルが自動判定しない

分類（core／非core／些末）は**必ず利用者へ問う**。スキルが本文の変更内容から自動で core/非core を決めない。安全則「迷ったら core 寄り」に従い、既定選択肢を core とする。

`上書き済み`／`廃止済み` に退役した ADR は凍結された歴史的成果物であり、本フローの対象外である。**この凍結原則の射程は決定内容の編集である。** front-matter（状態）にも `## Decision`（内容）にも触れず、状態の重複表示を消すだけの是正は、決定の記録を書き換えないため射程外であり、退役 ADR に対しても実施してよい。除去しようとする表示が固有の情報を持つ場合、それを `## 関連ADR` へ移してから除去する操作（＝移設。除去の直前に重複を作る）も、情報の総体が保存され決定の記録が変わらない限り同じく射程外である。逆に、決定内容に触れる編集は退役 ADR に加えてはならない。以下の判定フロー（core／非core／些末 の3段構え）は `有効` な ADR への決定内容の編集を振り分けるものであり、退役 ADR には適用しない（退役 ADR の決定本文は、非core と判定できたとしても編集しない）。

## 判定基準（decision tree・境界アンカー＝「再承認が要るか」）

1. その編集で「決定の結論・方向・前提」が変わるか？
   - **YES**（結論の反転・前提の失効・後継への置換）→ **core**（再承認が要る）
   - NO → 2 へ
2. 決定内容に触れるか？（パラメータ値・適用範囲・条件・理由付け）
   - **YES**（骨子は保つが決定関連情報を更新）→ **非core**（既承認の枠内）
   - **NO**（typo・整形・リンク・語句・体裁のみ、決定内容ゼロ）→ **些末**

**安全則: 判定に迷ったら core 寄り（新規 ADR）に倒す。** core を非core と誤判定すると、再承認イベントを飛ばして core 変更を権威記録に焼き込む不可逆な不整合が残る。逆に非core を core と誤って新規 ADR にしても過剰なだけで可逆（後で統合できる）。可逆性の非対称から core 寄りが安全である。

## AskUserQuestion の問い設計

編集操作を起動したら、`AskUserQuestion` で変更種別を問う。

- **質問**: 「この編集は core／非core／些末 のどれか？（迷う場合は core を選ぶ）」
- **選択肢（既定を先頭・core）**:
  1. **core（既定・推奨）** — 決定の結論・方向・前提が変わる。再承認が要る。
  2. **非core** — 骨子は保つが決定内容（パラメータ・適用範囲・条件・理由）を更新する。
  3. **些末** — typo・整形・リンク・語句・体裁のみで決定内容はゼロ。
- 既定（先頭）を core とし、安全則「迷ったら core」を選択肢説明に明示する。

## 3段構え編集機構の対応表

分類ごとに、選ぶ操作と変更履歴を残す先が決まる。

| 変更の性質 | 対応 | 変更履歴の在り処 |
|---|---|---|
| **core**（再承認が要る。決定の結論・方向・前提が変わる） | 新規 ADR 起票操作 ＋ 旧 ADR 上書き操作（5遷移の起票＋上書きを実行し、旧 ADR の有効性を `上書き済み` にする） | 別 ADR ファイル＋双方向相互参照 |
| **非core**（骨子は保つが決定に関わる情報を更新: パラメータ値・適用範囲・条件・理由付け） | 直接編集 ＋ 本文 `## 変更履歴` に1行追記（front-matter の `status`／`validity` は不変） | ファイル内 `## 変更履歴` |
| **些末**（typo・整形・リンク修正・語句・体裁のみ、決定内容ゼロ） | 直接編集のみ（front-matter・`## 変更履歴` は変更しない） | git のみ |

**原則: ADR ファイルは、あらゆる consumption 面（Wiki・Confluence・artifact・抽出ファイル）に対して自己完結する。** git は repo の実装詳細であり、決定関連情報の SSoT にはしない。3段構えのうち履歴を git のみに委ねてよいのが些末に限られるのは、この原則による（些末は決定関連情報を含まないため、git の外に痕跡が残らなくても自己完結性が損なわれない）。

## 記録の参照原則（参照を書く／直す編集の判定）

**記録（ADR）は、可変文書を現在の参照先として指さない。** 指す必要がある場合は、時点を固定した過去形の記述にするか、参照先の実質を自己完結的に述べる。本原則は分類（core／非core／些末）を問わず、ADR 本文へ参照を書く編集・既存の参照に手を入れる編集のすべてに効く（退役 ADR の決定本文は前記の凍結原則により対象外）。

**判定テスト**: 参照先を空にした状態でその記述を読み直す。**偽になるなら適用対象**であり、時点固定または自己完結化を要する。**真のままなら対象外**であり、記録として既に閉じている。

- **適用対象の例**: 「その規定は現に運用されている」のように、参照先が現にその内容を保持していることを現在形で主張する記述。参照先がその内容を保たなくなれば主張が偽になるため、「当時の」等を補って時点を固定するか、実質を書き下ろす。
- **対象外の例**: 廃止・変更の対象を同定する記述や、当時どこへ委譲したかという予定の記録。参照先が現在何を保持しているかを主張しないため、指し先が空になっても真のまま保たれる。

判定テストは原則の禁止範囲を狭めるものではない。ある記述が「現在の参照先として指す」に当たるかを個別に見分けるための道具である。

## 分類ごとの補足

- **core**: 起票（`status: 提案中`）と上書き（旧 `validity: 上書き済み`＋後継への双方向相互参照）の2操作を選ぶ。各操作の写入手順は SKILL.md 本体が一元的に参照する遷移手順にあるため、本ファイルでは手順を繰り返さない。旧 ADR が多決定で生存決定が残る場合（1→N の部分上書き＝分割）の実行手順も、同じ遷移手順に収録済みである。ただし分割は core／非core／些末 に続く第4の分類ではなく、決定のファイル間再配置という構造操作であり、状態遷移モデルの中でどう位置づけるかの統合は未了で別 Issue の管轄とする。したがって本ファイルの分岐が扱うのは当面 1→1 の上書きのみとする。
- **非core**: 対象 ADR 本文を直接編集し、本文 `## 変更履歴` に変更内容の1行を追記する。front-matter（`status`／`validity`／`superseded-by`）は変更しない。
- **些末**: 対象 ADR 本文のみを直接編集する。front-matter・`## 変更履歴` はいずれも変更しない（変更の履歴は git のコミットのみが持つ）。

非core と些末の差は `## 変更履歴` の追記有無にある（非core＝1行追記あり、些末＝追記なし）。この区別を取り違えないこと。
