# 台帳仕様

調査の状態を user-local に保持し、セッションを跨いだ再開を可能にする台帳ファイルの仕様。

## 置き場

台帳は `~/.claude/projects/<project-id>/inquiry/{slug}.md` へ置く。`memory/`・`growth/` と同階層に `inquiry/` を新設する。

この置き場はリポジトリの work tree の**外**にある。したがってリポジトリの `git status` には原理上現れず（AC3）、どのブランチのトラッキング対象にもならない。かつ worktree の削除では消えない（AC4）。この2つの受入条件は、置き場の選定そのものによって構成的に保証される。

## project-id とパスの解決手順

> **出典注記**: 本節は `plugins/growth/references/personal-store-spec.md`「project-id とパスの解決手順」からの**複写**である。growth は独立プラグイン（ADR-20260626）であり `${CLAUDE_PLUGIN_ROOT}` による cross-plugin 参照が成立しないため、両プラグインが独立して動作することを優先して複写している（Issue #511「制約」）。git のバージョン要件や置換規約を改修する際は複写元と本節の2箇所を追随させる必要がある。

`<project-id>` と台帳パスは以下の手順で解決する。

**リポジトリルートと project-id**:

```bash
git rev-parse --path-format=absolute --git-common-dir
```

このコマンドは worktree・通常リポジトリの両方で共通の `.git` ディレクトリの**絶対パス**を返す（例: `/home/user/myproject/.git`）。`--path-format=absolute` は Git 2.31 以降で利用可能。末尾の `/.git` を除いたパスをリポジトリルートとし、全 `/` を `-` に置換して `<project-id>` とする（例: `-home-user-myproject`）。

> `--git-common-dir` 単体は CWD に応じて相対パスを返す場合があるため `--path-format=absolute` を必ず併用する。
> `git rev-parse --show-toplevel` は worktree 内では worktree 固有パスを返すため使用しない（worktree で開始した調査を main から同一台帳で再開できなくなる。AC4）。
> `git rev-parse` が失敗する（git リポジトリ外・Git 2.31 未満で `--path-format` が不在）場合は台帳パスを解決できないため、その旨を提示して終了する。推測したパスへ書かない。

**台帳パスの組み立て**: `~/.claude/projects/<project-id>/inquiry/{slug}.md`。ディレクトリが不在なら `mkdir -p ~/.claude/projects/<project-id>/inquiry/` で作成する（初回起動時の正常系）。`mkdir` は必ず `~` 展開前の literal 形式 `mkdir -p ~/.claude/projects/<project-id>/inquiry/` で発行する。

## slug 規則

- **Issue 経路**: `issue-{番号}`。決定論的に定まるため、再開の同定は自明である。
- **フリーテキスト経路**: 根の問いから kebab-case で生成する。この slug は生成のたびに揺れうる**非決定的**な値であるため、**slug の一致で再開を判定しない**。同一の検討課題でもセッションを跨ぐと同一 slug に落ちる保証がないためである。既存台帳との同定は、起動時に台帳ディレクトリを列挙して利用者へ委ねる（列挙の実行タイミングと提示形式はフロー側の責務）。

## エントリ形式

台帳は**行ベースの平文規約**でパースする（YAML パーサによる厳密解釈を前提としない。growth の store 規約に倣う）。次の3セクション構成とする。

- `## 根の問い`: 根の問いの**原文**を verbatim 保持する（下記「原文の定義」）。
- `## 問い一覧`: 配下に `### Q{n}: {問いの文}` の見出しを持つ問いエントリを列挙する。各エントリは直下に `- key: value` の単一行メタ欄を持つ。
- `## 統合`: 根の問いへの解を書く。

**メタ欄**（`- key: value` の単一行形式）: `state`・`kind`・`implies`・`pivot`・`pivot-rationale`・`answer`。

**`## 根の問い` が保持する「原文」の定義**: ステップ0 の**終了時点で確定した文**である。

- 入力に疑問マーカーを含む文がある場合（起動時の擦り合わせが非発火）は、入力原文がそのまま原文になる。
- 起動時の擦り合わせが発火した場合は、**利用者が確定させた文**が原文になる。
- いずれの場合も、書き込み後は改変しない（AC6）。
- 再開時は、この節の既存の値が原文である。擦り合わせによる上書きを行わない。

## メタ欄の生成タイミング

各欄がいつ埋まるかを仕様として固定する（実装者の解釈に委ねない）。

- `state` と `kind`: **エントリ生成時（ステップ1 の分解）に必ず同時に付与する。空のままエントリを作らない。**
- `implies`: ステップ2（含意グラフ構築）。
- `pivot`・`pivot-rationale`: ステップ3（pivot 特定）。
- `answer`: ステップ4（pivot を解く）。ステップ4 は `kind` を**台帳から読むのみで再判定しない**。

人間ゲート1 の却下でステップ1 へ戻り台帳を書き換える場合も、追加・書き換えの対象となる全エントリがこの規則に従う（`state` と `kind` を同時付与する）。

## 状態値域

`state` は `未決` / `解いた` の**2値**とする。全ての問いが `state` を持ち、空にしない（AC5）。値域外の値を作らない。

> 子3（#512）が `推移的に決定済み` / `前提消滅` を追加して4値へ拡張する予定である。本 Issue #511 の時点では実装しない（値域は上記2値に限る）。この前方注記は将来の拡張地点を示すものであり、2値を超える値を本 Issue の台帳へ書くことを許すものではない。

## `kind` の値域

`kind` は単一値であり、値域は `SKILL.md` 本体の「調査の種類（`kind`）の値域」節に従う（本ファイルへ値を列挙して二重管理しない。参照のみ）。値域は汎用値を含む全域であるため、どの問いにも必ず1つの `kind` が決まる。全ての問いが `kind` を持ち、空にしない。値域にない値を作らない（AC9）。

## 含意関係の表現

各問いの `implies` 欄を**単一出典**とし、独立した含意グラフ表を持たない（同一情報の二重管理によるドリフトを避ける。グラフ全体は各エントリの `implies` から導出できる）。`implies` には `Qi → Qj`（Qi が決まれば Qj が決まる／消える）の形で、当該問いが決める問いを列挙する。

## 記述例

以下は本仕様のパース規約だけで、問い・状態・`kind`・含意関係・pivot 判定結果を抽出できる最小例である（`kind` の値は例示）。

```markdown
## 根の問い

認証基盤を A 方式と B 方式のどちらで刷新すべきか。

## 問い一覧

### Q1: A 方式と B 方式で既存コードへの改修量はどちらが小さいか
- state: 解いた
- kind: コードベース内の事実調査
- implies: Q1 → Q3
- pivot: yes
- pivot-rationale: Q3（総コスト）を決めるため到達数が最大
- answer: A 方式は既存の認証ミドルウェアを流用でき改修量が小さい

### Q2: 各方式の運用実績・既知の脆弱性はどうか
- state: 解いた
- kind: Web・文献調査
- implies: Q2 → Q3
- pivot: no
- pivot-rationale: -
- answer: 両方式とも実績あり。B 方式に軽微な既知課題

### Q3: どちらが総コスト（改修＋運用）で優れるか
- state: 未決
- kind: 汎用
- implies:
- pivot: no
- pivot-rationale: -
- answer:

## 統合

（各問いの answer を根の問いへの解へ統合する欄。ステップ6・7 で埋める）
```

## 構成上の保証と検証手段（AC3・AC4）

置き場が work tree 外であることは、次のコマンドで確認できる。

- **AC3**（台帳が未追跡ファイルとして現れない）: 台帳生成後に `git status --porcelain` の出力が空であること。
- **AC4 前半**（project-id の一致）: worktree 内で解決した `<project-id>` と main の作業ディレクトリで解決した `<project-id>` が一致すること（上記コマンドが両方で main の `.git` 絶対パスを返す）。
- **AC4 後半**（worktree 削除後の残存）: worktree を削除した後も `~/.claude/projects/<project-id>/inquiry/{slug}.md` が残存し、main から同一パスで読み出せること。
