## ステップ0: 根の問いの受領・固定

- **解決する問題**: 何を調べるのか（根の問い）を一意に確定させ、以降の全工程で改変されない原文を得る。

**入力2経路の判定**は「入力が Issue 番号（または Issue URL）の形をしているか否か」のみで行い、**ラベルの有無を分岐条件にしない**（spike ラベルから inquiry への振り分けは子4 のスコープであり、振り分け済みの入力を inquiry 側で再判定しない）。

- **Issue 経路**: `gh issue view {番号} --json number,title,body` の取得結果（タイトル＋本文）を入力テキストとする。ラベルは取得しない。`gh issue view` が失敗した場合（Issue 不在・権限なし・未認証）は、根の問いの原文を取得できないため、その旨を提示して終了する。Issue 番号や表題から根の問いを推測して代替しない（AC6 の verbatim 保持が成立しないため）。
- **フリーテキスト経路**: 入力文そのものを入力テキストとする。

続いて下記の**起動時停止点**を経て根の問いの原文を確定させ、台帳の `## 根の問い` へ verbatim 保持する。以降の全ステップの入力に原文を含める（全ての委譲プロンプトへ根の問いの原文を埋め込む方式）。

## ステップ0 の起動時停止点（確認A・確認B）

本ファイルが発火判定規則・提示文言・選択肢構成を持つ。**起動時に最大1回だけ停止**し、発火した確認事項のみを**1回の `AskUserQuestion`** でまとめて問う（確認A・確認B を別々の停止点に分けない）。

**これは人間ゲートではない**。ADR-20260716 決定1 の人間ゲート3つは、スキルが下した判定結果を利用者が承認・却下するゲートである。本停止点は入力受領時に不足している情報を補う擦り合わせであり、性質が異なる。したがって**ゲート番号を付さない**（ゲート2 は子3 のスコープのまま欠番）。

### 確認A: 既存台帳の同定

- **発火条件**: **フリーテキスト経路** かつ 台帳ディレクトリを列挙した結果が**1件以上**。
- **非発火**: Issue 経路（slug が `issue-{番号}` で決定論的に定まり同定が自明）、または既存台帳が0件。
- **提示**: 既存台帳それぞれの `## 根の問い` の見出しを一覧提示し、再開か新規かを問う。フリーテキスト経路の slug は非決定的であるため、**slug の一致で再開を判定しない**。

### 確認B: 根の問いの擦り合わせ

- **発火条件**: 入力テキストに**疑問マーカーを含む文が1つも無い**（下記の判定規則）。
- **非発火**: 疑問マーカーを含む文が1つ以上ある、または確認A で既存台帳の再開が選ばれた。
- **提示**: 入力から**推定した根の問い**を、推定である旨を明示して提示し、「この問いで確定してよいか／修正して指定するか」を問う。確定した文が根の問いの原文になる（AC6）。

**発火判定規則（閉じた基準・安全側）**: 入力テキスト（Issue 経路はタイトル＋本文、フリーテキスト経路は入力文そのもの）に、次の**3つの表層マーカーのいずれか**を含む文が1つ以上あるかで判定する。マーカーの集合はこの3つで**閉じており、拡張を認めない**。

| マーカー | 定義（表層形） | 例 |
|---|---|---|
| M1 | 文が疑問符（`?` または `？`）を含む | 「A と B のどちらが速いか？」 |
| M2 | 文末が疑問の終助詞「か」である（「〜か。」「〜か」「〜のか」「〜だろうか」） | 「A と B のどちらが速いか。」 |
| M3 | 「か」の直後に助詞「を」「は」「が」「について」のいずれかが続き、疑問節を閉じている | 「どちらを採用すべき**かを**決める」「何が原因**かは**未確認である」 |

1つ以上あれば**非発火**（入力原文をそのまま根の問いとして verbatim 固定する）。1つも無ければ**発火**（推定した根の問いを提示して確定を求める）。

**判定が割れる場合の規則（全域性の担保）**: あるマーカーに該当するか迷う文は「**該当しない**」と扱う（＝発火へ倒す・安全側）。これにより規則は全域かつ閉じたものになる。安全側が発火である根拠は費用の非対称にある — 誤発火の代償は確認1回だが、発火すべき入力を非発火にした代償は、根の問いが推定のまま固定され以降の全工程がその推定の上に乗り、かつ AC6 の verbatim 保持により誤った根の問いが工程を通じて忠実に増幅され検知されないことである。

**境界例の確定値**（実装者の裁量に委ねない・再解釈しない）:

| 入力に現れる文 | 該当マーカー | 判定 |
|---|---|---|
| 「A と B のどちらを採用すべきか？」 | M1 | 非発火 |
| 「どの方式が最も速いか。」 | M2 | 非発火 |
| 「どちらを採用すべきかを決める」 | M3 | 非発火 |
| 「X について検討する」 | なし | **発火** |
| 「X を確認する」 | なし | **発火** |
| 「X を調査する」 | なし | **発火** |
| 「X を実装する」「X を追加する」 | なし | 発火 |
| 「A か B を選ぶ」（「か」の直後が名詞句で M2〈文末〉にも M3〈「か」＋助詞〉にも当たらない選択の格助詞） | なし | **発火** |

「X を調査する／検討する／確認する」を発火側とする理由: これらの文は調査対象（トピック）を述べるが、**何を決めたいのか（問い）**を述べていない。verbatim 固定すると「X の何を知りたいのか」が未確定のまま分解へ入り、擦り合わせが防ごうとする失敗（推定の上に全工程が乗る）がそのまま起きる。

**発火判定はラベルを参照しない**: 判定は入力テキストの表層形のみで行う。したがって**疑問マーカーを含まない spike Issue では発火し、疑問マーカーを含む非 spike Issue では発火しない**（決定論点「Issue 経路での spike ラベルの扱い」の案 A を退行させない）。ラベルは `gh issue view` の取得項目にも含めない。

### 選択肢の構成

- **確認A のみ発火**: 「台帳 X を再開／台帳 Y を再開／…／新規で開始」。
- **確認B のみ発火**: 「推定した問いで確定／修正して指定」。
- **両方発火**: 「台帳 X を再開／…／新規で開始（根の問い＝推定した問い）／新規で開始（根の問いを修正して指定）」。**既存台帳の再開が選ばれた時点で推定した根の問いは破棄**し、台帳の `## 根の問い` の原文をそのまま継承する（＝再開時は確認B を再発火させない）。
- **両方非発火**: **停止せず**ステップ1 へ進む（無条件の確認にしない）。

**再開を選択した場合**: 台帳の `## 根の問い` を原文として継承し、未決の問いから継続する。

## ステップ1: 問いへの分解

- **解決する問題**: 根の問いを、作業ではなく**問い**の集合へ分解する。

作業への分解を弾く（動詞が「調べる／決める」等、問いの形になっているかを見る。「実装する／追加する」のような作業は問いに直さない）。分解結果を台帳へ列挙する際、**各エントリに `state: 未決` と `kind` を同時に付与する**。`kind` の決め方は「`SKILL.md` 本体の『調査の種類（`kind`）の値域』節の判定の手がかりを引いて、値域の値を1つ選ぶ」。どの種別にも明確に当てはまらない問いは汎用値の `kind` を付し、`kind` を空にしない・値域にない値を作らない。人間ゲート1 の却下でこのステップへ戻り台帳を書き換える場合も、追加・変更した全エントリが同じ規則で `state` と `kind` を持つ。

## ステップ2: 含意グラフ構築

- **解決する問題**: 問い同士の含意構造（どの問いがどの問いを決めるか）を可視化する。

各問いの `implies` 欄へ `Qi → Qj`（Qi が決まれば Qj が決まる／消える）を書く。

## ステップ3: pivot 特定

- **解決する問題**: 決定的な問い（これを決めれば他が推移的に決まる／消える問い）を特定する。

**台帳の未決の問い集合と `implies` 欄のみを入力とする再入可能な手順**として記述する（初回実行と再実行で手順を分岐させない。子3 のステップ5 からの復帰辺が本ステップを書き換えずに機能するため）。`implies` の到達数が多い問いを pivot とし、各問いの `pivot` 欄（`yes`／`no`）と `pivot-rationale` 欄を埋める。

**循環検出**: `implies` に循環がある（例: Q1 → Q2 かつ Q2 → Q1）場合は循環を検出し、循環内の代表を1つ pivot とするか、分解のやり直し（ステップ1）へ戻す。循環に陥ったまま pivot 特定が停止・無限ループしないこと（停止性を保つ）。

## 人間ゲート1（分解の妥当性・pivot 判定）

- `AskUserQuestion` で「分解は根の問いを覆っているか／pivot 判定は妥当か」を問う。選択肢は「承認（次へ）」「却下（理由を添える）」。
- **却下時はステップ1 へ戻る**（台帳を書き換える）。却下されたままの分解で解決のステップへ進まない。

## ステップ4: pivot を解く

- **解決する問題**: pivot を優先的に解き、その解を台帳へ記録する。

問いと、**台帳のエントリが既に持つ `kind`**（ステップ1 で付与済み。ここで**再判定しない**・台帳から読むのみ）を入力に、**`SKILL.md` 本体の「委譲」節の規則に従って委譲する**（テーブルの引き方・起動手段・縮退先は本体の委譲節が規定する）。委譲プロンプトに根の問いの原文を verbatim 埋め込む。委譲先が実行環境に不在の場合も本体の委譲節の縮退規則に従い、調査を継続する（異常終了しない）。

結果を `answer` 欄へ書き、`state` を `解いた` へ更新する。**委譲が解を出せなかった場合は `state` を `未決` のまま残し**、`answer` へ解が出なかった旨を記す（未決を「解いた」に見せかけない）。この未決の経路が、ステップ6 での未決提示の入力になる。pivot を優先順に解いた後、残りの未決の問いも解く（本 Issue #511 では枝刈りを行わない）。

## ステップ6: 根の問いへの統合

- **解決する問題**: 各問いの解を根の問いへの1つの解へまとめる。

各問いの `answer` を根の問いへの解へ統合する。**統合の出力前に台帳の `state` を走査し、`未決` の問いが1件でも残っている場合は、どの問いが未決かを利用者へ明示的に提示する**（未決を伏せたまま人間ゲート3 へ進まない）。

## 人間ゲート3（統合結果の確認）

- `AskUserQuestion` で統合結果を確定させる前に確認を求める。選択肢は「承認（確定）」「却下（理由を添える）」。
- **却下時はステップ6 へ戻る**。

## ステップ7: 成果物化

- **解決する問題**: 確定した解を台帳と利用者へ残す。

台帳の `## 統合` 欄へ書き込み、利用者へ提示する。

## 子3 の挿入点

- ステップ4 とステップ6 のあいだに**ステップ5（再評価・枝刈り）と人間ゲート2**が入り、**ステップ5 からステップ3 への復帰辺**が張られる。本ファイルのステップ3 が再入可能に書かれていることがその前提である。
- ステップ5 が適応的ループの中で台帳へ新しい問いを追加する場合も、**エントリ生成時に `state` と `kind` を同時付与する規則（ステップ1 と同一）**が適用される。`kind` 付与の責務はステップ1 に閉じており、子3 が新たな付与地点を作らずに済む。
