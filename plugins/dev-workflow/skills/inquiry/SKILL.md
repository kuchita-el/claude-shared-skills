# 問いの統治（inquiry）

大きめの検討課題について、決定的な問いを特定して優先順に解き、元の問いへの解を出す。細分化した問いを解くこと自体の目的化を防ぐ。

## 正本（唯一の出典）

`docs/adr/ADR-20260716-inquiry-skill-question-governance.md` が本スキルの正本。コアフロー（ステップ0〜7）・委譲境界・人間ゲート3箇所をそこから読み込み、スキル独自の語彙を導入しない。値域拡張の関連は `docs/adr/ADR-20260716-2-completion-judgment-inquiry-value-range.md`（本 Issue の状態値域は2値であり、拡張は子1 のスコープ）。

本 Issue #511 の実装範囲はステップ0〜4・6〜7 と人間ゲート1・3 である。ステップ5（再評価・枝刈り）と人間ゲート2 は範囲外（子3）。

## 引数と入力2経路

入力は Issue 番号（または Issue URL）／フリーテキストの2経路。判定は「入力が Issue 番号の形をしているか否か」のみで行い、**ラベルを分岐条件にしない**。

- **Issue 経路**: `gh issue view {番号} --json number,title,body` でタイトル＋本文を取得する（ラベルは取得しない）。
- **フリーテキスト経路**: 入力文そのものを用いる。

spike Issue から inquiry への振り分けは子4 のスコープであり、本スキルは振り分け済みの入力を再判定しない。

## 台帳

調査状態を `~/.claude/projects/<project-id>/inquiry/{slug}.md`（リポジトリの work tree 外）へ保持し、セッションを跨いで再開できる。置き場・パス解決手順・slug 規則・エントリ形式・状態値域は `${CLAUDE_SKILL_DIR}/references/ledger-spec.md` に従う。

## 調査の種類（`kind`）の値域

`kind` の値域と判定の手がかりの**単一出典**。3つの references はいずれもこの節を参照する。列挙するのは調査種別名のみで、委譲先の具体ツール名を含まない。

| `kind`（値域） | 判定の手がかり（この種類へ落ちる問いの特徴） |
|---|---|
| コードベース内の事実調査 | 既存コードの構造・呼び出し関係・実装の所在を問うもの |
| Web・文献調査 | リポジトリ外の事実・仕様・先行事例を問うもの |
| 依存・互換性調査 | パッケージ更新の影響・バージョン間の互換を問うもの |
| ドメイン理解・モデリング | 業務概念・状態遷移・境界の定義を問うもの |
| 履歴・経緯調査 | 過去の決定・変更がなぜそうなったかを問うもの |
| 実験・PoC | 実際に動かして観測しないと決まらないもの |
| 汎用 | 上記のいずれにも明確に当てはまらないもの（値域を全域にする受け皿） |

汎用値により値域は全域であり、どの問いにも必ず1つの `kind` が決まる。

## 手順

各ステップの詳細・人間ゲートの問い設計・起動時停止点の発火判定規則は `${CLAUDE_SKILL_DIR}/references/inquiry-flow.md` に従う。ステップ番号は正本のコアフローに一致させ、ステップ5 は欠番・範囲外とする。

### ステップ0: 根の問いの受領・固定（起動時停止点を含む）

入力2経路から根の問いの原文を受領し、以降の全工程で verbatim 保持する。台帳の `## 根の問い` へ保持する。

**起動時停止点**（骨格。詳細はフロー手順が持つ）:

1. 起動時に**最大1回**停止し、発火した確認のみを1回の `AskUserQuestion` でまとめて問う。
2. **確認A**: フリーテキスト経路かつ既存台帳が1件以上あるとき、既存台帳の根の問い見出しを一覧提示して再開／新規を問う。
3. **確認B**: 入力に疑問マーカーを含む文が1つも無いとき、推定した根の問いを提示して確定を求める。**発火判定は入力テキストの表層形のみで行い、ラベルを参照しない**（マーカーの定義・境界例・提示文言・選択肢は `${CLAUDE_SKILL_DIR}/references/inquiry-flow.md` が持つ）。
4. **再開時**は台帳の根の問いを継承し、確認B を再発火させない。
5. 両方非発火なら**停止せず**ステップ1 へ進む。
6. これは人間ゲートではなく入力受領時の擦り合わせであり、**ゲート番号を付さない**（正本の人間ゲートは3つ。欠番のゲート2 は子3）。

### ステップ1: 問いへの分解

作業ではなく**問い**へ分解し、台帳へ列挙する。各エントリ生成時に `state: 未決` と `kind` を**同時に付与**する（`kind` は上記「調査の種類（`kind`）の値域」節の判定の手がかりで1つ選ぶ。空にしない・値域外を作らない）。

### ステップ2: 含意グラフ構築

各問いの `implies` 欄へ `Qi → Qj`（Qi が決まれば Qj が決まる／消える）を書く。

### ステップ3: pivot 特定

台帳の未決の問い集合と `implies` のみを入力に、決定的な問い（pivot）を特定する（再入可能な手順）。循環は検出して停止しない。

### 人間ゲート1

`AskUserQuestion` で分解の妥当性と pivot 判定を利用者へ確認する。却下時はステップ1 へ戻る。

### ステップ4: pivot を解く

台帳のエントリが持つ `kind`（ステップ1 で付与済み・再判定しない）で下記「委譲」節の規則に従い委譲する。委譲プロンプトに根の問いの原文を verbatim 埋め込み、結果を `answer` 欄へ、`state` を `解いた` へ更新する。解が出なければ `state` は `未決` のまま残す。

### （ステップ5 は欠番・範囲外＝子3）

### ステップ6: 根の問いへの統合

各問いの `answer` を根の問いへの解へ統合する。統合前に `未決` の問いが残っていればどの問いかを利用者へ明示的に提示する（未決を伏せて進まない）。

### 人間ゲート3

`AskUserQuestion` で統合結果を確定させる前に確認する。却下時はステップ6 へ戻る。

### ステップ7: 成果物化

台帳の `## 統合` 欄へ書き込み、利用者へ提示する。

## 委譲

問いを解くときは、台帳のエントリが持つ `kind` で `${CLAUDE_SKILL_DIR}/references/delegation-table.md` を引き、当該行の**起動手段列**に従って起動する。委譲の入力は「問い ＋ `kind`」の対とし、根の問いの原文を verbatim で添える。当該行の委譲先が実行環境に不在の場合は**汎用の行**の委譲先へ縮退し、調査を継続する（異常終了しない）。委譲先・起動手段・縮退先はすべてテーブルのデータであり、本体は分岐を持たない。`kind` の判定基準は上記の値域節にあり、委譲時には判定しない。
