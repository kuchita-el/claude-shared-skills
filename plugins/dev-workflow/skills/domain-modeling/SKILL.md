# 関数型ドメインモデリング

ドメインのデータ型・状態・コマンド・イベント・ワークフローを記法ガイド準拠の擬似コードで形式化する対話型スキル。Scott Wlaschin『Domain Modeling Made Functional』の AND / OR 表記を採用し、実装言語に依存しない設計ドキュメントを生成する。

## 引数

- `$ARGUMENTS` の解析ルール:
  - **ドメイン名のみ** → Interactive モード
    - 例: `order`, `在庫管理`
    - ドメイン名はディレクトリ名としても使用する
  - **ドメイン名 `--auto <file>`** → Auto モード
    - 例: `order --auto docs/order/overview.md`
    - 入力ファイル不在時はエラー終了

## モード

| モード | 起動条件 | 動作 |
|---|---|---|
| Interactive | デフォルト | 各フェーズでユーザー確認を挟んでから次に進む |
| Auto | `--auto <file>` | 入力ファイルからドメイン概要を読み込み、全フェーズを一括実行する |

**Auto モードの動作ルール:**

- 各フェーズの「ユーザーに確認する」「修正を求める」等の対話ステップはスキップし、AI 判断で次フェーズに進む
- 入力ファイルと `docs/{domain}/event-storming.md`（存在する場合）をもとに各フェーズの成果物を自律的に作成する
- 全フェーズ完了後にドキュメントを生成し、最終確認のみユーザーに求める
- `event-storming.md` 不在時は入力ファイルの内容のみで最小情報のまま進める

## フェーズ構成

| Phase | 概要 |
|---|---|
| 0. 入力の読み込み | `docs/{domain}/event-storming.md` と `docs/big-picture.md` の有無を確認し、前提を揃える |
| 1. 値の定義 | ID・制約付き値・列挙型を定義する |
| 2. 状態の定義 | 集約の状態型を OR で定義し、無効状態を型で排除する |
| 3. 集約の定義 | 状態型・状態遷移・コマンド・ポリシーを集約単位で束ねる |
| 4. ワークフローの定義 | 外部公開ユースケースの契約を定義する |
| 5. コンテキスト境界の定義 | 外部イベント・外部コマンドを `外部_` プレフィックスで明示する |
| 6. ドキュメント生成 | `docs/{domain}/domain-model.md` を出力する |

**詳細手順:** `${CLAUDE_SKILL_DIR}/references/domain-modeling-flow.md` を参照

**記法規約:** `${CLAUDE_SKILL_DIR}/references/domain-model-notation.md` を参照（型定義・命名規約・成果物構成）

## 出力

- `docs/{domain}/domain-model.md` — データ型・状態遷移・コマンド・イベント・ポリシー・ワークフローの定義（記法ガイド「成果物構成」準拠）

## 対話の原則

- **ユーザーの言葉を尊重する**: ドメイン用語はユーザーが使った表現をそのまま型名に採用する。勝手に英訳・一般化しない
- **段階的に進める**: 各フェーズの結果をユーザーに確認してから次に進む（Auto モード除く）
- **不正な状態を型で排除する**: nullable・フラグで状態を区別する設計を避け、OR で状態を分けて無効組合せを排除する
- **失敗をドメイン語彙として表現する**: 曖昧な文字列エラーではなく、失敗理由を OR で列挙してプロダクトチーム全体で合意する対象とする

## event-storming スキルとの関係

- event-storming は「何が起きるか」（イベント・コマンド・集約）を洗い出す
- 本スキルは「データがどんな形を持ち、ワークフローがどんな変換か」を型で定義する
- 入力として `docs/{domain}/event-storming.md` を活用し、ホットスポット（未解決事項）を型化の対象とする
- 出力は同じ `docs/{domain}/` 配下に並び、両文書を組み合わせてドメインの全体像を構造的に表現する

## 注意事項

- **フレームワーク・言語非依存**: 出力は擬似コードによる設計ドキュメントであり、F#・TypeScript 等の言語固有構文を含めない
- **集約単位で型をまとめる**: 1つの集約のライフサイクルをそのセクション内で完結して読めることを重視する
- **発火イベントは1箇所のみ定義**: 発火元集約のセクションで定義し、受け手集約は名前と参照先のみ記載する
- **共通値の判定基準**: 2つ以上の集約で参照される型のみを共通値に置く。判断に迷う場合は集約固有とし、必要に応じて後から共通に昇格する
