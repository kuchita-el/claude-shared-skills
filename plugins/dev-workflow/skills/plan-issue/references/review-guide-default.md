# プランレビュー基準（デフォルト）

実装プランの品質をレビューするための基準。11の観点でプランを評価する。

## レビュー観点

| # | 観点 | チェック項目 |
|---|---|---|
| 1 | 実現可能性 | 各ステップで変更対象のファイル/モジュールが実在するか。存在しないAPIや未導入ライブラリを前提にしていないか |
| 2 | スコープ適切性 | Issueの要件に対して過不足がないか。Issue外の改善を混ぜていないか |
| 3 | 依存関係 | ステップ間の順序制約が明示されているか。外部サービスやCI/CDへの影響が考慮されているか |
| 4 | ステップ粒度 | 各ステップが1コミット程度の粒度か。「〜を適切に実装する」のような曖昧な記述がないか |
| 5 | 検証方法 | 完了を確認する手段（テスト、動作確認コマンド等）が各ステップに対して明示されているか |
| 6 | リスク | 破壊的変更、マイグレーション、後方互換性への影響が識別されているか |
| 7 | 検証方針の妥当性 | 検証方針のピックアップ基準（ACから自明でないもの・修正コストが高い振る舞い）が妥当か。重要な振る舞いの見落としや、自明なケースの過剰な列挙がないか |
| 8 | タスクの自己完結性 | 各タスクがサブエージェント1セッションで完結できるか。参照コンテキストが広すぎる場合、事前調査ステップが計画に含まれているか |
| 9 | アーキテクチャ準拠 | DBスキーマ変更・API変更・新規依存ライブラリが既存のアーキテクチャ方針と整合しているか。実装設計セクションが必要十分に記載されているか |
| 10 | テストケース対応表 | 「テストケース対応表」セクションが存在するか。AC全項目との対応・観点・採用技法・テストレベルが付与されているか。抽象度がコード非依存か |
| 11 | 判断明示セクションの存在 | `## 判断依頼` と `## 決定論点` の両セクションが存在するか。各セクションに内容または「該当なし」が明示されているか。「判断依頼」の各エントリが4フィールド（観点ID・説明・検出フェーズ・判断者ロール）を備えているか |

## 各観点の詳細

### 1. 実現可能性

- 変更対象ファイルがGrep/Globで実在確認されているか
- 前提としている機能・ツール・仕組みが既に利用可能か
- 新たに必要なものがある場合、そのステップが計画に含まれているか

### 2. スコープ適切性

- Issueの受け入れ条件と計画の成果物が1:1で対応しているか
- 「ついでに」的な改善・リファクタリングが混入していないか
- 必要な変更が漏れていないか（受け入れ条件の各項目がカバーされているか）

### 3. 依存関係

- タスク間の実行順序に矛盾がないか（依存先のタスクが先に実行される構成か）
- 外部サービス・CI/CDパイプラインへの影響が識別されているか
- 並行作業可能なタスクが明示されているか

### 4. ステップ粒度

- 各タスクが1コミット（数十分〜数時間）で完了できる規模か
- 実装内容が具体的に記述されているか（「適切に」「必要に応じて」等の曖昧表現がないか）
- 変更するファイルが明示されているか

### 5. 検証方法

- 各タスクに完了条件が明示されているか
- テスト実行、コマンド実行、目視確認等の具体的な検証手段が記載されているか
- 検証手段が再現可能か（誰が実行しても同じ結果を得られるか）

### 6. リスク

- 既存機能への影響が識別されているか
- 破壊的変更がある場合、影響範囲と対処方針が記載されているか
- 後方互換性を損なう変更がある場合、移行手順が考慮されているか

### 7. 検証方針の妥当性

- ピックアップ基準が妥当か（ACから自明でないもの・修正コストが高い振る舞いが選ばれているか）
- 重要な振る舞いの見落としがないか（リスクの高い境界条件・異常系が含まれているか）
- 自明なケース（単純なCRUD、設定値の反映等）が過剰に列挙されていないか

### 8. タスクの自己完結性

- 各タスクの説明だけで、実装に必要な情報が揃っているか
- 広範囲のコードベース理解を前提としていないか
- コンテキストが大きい場合、事前調査ステップ（Grep/Globで対象を特定する等）が含まれているか

### 9. アーキテクチャ準拠

- DBスキーマ変更がある場合、既存のテーブル設計・命名規則と整合しているか
- マイグレーション方針（破壊的変更の有無、ロールバック手順）が明記されているか
- API変更がある場合、既存のエンドポイント設計・命名規則・レスポンス形式と整合しているか
- 新規依存ライブラリがある場合、選定理由と代替案との比較が記載されているか
- 既存の技術スタックや方針に反する選択をしていないか（している場合はその根拠が示されているか）

### 10. テストケース対応表

- 「テストケース対応表」セクションが存在するか（欠落時はFAIL）
- AC全項目に最低1テストケースが対応付けられているか（未対応AC1件以上でFAIL）
- 各テストケースに「観点」「採用技法」「テストレベル」が付与されているか（欠落1件でFAIL）
- 観点が指定のクローズドリスト（典型ケース / 境界値 / 異常系 / 状態遷移 / 組み合わせ）内の値か（範囲外はFAIL）
- テストケースの抽象度がコード非依存か（関数名・テスト名・モック設定が混入していないか）
- AC未定義のIssueの場合、「該当なし。AC定義後に再生成すること」と明記されているか

### 11. 判断明示セクションの存在

- `## 判断依頼` セクションが存在するか（欠落時はFAIL）
- `## 決定論点` セクションが存在するか（欠落時はFAIL）
- 各セクションに内容または「該当なし」が明示されているか（空のままはFAIL）
- 「判断依頼」の各エントリが以下の4フィールドを備えているか（1件でも欠落時はFAIL）
  - `観点ID`（J{n} 形式、Issue内一意）
  - `説明`（判断対象として認識できる粒度）
  - `検出フェーズ`（plan-issue 生成時は `plan`）
  - `判断者ロール`（ADR-20260621-human-judgment-observation.md の例示列挙内）
- 「該当なし」が明示されている場合はPASS（セクション欠落と区別すること）

判定基準（ディシジョンテーブル）:

| セクション状態 | レビュー結果 |
|---|---|
| セクション自体が存在しない | FAIL |
| セクション存在 + 「該当なし」明示 | PASS |
| セクション存在 + エントリあり + 4フィールド全て揃っている | PASS |
| セクション存在 + エントリあり + 4フィールドに欠落あり | FAIL |

## レビュー手順

1. プランの各セクション（実装設計、タスク分解等）を上記11観点で確認する
2. 問題がある場合、具体的な指摘と修正案を記録する
