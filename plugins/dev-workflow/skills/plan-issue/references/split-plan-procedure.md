# 複数 PR 分割対応の手順（plan-issue ハイブリッドモード）

plan-issue SKILL.md のステップ1（モード判定）から参照される手順。**分割モードへの移行は明示的なトリガーがある場合のみ**行い、通常の単一プラン実行には影響しない。

## 分割モードに入るトリガー

以下のいずれかが明示されている場合のみ、本手順（ハイブリッドモード）を適用する:

- 補足指示または Issue 本文に「分割」「複数 PR」「split」等の分割要求がある
- 引数に k 指定（`k=2`・`2本目` 等）があり、`issue-{番号}-plan.md` が存在する

**いずれも該当しない場合は通常の単一プランフロー**（本ファイルを参照せずステップ2以降を継続する）。

## 分割計画マスターの有無による初回/N回目判定

分割モードと確定した後、**分割計画マスターファイル**（`issue-{番号}-plan.md`）の有無でさらに判定する。マスターの有無は `${CLAUDE_PLUGIN_ROOT}/references/plan-location-resolution.md` の解決順序（現作業ディレクトリの `docs/plans` を先に Glob、見つからなければメイン worktree の `docs/plans` を Glob。メイン worktree は `git worktree list --porcelain` 先頭エントリで解決）で確認し、見つかったディレクトリを**解決ディレクトリ**とする。これにより、別 worktree で実行してもメイン側のマスターを取りこぼして誤って初回モードへ落ち、分割計画を丸ごと再生成してしまう事故を防ぐ:

| 分割計画マスターファイルの状態 | 判定 |
|---|---|
| `issue-{番号}-plan.md` がどちらのディレクトリにも存在しない | **初回モード**（新規。書き込み先は現作業ディレクトリ） |
| `issue-{番号}-plan.md` が解決ディレクトリに存在する | **N回目モード**（解決ディレクトリへ in-place） |

### 既存ステップ5（連番サフィックス衝突回避）との区別

ステップ5は `issue-{番号}.md`（詳細プランファイル）の有無を判定して連番サフィックスを付与する（人間レビュー却下後の再生成リカバリ用）。本手順は `issue-{番号}-plan.md`（分割計画マスター）の有無を判定する。**判定対象ファイルが異なるため両者は衝突しない**:

- `issue-{番号}-plan.md` の有無 → 本手順（初回/N回目モード）の判定に使用
- `issue-{番号}.md` の有無 → ステップ5（連番リカバリ）の判定に使用

## 初回モード（分割計画マスターが存在しない場合）

N プラン分割計画と1本目の詳細プランを生成する。**初回モードもステップ5の通常フロー（`issue-{番号}.md` 基本名 + 連番サフィックス）を使わず、直接 `issue-{番号}-1.md` を保存先とする**（N回目モードとの命名一貫性のため、および `issue-{番号}.md` が既存でも連番リカバリが発動しないようにするため）:

1. AC↔PR マトリクスを含む **N プラン分割計画**を **メインが直接生成・Write** して `docs/plans/issue-{番号}-plan.md` に保存する（分割計画マスターは AC↔PR マトリクスの小型俯瞰計画であり、詳細プランと異なり plan エージェント直接 Write の対象外。生成主体はメインのまま据え置く）
   - 全 AC をどの PR が担当するかのマトリクス
   - PR 間の前後依存
   - 各 PR のスコープ概要（担当 AC・主な変更対象）
2. **1本目の詳細プラン**を生成する。まずメインが `docs/plans/` を Glob で列挙して書き込み先パス **`docs/plans/issue-{番号}-1.md`** を解決し（既存の場合はステップ5と同じ連番サフィックス `issue-{番号}-1-2.md` 等の空きパスを採用）、解決済みパスをステップ6の plan サブエージェントへ渡してエージェントが直接 Write する（plan 本文はメインに載せない）
   - 出力フォーマットの「分割の文脈」セクションを含める（`${CLAUDE_SKILL_DIR}/references/plan-output-format.md` 参照）

**遷移例**: 分割要求あり・`docs/plans/issue-268-plan.md` が存在しない状態で `plan-issue 268` を実行
→ `issue-268-plan.md`（3PR 分割計画マスター）と `issue-268-1.md`（PR-A 詳細プラン）を直接生成

## N回目モード（分割計画マスターが存在する場合）

引数で指定された k 本目のプランを詳細化する。**N回目モードはステップ5の通常フローを使わず、直接 `issue-{番号}-k.md` を保存先とする**（ステップ5の `issue-{番号}.md` 基本名 + 連番サフィックスの仕組みとは独立したパス）:

1. 解決ディレクトリの `docs/plans/issue-{番号}-plan.md` を Read して分割計画を確認する（解決ディレクトリは上記「分割計画マスターの有無による初回/N回目判定」で特定済み）
2. k 本目の担当 AC・スコープ・前後依存を分割計画から読み取る
3. k 本目の詳細プランの書き込み先パスをメインが Glob で解決する。**解決ディレクトリの `issue-{番号}-k.md`**（k は数値: 2, 3 …）を基本名とし、既存の場合はステップ5と同じ連番サフィックス（`issue-{番号}-k-2.md` 等）の空きパスを解決ディレクトリ内で採用する（in-place。現作業ディレクトリへコピーしない）
4. 解決済みパスをステップ6の plan サブエージェントへ渡し、エージェントが k 本目の詳細プランを直接 Write する（plan 本文はメインに載せない）
   - 「分割の文脈」セクションに k 本目の情報を記入する

**遷移例**: `docs/plans/issue-268-plan.md` が存在する状態で k=2 を指定して `plan-issue 268` を実行
→ `issue-268-2.md`（PR-B 詳細プラン）を直接生成（ステップ5の連番リカバリとは別経路）

## k（何本目か）の受け取り方

N回目モード起動時、補足指示として k を受け取る（例: `plan-issue 268 "k=2"` や `plan-issue 268 2本目`）。k の指定がない場合は、分割計画マスターを参照して未詳細化の PR をユーザーに提示し、`AskUserQuestion` でどの PR を詳細化するか確認する。
