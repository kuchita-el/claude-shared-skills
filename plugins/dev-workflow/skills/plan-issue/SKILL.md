# プラン作成（Plan Issue）

Issueの実装プランを作成し、プランファイル（保存先はモードで分岐。ステップ5参照）に保存する。プランには技術設計・タスク分解に加え、AC全項目と対応する**テストケース対応表**を含める。
plan サブエージェント（計画骨格を superpowers `writing-plans` へ委譲）でプラン作成 → 独立レビュアーエージェントによるレビュー → 修正のループを最大2周実行し、要確認事項をユーザーに質問する。

Issue番号を起点とする**通常モード**のほか、Issueを立てずにフリーテキストの補足指示だけでプランを作成する**Issueなしモード**を備える。

## 引数

- `$ARGUMENTS`: プラン作成の起点となる入力。Issue番号/URL、またはフリーテキストの補足指示
  - 通常モード（特定Issueのプラン）: `21`, `#21`, `https://github.com/owner/repo/issues/21`, `#21 のレビュー指摘を反映するプラン`
  - Issueなしモード（フリーテキスト起点）: `MUI v7→v9 アップグレードの実装プラン`, `#21 を参考にした移行プラン`
  - `--input <path>`: JSONファイルからIssue情報を読み込む（`gh issue view` をスキップ）

モード判定はステップ1を正本とする。判定は構文位置ではなく**入力の意図**に基づき、`--input` のみ構文で最優先判定する。Issue番号/URLが入力に含まれても、それが「対象」ではなく「参考」であれば通常モードには入らない。

## 手順

### 1. 引数の解析とモード判定

`$ARGUMENTS` を以下の手順でモード判定する。判定は構文位置ではなく**入力の意図**に基づく（このスキルの実行主体はLLMであり、構文パースより意図解釈が自然なため）:

1. **`--input <path>` あり** → **固定入力モード**（最優先・構文判定）。JSONファイルパスを取得する。自動化/プログラム呼び出し向けの明示指定のため、意図解釈の対象外とする
2. **引数が空** → エラー終了
3. **上記以外** → 全文をフリーテキストとして読み、意図を解釈してモードを決定する:
   - 入力の意図が「**特定の既存Issueの実装プランを作る**」ことであり、**対象Issueが一意に特定できる**（番号/URLが、参照ではなく対象として指されている） → **通常モード**。当該Issue番号を抽出し、残りのテキストは補足指示として扱う
   - それ以外（Issueへの言及が参考・引用にとどまる／対象Issueが曖昧／Issueと無関係な作業のプラン） → **Issueなしモード**。全文をフリーテキスト補足指示として扱う
   - 意図が「特定Issueのプラン作成」か判然としない、または対象Issueを一意に特定できない → `AskUserQuestion` でユーザーに確認する（黙って推測しない）

**判定例**: `${CLAUDE_SKILL_DIR}/references/mode-detection-examples.md` を参照（通常モード／Issueなしモード／確認の 7 ケース）。

**通常モードの場合の分割計画判定**: 通常モードと判定された後、補足指示等に分割要求（「分割」「複数 PR」「split」等）がある場合は分割モードに入る。分割モード内で `docs/plans/issue-{番号}-plan.md`（分割計画マスター）の有無により初回/N回目をさらに判定する。詳細手順は `${CLAUDE_SKILL_DIR}/references/split-plan-procedure.md` を参照。

### 2. レビュー基準の読み込み

以下の順序でReadツールで読み込み、両方の内容をマージしてレビュー基準とする:

1. **デフォルト（必須）**: `${CLAUDE_SKILL_DIR}/references/review-guide-default.md`
2. **プロジェクト固有（任意）**: `{プロジェクトルート}/.claude/plan-issue/review-guide.md`

デフォルトは常に読み込む。プロジェクト固有ファイルが存在する場合（Readが成功した場合）、その内容を追加のレビュー観点として加える。

### 3. Issue情報の取得

**固定入力モード（`--input`）:**

ReadツールでJSONファイルを読み込む。`gh` コマンドは実行しない。

**通常モード:**

```bash
gh issue view {issue_number} --json number,title,body,labels,comments
```

**Issueなしモード:**

`gh` コマンドは実行しない。ステップ1で分離したフリーテキスト補足指示をプラン作成の入力源とする。

### 4. ベースブランチの決定

以下の優先順位でベースブランチを決定する:

1. **コンテキストからの読み取り**: CLAUDE.md（ユーザースコープ・プロジェクトルート）、プロジェクトのブランチ運用ルール、ユーザーの補足指示にブランチ運用の記載があれば、そこからベースブランチを特定する
2. **デフォルトブランチへのフォールバック**: コンテキストから判断できない場合、以下のコマンドでリポジトリのデフォルトブランチを取得する

```bash
gh repo view --json defaultBranchRef --jq '.defaultBranchRef.name'
```

コマンドが失敗した場合は `main` を仮設定し、プランに「デフォルトブランチの取得に失敗したため `main` を仮設定。実装時に確認すること」と注記する。

### 5. プランファイルのパス解決（連番衝突回避命名）

ステップ6でエージェントへ渡す書き込み先パスを、メインが Glob 単独で決定論的に解決する。plan 本文をメインで Write したり実行ログを除去したりするステップは持たない（Write はエージェントが行い、実行ログはファイルに載らない）。

探索・書き込みの対象ディレクトリ（**解決ディレクトリ**）と書き込み先ルールは `${CLAUDE_PLUGIN_ROOT}/references/plan-location-resolution.md` の共通規約に従う。すなわち、既存プランの有無は「現作業ディレクトリの `docs/plans` を先に Glob、見つからなければメイン worktree の `docs/plans` を Glob（メイン worktree は `git worktree list --porcelain` 先頭エントリで解決）」の順で確認し、連番付与（既存特定後の in-place 書き込み）はその解決ディレクトリ内で行う。連番計算の基準ディレクトリは解決ディレクトリであり、現作業ディレクトリ固定ではない（メイン側の旧 plan を取りこぼして連番が逆行するのを防ぐため）。保存先はモードで分岐する:

- **通常モード / 固定入力モード**: `issue-{番号}.md` を基本名とする。解決ディレクトリの `docs/plans` を Glob で列挙して同名ファイルの有無を確認し、既に存在する場合は上書きせず連番サフィックス（`issue-{番号}-2.md`, `issue-{番号}-3.md` …）を付与して**解決ディレクトリ内**の空きファイル名を採用する。これは「人間レビュー却下後のリカバリ」での再生成時に旧 plan ファイルへのパス参照を保つための共通衝突回避機構であり、旧 plan を上書きしない不変条件はメイン側で決定論的に担保する（存在確認は Glob で足り、旧 plan 本文を Read してはならない）。既存プランがどちらのディレクトリにも無い場合は新規作成とし、書き込み先は現作業ディレクトリの `docs/plans` とする
- **Issueなしモード**: フリーテキスト入力の要約から kebab-case の slug を生成し `{slug}.md` を基本名とする。同様に解決ディレクトリの `docs/plans` を Glob で同名ファイルの有無を確認し、既に存在する場合は上書きせず連番サフィックス（`{slug}-2.md`, `{slug}-3.md` …）を付与して解決ディレクトリ内の空きファイル名を採用する

### 6. plan サブエージェントによるプラン作成

ステップ2のレビュー基準・ステップ3のIssue情報（またはフリーテキスト補足指示）・ステップ4のベースブランチ・ステップ5で解決したプランファイルパスを統合したプロンプトを構築し、plan サブエージェントを起動する。サブエージェントは渡された解決済みパスへプラン本文を直接 Write し、メインへはパス＋構造化サマリのみを返す（plan 本文全文はメイン context に載せない）。

**プロンプト構築・サブエージェント起動の詳細手順**: `${CLAUDE_SKILL_DIR}/references/agent-prompt-construction.md` を参照（3 種モード別テンプレート、起動手順、フォールバック）。

### 7. レビュアーエージェントによるレビュー → 修正ループ（最大2周）

プラン作成のコンテキストを持たない独立したレビュアーエージェントでプランを評価する。

> 本ステップはサブエージェントによる**自動レビュー**のループ。人間レビュアーによる plan ファイルへの却下時のリカバリは `## 人間レビュー却下後のリカバリ` を参照。

**7a. レビュアープロンプトの構築:**

登録済みサブエージェント（`subagent_type: dev-workflow:plan-reviewer`）に渡すプロンプトを構築する。以下を本文に含める:

- **プランファイルのパス**: ステップ5で解決し、ステップ6でエージェントが Write したプランファイルのパス
- **レビュー基準**: ステップ2で読み込んだレビュー基準の全文（デフォルト＋プロジェクト固有）

**7b. レビュアーエージェントの起動:**

Agent tool（`subagent_type: dev-workflow:plan-reviewer`）で起動する。モデルは定義の `inherit` に従う。Agent toolが使えない場合や起動に失敗した場合は、`plugins/dev-workflow/agents/plan-reviewer.md` の定義内容をプロンプト本文へ埋め込んでインラインで直接実行する（サブエージェント側からの定義ファイル再Readは行わない）。

**7c. レビュー結果の処理:**

- レビュー結果が `PASS` → ステップ8へ
- レビュー結果が `FAIL` → メインが Edit するのではなく、**新しい plan エージェントインスタンスを編集モードで起動**し、プランファイルのパス＋レビュアーの指摘テーブルを渡して修正させる（エージェントが自前で Read → 該当箇所を Edit → 変更サマリを返却。plan 本文はメインに載らない）。修正後 7b に戻る（最大2周）。ここで渡すプランファイルのパスは、ステップ5で `${CLAUDE_PLUGIN_ROOT}/references/plan-location-resolution.md` の解決順序に従って解決した解決ディレクトリ内のパスを用いる（編集は in-place で行い、現作業ディレクトリへコピーしない）

**ループ終了条件:**

- `PASS` → ステップ8へ
- 2周で収束しない → 残った指摘事項を根本的な問題としてステップ8で質問する

### 8. 要確認事項の質問

> 本ステップはプラン作成中のサブエージェント自動レビュー由来の確認事項を扱う。plan ファイル受け渡し後の人間レビュアーによる却下は `## 人間レビュー却下後のリカバリ` を参照。

以下の場合、`AskUserQuestion` でユーザーに質問する:

- レビュアーエージェントの指摘で解決できない問題がある場合
- Issueの要件が曖昧でプランの品質を担保できない場合
- 設計上の選択肢があり、判断が必要な場合

質問がない場合は、プランファイルのパスを報告して完了する。

## 出力フォーマット

プランファイル（保存先はステップ5参照）に `${CLAUDE_SKILL_DIR}/references/plan-output-format.md` の形式で出力する。

## 人間レビュー却下後のリカバリ

ステップ7（レビュアーエージェントによる自動レビュー）が PASS した後、ステップ6でエージェントが Write した plan ファイル（書き込み先はステップ5で解決）を**人間レビュアー**が読み、設計判断やスコープに不同意して却下した場合のリカバリパス。本セクションはステップ7のサブエージェント自動レビューとは別フローであり、契機が異なる（自動レビュー＝サブエージェントの指摘、本リカバリ＝人間レビュアーの却下）。

### 分岐

- **再生成（既定）**: 却下理由が方針・設計判断・スコープ・タスク分解の粒度・検証方針の選択に関わる場合
- **手動修正（軽微指摘のみ）**: 却下理由が文言の微修正・誤字・テーブル体裁・参照リンクの追加/修正のみの場合

### 軽微の判定基準

却下理由の性質で分類する（観測可能な判定）:

| 分類 | 該当する却下理由の例 | 対応パス |
|---|---|---|
| 方針・設計判断 | 集約境界の解釈、ユースケース分解、設計アプローチ | 再生成 |
| スコープ | IN/OUT の判断、関連 Issue との切り分け | 再生成 |
| タスク分解の粒度 | タスクの分割数、依存関係、順序 | 再生成 |
| 検証方針の選択 | テストレベル、検証手法、検証すべき振る舞いのピックアップ | 再生成 |
| 文言・体裁 | 誤字、表現の微修正、テーブルの整形 | 手動修正 |
| 参照リンク | 関連ドキュメントの追加、リンクの修正 | 手動修正 |

**境界が曖昧な場合は再生成側に倒す**。誤った設計のまま手動微修正で確定すると、下流の実装コストを破壊するため。

### 再生成パスの操作手順

**plan-issue スキルを再起動する**（ステップ1から再実行＝新規 plan インスタンスを生成）:

1. 却下理由をフリーテキスト補足指示として整理する（例: `集約境界の解釈を再検討。サブドメインAをBに包含するアプローチで再計画`）
2. plan-issue スキルを再起動する: `/dev-workflow:plan-issue {番号} "{補足指示}"`（plan-issue の通常モードは「Issue番号 + 補足指示」を引数で受け取れる。SKILL.md ステップ1の判定例参照）
3. ステップ5の連番命名解決により新規 plan は連番サフィックス付きパス（`issue-{番号}-2.md`, `issue-{番号}-3.md` …）へ書き込まれ（メインが Glob で空きパスを解決→新規 plan エージェントインスタンスが直接 Write）、旧 `issue-{番号}.md` は上書きされず残る（明示的に削除しない）。Issue・PR・dogfood ログから旧 plan へのパス参照は維持される

この経路は `docs/development/event-storming.md` P2（変更要求対応 → 新インスタンス作成）と整合する。

### 手動修正パスの操作手順

開発者が Edit で plan ファイルを直接修正する。修正対象の plan ファイルは `${CLAUDE_PLUGIN_ROOT}/references/plan-location-resolution.md` の解決順序（現作業ディレクトリ → メイン worktree フォールバック）で特定し、特定したディレクトリ内のファイルを in-place で修正する（現作業ディレクトリへコピーしない）。必要に応じてステップ7のレビュアーエージェントを再起動して整合性を再確認してもよい。

### 整合性・参照

- **event-storming P2**: 変更要求対応 → 新インスタンス作成。再生成パスはこれと完全整合する
- **event-storming H7**: 実装プランの粒度は開発者判断（解消済み）。本セクションの「軽微」の判定主体も開発者である
- **横断判断表（#97）**: 問題の深刻度 × 検出タイミング → 差し戻し先の判断軸として [Issue #97](https://github.com/kuchita-el/claude-shared-skills/issues/97) を参照する。#97 確定前は本リンクが前方互換アンカーとして機能し、判断表本体の転記は #97 完了後の別 PR で行う
