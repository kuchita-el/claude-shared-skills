# Phase 0 詳細手順（インプット種別判定・パス判定・契約導出）

implementation の Phase 0「インプットの理解」で実施する各小手順の詳細。本体 SKILL.md の Phase 0 節からポインタ参照される。

## エスカレーション履歴の確認

インプットがIssueの場合、Issueコメントから `<!-- implementation:escalation -->` マーカーを検索する。未回答のエスカレーション（マーカー後にコメントがない）があれば報告して終了する。回答済みのエスカレーション内容は作業リストの入力コンテキストに含める。

## 実行環境の検知

現在のブランチ名から以下を判定する。

- **worktree環境**: worktree内で実行されている場合、ブランチ切り替えに制約がある（他worktreeで使用中のブランチにはcheckoutできない等）。制約に遭遇したら、現在のブランチ上で作業を進め、必要に応じて手動マージ手順を報告する
- **別セッション再開**: 作業ブランチ上にいる場合、コミット履歴と差分から現在の実装状況を把握し、未完了の作業を継続する

## ブランチの準備

**新規作業の場合（既存の作業ブランチ上にいない場合）に限り実施する**。別セッション再開（既存の作業ブランチ上にいる場合。「実行環境の検知」参照）では本手順を実行せず、ベースの最新化やブランチの切り直しを行わない（進行中の作業を上書きしないため）。

1. **ベースブランチの確定**: CLAUDE.md、CONTRIBUTING.md 等からベースブランチ（develop、main 等）を確認する。指定がなければデフォルトブランチ（main/master 等）をベースにする。
2. **ベースブランチの最新化（接続契約）**: 新しいブランチを切る前に、`git fetch` でリモートを取得し、ローカルのベースブランチを `origin/<base>` へ fast-forward する。これにより現 HEAD ＝最新ベースを保証する。最新化は委譲の有無に関わらず本スキルが接続契約として担う（委譲先 `superpowers:using-git-worktrees` は現 HEAD から枝を切るため、委譲前に最新化済みのベースブランチへ切り替えておく）。baseRef は `fresh`（`origin/<base>` 起点）を既定とする。
3. **隔離ワークスペースの確保とブランチ作成**:
   - superpowers 導入時: `superpowers:using-git-worktrees` に委譲する（参照機構②: `Skill` ツール呼び出し）。手順2で最新化済みのベースを基点に隔離ワークスペースを確保する。
   - 非導入時: 最新化済みの `origin/<base>` を起点に新しいブランチを作成する（古いローカルの `<base>` からは切らない）。

## 正規パス/簡易パスの判定

インプット種別判定の後、計画ファイルに検証すべき振る舞いが具体的に定義されているかどうかで正規パスか簡易パスかを判定する。

| インプット種別 | 判定方法 | パス |
|---|---|---|
| Issue | `docs/plans/issue-{番号}.md`（基本名）と `docs/plans/issue-{番号}-*.md`（連番サフィックス付き）の **2 パターンを別 Glob で照合**して列挙する（単一の `issue-{番号}*.md` パターンは桁またぎ Issue 番号 #12 vs #123 等で誤マッチするため使わない）。各ファイル名から連番を抽出（基本名は 0、`-N` は N）し、**整数として比較**して最大値のファイルを Read で読み込む（文字列比較は `-10 < -2 < -9` となりリカバリ10回到達で誤動作するため使わない）。これは plan-issue「人間レビュー却下後のリカバリ」ステップ6 の衝突回避メカニズム（再生成時に新 plan へ連番付与）と対称で、リカバリ実行後の最新 plan を確実に拾うため。ファイルが存在し、検証すべき振る舞いが具体的に定義されていれば正規パス | 正規 or 簡易 |
| 計画ファイル | 入力されたファイルに検証すべき振る舞いが具体的に定義されていれば正規パス | 正規 or 簡易 |
| PRレビュー指摘 | 常に簡易パス | 簡易 |
| 会話コンテキスト | 常に簡易パス | 簡易 |

## 作業リストとレビュー契約の作成

仕様に曖昧さや設計判断が必要な箇所があればエスカレーションする。問題なければ、判定されたパスに応じて作業リストとレビュー契約を作成する。

レビュー契約はインプットの受け入れ条件・完了条件から導出した具体的なチェック項目のリストであり、Phase 2 で `superpowers:requesting-code-review` に渡され、機能要件の検証基準として使用する。

### 正規パス（計画ファイルに検証すべき振る舞いが定義されている場合）

作業リスト: 計画ファイルのタスク分解セクションから導出する。Phase 0での自力導出はスキップ。

レビュー契約の導出:
1. **検証方針（主）**: 検証すべき振る舞いの各エントリを検証可能なチェック項目に変換する
2. **Issue AC（補完）**: ACの各項目のうち、検証方針由来のチェック項目でカバーされていないものを追加する
3. **矛盾チェック**: 検証方針とACに明らかに相反する記述、または暗黙の前提の違いがある場合、エスカレーションする。粒度の差異は矛盾とみなさない
4. **重複排除**: 検証方針由来とAC由来で同一内容のチェック項目を統合する

### 簡易パス（従来通り）

作業リスト: 実装/修正すべき事項の一覧を自力導出する。

レビュー契約の導出:
- インプットに受け入れ条件がある場合（Issue/計画ファイル）、各条件を検証可能なチェック項目に変換する
- 受け入れ条件がない場合、作業リストの各項目から「何が完了していれば正しいか」を導出する
- 各項目は「〜が存在する」「〜が返される」「〜が呼ばれる」等、差分から検証可能な形式で記述する
