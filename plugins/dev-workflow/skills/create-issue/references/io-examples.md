# 入出力例（create-issue）

create-issue SKILL.md のドラフト先行生成手順から参照される入出力例集。標準例（refine-issue 出力簡潔化）、指定的HOW退避の例（WebSocket通知）、bug 種別の例（保存ボタン連打）の 3 例を収録する。

---

**入力**（アイデア、粗いメモ）:
```
refine-issueの出力が、状態が整ったIssueに対して過剰に冗長になることがある。指摘ゼロでも長い前置きが出る。簡潔化したい。
```

**出力**（生成されるIssue本文の骨子。抜粋）:
```markdown
## 背景・課題
refine-issue は、DoRをほぼ満たしたIssueに対しても定型の前置き・全項目の網羅出力を行い、出力が冗長になる。

## 目的
精査結果の情報量をIssueの状態に比例させ、Ready に近いIssueでは出力を簡潔化する。

## ゴール
Ready 判定のIssueで、指摘ゼロ時の出力が現状比で短縮され、必要な指摘のみが提示される。

## スコープ
### IN
- 1件モードの出力テンプレートの簡潔化（指摘ゼロ時の省略規則）
### OUT
- 全件モードの集計フォーマット変更

## 受入条件（AC）
1. Ready かつ指摘ゼロのIssueを 1件モードで精査した出力に、空セクション（確認事項・改善提案）が含まれない。
2. Not Ready のIssueでは従来通り全不足項目が列挙される（後退がない）。

## ブロッカー
なし
```

このように、アイデアの「簡潔化したい」という曖昧な要望を、観測可能な合否境界（空セクションが含まれない／全不足項目が列挙される）を持つACへ変換するのがドラフトの要点。

**指定的HOWの退避の例**:

入力（指定的HOWを含むアイデア）:
```
ログイン失敗時の通知をWebSocketで実装して、画面にリアルタイム表示したい。
```

出力（HOWを本務から排除し参考欄へ退避した骨子。抜粋）:
```markdown
## 目的
ログイン失敗を利用者が即座に認識でき、再試行や対処に素早く移れる状態にする。

## ゴール
ログイン失敗が、画面上で利用者に遅滞なく伝わる。

## 受入条件（AC）
1. ログイン失敗の発生から利用者への表示まで、知覚可能な遅延なく（例: 1秒以内に）通知が画面へ反映される。

## 制約
なし

## 参考: 実現の手がかり（拘束しない）
- リアルタイム配信の実装案（例: WebSocket）。採用は設計・実装フェーズで判断する。
```

入力の「WebSocketで実装」という指定的HOWは、目的・ゴール・ACでは「即時に伝わる」という成果表現へ書き換えられ、HOW自体は非拘束の「参考」欄にのみ残る。タイトル案も同様に「ログイン失敗の即時通知」のような成果表現とし、「WebSocket」を含めない。

**種別固有セクションの例**（bug 種別）:

入力（`--label type:bug`、または「再現」「不具合」等を含むアイデア）:
```
保存ボタンを連打すると同じレコードが二重登録される。再現する。
```

出力（bug プロファイルの追加必須セクション「再現手順」が上乗せされた骨子。抜粋）:
```markdown
## 背景・課題
保存処理に多重送信防止がなく、ボタン連打で同一レコードが重複登録される。

## 再現手順
1. 環境: （対象環境・バージョン）
2. 手順: 入力フォームで保存ボタンを短時間に複数回押下する
3. 期待挙動: レコードは1件のみ登録される
4. 実際の挙動: 押下回数分のレコードが重複登録される

## 受入条件（AC）
1. 保存ボタンを連続して複数回押下しても、登録されるレコードが1件に留まる。

## ブロッカー
なし
```

bug 種別では「再現手順」が追加必須セクションとしてドラフトされ、AC は再現条件と期待挙動の対で表現する。原因・修正方法は完了条件に含めない（種別プロファイルに従う）。
