# Issue起票（Create Issue）

粗いアイデアを入力に、DoR（Definition of Ready）を満たしたIssue本文を対話生成し `gh issue create` で起票する。

このスキルの狙いは **shift-left** にある。Issueを雑に起票してから `refine-issue` で精査すると、毎回「目的・ゴール・スコープ・ブロッカーが書かれていない（欠落系）」「ACが検証可能でない／主観的（品質系）」という同種の指摘が出て、作成→精査の往復が無駄になる。作成時にDoRを充足させれば、この往復を構造的に消せる（born Ready）。

DoRの判定基準は `refine-issue` と**同一の共有定義**を参照する。基準が二重管理されてドリフトすることを避けるため、DoR項目を本スキル本文に転記しない。常に共有定義を読み込み、それに従う。

## 引数

- `$ARGUMENTS`: 起票したいアイデア（必須。粗くてよい。1行のメモでも段落でも可）
  - `--repo <owner/repo>`: 起票先リポジトリ（省略時はカレントリポジトリ）
  - `--label <name>`: 付与するラベル（カンマ区切りで複数可）
  - `--dry-run`: 起票せず、生成したIssue本文をプレビュー出力して停止する（内容確認・検証用）

## 手順

### 1. 引数の解析

`$ARGUMENTS` からオプション（`--repo` / `--label` / `--dry-run`）を分離し、残りをアイデア本体として扱う。アイデア本体が空の場合は、何を起票したいか1問だけ質問してから進む。

### 2. DoR定義の読み込み

以下の優先順位でReadし、**読み込めた最初の1つ**をDoR定義として採用する:

1. プロジェクト固有: `{プロジェクトルート}/.claude/dor/definition.md`（存在すれば優先）
2. プラグイン共有デフォルト: `${CLAUDE_PLUGIN_ROOT}/references/dor-default.md`

採用したDoR定義の「チェック項目」と「受入条件の検証可能性・客観性 判定基準」節を、以降のドラフト生成・AC作成の基準とする。項目内容は定義ファイルに従い、本SKILL.mdには書かない。

加えて、**種別プロファイル定義**を以下の優先順位でReadし、**読み込めた最初の1つ**を採用する（DoR定義と同一の優先順位。詳細は ADR-20260604）:

1. プロジェクト固有: `{プロジェクトルート}/.claude/dor/type-profiles.md`（存在すれば優先）
2. プラグイン共有デフォルト: `${CLAUDE_PLUGIN_ROOT}/references/issue-type-profiles.md`

採用した種別プロファイルの「種別検出ロジック」と「種別プロファイル」節を、次のドラフト生成での種別固有セクションの追加基準とする。種別プロファイル資源が読み込めない場合は種別差分を適用せず、サイズ軸（DoR）のみでドラフトする（ソフトデグレード）。

### 3. ドラフト先行生成（draft-first）

アイデアから**埋められる項目を先に全て埋める**。ここで安易に質問しないのが肝心で、Claudeが文脈から妥当に推論できる項目（目的の言い換え、スコープの輪郭、明らかなブロッカー有無など）を質問で往復するのはユーザーの時間を浪費する。アイデアに含意された情報を最大限ドラフトに変換する。

まず**種別を検出する**（`--label` の `type:*` を優先し、無ければアイデア本文を種別プロファイルの推論キーワードに照らす。一意に定まらなければ種別差分を適用しない）。検出した種別のプロファイルが定める**追加必須セクション**を本文ドラフトへ反映する（例: bug 判定時は「再現手順」セクションをドラフトに含める）。AC形・完了定義も種別プロファイルに従う（例: spike では観測可能な入出力対を強制しない）。

DoR項目に対応する本文セクションをドラフトする（出力形式は後述の「生成するIssue本文の構造」に従う）。受入条件（AC）は、DoR定義の「受入条件の検証可能性・客観性 判定基準」に照らして**検証可能（観測可能な事象・定量閾値・合否境界を含む）かつ客観的（主観的形容詞を含まない）**な文言でドラフトする。判定基準そのものはDoR定義側にあるので、ここでは繰り返さない。

**指定的HOWの扱い（適正な抽象度の確保）**: アイデアに含まれる指定的HOW（特定の実装方式・技術選定・ライブラリ名を解として固定する記述。例「Yライブラリで実装する」「このクラスを継承する」）は、本務セクション（タイトル・目的・ゴール・AC）からは排除し、次の3分岐で振り分ける:

1. **成果表現へ書換**: HOWを、それが達成する成果/課題の状態へ書き換えて本務セクションに置く（例「WebSocketで通知を実装」→「ログイン失敗が即時に利用者へ伝わる」）。
2. **参考欄へ退避**: 書換後も有用な手がかり（採用を強制しない実装案）は「参考: 実現の手がかり（拘束しない）」欄へ退避する。
3. **制約欄へ分離**: 解を縛る硬い境界（変更不可・後方互換維持等）は「制約」欄へ分離する。制約とHOWの判別: 「Yは変更不可」＝制約（正当）、「Yを実装せよ」＝解の指定（書換対象）。

**タイトル案も本務の対象**: タイトル案は本テンプレート外で生成し手順5で提示するが、上記の書換対象に含む。タイトルは成果/課題表現とし、ライブラリ名・実装方式等のHOWを含めない（例「〜をReactで再実装する」→「〜の表示を刷新し操作性を改善する」）。

この書換/退避の**厳しさは種別に連動する**。検出した種別プロファイルの「適正な抽象度の厳しさ」値に従い、feature（高）は最も厳しくHOWを排除し、refactor / chore（低）は実装方式への言及を許容、spike / docs（適用外）はHOW検討自体が成果物のため書換しない。判定基準・厳しさ値は DoR定義「適正な抽象度 判定基準」と種別プロファイルにあるので、ここでは転記しない。

### 4. 欠落項目の最小質問

ドラフトしてもなお埋まらない項目（アイデアから推論できない事実）**だけ**を `AskUserQuestion` で質問する。すでにドラフトで埋めた項目を確認のために聞き直さない。質問は欠落の解消に必要な最小限に絞る。

埋まらない項目が「ブロッカーの有無」「スコープのOUT境界」など合否を左右しないものなら、仮の値（例: ブロッカー「なし」）をドラフトに明記し、質問せず進めてよい。

### 5. 本文の確定とプレビュー

回答を反映してIssue本文を確定する。CLAUDE.md規約により複数行本文をCLIへ直接渡せないため、確定本文を Write で一時ファイル `/tmp/issue-body-{timestamp}.md` に書き出す。

本文（タイトル案を含む）をユーザーにプレビュー提示する。

### 6. 起票

- `--dry-run` 指定時: 一時ファイルのパスと本文プレビューを示して**停止する**（`gh issue create` は実行しない）。
- 通常時: 以下で起票する。

```bash
gh issue create --title "{タイトル}" --body-file /tmp/issue-body-{timestamp}.md [--repo <owner/repo>] [--label <name>]
```

起票が失敗（認証・権限・ネットワーク等）した場合は、一時ファイルを削除せず残し、パスを示して再実行可能にする。成功時は起票されたIssueのURL/番号を報告する。

## 生成するIssue本文の構造

DoRの各チェック項目が本文のどこで満たされるかが一目でわかる構造にする。以下を基本テンプレートとし、アイデアに応じてセクションを増減してよい（ただしDoR項目に対応するセクションは省略しない）。検出した種別の**追加必須セクション**は、種別プロファイルに従ってこのテンプレートに上乗せする（例: bug→「再現手順」、refactor→「対象範囲・振る舞い不変の宣言」、spike→「調査の問い・完了の成果物」、docs→「記述対象・読者・配置」）。

「制約」と「参考: 実現の手がかり（拘束しない）」は役割が異なる別セクションである。**制約**＝解を縛る硬い境界（守らねばならない外枠。設計・実装はこの内側でしか解を選べない）。**参考**＝任意の非拘束ヒント（実装方式の案・参考実装等。採否は設計・実装フェーズが自由に判断してよい）。HOW（実現方法の指定）は本務セクション（目的・ゴール・AC）に置かず、硬い制約なら「制約」へ、任意の手がかりなら「参考」へ振り分ける。「制約」は無ければ「なし」、「参考」は無ければ省略してよい。

```markdown
## 背景・課題
[何が問題か、なぜ必要か]

## 目的
[このIssueで何を実現するか。1-2文]

## ゴール
[Issueの目的が達成された状態の一文要約（WHY/価値）。検証条件は受入条件（AC）に記述し、ここでは列挙しない]

## スコープ
### IN
- [対象に含むもの]
### OUT
- [対象に含まないもの]

## 受入条件（AC）
1. [検証可能・客観的なAC]
2. [...]

## ブロッカー
[作業開始を阻む依存・前提。なければ「なし」]

## 制約
<!-- 解を縛る硬い境界（変更不可・後方互換維持・遵守必須の規約等）。解空間の外枠として明示する。なければ「なし」 -->
[守るべき硬い制約。なければ「なし」]

## 参考: 実現の手がかり（拘束しない）
<!-- 任意の非拘束ヒント（実装方式の案・参考実装・想定アプローチ等）。ここに書いた内容は解を縛らず、採否は設計・実装フェーズが自由に判断してよい。指定的HOWはこの欄にのみ置く。手がかりが無ければ本セクションごと省略してよい -->
- [実現の手がかり（採用を強制しない）]
```

**入出力例**: `${CLAUDE_SKILL_DIR}/references/io-examples.md` を参照（標準例・指定的HOW退避・bug 種別例）。

## 検証（born Ready の確認）

生成本文がDoRを満たすことは、`refine-issue` 自身で検証できる。手順:

1. 生成したIssue本文を、`refine-issue --input` が受け付ける **Issue JSON** 形式に整形して一時ファイルへ書き出す。JSONのフィールドは `number`（仮で `0` 等）・`title`・`body`（生成本文）・`labels`（配列）・`comments`（空配列 `[]`）。
2. `refine-issue --input /tmp/issue-check-{timestamp}.json` を実行する。
3. 出力の **DoRチェック結果** 表で、課題・ゴール・受入条件・スコープ・ブロッカーが全て ✅ であり、**受入条件の改善提案** セクションが出力されない（＝AC品質の指摘がない）ことを確認する。

合格境界は「**欠落系**（目的・ゴール・スコープ・ブロッカーの記述漏れ）および**品質系**（ACの検証可能性・客観性）の指摘がゼロ」に限定する。プロジェクト固有DoRのMedium項目（見積もり可能・スコープ適切 等）はアイデアの内容に依存するため、ここでの合否には含めない。

**検証の入出力例**: 上記「入出力例」で生成した本文をJSON化して `refine-issue --input` に渡すと、DoRチェック結果が全項目 ✅・改善提案セクションなしとなり、欠落系・品質系の指摘ゼロを満たす。
