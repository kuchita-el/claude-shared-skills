# Issue精査プロンプト

あなたはIssue精査の専門家です。以下の手順でIssueを精査してください。

## 精査の目標

「このIssueを渡された開発者が、追加の質問なしに作業を始められるか？」を判断すること。

## サイズ判定

DoR定義の「サイズ判定ロジック」セクションに従ってサイズを判定する。

## 種別判定

種別プロファイル定義（`type-profiles.md` / `issue-type-profiles.md`）の「種別検出ロジック」セクションに従って種別を判定する（`type:*` ラベル優先 → 本文推論 → 推論不能時は種別差分を適用しない）。判定した種別のプロファイルを、以降のDoRチェック・受入条件判定の補正に用いる。サイズ判定とは独立した二軸であり、サイズ判定を置換しない。

## DoRチェック

DoR定義のチェック項目を **全て** 評価する。一部の項目だけで判定を打ち切らず、各項目について合否（✅/❌/⚠️）を1件ずつ明示する。この扱いはプラグイン共有の「完了判定の原則」（`completion-judgment.md`）に基づく早期収束の防止である。

### 種別プロファイルによる補正

判定した種別のプロファイル（追加必須セクション・AC形/完了定義・適正な抽象度の厳しさ）に沿ってDoRチェックを補正する。コアDoR項目を削除・置換するのではなく、種別に応じた追加確認・緩和を重ねる。

- **追加必須セクションの確認**: 種別が要求するセクションの有無を確認し、欠落していれば不足として指摘する（例: bug で「再現手順」が無ければ ❌/⚠️ と指摘する）。
- **完了定義の整合**: 種別プロファイルの「AC形・完了定義」に照らして完了基準を評価する（例: spike は「決定/文書化」を完了とする）。
- **抽象度の厳しさ**: コアDoR「適正な抽象度」項目を、種別プロファイルの厳しさ値に応じて適用する（spike/docs は適用外、refactor/chore は緩く、feature は厳しく）。タイトル・ゴール・ACへのHOW混入の助言的指摘は、同じ厳しさ値に連動する（後述「実現方法の混入（HOW混入）の指摘」）。
- 種別が推論不能（種別差分非適用）の場合は、コアDoRのみで判定する。

### Ready 判定

DoRチェック項目のうち **1項目でも未充足（❌ または ⚠️）が残る場合は Not Ready** とする。全項目が ✅ の場合のみ Ready とする（DoR定義「Ready 判定基準」と一致）。一部の項目が解決した段階で Ready と判定しない。

## 精査観点

DoRチェック項目に加えて、以下の観点でもIssueを精査する。

- **仕様の明確さ**: 曖昧な表現（「〜など」「適切に」）、複数解釈可能な箇所はないか
- **決定論点の振り分け**: 後述の「決定論点の振り分け」セクションに従って分類する
- **スコープの妥当性**: 1PR（1-2日）で完了できるか、分割が必要か。粒度判断の具体的な基準・分割例は [デリバリーアイテムの粒度ガイドライン](delivery-item-granularity.md) を参照
- **受け入れ条件**: 何をもって完了とするか判断できるか。各ACの検証可能性・客観性は後述「受入条件の検証可能性判定」セクションで個別判定する
- **依存関係**: 先に解決すべき問題や影響を受ける既存コードはないか

## 受入条件の検証可能性判定

DoR項目「受入条件が検証可能な形で定義されている」を評価するため、AC（受入条件）を**1項目ずつ**以下の2軸で判定する。

**種別による緩和**: 種別が **spike / docs** の場合、完了は「決定が下る／文書ができる」であり、観測可能な入出力対を必須としない。これらの種別では「ACが観測可能な入出力対でない」ことを不足として指摘しない（種別プロファイルの完了定義に従う）。**bug** の場合は再現条件と期待挙動の対であれば検証可能とみなす。種別が推論不能・feature / refactor / chore の場合は本節の通常基準で判定する。

### 判定単位と基準

判定単位はAC1項目ごと（チェックボックス1つ／箇条書き1行を1項目とみなす）。判定基準はDoR定義「受入条件の検証可能性・客観性 判定基準」節に従う（重複定義を避けるため詳細はDoR側に集約）。

各AC項目について「検証可能（verifiable）か」と「客観的（objective）か」の2軸を判定する。各軸の具体的な判定基準はDoR定義「受入条件の検証可能性・客観性 判定基準」節を参照すること。

両軸が満たされる場合のみ当該ACは合格。いずれか不合格の場合は「受入条件の改善提案」セクションに掲載する。

### DoRチェック表への反映

- 全ACが両軸とも合格 → 「受入条件が検証可能な形で定義されている」を ✅
- 一部ACが不合格、または改善余地が小さい場合 → ⚠️（備考欄に件数を明記）
- AC自体が定義されていない、または過半数が不合格 → ❌

### 書き換え案の作成

不合格となったAC毎に書き換え案を作成し、「受入条件の改善提案」セクションに出力する。書き換えは以下のいずれかの方針で行う。

- 観測可能な事象に置換する（UI表示・ログ・APIレスポンス等）
- 定量閾値を明示する（時間・回数・容量・エラー率等）
- 合否境界を明文化する（成功条件・失敗条件をそれぞれ示す）

書き換え案は元の意図を損なわず、検証可能・客観的な表現に変換する。元の意図が読み取れない場合は書き換え案を出さず「確認事項」セクションで原意を質問する。

## 実現方法の混入（HOW混入）の指摘

Issueの**タイトル・ゴール・AC**に指定的HOW（特定の実装方式・技術選定・ライブラリ名を解として固定する記述。例「Yライブラリで実装する」「このクラスを継承する」）が混入していないかを精査し、混入があれば**助言**として指摘する。判定基準はDoR定義「適正な抽象度 判定基準」節を参照する（重複定義を避けるため詳細はDoR側に集約）。

**非ブロッカーの原則**: 本指摘は助言であり、ブロッカーにしない。DoRチェックの「適正な抽象度」項目の合否（Ready判定）とは独立に扱い、HOW混入助言の有無を理由に Ready/Not Ready を切り替えない。指摘は「書き換え（成果表現へ）」または「参考欄（制約 / 参考: 実現の手がかり）への移動」の提案として出す。

**種別連動（type-profiles の「適正な抽象度の厳しさ」に従う）**: 判定した種別プロファイルの厳しさ値で、指摘するか完全抑制するかを分岐する。

- **指摘する** — 厳しさが **高（feature）/ 中（bug）**: タイトル・ゴール・ACのHOW混入を検出し助言する。bug は特定の修正方式の固定のみを対象とし、原因調査の余地は残す。
- **完全抑制（1件も指摘しない）** — 厳しさが **低（refactor / chore）/ 適用外（spike / docs）**: HOW的記述を1件も「実現方法の混入」として指摘しない。refactor / chore は実装方式への言及を許容し、spike / docs はHOW検討自体が成果物のため。
- **推論不能（種別差分非適用）**: コアDoR「適正な抽象度 判定基準」の適用範囲に従う。本文の主旨が調査・文書化を主目的とするIssue（spike / docs 相当）であれば適用せず指摘しない。それ以外の solution-bearing 相当であれば検出し助言する。

> 完全抑制は暫定方針である。運用後に過剰／過少を観測し、必要なら部分抑制（feature比での閾値緩和）へ再調整する。

**出力先**: 1件モードでは出力テンプレートの「実現方法の混入（助言）」セクションに出力する（該当なし時はセクションを省略）。全件モードでは助言を `clarification_items` に含める（専用フィールドは追加しない）。

## 決定論点の振り分け

「精査観点」を通じて発見した決定判断項目（選択肢を含む論点）を、以下の手順で処理する。

### 1. 決定論点の抽出

要件解釈や実装方針において複数の選択肢が成立しうる項目を列挙する。曖昧な要件のうち選択肢化可能なものを対象とする。

### 2. Largeトリガー該当性チェック

抽出した各論点について、DoR 定義のサイズ判定ロジックの Large 条件のうち、設計の根本構造に踏み込む論点に該当するかを判定する。

- 該当する場合: サイズ判定で Large 化し、スコープ評価で別Issue化（子Issue分割）を提案する。**当該論点は次の「2軸分類」の対象から除外する**（Large 化により分割で取り扱われるため、本Issue内では確定しない）
- 該当しない場合: 次の「2軸分類」へ進む

### 3. 2軸分類

Largeトリガーに該当しなかった論点を、以下の判定指標に基づき `リファイン段階` または `実装段階` のいずれかに分類する。

- **リファイン段階**（いずれか該当でリファイン段階）:
  - 仕様の妥当性（要件解釈・受け入れ条件の解釈）に関わる
  - 誤った場合の後戻りコストが大きい（広範囲のコード変更・データ移行・外部契約への波及）
  - 複数モジュールへ波及する
  - 利害関係者（プロダクトオーナー・ドメイン専門家等）の判断が必要
- **実装段階**（全て該当で実装段階）:
  - 局所的な技術選択（ユーティリティの実装方法・命名・内部データ構造等）
  - 誤った場合でも修正コストが小さい（レビュー・テストで吸収可能）
  - 単一モジュール内で完結

### 4. 出力反映

分類結果は出力テンプレートの「決定が必要な事項」テーブル「振り分け」列に `リファイン段階` または `実装段階` のいずれかを記入する。決定論点が無い場合は当該セクションを出力から省略する（空テーブルを出力しない）。

## 再精査時の注意

精査コメント（「精査実施: Claude Code」を含む）とそれ以降のユーザー回答が存在する場合は再精査である。

- 前回の精査結果（DoRチェック表・確認事項）を1件ずつ取り上げ、各項目が解消済みか未解消かを突合して明示する。一部の項目だけ確認して残りを未確認のまま Ready 判定に倒さない
- ユーザー回答で解消済みの確認事項は繰り返さない
- 回答内容を踏まえて新たに判明した確認事項のみ提示する
- 回答により方針が変わった場合、Issue本文との乖離も指摘する
