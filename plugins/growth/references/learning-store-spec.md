# 学び置き場（配布物）仕様

growth プラグインの学習ループ（`[Capture] → [Distill] → [Route] → [Promote] → [Distribute]`）の終端、検証済みの学びを収める**配布物**＝学び置き場の、形式・スキーマ・配置・ライフサイクルを定義する。

## 位置づけ

- `DESIGN.md` 決定事項1「学び置き場の形式（単一の人間可読ファイル）」を具体化する文書。個人ローカル store（[`personal-store-spec.md`](personal-store-spec.md)、#345）が学習ループの起点（生捕捉）を定義するのに対し、本仕様は**終点（配布される成果物）**を定義する。
- 本ファイルは committed な配布物であり、未検証の生捕捉を貯める個人ローカル store とは別物。検証を経て共有価値が確認された学びだけがここへ昇格する（`DESIGN.md` 保存設計・二段ゲート）。
- **本仕様が定義するのは「最終の置き場」だけ**。store からの蒸留・行先選定（Route）・昇格ゲート（Promote）は上流（別 Phase / 別 Issue）の責務であり、本仕様はその着地先の形式に徹する。

## 2面モデル（origin / consumer）

学び置き場には2つの面があり、書き込みとフィードバックは非対称に流れる。

| 面 | 実体 | 役割 |
|---|---|---|
| **origin 面** | 配布しようとしている権威ファイル（このリポジトリ等、origin が編集する側） | ライフサイクル＝状態機械の住処。昇格（追加）・撤回・整理（除去）はすべて origin への編集（PR 等）として起きる |
| **consumer 面** | 配布された先の読み取り専用ミラー（各利用者のプラグインキャッシュ等） | origin の最新状態のスナップショット。独立したライフサイクルを持たず、配布で同期されるだけ |

- **書き込みは origin で一度きり（fan-out）**。consumer は配布ファイルに書き込まない。
- consumer 面で観測される発火・効果は、配布ファイルへ書き戻さず、**別チャネル（fan-in）で origin へ還流**する。fan-in 機構（使用台帳→集約ダイジェスト）は Phase 3 の責務であり、本仕様の対象外。
- この非対称（fan-out 配布と fan-in フィードバックの分離）ゆえ、配布ファイルが持つ値は **origin 権威の単一値**に限る。使用観測の集約値（発火回数・効果測定）は配布ファイルに持たせない。

## 2空間モデル（共有境界＝空間）

最終の置き場には、相互に独立した2つの空間がある。**空間そのものが共有境界**を表す（`DESIGN.md`「母集団＝共有境界」）。

| 空間 | 共有境界 | 実体 | 本リポジトリでの扱い |
|---|---|---|---|
| **パブリック/グローバル空間** | 全世界 × 全プロジェクト | 公開プラグインリポジトリ（本リポジトリ） | 具体インスタンスをここに置く |
| **閉じた空間** | チーム / プロジェクト | 利用者が管理するリポジトリ／ストレージ（例: 共有ドライブのローカルマウント） | 同一フォーマットの第2インスタンス。本リポジトリは規約を文書化するのみで、具体ストレージは持たない |

- 2空間は**独立**であり、**空間（リポジトリ）をまたぐ参照を持たない**。理由は「ライフサイクル」節の参照規約を参照。
- store からの蒸留は「project-local 観点」と「universal 観点」の2系統に分かれ、それぞれを対応する空間へ Route する（Route は上流の責務）。
- 本リポジトリはオーナー所有の公開リポジトリであり、特定チーム/プロジェクト専用ではない。したがって閉じた空間の実体（生データ・チーム限定の学び）は本リポジトリには置かない。

## 配置

パブリック/グローバル空間の具体インスタンスを以下に置く。

```
plugins/growth/learnings.md
```

- 単一の人間可読ファイル（`DESIGN.md` 決定事項1）。複数ファイル分割・DB 化はしない。
- 閉じた空間のインスタンスは、利用者が管理するストレージ上の任意パスに同一フォーマットで置く。単一可読ファイルであるため git リポジトリでも共有ドライブでも成立する。
- 配布ファイルがどう consumer のコンテキストへロードされるか（活性化モデル）は Phase 3 の決定対象（#350、旧 #357）であり、本仕様の対象外。本仕様は配置と形式のみを定義する。

## エントリのスキーマ

学び置き場は Markdown ファイルであり、1学びを1セクション（`##` 見出し）として持つ。

### 1欄スキーマ

**per-entry のスキーマは「振る舞い差分（規範）」の1要素のみ**とする。メタフィールドを持たない。

| 要素 | 内容 |
|---|---|
| 見出し（`## <規範の短い見出し>`） | その学びが命じる振る舞い差分の一文要約（規範） |
| 本文 | 規範差分の具体・理由（なぜその振る舞いを取るか）。複数行可 |

共有境界・provenance（出自）・撤回追跡といった残りの情報は、**メタ欄を足さず、空間（どのファイル/リポジトリに在るか）とエントリ本文と git 履歴**が担う。これは `DESIGN.md` 原理1（保存対象は気付きではなく振る舞い差分そのもの）と一致し、足場を最も薄く保つ。

明示的に**持たない**もの（いずれも別の機構が担う）:

- 検証状態（verified/deprecated/retired 等）: 配布ファイルは検証済みのみを載せる（載っている＝verified）。降格・撤回は物理除去で表現する（「ライフサイクル」節）。
- 安定 ID: fan-in ダイジェスト等の下流参照が現れる Phase 3 で必要になれば導入する。現時点では利用者が居らず投機的なため持たない。
- 発火/効果メモ: 観測値は fan-in 側（Phase 3）に置く。
- 畳み込み先参照・導入元参照: 除去・移管・出自は git 履歴で辿る（「ライフサイクル」節の参照規約）。

### 記法ルール（規範の強制）

- 見出し・本文は**実行可能な振る舞い差分（規範）**を表すこと。「次回どう違う行動を取るか」が読み取れない、空または純記述的（事実の記述に留まり行動を命じない）な内容は受け付けない。
- 規範でない内省（感想・分類・未検証の気付き）は学び置き場に書かない。それらは個人ローカル store（生捕捉）に留め、検証を経るまで昇格させない。

### 記法例

```
## ファイル復元には git restore を使う
ファイルを復元するとき git checkout ではなく git restore を使う。git checkout はブランチ切り替えと復元が多重定義されており、誤操作で別ブランチへ移る事故を招くため。

## 長文は CLI 引数に直接渡さず一時ファイル経由にする
Markdown 等の長文を CLI オプションに直接渡さない。ファイルへ書き出し `--body-file` 等で渡す。シェルのクォート/ヒアドキュメント制約による破損と許可プロンプトの中断を避けるため。
```

## ライフサイクル

学び置き場のエントリは、昇格で生まれ、整理で除去される。状態機械はすべて origin 面の編集として起こる。

### 昇格（エントリの追加）

- store からの蒸留・検証・行先選定（Distill → Route → Promote）を経た学びが、対応する空間の学び置き場へエントリとして追加される。
- 昇格は二段ゲートの「仕組み化」に当たり、承認またはマルチエージェントレビューを経る（L2、`DESIGN.md` 二段ゲート）。Route/Promote の機構自体は上流の責務。

### 忘却＝整理（エントリの物理除去）

忘却は**物理除去**（origin ファイルからエントリを削除する編集）で実現する。足場を痩せさせる（原理5）・via negativa（原理7）に忠実な形であり、可逆性は git 履歴が担保する。中間状態（deprecated/retired 等）をファイルに残さない。

除去には3つのモード（判定軸）があり、いずれも出口は物理除去で同じだが、性質が異なる。

| モード | 性質 | 対応原理 |
|---|---|---|
| ①反証/陳腐化 | 学びの予測が反証された、または前提が古びた | 原理5（蒸留・更新） |
| ②不使用/無効果 | 発火しない・効かない（fan-in 由来の信号、Phase 3） | 原理5 |
| ③畳み込み（整理） | 具体ルールが上位の原理・より強いキャリアへ昇華し、個別エントリが不要になった | 原理5・原理7 |

- ③は単なる削除ではなく**整理（蒸留統合）**である。具体が抽象（上位原理）や、より強制力の高いキャリア（skill / hook / lint・test 等の構造、原理4）へ昇華し、その結果として学び置き場から出ていく。情報は失われず、より強い形へ移送される。
- 忘却の**閾値・自動化・実行タイミング**、および②の信号源（使用台帳→集約ダイジェスト）は Phase 3 の対象であり、本仕様の対象外。本仕様は判定軸と除去の形（物理除去）を定義するに留める。

### 撤回・移管・出自の追跡（git 履歴に委ねる）

- 撤回（学びを取り下げる）は origin 側のファイル編集＝**エントリの除去**として実現する。なぜ除去したか・どこへ移管したか・どの議論で起票したかは、**除去/追加した commit と紐づく PR・Issue（git 履歴）**で辿る。これらのためのメタ欄をエントリに持たせない。

### 参照規約

- エントリ間・キャリア間の参照は、**同一空間（同一リポジトリ）内に閉じる場合のみ許容**する。空間をまたぐ参照は持たない。
- 理由: 同一リポジトリ内に閉じた参照なら、あるエントリ（葉）を除去する際にリポジトリ内の知識全体を1つの変更（PR）で原子的に整理し直せるため、整合を保てる。リポジトリをまたぐ参照は相互依存を生み、原子的な整理が不能になって除去を妨げる。
- 関連する学び同士を結ぶネットワーキング（同一空間内のナレッジグラフ）は、上記規約の範囲で**任意機能**として実現しうるが、既定では持たない（YAGNI）。

## 一覧性と上限

- 学びを単一ファイルに集約し、1欄スキーマで薄く保つことにより、全体を一覧でき肥大が一目で分かる（`DESIGN.md` 決定事項1・原理5 の前提）。
- 肥大への対処は、③整理（具体を原理・構造へ畳んで除去する）を継続的に行い**足場を痩せさせ続ける**ことを定性方針とする。
- 定量的な上限（行数・件数の閾値）・肥大検知の自動化・実行タイミングは持たない（Phase 3 の対象）。

## 中間層（生の学びの置き場）について

- ローカル store と最終の置き場の間に、横断的にトラッキングするための中間層（「生の学びの置き場」）を**既定では設けない**。store から2観点（project-local / universal）で蒸留し、それぞれを最終の置き場へ直接 Route する。
- 生データからの横断トラッキングが必要な場合に限り、中間層を任意で挟みうるが、その実体は**利用者が管理するリポジトリ／ストレージ**であり、本プラグイン（公開リポジトリ）には置かない。

## スコープ境界

| 本仕様が定義する（IN） | 本仕様が定義しない（OUT） |
|---|---|
| 配置（パブリック空間の確定パス・単一可読ファイル） | 活性化（ロード）モデル＝配布ファイルが consumer コンテキストへ載る仕組み（Phase 3 / #350） |
| 2面モデル（origin / consumer）と fan-out / fan-in 分離 | fan-in 機構（使用台帳→集約ダイジェスト）の実装（Phase 3） |
| 2空間モデル（独立・空間＝共有境界・空間横断参照の禁止） | store からの蒸留・行先選定（Route）・昇格ゲート（Promote）の機構 |
| エントリの1欄スキーマ・記法ルール・記法例 | 忘却の閾値・自動化・実行タイミング（Phase 3） |
| 忘却の判定軸（3モード）と除去の形（物理除去） | 効果測定の自動計測（Phase 3 以降） |
| 同一空間内に限る参照規約 | トリガ欄の要否・簡潔さ制約の最終強度（活性化モデル従属、Phase 3 で追補） |
