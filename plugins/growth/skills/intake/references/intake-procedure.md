# 取り込み手順（対象特定→重複検出→裁定提案→承認→起票/追記→取り込み時クローズ）

intake スキルの各段の判定基準・コマンド・本文書式の詳細。SKILL.md の手順 overview から参照される単一出典。worked example は [`intake-examples.md`](intake-examples.md) を、取り込み Issue の構造・裁定結果形式・不変条件は [`intake-issue-spec.md`](../../../references/intake-issue-spec.md) を参照する。

## 0. 目的・責務境界

- **目的**: 複数の `growth:promote` 仮説 Issue を取り込み Issue へ束ね、人間の career 裁定を経て、仮説を取り込み時クローズする。
- **責務境界**: intake が担うのは「対象特定 → 重複検出 → 裁定提案 → 承認 → 起票/追記 → 取り込み時クローズ」の6段。次は**行わない**:
  - 仮説 Issue の起票（promote の責務）・仮説テンプレートの定義（promotion-issue-spec.md #382）
  - 各 career の成果物への実書き込み（learnings.md 変換 #383 / 残りキャリア #384 / 成果物 PR の作成）
  - career 仮説の生成（distill の責務。本スキルは仮説を参照するのみ）
  - dev-workflow スキルの直接呼び出し（疎結合。起票・編集・クローズは `gh` 直接のみ）
  - 取り込み Issue 自身の連鎖クローズ機構の設置（cascade-close を新設しない。intake-issue-spec.md）
- **裁定ゲートの位置**: §4 の承認が career を確定する人間裁定ゲート。promote の検証段（未検証フィルタ）とは別物で、ここは「career を締める」L2 相当の明示ゲート。

## 1. 取り込み対象の特定

1. **対象の入力**: 引数で仮説 Issue 番号が渡ればそれを取り込み対象にする。渡らなければ `gh issue list --label growth:promote --state open` で inbox を列挙し、どれを取り込むかを人間に選ばせる（§4 の承認と同じ AskUserQuestion にまとめてもよい）。
2. **仮説本文の読取**: 各仮説を `gh issue view <番号> --json number,title,body,labels` で読む。本文の振る舞い差分・`## スコープ`（および存在すれば career 仮説欄）を §3 の裁定提案の材料にする。
3. **対象の妥当性**: `growth:promote` ラベルを持たない Issue・既に closed の Issue が対象に混じっていたら、対象から除外して報告する（誤った取り込みを防ぐ）。0件なら §7 のエラー処理へ。

## 2. 既存取り込み先の検出（合流 or 新規）

「集約先は複数ありうる」不変条件（intake-issue-spec.md）に従い、束ね先を引数で固定せず内容重複で判定する。

1. **既存取り込み Issue の列挙**: `gh issue list --label growth:intake --state open --json number,title,body` で open の取り込み Issue を列挙する。
2. **意味的照合**: 各取り込み対象仮説の振る舞い差分のトピックを、列挙した取り込み Issue の「取り込んだ仮説」「裁定結果」内容と意味的に照合する。同種・関連する学びを束ねている取り込み Issue があれば**合流先候補**とする。
3. **束ね先の決定（提案）**: 合流先候補があれば「既存 #X へ合流」、無ければ「新規作成」を束ね先として §4 の承認サマリに含める。複数の合流先候補が競合する場合は最も重複の濃いものを第一提案にし、人間が §4 で選べるようにする。

> 照合は意味判断であり機械一致ではない。確信が持てない場合は新規作成を既定とし、合流可否を §4 で人間に委ねる(過剰合流より分割を既定に倒す)。

## 3. 裁定提案の生成

各仮説について career（#349 D1 の4分類）と空間を**提案**する。確証ではなく、人間裁定（§4）のたたき台。

- **career の提案**: 仮説本文の振る舞い差分を読み、4分類のいずれかを提案する。
  - learnings.md（テキスト規範として置く汎用ルール）
  - ADR 差分（後戻りコスト高・横断再発する構造的設計判断）
  - 改善還元 Issue（任意プラグイン・コミュニティの改善。dev-workflow 含む。ADR-20260628-2 決定6 で一般化）
  - 強キャリア（skill / hook / lint / test へ構造変換可能）
  - 仮説本文に distill の career 仮説欄があればそれを第一提案の種にする(career 仮説配線は別 Issue のため、無い場合は振る舞い差分から推定する)。
- **空間の提案**: 仮説の `## スコープ`（`universal` / `project-local`）を公開ゲートに流用する（intake-issue-spec.md 不変条件3）。`universal` → パブリック、`project-local` → 閉じた を提案する。
- 提案はあくまで仮説。人間が §4 で career・空間を変更でき、public への昇格を拒否（パブリック→閉じた へ変更）できる。

## 4. 裁定サマリの提示と承認（人間裁定ゲート）

AskUserQuestion で、次をまとめて提示し承認を得る。承認されるまで §5 以降（起票・クローズ）へ進めない。

提示内容:

- **裁定対象**: 取り込む仮説 Issue のリスト（番号・タイトル）。
- **束ね先**: 「新規取り込み Issue を作成」または「既存 #X へ合流」（§2 の決定。複数候補があれば選択肢にする）。
- **裁定結果の提案**: 仮説ごとの career・空間（§3）。

人間の応答で確定する事項:

- 各仮説の career（提案を承認 or 変更）。
- 各仮説の空間（パブリック / 閉じた。public 昇格の拒否を含む）。
- 束ね先（新規 / 既存 #X）。
- 取り込みから除外する仮説があれば除外する。

> 対話セッション中は AskUserQuestion で即時確認する。非対話（自動実行）で承認が取れない場合は、起票・クローズを行わず「裁定承認が必要」と報告して終了する（仮説は inbox に残す）。裁定を握り潰して自動確定しない。

## 5. 正式起票／追記

承認された裁定で取り込み Issue を materialize する。本文構造は intake-issue-spec.md「取り込み Issue の構造」に従う。

1. **本文の組み立て**: 次の3セクションを含む本文を組み立てる。
   - `## 取り込んだ仮説`: 束ねた仮説の `#番号` 参照リスト。**1行1仮説**(`- #401` を改行区切りで列挙。カンマ区切り1行にしない)とし、仮説はクローズされるため checkbox は付けない。
   - `## 裁定結果`: 仮説・career・空間・備考の4列テーブル(intake-issue-spec.md「裁定結果の記録形式」)。
   - `## 成果物`: career を実体化する成果物の task list(`- [ ] <成果物の説明>（#仮説番号 由来）`)。
2. **本文の受け渡し**: 複数行本文を CLI 引数へ直接渡さず、Write で一時ファイル(例: `/tmp/intake-issue-<連番>.md`)へ書き出してから `--body-file` で渡す(CLAUDE.md 規約。シェルのクォート/ヒアドキュメント制約による破損を避ける)。
3. **起票 or 追記**:
   - **新規**: `gh issue create --title "<取り込みの主題>" --body-file <一時ファイル> --label growth --label growth:intake` で起票する。`growth:intake` ラベルが未作成の環境ではラベル指定が失敗しうるため、失敗時はラベルなしで起票し「`growth:intake` 未作成。セットアップでラベル作成・付与が必要」と報告する(ラベル実体作成は #385/セットアップへ委譲。intake-issue-spec.md)。
   - **既存合流**: `gh issue view <既存#X> --json body` で現本文を取得し、「取り込んだ仮説」「裁定結果」「成果物」へ今回分を追記した本文を組み立て、`gh issue edit <既存#X> --body-file <一時ファイル>` で更新する(追記であり既存記載を破壊しない)。
4. **dev-workflow スキルを呼ばない**(疎結合)。起票・編集は `gh` 直接のみ。

> **疎結合の静的保証**: 経路は `Bash(gh issue create*)` / `Bash(gh issue edit*)` のみ。SKILL.md の `allowed-tools` に Skill ツール(dev-workflow 呼び出し経路)を含めないことで疎結合を許可ツールの面から保証する。

## 6. 取り込み時クローズ

§5 の起票/追記が**成功した後にのみ**、束ねた各仮説を閉じる。取り込み Issue 番号が確定してからでないとリンクコメントを書けないため、順序は §5 → §6 を厳守する。

各仮説に対し、次を**この順**で行う:

1. **リンクコメント**: `gh issue comment <仮説番号> --body "取り込み Issue #<取り込み番号> へ取り込みました（career 裁定の集約点）。"`(複数行にする場合は Write + `--body-file`)。どの取り込み Issue へ吸収されたかを仮説側に残す。
2. **not planned クローズ**: `gh issue close <仮説番号> --reason "not planned"`。作業放棄ではなく「配送伝票としてのトリアージ完了」を表す(ADR-20260628-2 決定4)。

全仮説のクローズ後、`gh issue list --label growth:promote --state open` に取り込んだ仮説が現れないこと(inbox から外れたこと)を確認する(AC3)。

**ディシジョンテーブル(取り込み時クローズ)**:

| 起票/追記成功 | 仮説のクローズ |
|---|---|
| No(失敗) | しない(仮説は inbox に残す。再実行可能) |
| Yes | する(リンクコメント → not planned。束ねた全仮説) |

- リンクコメント付与に失敗した仮説は、クローズせず報告する(どの取り込み Issue へ吸収されたか仮説側に残らないクローズを避ける)。
- 一部仮説のクローズに失敗しても、起票済み取り込み Issue と成功済みクローズは維持する(冪等。失敗仮説のみ再実行で閉じられる)。

## 7. エラー・境界処理

| 状況 | intake の振る舞い |
|---|---|
| 取り込み対象が0件(引数なし・inbox も空) | 「取り込む仮説がありません(確認: `gh issue list --label growth:promote --state open`)」と報告して終了。起票0・クローズ0 |
| 対象に `growth:promote` 非保持 / 既 closed が混在 | 当該を対象から除外して報告し、残りで続行(0件になれば上行と同じ) |
| 一部仮説の `gh issue view`(§1.2)が失敗(存在しない番号・権限等) | 本文を取得できない仮説を対象から除外して報告し、残りで続行(0件になれば取り込み対象0件と同じ)。本文なしで §3 の裁定提案を生成しない |
| §4 の承認が得られない(非対話・拒否) | 起票・クローズを行わず「裁定承認が必要」と報告して終了。仮説は inbox に残す |
| `gh issue create` / `gh issue edit` が失敗(認証・権限・ネットワーク等) | 一時ファイルを残しパスを示して再実行可能にする。取り込み時クローズは行わない(仮説は inbox のまま) |
| `growth:intake` ラベルが未作成 | ラベルなしで起票し「ラベル未作成。セットアップで作成・付与が必要」と報告(#385/セットアップへ委譲)。**重複起票リスク**: ラベルなしで起票すると次回実行の §2 重複検出(`--label growth:intake`)で当該取り込み Issue を発見できず、別の取り込み Issue が重複起票されうる。セットアップ(`gh label create growth:intake`)→ 既存取り込み Issue への手動ラベル付与を済ませてから再実行することを推奨 |
| 一部仮説のクローズ / リンクコメントが失敗 | 成功分は維持し、失敗仮説を報告(再実行で閉じられる。inbox に残る) |

いずれも「起票/追記成功を確認してから仮説を閉じる」順序を厳守し、取り込み先が確定しないまま仮説を inbox から消さないことを保証する。
